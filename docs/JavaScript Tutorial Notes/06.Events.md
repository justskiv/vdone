# Events

## 1. The EventTarget Interface

An event is essentially a way for the various parts of a program to communicate with each other, and it is also one implementation of asynchronous programming. The DOM supports a large number of events. This chapter begins our introduction to **DOM event programming**.

### 1. Overview

**DOM event operations (listening and triggering) are all defined on the `EventTarget` interface.** All node objects implement this interface, and some other browser built-in objects that need event communication (such as `XMLHttpRequest`, `AudioNode`, and `AudioContext`) also implement it.

This interface mainly provides three instance methods.

- `addEventListener`: binds a listener function for an event
- `removeEventListener`: removes a listener function for an event
- `dispatchEvent`: triggers an event



### 2. EventTarget.addEventListener()

`EventTarget.addEventListener()` is used to **define a listener function for a specific event on the current node or object**. **Once this event occurs, the listener function will be executed.** This method has no return value.

```js
target.addEventListener(type, listener[, useCapture]);
// event name, listener function [, whether to trigger during the capture phase]
```

This method accepts three parameters.

- `type`: the event name, case-sensitive.
- `listener`: the listener function. When the event occurs, this listener function will be called.
- `useCapture`: a boolean indicating whether the listener function is triggered during the capture phase (see the "Event Propagation" section below). The default is `false` (the listener function is only triggered during the bubbling phase). This parameter is optional.

Here is an example.

```js
function hello() {
  console.log('Hello world');
}

var button = document.getElementById('btn');
button.addEventListener('click', hello, false);
```

In the code above, the `addEventListener` method of the `button` node binds the listener function `hello` to the `click` event, and that function is only triggered during the bubbling phase.

There are two things to note about the parameters.

First, the second parameter, besides being a listener function, can also be an object with a `handleEvent` method.

```js
buttonElement.addEventListener('click', {
  handleEvent: function (event) {
    console.log('click');
  }
});
```

In the code above, the second parameter of the `addEventListener` method is an object with a `handleEvent` method.

Second, **the third parameter, besides being the boolean `useCapture`, can also be a configuration object**. This object has the following properties.

> - `capture`: a boolean indicating whether the event triggers the listener function during the `capture phase`.
> - `once`: a boolean indicating whether the listener function is triggered only once and then automatically removed.
> - `passive`: a boolean indicating that the listener function will not call the event's `preventDefault` method. If the listener function does call it, the browser will ignore that request and output a warning line in the console.

If you want the event listener function to **execute only once**, you can enable the `once` property of the configuration object.

```js
element.addEventListener('click', function (event) {
  // code that runs only once
}, {once: true});
```

The `addEventListener` method can add multiple different listener functions for the same event on the current object. These functions are triggered in the order they were added, that is, first added, first triggered. If you add the same listener function multiple times for the same event, the function will only execute once, and the redundant additions will be automatically removed (you do not need to use `removeEventListener` to remove them manually).

```js
function hello() {
  console.log('Hello world');
}

document.addEventListener('click', hello, false);
document.addEventListener('click', hello, false);
```

Running the code above, clicking on the document will only output a single line of `Hello world`.

If you **want to pass parameters to the listener function, you can wrap it in an anonymous function.**

```js
function print(x) {
  console.log(x);
}

var el = document.getElementById('div1');
el.addEventListener('click', function () { print('Hello'); }, false);
```

The code above passes a parameter to the listener function `print` through an anonymous function.

Inside the listener function, **`this` points to the object on which the current event occurs.**

```js
// HTML code is as follows
// <p id="para">Hello</p>
var para = document.getElementById('para');
para.addEventListener('click', function (e) {
  console.log(this.nodeName); // "P"
}, false);
```

In the code above, the `this` inside the listener function points to the object `para` where the event occurs.



### 3. EventTarget.removeEventListener()

The `EventTarget.removeEventListener` method is **used to remove a listener function added by the `addEventListener` method**. This method has no return value.

```js
div.addEventListener('click', listener, false);
div.removeEventListener('click', listener, false);
```

The parameters of the `removeEventListener` method are exactly the same as those of the `addEventListener` method. Its first parameter, the "event type", is case-sensitive.

Note that the listener function removed by the `removeEventListener` method must be the very same listener function added by the `addEventListener` method, and it must be on the same element node, otherwise it has no effect.

```js
div.addEventListener('click', function (e) {}, false);
div.removeEventListener('click', function (e) {}, false);// removal fails because it is not the same listener function
```

In the code above, the `removeEventListener` method has no effect, because the listener function is not the same anonymous function.

```js
element.addEventListener('mousedown', handleMouseDown, true);
element.removeEventListener("mousedown", handleMouseDown, false); // removal fails because the third parameter differs
```

In the code above, the `removeEventListener` method also has no effect, because the third parameter differs.



### 4. EventTarget.dispatchEvent()

The `EventTarget.dispatchEvent` method **triggers a specified event on the current node, thereby triggering the execution of the listener functions. This method returns a boolean**: as long as any listener function calls `Event.preventDefault()`, the return value is `false`, otherwise it is `true`.

```js
target.dispatchEvent(event)
```

The **parameter of the `dispatchEvent` method is an instance of an `Event` object** (see the "Event Object" chapter for details).

```js
para.addEventListener('click', hello, false);
var event = new Event('click');
para.dispatchEvent(event);
```

The code above triggers a `click` event on the current node.

If the parameter of the `dispatchEvent` method is empty or is not a valid event object, an error will be thrown.

The code below uses the return value of the `dispatchEvent` method to **determine whether the event was canceled**.

```js
var canceled = !cb.dispatchEvent(event);
if (canceled) {
  console.log('event canceled');
} else {
  console.log('event not canceled');
}
```



## 2. The Event Model

### 1. Listener Functions

The browser's event model responds to events through listener functions (listeners). After an event occurs, when the browser detects this event, it executes the corresponding listener function. This is the main programming approach of the event-driven programming model.

JavaScript provides three ways to bind a listener function to an event.



#### 1.2 HTML on- Attributes

The HTML language allows you to define the listening code for certain events directly in an element's attributes.

```html
<body onload="doSomething()"> <!-- include the parentheses -->
<div onclick="console.log('event triggered')">
```

The code above specifies the listening code for the `load` event of the `body` node and the `click` event of the `div` node. Once the event occurs, this code is executed.

An element's event listener attributes are all `on` plus the event name, for example `onload` is `on + load`, representing the listening code for the `load` event.

Note that the value of these attributes is the code that will be executed, not a function.

```html
<!-- correct -->
<body onload="doSomething()">

<!-- wrong -->
<body onload="doSomething">
```

Once the specified event occurs, the value of the `on-` attribute is passed verbatim to the JavaScript engine for execution. Therefore, if you want to execute a function, **don't forget to add a pair of parentheses**.

The listening code specified this way is only triggered during the **bubbling phase**.

```html
<div onClick="console.log(2)">
  <button onClick="console.log(1)">Click</button>
</div>
```

In the code above, `<button>` is a child element of `<div>`. The `click` event of the `<button>` will also trigger the `click` event of the `<div>`. Since the listening code in the `on-` attribute is only triggered during the bubbling phase, the result of clicking is to first output `1`, then output `2`, that is, the event **bubbles from the child element up to the parent element.**

Setting the `on-` attribute directly has the same effect as setting the `on-` attribute through the element node's `setAttribute` method.

```js
el.setAttribute('onclick', 'doSomething()');
// equivalent to
// <Element onclick="doSomething()">
```



#### 1.2 Element Node Event Properties (e.g., el.onclick)

An element node object's **event properties** can also specify a listener function.

```js
window.onload = doSomething; // note this is the function name, without parentheses

div.onclick = function (event) { // onclick is a property of the element, pointing to a function that executes when a click is triggered
  console.log('event triggered');
};
```

The listener function specified this way is also only triggered during the **bubbling phase**.

Note that the difference between this method and HTML's `on-` attribute is that its value is the function name (`doSomething`), unlike the latter, which must provide the complete listening code (`doSomething()`).



#### 1.3 EventTarget.addEventListener()

**All DOM node instances have the `addEventListener` method**, which is used to define a listener function for an event on that node.

```js
window.addEventListener('load', doSomething, false); // note this is the function name, without parentheses
```

For a detailed introduction to the `addEventListener` method, see the `EventTarget` chapter.



#### 1.4 Summary

Of the three methods above,

The first, "HTML's on- attribute", violates the principle of separating HTML and JavaScript code by writing the two together, which is not conducive to a clear division of code labor, so it is **not recommended**.

The **drawback** of the second, "element node event properties", is that **only one listener function can be defined per event**. That is, if you define the `onclick` property twice, the latter definition overrides the former. Therefore, it is **also not recommended**.

The third, `EventTarget.addEventListener`, is the **recommended way to specify a listener function**. It has the following advantages:

- Multiple listener functions can be added for the same event.
- You can specify in which phase (capture phase or bubbling phase) the listener function is triggered.
- Besides DOM nodes, other objects (such as `window`, `XMLHttpRequest`, etc.) also have this interface, making it effectively a unified listener function interface across all of JavaScript.



### 2. The Reference of this (Points to the Element Node)

**Inside a listener function, `this` points to the element node that triggered the event. (For all three event-binding methods, `this` always points to the element node.)**

```html
<button id="btn" onclick="console.log(this.id)">Click</button>
```

Running the code above, clicking will output `btn`.

For the other two ways of writing listener functions, the reference of `this` is the same.

```js
// HTML code is as follows
// <button id="btn">Click</button>
var btn = document.getElementById('btn');

// approach one
btn.onclick = function () {
  console.log(this.id);
};

// approach two
btn.addEventListener(
  'click',
  function (e) {
    console.log(this.id);
  },
  false
);
```

With both of the above approaches, clicking the button will also output `btn`.



### 3. Event Propagation (Capture, Target, Bubbling)

After an event occurs, it **propagates between child and parent elements** (propagation). This propagation is divided into three phases.

- **First phase**: from the `window` object down to the target node (from the upper layer to the lower layer), called the "**capture phase**" (capture phase).
- **Second phase**: triggered on the target node, called the "**target phase**" (target phase).
- **Third phase**: from the target node back up to the `window` object (from the lower layer back to the upper layer), called the "**bubbling phase**" (bubbling phase).

This three-phase propagation model means that the same event is triggered on multiple nodes.

```html
<div>
  <p>Click</p>
</div>
```

In the code above, there is a `<p>` node inside the `<div>` node.

If we set listener functions for the `click` event on both of these nodes (one listener function for each node's capture phase and bubbling phase), that is four listener functions in total. Then, clicking on the `<p>` will trigger the `click` event four times.

```js
var phases = {
  1: 'capture',
  2: 'target',
  3: 'bubble'
};

var div = document.querySelector('div');
var p = document.querySelector('p');

div.addEventListener('click', callback, true); // true means trigger during the capture phase
p.addEventListener('click', callback, true); // true means trigger during the capture phase
div.addEventListener('click', callback, false);// false means trigger during the bubbling phase
p.addEventListener('click', callback, false); // false means trigger during the bubbling phase

function callback(event) {
  var tag = event.currentTarget.tagName; // tag name of the current target object
  var phase = phases[event.eventPhase]; // the phase that was triggered
  console.log("Tag: '" + tag + "'. EventPhase: '" + phase + "'");
}

// result after clicking
// Tag: 'DIV'. EventPhase: 'capture'  capture phase
// Tag: 'P'. EventPhase: 'target' target phase
// Tag: 'P'. EventPhase: 'target' target phase
// Tag: 'DIV'. EventPhase: 'bubble' bubbling phase
```

The code above shows that the `click` event was triggered four times: once each during the capture phase and bubbling phase of the `<div>` node, and twice during the target phase of the `<p>` node.

1. Capture phase: as the event propagates from `<div>` to `<p>`, it triggers the `click` event of the `<div>`;
2. Target phase: when the event reaches `<p>` from `<div>`, it triggers the `click` event of the `<p>`;
3. Bubbling phase: as the event propagates back from `<p>` to `<div>`, it triggers the `click` event of the `<div>` again.

Among these, the `<p>` node has two listener functions (a difference in the third parameter of the `addEventListener` method causes two listener functions to be bound), so they are both triggered once by the `click` event. Therefore, the `<p>` produces two outputs during the `target` phase.

**Note** that **the browser always assumes that the target node of a `click` event is the most deeply nested node at the click position** (in this example, the `<p>` node inside the `<div>` node). So, **for the `<p>` node, both the capture phase and the bubbling phase are shown as the `target` phase.**

The topmost object in event propagation is `window`, followed in order by `document`, `html` (`document.documentElement`), and `body` (`document.body`). That is, the propagation order of the event in the example above is `window`, `document`, `html`, `body`, `div`, `p` during the capture phase, and `p`, `div`, `body`, `html`, `document`, `window` during the bubbling phase.



```
Capture phase:
window (browser object) --> document (document object) --> html --> body --> div --> p (click target, in the target phase)

Bubbling phase:
p (click target, in the target phase) --> div --> body --> html --> document (document object) --> window (browser object)
```



### 4. Event Delegation

Because an event bubbles up to its parent node during the bubbling phase, you can **define a child node's listener function on the parent node, letting the parent node's listener function handle the events of multiple child elements collectively. This approach is called event delegation** (delegation).

```js
var ul = document.querySelector('ul');

ul.addEventListener('click', function (event) {
  if (event.target.tagName.toLowerCase() === 'li') { // the browser assumes the click target is the most deeply nested element
    // some code
  }
});
```

In the code above, the listener function for the `click` event is defined on the `<ul>` node, but in practice it handles the `click` events of the child nodes `<li>`. The advantage of this is that you only need to define one listener function to handle the events of multiple child nodes, instead of defining a listener function on each `<li>` node. Moreover, if child nodes are added later, the listener function still works.



##### Stopping Event Bubbling with event.stopPropagation()

If you want an event to stop at a certain node and propagate no further, you can use the event object's `stopPropagation` method.

```js
// once the event propagates to the p element, it stops propagating downward
p.addEventListener('click', function (event) {
  event.stopPropagation(); // this method is on the event object
}, true); // true means bind the listener function during the capture phase

// once the event bubbles up to the p element, it stops bubbling upward
p.addEventListener('click', function (event) {
  event.stopPropagation();
}, false); // false means bind the listener function during the bubbling phase (the default)
```

In the code above, the `stopPropagation` method stops the event propagation during the capture phase and the bubbling phase, respectively.

However, **the `stopPropagation` method only stops the event's propagation; it does not prevent the event from triggering the `<p>` node's other `click` event listener functions**. That is, it does not completely cancel the `click` event.

```js
p.addEventListener('click', function (event) {
  event.stopPropagation();
  console.log(1);
});

p.addEventListener('click', function(event) {
  // will be triggered
  console.log(2);
});
```

In the code above, the `p` element has two `click` event listener functions bound. The `stopPropagation` method can only stop the propagation of this event; it cannot cancel the event, so the second listener function will be triggered. The output will be 1 first, then 2.

If you **want to completely cancel the event** and no longer trigger any of the subsequent `click` listener functions, **you can use the `stopImmediatePropagation()` method**.

```js
p.addEventListener('click', function (event) {
  event.stopImmediatePropagation();
  console.log(1);
});

p.addEventListener('click', function(event) {
  // will not be triggered
  console.log(2);
});
```

In the code above, the `stopImmediatePropagation` method can completely cancel this event, so that all `click` listener functions bound afterward are no longer triggered. Therefore, only 1 is output, not 2.





## 3. The Event Object

### 1. Overview

**After an event occurs, an event object is produced and passed as a parameter to the listener function.** The browser natively provides an `Event` object, and all events are instances of this object, or rather, they inherit from the `Event.prototype` object.

**The `Event` object is itself a constructor that can be used to generate new instances.**

```js
event = new Event(type, options);// first parameter, the event name; second parameter, the event object's configuration object
```

The `Event` constructor accepts two parameters. The first parameter, `type`, is a string representing the event name; the second parameter, `options`, is an object representing the event object's configuration. This object mainly has the following two properties.

- `bubbles`: a boolean, optional, defaulting to `false`, indicating **whether the event object bubbles**.
- `cancelable`: a boolean, optional, defaulting to `false`, indicating **whether the event can be canceled**, that is, whether `Event.preventDefault()` can cancel this event. Once an event is canceled, it is as if it never occurred, and the browser's default behavior for that event will not be triggered.

```js
var ev = new Event( // create a new event instance
  'look', // event name
  { // event configuration
    'bubbles': true, // whether the event bubbles
    'cancelable': false // whether the event can be canceled
  }
);
document.dispatchEvent(ev); // trigger the ev instance, which is a look event
```

The code above creates a new `look` event instance, then triggers the event using the `dispatchEvent` method.

Note that if the `bubbles` property is not explicitly set to `true`, the generated event can only trigger listener functions during the "capture phase".

```js
// HTML code is
// <div><p>Hello</p></div>
var div = document.querySelector('div');
var p = document.querySelector('p');

function callback(event) {
  var tag = event.currentTarget.tagName;
  console.log('Tag: ' + tag); // no output
}

div.addEventListener('click', callback, false); // this listener function is for the bubbling phase

var click = new Event('click');
p.dispatchEvent(click); // dispatch a click event on p; this event does not bubble by default, so it will not trigger the parent element div's click event
```

In the code above, the `p` element dispatches a `click` event that does not bubble by default. The `div.addEventListener` method specifies listening during the bubbling phase, so the listener function will not be triggered. If it were written as `div.addEventListener('click', callback, true)`, then it could listen for this event during the "capture phase".

On the other hand, if this event is triggered on the `div` element.

```js
div.dispatchEvent(click);
```

Then, regardless of whether the `div` element listens during the bubbling phase or the capture phase, the listener function will be triggered. Because at this point the `div` element is the target of the event, there is no question of whether it bubbles; the `div` element always receives the event, so the listener function takes effect.



### 2. Instance Properties

#### 2.1 Event.bubbles (Whether It Bubbles, Read-Only); Event.eventPhase (Returns an Integer Representing the Event's Phase, Read-Only)

The `Event.bubbles` property **returns a boolean indicating whether the current event bubbles**. This property is **read-only** and is generally used to learn whether an Event instance can bubble. As mentioned earlier, unless explicitly declared, an event generated by the `Event` constructor does not bubble by default.

The `Event.eventPhase` property **returns an integer constant indicating the phase the event is currently in. This property is read-only.**

```js
var phase = event.eventPhase;
```

The return value of `Event.eventPhase` has four possibilities.

- 0, the event is **currently not occurring**.
- 1, the event is currently in the **capture phase**, that is, in the process of propagating from ancestor nodes toward the target node.
- 2, the event has **reached the target node**, that is, the node pointed to by the `Event.target` property.
- 3, the event is in the **bubbling phase**, that is, in the process of reverse propagation from the target node toward ancestor nodes.



#### 2.2 Event.cancelable (Whether the Default Behavior Can Be Canceled); Event.cancelBubble (Whether Bubbling Is Stopped); event.defaultPrevented (Whether the Default-Behavior-Canceling Method Was Called)

The `Event.cancelable` property **returns a boolean indicating whether the event can be canceled**. This property is **read-only** and is generally used to learn about the characteristics of an Event instance.

**Most native browser events can be canceled.** For example, canceling a `click` event makes clicking a link have no effect. But unless explicitly declared, an event generated by the `Event` constructor **cannot be canceled by default**.

```js
var evt = new Event('foo');
evt.cancelable  // false, not cancelable by default
```

When the `Event.cancelable` property is `true`, calling `Event.preventDefault()` can cancel this event, preventing the browser's default behavior for that event.

If the event cannot be canceled, calling `Event.preventDefault()` has no effect. So before using this method, it is best to use the `Event.cancelable` property to check whether it can be canceled.

```js
function preventEvent(event) {
  if (event.cancelable) {
    event.preventDefault();
  } else {
    console.warn('This event couldn\'t be canceled.');
    console.dir(event);
  }
}
```

The `Event.cancelBubble` property is a boolean. If set to `true`, it is equivalent to executing `Event.stopPropagation()`, and can **stop the propagation of the event.**

The `Event.defaultPrevented` property returns a boolean indicating whether the event **has called the `Event.preventDefault` method. This property is read-only.**

```js
if (event.defaultPrevented) {
  console.log('this event has already been canceled');
}
```



#### 2.3 Event.currentTarget (The Node Currently Being Passed Through); Event.target (The Target Node)

After an event occurs, it goes through the capture and bubbling phases, passing through multiple DOM nodes in turn. Therefore, at any given moment there are two nodes related to the event: one is the event's **original trigger node** (`Event.target`), and the other is the node the event is **currently passing through** (`Event.currentTarget`). **The former is usually a descendant node of the latter.**

The `Event.currentTarget` property returns the node where the event **is currently located**, that is, the node the event is currently passing through, which is also the node where the currently executing listener function resides. **As the event propagates, the value of this property changes.**

The `Event.target` property returns **the node that originally triggered the event**, that is, the node where the event first occurred. This property does not change as the event propagates.

During event propagation, the values of the `Event.target` and `Event.currentTarget` properties inside the listener functions of different nodes are different.

```js
// HTML code is
// <p id="para">Hello <em>World</em></p>
function hide(e) {
  // whether you click Hello or World, always returns true
  console.log(this === e.currentTarget);

  // clicking Hello returns true
  // clicking World returns false
  console.log(this === e.target);
}

document.getElementById('para').addEventListener('click', hide, false);
```

In the code above, `<em>` is a child node of `<p>`. Clicking `<em>` or clicking `<p>` will both cause the listener function to execute. At this point, `e.target` always points to the node at the original click position, while `e.currentTarget` points to the node being passed through during event propagation. Since the listener function is only triggered when the event passes through, `e.currentTarget` always equals the `this` inside the listener function.



#### 2.4 Event.type (The Event Type, e.g., 'click')

The `Event.type` property **returns a string representing the event type**. The type of an event is specified when the event is generated. This property is read-only.

```js
var evt = new Event('foo');
evt.type // "foo"
```



#### 2.5 Event.timeStamp (Millisecond Timestamp Relative to When the Page Was Opened)

The `Event.timeStamp` property returns a millisecond timestamp indicating the time the event occurred. It is calculated relative to when the page finished loading.

```js
var evt = new Event('foo');
evt.timeStamp // 3683.6999999995896
```

Its return value may be an integer or a decimal (a high-precision timestamp), depending on the browser's settings.

Below is an **example of calculating the speed of mouse movement**, displaying the number of pixels moved per second.

###### Example: Calculating Mouse Movement Speed

```js
var previousX;
var previousY;
var previousT;

window.addEventListener('mousemove', function(event) {
  if (
    previousX !== undefined &&
    previousY !== undefined &&
    previousT !== undefined
  ) {
    var deltaX = event.screenX - previousX;
    var deltaY = event.screenY - previousY;
    var deltaD = Math.sqrt(Math.pow(deltaX, 2) + Math.pow(deltaY, 2));

    var deltaT = event.timeStamp - previousT;
    console.log(deltaD / deltaT * 1000);
  }

  previousX = event.screenX;
  previousY = event.screenY;
  previousT = event.timeStamp;
});
```



#### 2.6 Event.isTrusted (Whether the Event Was Generated by a Real User Action)

The `Event.isTrusted` property returns a **boolean** indicating **whether the event was produced by a real user action**. For example, a user clicking a link produces a `click` event that is generated by the user; an event generated by the `Event` constructor is generated by a script.

```js
var evt = new Event('foo');
evt.isTrusted // false
```

In the code above, the `evt` object is generated by a script, so the `isTrusted` property returns `false`.



#### 2.7 Event.detail (Details of the Event, e.g., Single-Click or Double-Click)

The `Event.detail` property is only present for the browser's UI (user interface) events. This property returns a number representing some information about the event. Its specific meaning depends on the event type. For example, for `click` and `dblclick` events, `Event.detail` is the number of times the mouse was pressed (`1` for a single click, `2` for a double click, `3` for a triple click); for mouse wheel events, `Event.detail` is the distance the wheel scrolled in the positive direction, with a negative value being the distance scrolled in the negative direction, and the return value is always a multiple of 3.

```js
// HTML code is as follows
// <p>Hello</p>
function giveDetails(e) {
  console.log(e.detail); // the number of rapid clicks
}

document.querySelector('p').onclick = giveDetails;
```



### 3. Instance Methods

#### 3.1 Event.preventDefault() — Cancel the Browser's Default Behavior for the Current Event

The `Event.preventDefault` method **cancels the browser's default behavior for the current event**. For example, after clicking a link, the browser by default navigates to another page; after using this method, it will not navigate. Or, for example, pressing the spacebar scrolls the page down a certain distance; after using this method, it will not scroll. The prerequisite for this method to take effect is that the event object's `cancelable` property is `true`; if it is `false`, calling this method has no effect.

**Note that this method only cancels the event's default effect on the current element; it does not stop the event's propagation.** If you want to stop propagation, you can use the `stopPropagation()` or `stopImmediatePropagation()` methods.

```js
// HTML code is
// <input type="checkbox" id="my-checkbox" />
var cb = document.getElementById('my-checkbox');

cb.addEventListener(
  'click',
  function (e){ e.preventDefault(); },
  false
);
```

In the code above, the browser's default behavior is that clicking selects the checkbox; canceling this behavior makes it impossible to select the checkbox.

Using this method, you can set validation conditions for a text input box. If the user's input does not meet the conditions, the character cannot be entered into the text box.

###### Example: An Input Box That Only Accepts Letters

```js
// HTML code is
// <input type="text" id="my-input" />
var input = document.getElementById('my-input');
input.addEventListener('keypress', checkName, false);

function checkName(e) {
  if (e.charCode < 97 || e.charCode > 122) {
    e.preventDefault();
  }
}
```

After setting a listener function for the text box's `keypress` event, the code above only allows lowercase letters to be entered; otherwise the default behavior of the input event (writing into the text box) is canceled, so nothing can be entered into the text box.



#### 3.2 Event.stopPropagation() — Stop Event Propagation

The `stopPropagation` method stops the event from continuing to propagate through the DOM, preventing it from triggering listener functions defined on other nodes, but it does not include the other event listener functions on the current node.

```js
function stopEvent(e) {
  e.stopPropagation(); // stop event bubbling
}

el.addEventListener('click', stopEvent, false);
```

In the code above, the `click` event will not bubble any further up to the parent node of the `el` node.



#### 3.3 Event.stopImmediatePropagation() — Prevent Other Listener Functions of the Same Event from Being Called

The `Event.stopImmediatePropagation` method prevents the other listener functions of the same event from being called, regardless of whether those listener functions are defined on the current node or other nodes. That is, this method stops the event's propagation more thoroughly than `Event.stopPropagation()`.

If the same node has multiple listener functions specified for the same event, these functions are called in turn according to the order they were added. As soon as one of the listener functions calls the `Event.stopImmediatePropagation` method, the other listener functions will no longer execute.

```js
function l1(e){
  e.stopImmediatePropagation();
}

function l2(e){ // will not be called
  console.log('hello world');
}

el.addEventListener('click', l1, false);
el.addEventListener('click', l2, false);
```

The code above adds two listener functions, `l1` and `l2`, for the `click` event on the `el` node. Because `l1` calls the `event.stopImmediatePropagation` method, `l2` will not be called.



#### 3.4 Event.composedPath() — An Array of the Target and the Bubbling Nodes

`Event.composedPath()` returns an array whose members are the event's bottommost node and all the upper-layer nodes it passes through in turn while bubbling.

```js
// HTML code is as follows
// <div>
//   <p>Hello</p>
// </div>
var div = document.querySelector('div');
var p = document.querySelector('p');

div.addEventListener('click', function (e) {
  console.log(e.composedPath());
}, false);
// [p, div, body, html, document, Window]
```

In the code above, the bottommost node of the `click` event is `p`, and going upward in turn are `div`, `body`, `html`, `document`, and `Window`.





## 4. Mouse Events

### 1. Types of Mouse Events

Mouse events are events related to the mouse, inheriting the `MouseEvent` interface. The specific events mainly include the following.

- `click`: triggered when the mouse is pressed (usually the primary button). **【Single click】**
- `dblclick`: triggered when the mouse is double-clicked on the same element. **【Double click】**
- `mousedown`: triggered when a mouse button is pressed. **【Press down】**
- `mouseup`: triggered when a pressed mouse button is released. **【Release】**
- `mousemove`: triggered when the mouse moves within a node. As the mouse continues to move, this event is triggered continuously. To avoid performance problems, it is recommended to add some limit to the listener function for this event, such as limiting it to run only once within a certain period of time. **【Pass over (multiple times)】**
- `mouseenter`: triggered when the mouse enters a node; entering a child node does not trigger this event (see below for details). **【Enter (once)】**
- `mouseover`: triggered when the mouse enters a node; entering a child node triggers this event again (see below for details). **【Enter + child nodes】**
- `mouseleave`: triggered when the mouse leaves a node; leaving a parent node does not trigger this event (see below for details). **【Leave (once)】**
- `mouseout`: triggered when the mouse leaves a node; leaving a parent node triggers this event (see below for details). **【Leave + child nodes】**
- `contextmenu`: triggered when the right mouse button is pressed (before the context menu appears), or when the "context menu key" is pressed. **【Right click】**
- `wheel`: triggered when the mouse wheel is scrolled; this event inherits the `WheelEvent` interface. **【Wheel】**

A `click` event refers to the user completing a `mousedown` action and then a `mouseup` action at the same position. Therefore, the trigger order is: `mousedown` is triggered first, `mouseup` is triggered next, and `click` is triggered last.

A `dblclick` event is triggered after `mousedown`, `mouseup`, and `click`.

The `mouseover` event and the `mouseenter` event are both triggered when the mouse enters a node. The difference between them is that the `mouseenter` event is triggered only once, while the `mouseover` event is triggered multiple times on child nodes as long as the mouse moves within the node.

```js
/* HTML code is as follows
 <ul>
   <li>item 1</li>
   <li>item 2</li>
  <li>item 3</li>
 </ul>
*/

var ul = document.querySelector('ul');

// after entering the ul node, the mouseenter event is triggered only once
// afterward, as long as the mouse moves within the node, this event will not be triggered again
// event.target is the ul node
ul.addEventListener('mouseenter', function (event) {
  event.target.style.color = 'purple';
  setTimeout(function () {
    event.target.style.color = '';
  }, 500);
}, false);

// after entering the ul node, as long as it moves over a child node, the mouseover event is triggered multiple times
// event.target is the li node
ul.addEventListener('mouseover', function (event) {
  event.target.style.color = 'orange';
  setTimeout(function () {
    event.target.style.color = '';
  }, 500);
}, false);
```

In the code above, entering a child node within the parent node does not trigger the `mouseenter` event, but it does trigger the `mouseover` event.

The `mouseout` event and the `mouseleave` event are both triggered when the mouse leaves a node. The difference between them is that when leaving a child element within a parent element, the `mouseleave` event is not triggered, but the `mouseout` event is.

```js
/* HTML code is as follows
 <ul>
   <li>item 1</li>
   <li>item 2</li>
  <li>item 3</li>
 </ul>
*/

var ul = document.querySelector('ul');

// after first entering the ul node, then moving within the node, the mouseleave event is not triggered
// only when leaving the ul node is mouseleave triggered once
// event.target is the ul node
ul.addEventListener('mouseleave', function (event) {
  event.target.style.color = 'purple';
  setTimeout(function () {
    event.target.style.color = '';
  }, 500);
}, false);

// after first entering the ul node, then moving within the node, the mouseout event is triggered multiple times
// event.target is the li node
ul.addEventListener('mouseout', function (event) {
  event.target.style.color = 'orange';
  setTimeout(function () {
    event.target.style.color = '';
  }, 500);
}, false);
```

In the code above, leaving a child node within the parent node does not trigger the `mouseleave` event, but it does trigger the `mouseout` event.



### 2. Overview of the MouseEvent Interface

The `MouseEvent` interface means that **all objects produced by mouse events are instances of `MouseEvent`**. In addition, **wheel events and drag events are also instances of `MouseEvent`.**

The `MouseEvent` interface **inherits the `Event` interface**, so it has all of `Event`'s properties and methods. It also has its own properties and methods.

The browser **natively provides a `MouseEvent` constructor** for creating a new `MouseEvent` instance.

```js
var event = new MouseEvent(type, options);// first parameter, the event name string; second parameter, the event configuration object
```

The `MouseEvent` constructor accepts two parameters. The first parameter is a string representing the **event name**; the second parameter is an **event configuration object**, and this parameter is optional. Besides the instance configuration properties of the `Event` interface, this object can configure the following properties, all of which are optional.

- `screenX`: a number, the horizontal position of the mouse relative to the **screen** (in pixels), defaulting to 0; setting this property does not move the mouse.
- `screenY`: a number, the vertical position of the mouse relative to the **screen** (in pixels); otherwise the same as `screenX`.
- `clientX`: a number, the horizontal position of the mouse relative to the **program window** (in pixels), defaulting to 0; setting this property does not move the mouse.
- `clientY`: a number, the vertical position of the mouse relative to the **program window** (in pixels); otherwise the same as `clientX`.
- `ctrlKey`: a boolean, whether the Ctrl key was pressed simultaneously, defaulting to `false`.
- `shiftKey`: a boolean, whether the Shift key was pressed simultaneously, defaulting to `false`.
- `altKey`: a boolean, whether the Alt key was pressed simultaneously, defaulting to `false`.
- `metaKey`: a boolean, whether the Meta key (the win key) was pressed simultaneously, defaulting to `false`.
- `button`: a number indicating which mouse button was pressed, defaulting to `0`, which means the primary button was pressed (usually the left mouse button) or the current event does not define this property; `1` means the auxiliary button was pressed (usually the middle mouse button), and `2` means the secondary button was pressed (usually the right mouse button).
- `buttons`: a number indicating which mouse buttons were pressed, a three-bit binary value defaulting to `0` (no button pressed). `1` (binary `001`) means the primary button (usually the left button) was pressed, `2` (binary `010`) means the secondary button (usually the right button) was pressed, and `4` (binary `100`) means the auxiliary button (usually the middle button) was pressed. Therefore, if `3` (binary `011`) is returned, it means the left and right buttons were pressed simultaneously.
- `relatedTarget`: a node object representing the event's related node, defaulting to `null`. For `mouseenter` and `mouseover` events, it represents the element node the mouse just left; for `mouseout` and `mouseleave` events, it represents the element node the mouse is entering.

Here is an example.

```js
var event = new MouseEvent('click2', {
    'bubbles': true,
    'cancelable': true
});
var cb = document.getElementById('checkbox');
cb.addEventListener('click2',function(){ // bind the event listener function
    console.log(22) // executed
})
cb.dispatchEvent(event);// trigger the event
```

The code above generates a mouse click event and triggers it.



### 3. Instance Properties of the MouseEvent Interface

#### 3.1 MouseEvent.altKey, MouseEvent.ctrlKey, MouseEvent.metaKey, MouseEvent.shiftKey

The four properties `MouseEvent.altKey`, `MouseEvent.ctrlKey`, `MouseEvent.metaKey`, and `MouseEvent.shiftKey` all return a boolean indicating whether the corresponding key was pressed when the event occurred. They are all **read-only** properties.

- `altKey` property: the Alt key
- `ctrlKey` property: the Ctrl key
- `metaKey` property: the Meta key (on a Mac keyboard it is a four-petaled little flower; on a Windows keyboard it is the Windows key)
- `shiftKey` property: the Shift key

```js
// HTML code is as follows
// <body onclick="showKey(event)">
function showKey(e) {
  console.log('ALT key pressed: ' + e.altKey);
  console.log('CTRL key pressed: ' + e.ctrlKey);
  console.log('META key pressed: ' + e.metaKey);
  console.log('SHIFT key pressed: ' + e.shiftKey);
}
```

In the code above, clicking the page will output whether the corresponding keys were pressed simultaneously.



#### 3.2 MouseEvent.button (Which Mouse Button); MouseEvent.buttons (Which Buttons Are Pressed Simultaneously)

The `MouseEvent.button` property returns **a number indicating which mouse button was pressed when the event occurred**. This property is **read-only**.

- 0: the primary button (usually the left button) was pressed, or this property was not initialized for the event (e.g., the `mousemove` event).
- 1: the auxiliary button (usually the middle button or wheel button) was pressed.
- 2: the secondary button (usually the right button) was pressed.

```js
// HTML code is
// <button onmouseup="whichButton(event)">Click</button>
var whichButton = function (e) {
  switch (e.button) {
    case 0:
      console.log('Left button clicked.');
      break;
    case 1:
      console.log('Middle button clicked.');
      break;
    case 2:
      console.log('Right button clicked.');
      break;
    default:
      console.log('Unexpected code: ' + e.button);
  }
}
```

The `MouseEvent.buttons` property returns a **three-bit value** indicating **which buttons are pressed simultaneously**. It is **used to handle the case where multiple mouse buttons are pressed at the same time**. This property is **read-only**.

- 1: binary `001` (decimal 1), means the left button is pressed.
- 2: binary `010` (decimal 2), means the right button is pressed.
- 4: binary `100` (decimal 4), means the middle button or wheel button is pressed.

When multiple buttons are pressed simultaneously, the bit corresponding to each pressed button will have a value. For example, pressing the left and right buttons simultaneously returns 3 (binary 011).

```js
document.body.addEventListener('mousemove',function(e){ // note: with click it is always 0
    console.log(e.buttons)
})
// 0 when no button is pressed
// left button pressed: 1  (001)
// right button pressed: 2  (010)
// middle button pressed: 4  (100)

// left and right buttons pressed: 3  (011)
// left and middle buttons pressed: 5  (101)
// right and middle buttons pressed: 6  (110)
// left, middle, and right buttons pressed: 7  (111)
```





#### 3.3 MouseEvent.clientX (X Coordinate Relative to the Browser); MouseEvent.clientY (Y Coordinate Relative to the Browser)

The `MouseEvent.clientX` property returns the horizontal coordinate of the mouse position relative to the top-left corner of the browser window (in pixels), and the `MouseEvent.clientY` property returns the vertical coordinate. Both of these properties are **read-only**.

```js
// HTML code is
// <body onmousedown="showCoords(event)">
function showCoords(evt){
  console.log(
    'clientX value: ' + evt.clientX + '\n' +
    'clientY value: ' + evt.clientY + '\n'
  );
}
```

These two properties also each have an **alias, `MouseEvent.x` and `MouseEvent.y`**.



#### 3.4 MouseEvent.movementX (X Distance from the Last Mouse Move Event); MouseEvent.movementY (Y Distance from the Last Mouse Move Event)

The `MouseEvent.movementX` property returns the horizontal distance between the current position and the previous `mousemove` event (in pixels). Numerically, it equals the following formula.

```js
currentEvent.movementX = currentEvent.screenX - previousEvent.screenX
```

The `MouseEvent.movementY` property returns the vertical distance between the current position and the previous `mousemove` event (in pixels). Numerically, it equals the following formula.

```js
currentEvent.movementY = currentEvent.screenY - previousEvent.screenY。
```

Both of these properties are **read-only**.



#### 3.5 MouseEvent.screenX (X Coordinate Relative to the Screen); MouseEvent.screenY (Y Coordinate Relative to the Screen)

The `MouseEvent.screenX` property returns the horizontal coordinate of the mouse position relative to the top-left corner of the screen (in pixels), and the `MouseEvent.screenY` property returns the vertical coordinate. Both of these properties are **read-only**.

```js
// HTML code is as follows
// <body onmousedown="showCoords(event)">
function showCoords(evt) {
  console.log(
    'screenX value: ' + evt.screenX + '\n',
    'screenY value: ' + evt.screenY + '\n'
  );
}
```



#### 3.6 MouseEvent.offsetX (X Offset); MouseEvent.offsetY (Y Offset)

The `MouseEvent.offsetX` property returns the horizontal distance from the mouse position to the left `padding` edge of the target node (in pixels), and the `MouseEvent.offsetY` property returns the vertical distance to the top `padding` edge of the target node. Both of these properties are read-only.

```js
/* HTML code is as follows
  <style>
    p {
      width: 100px;
      height: 100px;
      padding: 100px;
    }
  </style>
  <p>Hello</p>
*/
var p = document.querySelector('p');
p.addEventListener(
  'click',
  function (e) {
    console.log(e.offsetX); // includes padding
    console.log(e.offsetY);
  },
  false
);
```

In the code above, if the mouse clicks at the center position of the `p` element, it returns `150 150`. So the center position's distance from the left and top `padding` edges equals the width of the `padding` (100 pixels) plus half the width of the element's content area (50 pixels).



#### 3.7 MouseEvent.pageX (X Coordinate Relative to the Document); MouseEvent.pageY (Y Coordinate Relative to the Document)

The `MouseEvent.pageX` property returns the distance from the mouse position to the left edge of the document (in pixels), and the `MouseEvent.pageY` property returns the distance to the top edge of the document (in pixels). Their **return values include the parts of the document that are not visible**. Both of these properties are **read-only**.

```js
/* HTML code is as follows
  <style>
    body {
      height: 2000px;
    }
  </style>
*/
document.body.addEventListener(
  'click',
  function (e) {
    console.log(e.pageX);
    console.log(e.pageY);
  },
  false
);
```

In the code above, the page height is 2000 pixels, which produces a vertical scroll bar. After scrolling to the bottom of the page and clicking the mouse, the output `pageY` value will be close to 2000.



#### 3.8 MouseEvent.relatedTarget (The Event's Related Node)

The `MouseEvent.relatedTarget` property **returns the event's related node**. For events that have no related node, this property returns `null`. This property is **read-only**.

The table below lists the meanings of the `target` property value and the `relatedTarget` property value for different events.

| Event Name | target Property        | relatedTarget Property |
| :--------- | :--------------------- | :--------------------- |
| focusin    | the node receiving focus | the node losing focus  |
| focusout   | the node losing focus  | the node receiving focus |
| mouseenter | the node being entered | the node being left    |
| mouseleave | the node being left    | the node being entered |
| mouseout   | the node being left    | the node being entered |
| mouseover  | the node being entered | the node being left    |
| dragenter  | the node being entered | the node being left    |
| dragexit   | the node being left    | the node being entered |

Here is an example.

```js
/*
  HTML code is as follows
  <div id="outer" style="height:50px;width:50px;border-width:1px solid black;">
    <div id="inner" style="height:25px;width:25px;border:1px solid black;"></div>
  </div>
*/

var inner = document.getElementById('inner');
inner.addEventListener('mouseover', function (event) {
  console.log('entered ' + event.target.id + ' left ' + event.relatedTarget.id);
}, false);
inner.addEventListener('mouseenter', function (event) {
  console.log('entered ' + event.target.id + ' left ' + event.relatedTarget.id);
});
inner.addEventListener('mouseout', function () {
  console.log('left ' + event.target.id + ' entered ' + event.relatedTarget.id);
});
inner.addEventListener("mouseleave", function (){
  console.log('left ' + event.target.id + ' entered ' + event.relatedTarget.id);
});

// when the mouse moves from outer into inner, it outputs
// entered inner left outer
// entered inner left outer

// when the mouse moves from inner into outer, it outputs
// left inner entered outer
// left inner entered outer
```



### 4. Instance Methods of the MouseEvent Interface

#### 4.1 MouseEvent.getModifierState() — Whether a Specified Modifier Key Is Pressed

The `MouseEvent.getModifierState` method **returns a boolean indicating whether a specific modifier key is pressed. Its parameter is a string representing a [modifier key](https://developer.mozilla.org/en-US/docs/Web/API/KeyboardEvent/getModifierState#Modifier_keys_on_Gecko).**

```js
document.addEventListener('click', function (e) {
  console.log(e.getModifierState('CapsLock'));
}, false);
```

The code above can determine whether the user has pressed the Caps Lock key.



### 5. The WheelEvent Interface (Wheel)

#### 5.1 Overview

**The WheelEvent interface inherits the MouseEvent instance and represents the instance object of a mouse wheel event.** Currently, there is only one event related to the mouse wheel, the `wheel` event; when the user scrolls the mouse wheel, an instance of this event is generated.

The browser natively provides a `WheelEvent()` constructor for generating `WheelEvent` instances.

```js
var wheelEvent = new WheelEvent(type, options);
```

The `WheelEvent()` constructor can accept two parameters. The first is a string representing the event type, and for wheel events this value can currently only be `wheel`. The second parameter is the event's configuration object. Besides the configuration properties of `Event` and `UIEvent`, this object can accept the following properties, all of which are optional.

- `deltaX`: a number representing the horizontal scroll amount of the wheel, defaulting to 0.0.
- `deltaY`: a number representing the vertical scroll amount of the wheel, defaulting to 0.0.
- `deltaZ`: a number representing the Z-axis scroll amount of the wheel, defaulting to 0.0.
- `deltaMode`: a number representing the unit of the related scroll event, applicable to the three properties above. `0` means the scroll unit is pixels, `1` means the unit is lines, and `2` means the unit is pages, defaulting to `0`.



#### 5.2 Instance Properties

A `WheelEvent` event instance, besides having the instance properties and instance methods of `Event` and `MouseEvent`, also has some of its own instance properties, but it has no instance methods of its own.

The properties below are all read-only.

- `WheelEvent.deltaX`: a number representing the horizontal scroll amount of the wheel.
- `WheelEvent.deltaY`: a number representing the vertical scroll amount of the wheel.
- `WheelEvent.deltaZ`: a number representing the Z-axis scroll amount of the wheel.
- `WheelEvent.deltaMode`: a number representing the unit of the three properties above; `0` is pixels, `1` is lines, `2` is pages.





## 5. Keyboard Events

### 1. Types of Keyboard Events

Keyboard events are triggered by the user striking the keyboard. There are mainly three events: `keydown`, `keypress`, and `keyup`, all of which inherit the `KeyboardEvent` interface.

- `keydown`: triggered when a key on the keyboard is pressed. **【Press down】**
- `keypress`: triggered when a key with a value is pressed; that is, pressing a valueless key such as Ctrl, Alt, Shift, or Meta does not trigger this event. For a key with a value, pressing it first triggers the `keydown` event, then triggers this event. **【Press a key with a value】**
- `keyup`: this event is triggered when a key on the keyboard is released. **【Release】**

If the user holds down a key without releasing it, keyboard events are triggered continuously in the following order.

1. keydown
2. keypress
3. keydown
4. keypress
5. ... (repeat the above process)
6. keyup



### 2. Overview of the KeyboardEvent Interface

The `KeyboardEvent` interface is used to describe the user's interaction with the keyboard. This interface **inherits the `Event` interface** and defines its own instance properties and instance methods.

The browser natively provides a `KeyboardEvent` constructor for creating new keyboard event instances.

```js
new KeyboardEvent(type, options) // first parameter, the event type; second parameter, the event configuration object
```

The `KeyboardEvent` constructor accepts two parameters. The first parameter is a string representing the **event type**; the second parameter is an **event configuration object**, and this parameter is optional. Besides the properties provided by the `Event` interface, the following fields can also be configured, all of which are optional.

- `key`: a string, the key currently pressed, defaulting to an empty string. **【Key name】**
- `code`: a string representing the string form of the key currently pressed, defaulting to an empty string. **【Key code】**
- `location`: an integer, the position of the key currently pressed, defaulting to `0`.
- `ctrlKey`: a boolean, whether the Ctrl key is pressed, defaulting to `false`.
- `shiftKey`: a boolean, whether the Shift key is pressed, defaulting to `false`.
- `altKey`: a boolean, whether the Alt key is pressed, defaulting to `false`.
- `metaKey`: a boolean, whether the Meta key is pressed, defaulting to `false`.
- `repeat`: a boolean, whether the key is being repeated, defaulting to `false`.



### 3. Instance Properties of KeyboardEvent

#### 3.1 KeyboardEvent.altKey, KeyboardEvent.ctrlKey, KeyboardEvent.metaKey, KeyboardEvent.shiftKey 【Whether the Corresponding Key Is Pressed, a Boolean】

The following properties are all **read-only** properties that return a boolean indicating whether the corresponding key is pressed.

- `KeyboardEvent.altKey`: whether the Alt key is pressed
- `KeyboardEvent.ctrlKey`: whether the Ctrl key is pressed
- `KeyboardEvent.metaKey`: whether the meta key is pressed (on a Mac system it is a four-petaled little flower; on a Windows system it is the windows key)
- `KeyboardEvent.shiftKey`: whether the Shift key is pressed

Here is an example.

```js
function showChar(e) {
  console.log('ALT: ' + e.altKey);
  console.log('CTRL: ' + e.ctrlKey);
  console.log('Meta: ' + e.metaKey);
  console.log('Shift: ' + e.shiftKey);
}

document.body.addEventListener('keydown', showChar, false);
```



#### 3.2 KeyboardEvent.code (Key Code)

The `KeyboardEvent.code` property **returns a string representing the string form of the key currently pressed**. This property is **read-only**.

Below are the string forms of some common keys; for other keys, please consult the [documentation](https://developer.mozilla.org/en-US/docs/Web/API/KeyboardEvent/code#Code_values).

- Number keys 0 - 9: return `digital0` - `digital9`
- Letter keys A - z: return `KeyA` - `KeyZ`
- Function keys F1 - F12: return `F1` - `F12`
- Arrow keys: return `ArrowDown`, `ArrowUp`, `ArrowLeft`, `ArrowRight`
- Alt key: returns `AltLeft` or `AltRight`
- Shift key: returns `ShiftLeft` or `ShiftRight`
- Ctrl key: returns `ControlLeft` or `ControlRight`



#### 3.3 KeyboardEvent.key (Key Name)

The `KeyboardEvent.key` property **returns a string representing the name of the key pressed**. This property is **read-only**.

If the key pressed represents a printable character, it returns that character, for example a digit or letter.

If the key pressed represents an unprintable special character, it returns a predefined key value, for example Backspace, Tab, Enter, Shift, Control, Alt, CapsLock, Esc, Spacebar, PageUp, PageDown, End, Home, Left, Right, Up, Down, PrintScreen, Insert, Del, Win, F1–F12, NumLock, Scroll, etc.

If a control key and a symbol key are pressed simultaneously, the key name of the symbol key is returned. For example, pressing Ctrl + a returns `a`; pressing Shift + a returns the uppercase `A`.

If the key name cannot be recognized, it returns the string `Unidentified`.



#### 3.4 KeyboardEvent.location (Which Position the Key Is In, an Integer)

The `KeyboardEvent.location` property **returns an integer indicating which region of the keyboard the pressed key is in**. It can take the following values.

- 0: in the main region of the keyboard, or it cannot be determined which region it is in.

- 1: on the left side of the keyboard, applicable only to those keys that have two positions (such as the Ctrl and Shift keys).

- 2: on the right side of the keyboard, applicable only to those keys that have two positions (such as the Ctrl and Shift keys).

- 3: in the numeric keypad.



#### 3.5 KeyboardEvent.repeat (Whether the Key Is Held Down)

`KeyboardEvent.repeat` returns a **boolean indicating whether the key is being held down**, so as to determine whether the key is being repeated; that is, the browser continuously triggers the `keydown` and `keypress` events until the user releases their hand.



### 4. Instance Methods of KeyboardEvent

#### 4.1 KeyboardEvent.getModifierState() — Whether a Specified Modifier Key Is Pressed

The `KeyboardEvent.getModifierState()` method returns a **boolean indicating whether a specified modifier key is pressed or active**. Its common parameters are as follows.

- `Alt`: the Alt key
- `CapsLock`: the Caps Lock key
- `Control`: the Ctrl key
- `Meta`: the Meta key
- `NumLock`: the numeric keypad toggle key
- `Shift`: the Shift key

```js
if (
  event.getModifierState('Control') +
  event.getModifierState('Alt') +
  event.getModifierState('Meta') > 1
) {
  return;
}
```

The code above means that as long as any two or more of `Control`, `Alt`, and `Meta` are pressed simultaneously, it returns.



## 6. Progress Events



### 1. Types of Progress Events

**Progress events are used to describe the loading progress of resources.** They are mainly triggered by the loading of AJAX requests, `<img>`, `<audio>`, `<video>`, `<style>`, `<link>`, and other external resources, inheriting the `ProgressEvent` interface. They mainly include the following **events**.

- `abort`: triggered when the loading of an external resource is aborted (for example, the user cancels). If an error causes the abort, this event will not be triggered. **【Loading aborted】**
- `error`: triggered when an external resource cannot be loaded due to an error. **【Loading error】**
- `load`: triggered when an external resource loads successfully. **【Loading succeeded】**
- `loadstart`: triggered when an external resource begins loading. **【Loading started】**
- `loadend`: triggered when an external resource stops loading, occurring after events such as `error`, `abort`, and `load`. **【Loading stopped】**
- `progress`: triggered continuously during the loading of an external resource. **【Loading, triggered continuously】**
- `timeout`: triggered when loading times out. **【Loading timed out】**

Note that besides resource downloads, **file uploads also have these events**.

Here is an example.

```js
image.addEventListener('load', function (event) { // loaded successfully
  image.classList.add('finished');
});

image.addEventListener('error', function (event) { // loading error
  image.style.display = 'none';
});
```

The code above adds a `finished` class to the image element after the image element finishes loading. If loading fails, it sets the image element's style to not display.

Sometimes image loading completes before the script runs, especially when the script is placed at the bottom of the page, so the listener functions for the `load` and `error` events may not execute at all. Therefore, a more reliable approach is to first use the `complete` property to check whether loading has finished.

```js
function loaded() {
  // ...
}

if (image.complete) { // whether loading has finished
  loaded();
} else {
  image.addEventListener('load', loaded); // loaded-successfully event
}
```

Since DOM element nodes do not provide a property for whether a loading error occurred, the listener function for the `error` event is best placed in the HTML code of the `<img>` element, which guarantees that it will execute 100% of the time when a loading error occurs.

```html
<img src="/wrong/url" onerror="this.style.display='none';" />
```

The listener function for the `loadend` event can be used to replace the listener functions for the `abort`, `load`, and `error` events, because it always occurs after these events.

```js
req.addEventListener('loadend', loadEnd, false);

function loadEnd(e) {
  console.log('transfer finished, success or failure unknown');
}
```

The `loadend` event itself does not provide information about the reason the progress ended, but it can be used to do some operations that need to be done in all loading-end scenarios.

Additionally, the `error` event has a special characteristic: it does not bubble. So a child element's `error` event will not trigger the parent element's `error` event listener function.



### 2. The ProgressEvent Interface

#### 2.1 Overview

The `ProgressEvent` interface is mainly used to describe the loading progress of external resources, such as AJAX loading, `<img>`, `<video>`, `<style>`, `<link>`, and other external resource loading. Progress-related events all inherit this interface. **This interface inherits the Event interface.**

The browser natively provides a `ProgressEvent()` constructor for generating event instances.

```js
new ProgressEvent(type, options) // first parameter, the event type; second parameter, the configuration object
```

The `ProgressEvent()` constructor accepts two parameters. The first parameter is a string representing the event type, and this parameter is required. The second parameter is a configuration object representing the event's properties, and this parameter is optional. Besides being able to use the configuration properties of the `Event` interface, the configuration object can also use the following properties, all of which are optional.

- `lengthComputable`: a boolean indicating **whether the total amount to be loaded can be computed**, defaulting to `false`.
- `loaded`: an integer indicating **the amount already loaded**, defaulting to `0`.
- `total`: an integer indicating **the total amount that needs to be loaded**, defaulting to `0`.

#### 2.2 Instance Properties of ProgressEvent.

- `ProgressEvent.lengthComputable` **whether the total amount can be computed**
- `ProgressEvent.loaded` **the amount already loaded**
- `ProgressEvent.total ` **the total amount that needs to be loaded**

If `ProgressEvent.lengthComputable` is `false`, `ProgressEvent.total` is actually meaningless.

Here is an example.

```js
var p = new ProgressEvent('load', {
  lengthComputable: true,
  loaded: 30,
  total: 100,
});

document.body.addEventListener('load', function (e) {
  console.log('loaded: ' + (e.loaded / e.total) * 100 + '%');
});

document.body.dispatchEvent(p);
// loaded: 30%
```

The code above first constructs a `load` event, which after being dispatched is caught by the listener function.

Below is a practical example.

```js
var xhr = new XMLHttpRequest();

xhr.addEventListener('progress', updateProgress, false); // loading
xhr.addEventListener('load', transferComplete, false); // loaded successfully
xhr.addEventListener('error', transferFailed, false); // loading error
xhr.addEventListener('abort', transferCanceled, false); // loading aborted

xhr.open();

function updateProgress(e) { // loading
  if (e.lengthComputable) { // whether the total can be computed
    var percentComplete = e.loaded / e.total; // compute loading progress
  } else {
    console.log('cannot compute progress');
  }
}

function transferComplete(e) { // loaded successfully
  console.log('transfer finished');
}

function transferFailed(evt) { // loading error
  console.log('an error occurred during the transfer');
}

function transferCanceled(evt) { // loading aborted
  console.log('the user canceled the transfer');
}
```

The code above shows the progress events for the download process; there are also **progress events for the upload process**. In this case, all listener functions must be placed on the `XMLHttpRequest.upload` object.

```js
var xhr = new XMLHttpRequest();

xhr.upload.addEventListener('progress', updateProgress, false);
xhr.upload.addEventListener('load', transferComplete, false);
xhr.upload.addEventListener('error', transferFailed, false);
xhr.upload.addEventListener('abort', transferCanceled, false);

xhr.open();
```



## 7. Form Events

### 1. Types of Form Events

#### 1.1 The input Event (Triggered When the Value Changes, Triggered Continuously)

The `input` event is triggered when the **value of an `<input>`, `<select>`, or `<textarea>` changes**. For checkboxes (`<input type=checkbox>`) or radio buttons (`<input type=radio>`), it is also triggered when the user changes the selection. In addition, for an element with the `contenteditable` attribute turned on, the `input` event is also triggered whenever the value changes.

A characteristic of the `input` event is that it **is triggered continuously**; for example, **each time the user presses a key, the `input` event is triggered once**.

The `input` event object **inherits the `InputEvent` interface**.

This event is very similar to the `change` event; the difference is that **the `input` event occurs immediately after the element's value changes**, **whereas `change` occurs when the element loses focus**, at which point the content may have already changed multiple times. That is, if there are continuous changes, **the `input` event is triggered multiple times, while the `change` event is triggered only once when focus is lost.**

Below is an example with a `<select>` element.

```js
/* HTML code is as follows
<select id="mySelect">
  <option value="1">1</option>
  <option value="2">2</option>
  <option value="3">3</option>
</select>
*/

function inputHandler(e) {
  console.log(e.target.value)
}

var mySelect = document.querySelector('#mySelect');
mySelect.addEventListener('input', inputHandler);
```

In the code above, changing the dropdown option triggers the `input` event, thereby executing the callback function `inputHandler`.



#### 1.2 The select Event (Triggered When Text Is Selected)

The `select` event is triggered when **text is selected** within an `<input>` or `<textarea>`.

```js
// HTML code is as follows
// <input id="test" type="text" value="Select me!" />

var elem = document.getElementById('test');
elem.addEventListener('select', function (e) {
  console.log(e.type); // "select" the event type

  var _target = e.target;
  console.log(_target.value); // the entire value of the text box
  console.log(_target.selectionDirection); // the direction of the selection: 'forward' or 'backward'

  console.log(_target.selectionStart); // the index where the selection starts
  console.log(_target.selectionEnd); // the index where the selection ends
  // note: the start and end indices are independent of the selection direction; the start index is always the earlier one

  console.log(_target.value.slice(_target.selectionStart, _target.selectionEnd)) // the selected portion of the string
}, false);
```

The selected text can be obtained through the `selectionDirection`, `selectionEnd`, `selectionStart`, and `value` properties of the `event.target` element.



#### 1.3 The change Event (Triggered When the Value Changes, Once)

The `change` event is triggered when the **value of an `<input>`, `<select>`, or `<textarea>` changes**. Its biggest difference from the `input` event is that **it is not triggered continuously; it is only triggered when all modifications are complete**; on the other hand, an `input` event is necessarily accompanied by a `change` event. Specifically, it falls into the following cases.

- Triggered when a radio button (radio) or checkbox (checkbox) is activated.
- Triggered when the user submits. For example, completing a selection from a dropdown list (select), or completing a selection in a date or file input box.
- Triggered when the value of a text box or `<textarea>` element changes and it loses focus.

Here is an example.

```js
// HTML code is as follows
// <select size="1" onchange="changeEventHandler(event);">
//   <option>chocolate</option>
//   <option>strawberry</option>
//   <option>vanilla</option>
// </select>

function changeEventHandler(event) {
  console.log(event.target.value);
}
```

If you compare this with the `input` event example above, you will find that for a `<select>` element, the `input` and `change` events are essentially equivalent.



#### 1.4 The invalid Event (Triggered When a Form Submission Does Not Meet the Conditions)

**When the user submits a form, if the value of a form element does not meet the validation conditions, the `invalid` event is triggered.**

```html
<form>
  <input type="text" required oninvalid="console.log('invalid input')" />
  <button type="submit">Submit</button>
</form>
```

In the code above, the input box is required. If it is not filled in, when the user clicks the button to submit, the input box's `invalid` event is triggered, causing the submission to be canceled.



#### 1.5 The reset Event (Reset), The submit Event (Submit)

The `reset` event is triggered when **the form is reset (all form members revert to their default values).**

The `submit` event is triggered when **form data is submitted to the server**.



Note that **these two events occur on the form object `<form>`, not on the form's members**, because what is submitted is the form, not the form members.

```html
<form onreset="console.log('reset event triggered')" onsubmit="console.log('submit event triggered')" name="input" action="html_form_action.php" method="get">
    <input type="text" name="lname" value="Duck">

    <button type="reset">Reset</button>
    <button type="submit">Submit</button>
</form>
```



### 2. The InputEvent Interface (The Instance of the input Event)

**The `InputEvent` interface is mainly used to describe the instance of the `input` event.** This interface **inherits the `Event` interface** and also defines some of its own instance properties and instance methods.

The browser natively provides an `InputEvent()` constructor for generating instance objects.

```js
new InputEvent(type, options) // first parameter, the event name; second parameter, the configuration object
```

The `InputEvent` constructor can accept two parameters. The first parameter is a string representing the **event name**, and this parameter is required. The second parameter is a **configuration object** used to set the properties of the event instance, and this parameter is optional. Besides the configuration properties of the `Event` constructor, the configuration object can also set the following fields, all of which are optional.

- `inputType`: a string representing the **type of change that occurred** (see below).
- `data`: a string representing the **inserted string**. If there is no inserted string (for example, a delete operation), it returns `null` or an empty string.
- `dataTransfer`: returns a **DataTransfer object instance; this property is usually only valid when the input box accepts rich text input**.

The instance properties of `InputEvent` are mainly the three properties above, and all three instance properties are **read-only**.

#### (1) InputEvent.data (The Portion of the Content That Changed)

The `InputEvent.data` property returns a string representing **the changed content**.

```js
// HTML code is as follows
// <input type="text" id="myInput">
var input = document.getElementById('myInput');
input.addEventListener('input', myFunction, false);

function myFunction(e) {
  console.log(e.data);
}
```

In the code above, if you manually type `abc` into the input box, the console will first output `a`, then `b` on the next line, then `c` on the next line. Then if you select `abc` and delete it all at once, the console will output `null` or an empty string.



#### (2) InputEvent.inputType (The Type of Change)

The `InputEvent.inputType` property returns a string representing the **type of change** that occurred to the string.

For common cases, the Chrome browser's return values are as follows. For the complete list, refer to the [documentation](https://w3c.github.io/input-events/index.html#dom-inputevent-inputtype).

- Manually inserting text: `insertText`
- Inserting text by pasting: `insertFromPaste`
- Deleting backward: `deleteContentBackward`
- Deleting forward: `deleteContentForward`



#### (3) InputEvent.dataTransfer

The `InputEvent.dataTransfer` property **returns a DataTransfer instance. This property is only valid when the text box accepts pasted content (`insertFromPaste`) or dragged content (`insertFromDrop`).**



## 8. Touch Events

### 1. Overview of Touch Operations

The browser's touch API consists of three parts.

- Touch: an instance of a single touch point
- TouchList: an instance of a collection of multiple touch points
- TouchEvent: an instance of an event raised by a touch

The instance object of the `Touch` interface is used to represent a touch point (a finger or a stylus), including properties such as **position, size, shape, pressure, and target element**. Sometimes a touch action consists of multiple touch points (multiple fingers), and a collection of multiple touch points is represented by an instance object of the `TouchList` interface. The instance object of the `TouchEvent` interface represents an event raised by a touch; only a touchscreen can raise this type of event.

In many cases, **touch events and mouse events are triggered simultaneously**, even when the mouse is not actually used. This is so that code that only defines mouse events and not touch events can still work in the case of a touchscreen. If you want to avoid this situation, you **can use the `event.preventDefault` method to prevent the mouse events from being fired**.

### 2. The Touch Interface

#### 2.1 Overview of the Touch Interface (A Single Touch Point)

The Touch interface **represents a single touch point**. A touch point may be a finger or a stylus.

The browser natively provides a `Touch` constructor for generating `Touch` instances.

```js
var touch = new Touch(touchOptions);
```

The `Touch` constructor accepts a **configuration object as a parameter**, which has the following properties.

- `identifier`: required, of integer type, representing the unique ID of the touch point.
- `target`: required, of element-node type, representing the page element where the touch point started.
- `clientX`: optional, of number type, representing the horizontal distance of the touch point relative to the top-left corner of the browser window, defaulting to 0.
- `clientY`: optional, of number type, representing the vertical distance of the touch point relative to the top-left corner of the browser window, defaulting to 0.
- `screenX`: optional, of number type, representing the horizontal distance of the touch point relative to the top-left corner of the screen, defaulting to 0.
- `screenY`: optional, of number type, representing the vertical distance of the touch point relative to the top-left corner of the screen, defaulting to 0.
- `pageX`: optional, of number type, representing the horizontal position of the touch point relative to the top-left corner of the page (i.e., including the page's scroll distance), defaulting to 0.
- `pageY`: optional, of number type, representing the vertical position of the touch point relative to the top-left corner of the page (i.e., including the page's scroll distance), defaulting to 0.
- `radiusX`: optional, of number type, representing the X-axis radius of the elliptical area affected around the touch point, defaulting to 0.
- `radiusY`: optional, of number type, representing the Y-axis radius of the elliptical area affected around the touch point, defaulting to 0.
- `rotationAngle`: optional, of number type, representing the rotation angle of the ellipse of the touch area, in degrees, between 0 and 90 degrees, defaulting to 0.
- `force`: optional, of number type, in the range between `0` and `1`, representing the touch pressure. `0` represents no pressure, `1` represents the maximum pressure the hardware can recognize, defaulting to `0`.



#### 2.2 Instance Properties of the Touch Interface

##### (1) Touch.identifier (The Touch Point's ID)

The `Touch.identifier` property returns an integer representing the unique ID of the touch point. This value remains unchanged throughout the entire touch process, until the touch event ends.

```js
someElement.addEventListener('touchmove', function (e) {
  for (var i = 0; i < e.changedTouches.length; i++) {
    console.log(e.changedTouches[i].identifier);
  }
}, false);
```



##### (2) Touch.screenX, Touch.screenY, Touch.clientX, Touch.clientY, pageX, pageY (Coordinates Relative to the Screen, Browser, and Document)

The `Touch.screenX` and `Touch.screenY` properties represent the horizontal and vertical coordinates of the touch point relative to the top-left corner of the screen, respectively, regardless of whether the page is scrolled.

The `Touch.clientX` and `Touch.clientY` properties represent the horizontal and vertical coordinates of the touch point relative to the top-left corner of the browser viewport, respectively, regardless of whether the page is scrolled.

The `Touch.pageX` and `Touch.pageY` properties represent the horizontal and vertical coordinates of the touch point relative to the top-left corner of the current page, respectively, including the displacement caused by page scrolling.



##### (3) Touch.radiusX, Touch.radiusY, Touch.rotationAngle (The Radius and Angle of the Elliptical Touch Area)

The `Touch.radiusX` and `Touch.radiusY` properties return **the X-axis radius and Y-axis radius of the elliptical area affected around the touch point, in pixels. Multiplying by 2 gives the width and height of the touch area**.

The `Touch.rotationAngle` property represents the rotation angle of the ellipse of the touch area, in degrees, between `0` and `90` degrees.

These three properties together define **the area where the user contacts the screen**, which is helpful for describing imprecise touches such as a finger. When a fingertip contacts the screen, the touch range forms an ellipse, and these three properties are used to describe this elliptical area.

Here is an example.

```js
div.addEventListener('touchstart', rotate);
div.addEventListener('touchmove', rotate);
div.addEventListener('touchend', rotate);

function rotate(e) {
  var touch = e.changedTouches.item(0);
  e.preventDefault();

  src.style.width = touch.radiusX * 2 + 'px';
  src.style.height = touch.radiusY * 2 + 'px';
  src.style.transform = 'rotate(' + touch.rotationAngle + 'deg)';
};
```



##### (4) Touch.force (Touch Pressure)

The `Touch.force` property returns a number between `0` and `1` representing the touch pressure. `0` represents no pressure, and `1` represents the maximum pressure the hardware can recognize.



##### (5) Touch.target (The Element Where the Touch Started)

The `Touch.target` property returns an element node representing the element node where the touch occurred. Even if the touch point has already left this node, this property remains unchanged.



### 3. The TouchList Interface

The `TouchList` interface **represents a collection of touch points**. Its instance is an **array-like object** whose members are instances of `Touch`, representing all the touch points. When a user touches with three fingers, the resulting `TouchList` instance will contain three members, with each finger's touch point corresponding to one `Touch` instance object.

Its instances are mainly obtained through the touch event's `TouchEvent.touches`, `TouchEvent.changedTouches`, and `TouchEvent.targetTouches` properties.

It has only two instance properties and instance methods.

- `TouchList.length`: a number representing the number of members (i.e., the number of touch points).
- `TouchList.item()`: returns the member at the specified position; its parameter is the position number of that member (starting from zero).



### 4. The TouchEvent Interface

#### 4.1 Overview

The TouchEvent interface **inherits the Event interface** and represents an **event instance raised by a touch**, usually from a touchscreen or trackpad. In addition to the inherited properties, it also has some of its own properties.

The browser natively provides a `TouchEvent()` constructor for generating touch event instances.

```js
new TouchEvent(type, options)
```

The `TouchEvent()` constructor can accept two parameters. The first parameter is a string representing the event type; the second parameter is the event's configuration object, which is optional, and all of the object's properties are also optional. Besides the configuration properties of the `Event` interface, this interface also has some of its own configuration properties.

- `touches`: a `TouchList` instance representing all currently active touch points, defaulting to an empty array `[]`.
- `targetTouches`: a `TouchList` instance representing all touch points that are inside the touch's target element node and still active, defaulting to an empty array `[]`.
- `changedTouches`: a `TouchList` instance representing the touch points relevant to this touch event, defaulting to an empty array `[]`.
- `ctrlKey`: a boolean indicating whether the Ctrl key is pressed simultaneously, defaulting to `false`.
- `shiftKey`: a boolean indicating whether the Shift key is pressed simultaneously, defaulting to `false`.
- `altKey`: a boolean indicating whether the Alt key is pressed simultaneously, defaulting to `false`.
- `metaKey`: a boolean indicating whether the Meta key (or the Windows key) is pressed simultaneously, defaulting to `false`.



#### 4.2 Instance Properties

An instance of the TouchEvent interface has all the properties and methods of an `Event` instance, plus some of its own instance properties, all of which are read-only.

##### (1) TouchEvent.altKey, TouchEvent.ctrlKey, TouchEvent.shiftKey, TouchEvent.metaKey (Whether Certain Modifier Keys Are Pressed Simultaneously)

- `TouchEvent.altKey`: a boolean indicating whether the Alt key was pressed during the touch.
- `TouchEvent.ctrlKey`: a boolean indicating whether the Ctrl key was pressed during the touch.
- `TouchEvent.shiftKey`: a boolean indicating whether the Shift key was pressed during the touch.
- `TouchEvent.metaKey`: a boolean indicating whether the Meta key (or the Windows key) was pressed during the touch.

Here is an example.

```js
someElement.addEventListener('touchstart', function (e) {
  console.log('altKey = ' + e.altKey);
  console.log('ctrlKey = ' + e.ctrlKey);
  console.log('metaKey = ' + e.metaKey);
  console.log('shiftKey = ' + e.shiftKey);
}, false);
```



##### (2) TouchEvent.changedTouches (Collection of Touch Points; Meaning Differs by Touch Event)

The `TouchEvent.changedTouches` property returns a `TouchList` instance whose members are a group of `Touch` instance objects representing the touch points relevant to this touch event.

For **different events, the meaning of this property differs.**

- `touchstart` event: the touch points that were activated
- `touchmove` event: the touch points that changed
- `touchend` event: the touch points that disappeared (i.e., the points that are no longer being touched)

Here is an example.

```js
someElement.addEventListener('touchmove', function (e) {
  for (var i = 0; i < e.changedTouches.length; i++) {
    console.log(e.changedTouches[i].identifier);
  }
}, false);
```



##### (3) TouchEvent.touches (Collection of Still-Active Touch Points)

The `TouchEvent.touches` property returns a `TouchList` instance whose members are all the **touch points still active (i.e., being touched)**. Generally speaking, one finger is one touch point.

Here is an example.

```js
someElement.addEventListener('touchstart', function (e) {
  switch (e.touches.length) {
    // one-finger touch
    case 1: handle_one_touch(e); break;
    // two-finger touch
    case 2: handle_two_touches(e); break;
    // three-finger touch
    case 3: handle_three_touches(e); break;
    // other cases
    default: console.log('Not supported'); break;
  }
}, false);
```



##### (4) TouchEvent.targetTouches (Collection of Active Touch Points Within the Target Element)

The `TouchEvent.targetTouches` property returns a `TouchList` instance whose members are all the **touch points still active (i.e., being touched) inside the target element node of the touch event**.

```js
function touches_in_target(ev) {
  return (ev.touches.length === ev.targetTouches.length ? true : false);
}
```

The code above is used to determine whether all touch points are inside the target element.



### 5. Types of Touch Events

There are several types of events raised by touches. You can use the `TouchEvent.type` property to see which type of event actually occurred.

- `touchstart`: triggered when the user begins to touch; its `target` property returns the element node where the touch occurred. **【Touch start】**
- `touchend`: triggered when the user no longer contacts the touchscreen (or moves off the edge of the screen); its `target` property is consistent with the `touchstart` event, namely the element node where the touch started. Its `changedTouches` property returns a `TouchList` instance containing all the touch points that are no longer being touched (i.e., `Touch` instance objects). **【Touch end】**
- `touchmove`: triggered when the user moves a touch point; its `target` property is consistent with the `touchstart` event. If the touch's radius, angle, or force changes, this event is also triggered. **【Touch moving】**
- `touchcancel`: triggered when a touch point is canceled, for example when a modal window pops up in the touch area, when the touch point leaves the document area (entering the browser menu bar), or when the user has too many touch points exceeding the supported limit (the earlier touch points are automatically canceled). **【Touch point canceled】**

Here is an example.

```js
var el = document.getElementsByTagName('canvas')[0];
el.addEventListener('touchstart', handleStart, false);
el.addEventListener('touchmove', handleMove, false);

function handleStart(evt) {
  evt.preventDefault();
  var touches = evt.changedTouches;
  for (var i = 0; i < touches.length; i++) {
    console.log(touches[i].pageX, touches[i].pageY);
  }
}

function handleMove(evt) {
  evt.preventDefault();
  var touches = evt.changedTouches;
  for (var i = 0; i < touches.length; i++) {
    var touch = touches[i];
    console.log(touch.pageX, touch.pageY);
  }
}
```



## 9. Drag Events

### 1. Types of Drag Events

Dragging (drag) refers to **the user pressing and holding a mouse button on an object, dragging it to another position, then releasing the mouse button to place the object there**.

There are several kinds of objects that can be dragged, including **element nodes, images, links, selected text, and so on**. On a web page, except for element nodes which are not draggable by default, the others (images, links, selected text) can all be dragged directly. To make an element node draggable, you can set the node's `draggable` attribute to `true`.

```html
<div draggable="true">
  This area can be dragged
</div>
```

The `draggable` attribute **can be used on any element node**, but images (`<img>`) and links (`<a>`) can be dragged without adding this attribute. For them, this attribute is often used to set it to `false`, to prevent these two kinds of elements from being dragged.

**Note** that once an element node's `draggable` attribute is set to `true`, **you can no longer use the mouse to select the text or child nodes inside that node**.

When an element node or selected text is being dragged, **drag events are triggered continuously**, including the following events.

**Triggered on the dragged node:**

- `drag`: **during the dragging process**, triggered **continuously** on the dragged node (every few hundred milliseconds).
- `dragstart`: triggered on the dragged node when the user **starts dragging**; this event's `target` property is the dragged node. You should usually specify the drag data in this event's listener function.
- `dragend`: triggered on the dragged node when **dragging ends** (releasing the mouse button or pressing the ESC key); this event's `target` property is the dragged node. It is triggered on the same node as the `dragstart` event. Regardless of whether the drag crosses windows or is canceled midway, the `dragend` event always fires.



**Triggered on the node being dragged to:**

- `dragenter`: triggered once on the current node when the **drag enters the current node**; this event's `target` property is the current node. You should usually specify in this event's listener function **whether to allow the dragged data to be dropped on the current node**. If the current node has no listener function for this event, or the listener function does not perform any operation, it means the data is not allowed to be dropped on the current node. Showing visually that the drag has entered the current node is also set in this event's listener function.

- `dragover`: triggered **continuously** on the current node (every few hundred milliseconds) when **dragging over the current node**; this event's `target` property is the current node. The difference between this event and the `dragenter` event is that the `dragenter` event is triggered when entering the node, after which, as long as the drag does not leave the node, the `dragover` event is triggered continuously.

- `dragleave`: triggered on the current node when **the drag operation leaves the range of the current node**; this event's `target` property is the current node. If you want to show visually that the drag has left the current node, set it in this event's listener function.

- `drop`: triggered on the target node when **the dragged node or selected text is released onto the target node**. Note that if the current node does not allow `drop`, then even if you release the mouse button over that node, this event will not be triggered. If the user presses the ESC key to cancel the operation, this event will also not be triggered. This event's listener function is responsible for extracting the drag data and performing related processing.



The example below shows how to dynamically change the background color of the dragged node.

```js
div.addEventListener('dragstart', function (e) {
  this.style.backgroundColor = 'red';
}, false);

div.addEventListener('dragend', function (e) {
  this.style.backgroundColor = 'green';
}, false);
```

In the code above, when the `div` node is being dragged, its background color turns red, and when dragging ends, it turns back to green.

```html
<div class="div1" draggable="true">
    div1, this area can be dragged
</div>

<div class="div2">
    div2
</div>

<script>
    var div1 = document.querySelector('.div1');
    var div2 = document.querySelector('.div2');

    div1.addEventListener('dragstart', function (e){
        console.log('drag started')
    });
    div1.addEventListener('drag', function (e) {
        console.log('dragging') // triggered continuously
    });
    div1.addEventListener('dragend', function (e){
        console.log('drag ended')
    });


    div2.addEventListener('dragenter', function(){
        console.log('dragged onto div2')
    })
    div2.addEventListener('dragover', function(){
        //console.log('over div2') // triggered continuously
    })
    div2.addEventListener('dragleave', function(){
        console.log('left div2')
    })
    div2.addEventListener('drop', function() {
        console.log('released on div2')
    })
</script>
```



Below is an example showing how to drag a node from its current parent node into another parent node.

```js
/* HTML code is as follows
 <div class="dropzone">
   <div id="draggable" draggable="true">
     This node can be dragged
   </div>
 </div>
 <div class="dropzone"></div>
 <div class="dropzone"></div>
 <div class="dropzone"></div>
*/

// the dragged node
var dragged;

document.addEventListener('dragstart', function (event) {
  // save the dragged node
  dragged = event.target;
  // make the dragged node's background semi-transparent
  event.target.style.opacity = 0.5;
}, false);

document.addEventListener('dragend', function (event) {
  // restore the dragged node's background to normal
  event.target.style.opacity = '';
}, false);

document.addEventListener('dragover', function (event) {
  // prevent the drag effect from being reset, allowing the dragged node to be dropped into the target node
  event.preventDefault();
}, false);

document.addEventListener('dragenter', function (event) {
  // turn the target node's background color purple
  // since this event bubbles, we need to filter the nodes
  if (event.target.className === 'dropzone') {
    event.target.style.background = 'purple';
  }
}, false);

document.addEventListener('dragleave', function( event ) {
  // restore the target node's background color to normal
  if (event.target.className === 'dropzone') {
    event.target.style.background = '';
  }
}, false);

document.addEventListener('drop', function( event ) {
  // prevent the event's default behavior (such as opening links on certain element nodes),
  event.preventDefault();
  if (event.target.className === 'dropzone') {
    // restore the target node's background color
    event.target.style.background = '';
    // insert the dragged node into the target node
    dragged.parentNode.removeChild(dragged);
    event.target.appendChild( dragged );
  }
}, false);
```

There are a few things to note about drag events.

- During the dragging process, only the above drag events are triggered; even though the mouse is moving, **mouse events are not triggered**.
- Dragging a file from the operating system into the browser does not trigger the `dragstart` and `dragend` events.
- The listener functions for the `dragenter` and `dragover` events are used to extract the drag data (i.e., to allow the dragged element to be dropped). Since most areas of a web page are not suitable as target nodes for dropping dragged elements, these two events default to not allowing the current node to accept dragged elements. **If you want to allow data to be dropped on a target node, you must first prevent the default behavior of these two events.**

```html
<div ondragover="return false">
<div ondragover="event.preventDefault()">
```

In the code above, if you do not cancel the drag events or prevent the default behavior, you cannot drop the dragged node on the `div` node.



### 2. The DragEvent Interface

**All drag events inherit the `DragEvent` interface, which in turn inherits the `MouseEvent` interface and the `Event` interface.**

The browser natively provides a `DragEvent()` constructor for generating drag event instance objects.

```js
new DragEvent(type, options)
```

The `DragEvent()` constructor accepts two parameters. The first parameter is a **string representing the event type**, and this parameter is required; the second parameter is the event's **configuration object** used to set the event's properties, and this parameter is optional. Besides accepting the configuration properties of the `MouseEvent` interface and the `Event` interface, the configuration object can also **set the `dataTransfer` property to either `null` or an instance of the `DataTransfer` interface.**

The instance object of `DataTransfer` is used to read and write the data transmitted during a drag event; see the "The DataTransfer Interface" section below for details.



### 3. Overview of the DataTransfer Interface

**All drag event instances have a `DragEvent.dataTransfer` property used to read and write the data that needs to be passed.** The value of this property is an instance of the `DataTransfer` interface.

The browser natively provides a `DataTransfer()` constructor for generating `DataTransfer` instance objects.

```js
var dataTrans = new DataTransfer();
```

The `DataTransfer()` constructor **accepts no parameters**.

Drag data is divided into two aspects: **the kind of data (also called the format) and the value of the data**. The kind of data is a MIME string (such as `text/plain`, `image/jpeg`), and the value of the data is a string. Generally speaking, if you drag a piece of text, the data defaults to that text; if you drag a link, the data defaults to the link's URL.

At the start of a drag event, the developer can provide the data type and data value. During the dragging process, the developer checks the data type through the listener functions for the `dragenter` and `dragover` events to determine whether to allow the dragged object to be dropped. For example, in an area that only allows links to be dropped, check whether the dragged data type is `text/uri-list`.

**When the `drop` event occurs, the listener function extracts the drag data and processes it.**



### 4. Instance Properties of DataTransfer

#### 4.1 DataTransfer.dropEffect — Set the Effect of the Area That Accepts the Drag

The `DataTransfer.dropEffect` property is **used to set the effect when the dragged node is dropped**, which affects the shape of the mouse cursor when the drag passes over the relevant area. It can take the following values.

- copy: copy the dragged node
- move: move the dragged node
- link: create a link pointing to the dragged node
- none: the dragged node cannot be dropped

Besides these values, setting any other value is invalid.

```js
target.addEventListener('dragover', function (e) {
  e.preventDefault();
  e.stopPropagation();
  e.dataTransfer.dropEffect = 'copy';
});
```

In the code above, once the dragged element is dropped, the accepting area will copy that node.

The `dropEffect` property is generally **set in the listener functions of the `dragenter` and `dragover` events**; for the three events `dragstart`, `drag`, and `dragleave`, this property has no effect. This is because this property is only valid for the area that accepts the dragged node, and is invalid for the dragged node itself. After entering the target area, the drag behavior is initialized to the set effect.



#### 4.2 DataTransfer.effectAllowed — Set the Effects Allowed for the Dragged Node

The `DataTransfer.effectAllowed` property **sets the effects allowed in this drag**. It can take the following values.

- copy: copy the dragged node
- move: move the dragged node
- link: create a link pointing to the dragged node
- copyLink: allow `copy` or `link`
- copyMove: allow `copy` or `move`
- linkMove: allow `link` or `move`
- all: allow all effects
- none: the dragged node cannot be dropped
- uninitialized: the default value, equivalent to `all`

If a certain effect is not allowed, the user cannot achieve that effect on the target node.

**This property and the `dropEffect` property are two aspects of the same thing. The former sets the effects allowed for the dragged node, while the latter sets the effect of the area that accepts the drag; they are often used together.**

**The listener function of the `dragstart` event can be used to set this property.** Setting this property in the listener functions of other events is invalid.

```js
source.addEventListener('dragstart', function (e) { // set on the dragged node
  e.dataTransfer.effectAllowed = 'move';
});

target.addEventListener('dragover', function (e) { // set on the accepting-area node
  e.dataTransfer.dropEffect = 'move';
});
```

As long as either the `dropEffect` property or the `effectAllowed` property is `none`, the `drop` operation cannot be completed on the target node.



#### 4.3 DataTransfer.files (Local Files)

The `DataTransfer.files` property is **a FileList object containing a group of local files** that can be transmitted during a drag operation. If this drag does not involve files, this property is an empty FileList object.



##### Example: Receiving Dragged Files

```js
// HTML code is as follows
// <div id="output" style="min-height: 200px;border: 1px solid black;">
//   Drag files here
// </div>

var div = document.getElementById('output');

div.addEventListener("dragenter", function( event ) {
  div.textContent = '';
  event.stopPropagation();
  event.preventDefault();
}, false);

div.addEventListener("dragover", function( event ) {
  event.stopPropagation();
  event.preventDefault();
}, false);

div.addEventListener("drop", function( event ) {
  event.stopPropagation();
  event.preventDefault();
  var files = event.dataTransfer.files;
  for (var i = 0; i < files.length; i++) {
    div.textContent += files[i].name + ' ' + files[i].size + ' bytes\n';
  }
}, false);
```

In the code above, the information of the dragged files is read through the `dataTransfer.files` property. If you want to **read the file content, you need to use the `FileReader` object.**

```js
div.addEventListener('drop', function(e) {
  e.preventDefault();
  e.stopPropagation();

  var fileList = e.dataTransfer.files;
  if (fileList.length > 0) {
    var file = fileList[0];
    var reader = new FileReader();
    reader.readAsDataURL(file);// parse as a url
    reader.onloadend = function(e) {
      if (e.target.readyState === FileReader.DONE) {
        var content = reader.result;
        div.innerHTML = '<img src="'+ content +'"> File: ' + file.name + '\n\n' + content;
      }
    }
    reader.readAsBinaryString(file);
  }
});
```



#### 4.4 DataTransfer.types (Data Formats)

The `DataTransfer.types` property is **a read-only array** in which each member is a string representing **the format of the dragged data** (usually a MIME value). For example, if the dragged content is text, the corresponding member is `text/plain`.

Below is an example that decides whether to allow the `drop` operation on the current node by checking the type of the `dataTransfer` property.

```js
function contains(list, value){
  for (var i = 0; i < list.length; ++i) {
    if(list[i] === value) return true;
  }
  return false;
}

function doDragOver(event) {
  var isLink = contains(event.dataTransfer.types, 'text/uri-list');
  if (isLink) event.preventDefault();
}
```

In the code above, dropping on the current node is only allowed when the dragged node is a link.



#### 4.5 DataTransfer.items

The `DataTransfer.items` property **returns an array-like read-only object (a DataTransferItemList instance)** in which each member is one of the objects being dragged (a DataTransferItem instance). If this drag does not contain objects, it returns an empty object.

A DataTransferItemList instance has the following properties and methods.

- `length`: returns the number of members
- `add(data, type)`: adds a string with the specified content and type (such as `text/html` and `text/plain`) as a member
- `add(file)`: another usage of the `add` method, adding a file as a member
- `remove(index)`: removes the member at the specified position
- `clear()`: removes all members

A DataTransferItem instance has the following properties and methods.

- `kind`: returns the kind of the member (`string` or `file`).
- `type`: returns the type of the member (usually a MIME value).
- `getAsFile()`: if what is dragged is a file, returns that file, otherwise returns `null`.
- `getAsString(callback)`: if what is dragged is a string, passes that string to the specified callback function for processing. This method is asynchronous, so a callback function must be passed in.

Here is an example.

```js
div.addEventListener('drop', function (e) {
  e.preventDefault();
  if (e.dataTransfer.items != null) {
    for (var i = 0; i < e.dataTransfer.items.length; i++) {
      console.log(e.dataTransfer.items[i].kind + ': ' + e.dataTransfer.items[i].type);
    }
  }
});
```



### 5. Instance Methods of DataTransfer

#### 5.1 DataTransfer.setData() — Set the Data Carried by the Drag Event

The `DataTransfer.setData()` method is used to **set the data carried by the drag event**. This method has no return value.

```js
event.dataTransfer.setData('text/plain', 'Text to drag');
```

The code above adds plain text data to the current drag event.

This method accepts two parameters, both strings. The first parameter **represents the data type** (such as `text/plain`), and the second parameter is the **specific data**. If data of the specified type does not exist in the `dataTransfer` property, then this data is added; otherwise the existing data is replaced by the new data.

If you drag a text box or drag selected text, the corresponding text data is added to the `dataTransfer` property by default, without you needing to specify it manually.

```html
<div draggable="true">
  aaa
</div>
```

In the code above, dragging this `<div>` element automatically carries the text data `aaa`.

Using the `setData` method, you can replace the existing data.

```html
<div
  draggable="true"
  ondragstart="event.dataTransfer.setData('text/plain', 'bbb')"
>
  aaa
</div>
```

In the code above, the drag data is actually `bbb`, not `aaa`.

Below is how to add other types of data. Since `text/plain` is the most universally supported format, to ensure compatibility, it is recommended to always save a copy of the data in plain text format last.

```js
var dt = event.dataTransfer;

// add a link
dt.setData('text/uri-list', 'http://www.example.com');
dt.setData('text/plain', 'http://www.example.com');

// add HTML code
dt.setData('text/html', 'Hello there, <strong>stranger</strong>');
dt.setData('text/plain', 'Hello there, <strong>stranger</strong>');

// add an image URL
dt.setData('text/uri-list', imageurl);
dt.setData('text/plain', imageurl);
```

You can provide multiple formats of data at once.

```js
var dt = event.dataTransfer;
dt.setData('application/x-bookmark', bookmarkString);
dt.setData('text/uri-list', 'http://www.example.com');
dt.setData('text/plain', 'http://www.example.com');
```

In the code above, by storing three types of data on the same event, the drag event can `drop` different values on different objects. Note that the first format is a custom format that the browser cannot read by default, which means that only a node with specific code deployed can `drop` (read) this data.



#### 5.2 DataTransfer.getData() — Return Data of a Specified Type

The `DataTransfer.getData()` method accepts a string (representing the data type) as a parameter and **returns the data of the specified type carried by the event (usually data added with the `setData` method)**. If data of the specified type does not exist, it returns an empty string. Usually, data can only be extracted after the `drop` event is triggered.

Below is a listener function for a `drop` event, used to extract data of a specified type.

```js
function onDrop(event) {
  var data = event.dataTransfer.getData('text/plain');
  event.target.textContent = data;
  event.preventDefault();
}
```

The code above extracts the text data of the drag event and replaces the current node's text content with it. Note that at this point you must also cancel the browser's default behavior, because if the user dragged a link, the browser would by default open that link in the current window.

The `getData` method returns a string, and if it contains multiple items of data, you must parse them manually.

```js
function doDrop(event) {
  var lines = event.dataTransfer.getData('text/uri-list').split('\n');
  for (let line of lines) {
    let link = document.createElement('a');
    link.href = line;
    link.textContent = line;
    event.target.appendChild(link);
  }
  event.preventDefault();
}
```

In the code above, the `getData` method returns a group of links, which must be parsed manually.

If the type value is specified as `URL`, you can extract the first valid link.

```js
var link = event.dataTransfer.getData('URL');
```

The example below extracts data from multiple types of data.

```js
function doDrop(event) {
  var types = event.dataTransfer.types;
  var supportedTypes = ['text/uri-list', 'text/plain'];
  types = supportedTypes.filter(function (value) { types.includes(value) });
  if (types.length) {
    var data = event.dataTransfer.getData(types[0]);
  }
  event.preventDefault();
}
```



#### 5.3 DataTransfer.clearData() — Clear Specified or All Data

The `DataTransfer.clearData()` method accepts a string (representing the data type) as a parameter and deletes the data of the specified type carried by the event. If no type is specified, it deletes all data. If the specified type does not exist, calling this method has no effect.

```js
event.dataTransfer.clearData('text/uri-list');
```

The code above clears the `text/uri-list` type data carried by the event.

This method does not remove the dragged files, so after calling this method, the `DataTransfer.types` property may still return the `Files` type (provided that files are being dragged).

Note that this method **can only be used inside the listener function of the `dragstart` event**, because this is the only time the data of the drag operation is writable.



#### 5.4 DataTransfer.setDragImage() — Set the Image Shown During Dragging

During dragging (after the `dragstart` event is triggered), the browser **displays an image that follows the mouse**, representing the dragged node. This image is **created automatically**, usually showing the appearance of the dragged node, and you do not need to set it yourself.

The `DataTransfer.setDragImage()` method **can customize this image**. It accepts three parameters. The first is an `<img>` node or a `<canvas>` node; if omitted or `null`, the appearance of the dragged node is used. The second and third parameters are the horizontal and vertical coordinates of the mouse relative to the top-left corner of this image.

Here is an example.

```js
/* HTML code is as follows
 <div id="drag-with-image" class="dragdemo" draggable="true">
   drag me
 </div>
*/

var div = document.getElementById('drag-with-image');
div.addEventListener('dragstart', function (e) {
  var img = document.createElement('img');
  img.src = 'http://path/to/img';
  e.dataTransfer.setDragImage(img, 0, 0);
}, false);
```



## 10. Other Common Events

### 1. Resource Events

#### 1.1 The beforeunload Event (Called Before Closing the Window)

The `beforeunload` event is triggered **just before a window, document, or various resources are about to be unloaded**. It can be used to prevent the user from accidentally unloading resources.

If the event object's `returnValue` property is a non-empty string, the browser will pop up a dialog box asking the user whether they want to unload the resource. However, the string the user specifies may not be displayed, and the browser will show a predefined string. If the user clicks the "Cancel" button, the resource will not be unloaded.

```js
window.addEventListener('beforeunload', function (event) {
  event.returnValue = 'Are you sure you want to leave?';
});
```

In the code above, if the user **closes the window, the browser will pop up a window asking the user to confirm**.

Browsers behave very inconsistently with this event. Some browsers will pop up the dialog box when `event.preventDefault()` is called. The IE browser requires explicitly returning a non-empty string for the dialog box to pop up. Moreover, **most browsers do not display the specified text in the dialog box, only the default text**. Therefore, you can use the following approach to obtain the maximum compatibility.

```js
window.addEventListener('beforeunload', function (e) {
  var confirmationMessage = 'Confirm closing the window?';

  e.returnValue = confirmationMessage;
  return confirmationMessage;
});
```

Note that many mobile browsers ignore this event by default, and desktop browsers also have ways to ignore it. So it may not take effect at all, and you cannot rely on it to prevent the user from closing the window. In addition, once the `beforeunload` event is used, the browser will not cache the current page, and using the "back" button will request the page from the server again. This is because the purpose of listening to this event is generally to modify the initial state, and at that point caching the initial page is meaningless.

Basically, there is only one situation in which you should listen to the `unload` event, and you should not listen to it otherwise: **the user has modified a form and wants to leave before saving it.**



#### 1.2 The unload Event (Called When the Window Is About to Close)

The `unload` event is triggered **when the window is closed or the `document` object is about to be unloaded**. Its trigger order is after the `beforeunload` and `pagehide` events.

When the `unload` event occurs, the document is in a special state. All resources still exist, but they are all invisible to the user, and all UI interaction is invalid. This event cannot be canceled; even if you throw an error in the listener function, you cannot stop the document from unloading.

```js
window.addEventListener('unload', function(event) {
  console.log('the document is about to unload');
});
```

On mobile, the browser or system may directly discard the page, in which case this event will not occur at all. And just like the `beforeunload` event, once the `unload` event is used, the browser will not cache the current page, for the same reason as above. Therefore, in no case should you rely on this event. To specify code to execute when the page unloads, you may consider not using this event at all.



#### 1.3 The load Event, the error Event, the abort Event (Called When the Page or a Resource Loads Successfully/Fails/Is Canceled)

The `load` event is **triggered when the page or a resource loads successfully**. Note that **when the page or resource is loaded from the browser cache, the `load` event is not triggered**.

```js
window.addEventListener('load', function(event) {
  console.log('all resources have finished loading');
});
```

The `error` event is **triggered when the page or a resource fails to load**.

The `abort` event is **triggered when the user cancels loading**.

These three events actually belong to progress events, and occur not only on the `document` object but also on various external resources. Browsing a web page is a process of loading various resources: images (image), style sheets (style sheet), scripts (script), videos (video), audio (audio), Ajax requests (XMLHttpRequest), and so on. These resources, as well as the `document` object, the `window` object, and the XMLHttpRequestUpload object, all trigger the `load` event and the `error` event.

**The objects that trigger these three events include:**

* window
* document
* body
* img
* style
* script
* video
* audio
* Ajax
* and so on



### 2. Session History Events

#### 2.1 The pageshow Event (Page Display, Executed After Loading the Page), the pagehide Event (Triggered When Leaving the Current Page)

By default, the browser caches pages within the current session, and when the user **clicks the "forward/back" button, the browser loads the page from the cache.**

The pageshow event is **triggered when the page loads, including both the first load and loading from the cache**. If you want to specify code that runs every time the page loads (whether or not from the browser cache), you can put it in this event's listener function.

On the first load, its trigger order is after the `load` event. When loading from the cache, the `load` event is not triggered, because the way the page appears in the cache is usually the way it looks after the `load` event's listener function has run, so there is no need to run it again. Similarly, if the page is loaded from the cache, the JavaScript scripts that initialize the page (such as the listener function of the DOMContentLoaded event) will also not execute.

```js
window.addEventListener('pageshow', function(event) {
  console.log('pageshow: ', event);
});
```

The pageshow event **has a `persisted` property** that returns a boolean. When the page loads for the first time, this property is `false`; when the page loads from the cache, this property is `true`.

```js
window.addEventListener('pageshow', function(event){
  if (event.persisted) { // whether it loaded from the cache
    // ...
  }
});
```

The `pagehide` event is similar to the `pageshow` event; it is triggered when the user **leaves the current page via the "forward/back" button**. Its difference from the unload event is that if you define a listener function for the `unload` event on the window object, the page will not be saved in the cache, whereas using the `pagehide` event, the page will be saved in the cache.

A `pagehide` event instance also **has a `persisted` property; setting this property to `true` means the page is to be saved in the cache, and setting it to `false` means the page is not saved in the cache**. In the latter case, if a listener function for the unload event has been set, that function will run immediately after the pagehide event.

If the page contains a `<frame>` or `<iframe>` element, then the `<frame>` page's `pageshow` event and `pagehide` event are both triggered before the main page.

**Note that these two events are only triggered when the browser's `history` object changes, and have nothing to do with whether the page is visible.**



#### 2.2 The popstate Event (Triggered When the Current Record of the Browser's `history` Object Explicitly Changes)

The `popstate` event is **triggered when the current record of the browser's `history` object explicitly changes**. Note that calling `history.pushState()` or `history.replaceState()` does not trigger the `popstate` event. This event is only triggered when the user explicitly switches between `history` records, for example when **the mouse clicks the "back/forward" button, or when `history.back()`, `history.forward()`, or `history.go()` is called in a script**.

This event object has a `state` property that holds the `state` object added to the current record by the `history.pushState` method and the `history.replaceState` method.

```js
window.onpopstate = function (event) {
  console.log('state: ' + event.state);
};
history.pushState({page: 1}, 'title 1', '?page=1');
history.pushState({page: 2}, 'title 2', '?page=2');
history.replaceState({page: 3}, 'title 3', '?page=3');
history.back(); // state: {"page":1}
history.back(); // state: null
history.go(2);  // state: {"page":3}
```

In the code above, the `pushState` method adds two records to `history`, then the `replaceState` method replaces the current record. Therefore, calling the `back` method twice in a row returns the current entry to the original URL, which has no associated `state` object, so the event's `state` property is `null`; then going forward two records returns to the record added by the `replaceState` method.

Browsers handle whether the `popstate` event is triggered on a page's first load differently; Firefox does not trigger this event.



#### 2.3 The hashchange Event (Triggered When the hash Changes)

The `hashchange` event is **triggered when the hash part of the URL (i.e., the part after the `#` sign, including the `#` sign) changes**. This event is **generally listened for on the `window` object**.

A `hashchange` event instance has two unique properties: the `oldURL` property and the `newURL` property, representing the complete URL before and after the change, respectively.

```js
// URL is http://www.example.com/
window.addEventListener('hashchange', myFunction);

function myFunction(e) {
  console.log(e.oldURL);
  console.log(e.newURL);
}

location.hash = 'part2';
// http://www.example.com/
// http://www.example.com/#part2
```



### 3. Page State Events

#### 3.1 The DOMContentLoaded Event (Triggered After the DOM Content Has Finished Loading)

**After the web page has been downloaded and parsed, the browser triggers the DOMContentLoaded event on the `document` object.** At this point, only the parsing of the page is complete (the DOM for the entire page has been generated), and **all external resources (style sheets, scripts, iframes, etc.) may not yet have finished downloading**. That is, this event **occurs much earlier than the `load` event**.

```js
document.addEventListener('DOMContentLoaded', function (event) {
  console.log('DOM generated');
});
```

Note that a web page's JavaScript scripts execute synchronously, and once a script becomes blocked, it will delay the triggering of the `DOMContentLoaded` event.

```js
document.addEventListener('DOMContentLoaded', function (event) {
  console.log('DOM generated');
});

// this code will delay the triggering of the DOMContentLoaded event
for(var i = 0; i < 1000000000; i++) {
  // ...
}
```

#### 3.2 The readystatechange Event (Triggered When the `readyState` Property of the Document Object or the XMLHttpRequest Object Changes)

The `readystatechange` event is **triggered when the `readyState` property of the Document object or the XMLHttpRequest object changes**. `document.readyState` has three possible values: `loading` (the page is loading), `interactive` (the page has finished parsing, but external resources are still loading), and `complete` (the page and all external resources have finished loading, and the `load` event is about to be triggered).

```js
document.onreadystatechange = function () {
  if (document.readyState === 'interactive') {
    // ...
  }
}
```

This event can be regarded as another way of implementing the `DOMContentLoaded` event.





### 4. Window Events

#### 4.1 The scroll Event (Triggered When the Document or a Document Element Scrolls)

The `scroll` event is triggered when **the document or a document element scrolls**, mainly occurring when the user drags the scroll bar.

```js
window.addEventListener('scroll', callback);
```

This event **is triggered continuously and in large numbers**, so its listener function should not contain very computationally expensive operations. The recommended approach is to use `requestAnimationFrame` or `setTimeout` to control the trigger frequency of this event, and then combine it with `customEvent` to dispatch a new event.

```js
(function () {
  var throttle = function (type, name, obj) { // throttle function to control trigger frequency
    var obj = obj || window;
    var running = false;
    var func = function () {
      if (running) { return; }
      running = true;
      requestAnimationFrame(function() { // combine this method to limit it to 60 times per second
        obj.dispatchEvent(new CustomEvent(name));
        running = false;
      });
    };
    obj.addEventListener(type, func);
  };

  // redefine the scroll event as the optimizedScroll event
  throttle('scroll', 'optimizedScroll');
})();

window.addEventListener('optimizedScroll', function() {
  console.log('Resource conscious scroll callback!');
});
```

In the code above, the `throttle` function is used to control the event trigger frequency, and the `requestAnimationFrame` method ensures that the listener function for the `scroll` event is triggered only once per page repaint (60 times per second). That is, the method above limits the trigger frequency of the `scroll` event to 60 times per second. Specifically, as long as the `scroll` event's frequency is below 60 times per second, the `optimizedScroll` event is triggered, thereby executing the listener function for the `optimizedScroll` event.

Switching to the `setTimeout` method allows you to set a larger time interval.

```js
(function() {
  window.addEventListener('scroll', scrollThrottler, false);

  var scrollTimeout;
  function scrollThrottler() {
    if (!scrollTimeout) {
      scrollTimeout = setTimeout(function () {
        scrollTimeout = null;
        actualScrollHandler();
      }, 66);
    }
  }

  function actualScrollHandler() {
    // ...
  }
}());
```

In the code above, each `scroll` event executes the `scrollThrottler` function. Inside this function there is a timer `setTimeout` that triggers the actually-executed task `actualScrollHandler` every 66 milliseconds (15 times per second).

Below is a more general way of writing the `throttle` function.

```js
function throttle(fn, wait) {
  var time = Date.now();
  return function() {
    if ((time + wait - Date.now()) < 0) {
      fn();
      time = Date.now();
    }
  }
}

window.addEventListener('scroll', throttle(callback, 1000));
```

The code above limits the trigger frequency of the `scroll` event to once per second.

The [lodash](https://www.lodashjs.com/) function library provides a ready-made `throttle` function that can be used directly.

```js
window.addEventListener('scroll', _.throttle(callback, 1000));
```

This book introduced the concept of `debounce` earlier. The difference between `throttle` and it is that `throttle` is "throttling", ensuring it executes only once within a period of time, while `debounce` is "debouncing", executing only after the continuous operations end. Taking page scrolling as an example, `debounce` waits until the user stops scrolling before executing, while `throttle` still executes during the scrolling process if the user keeps scrolling the page.



#### 4.2 The resize Event (Triggered When the Window Size Changes)

The `resize` event is **triggered when the browser window size changes**, mainly **occurring on the `window` object**.

```js
var resizeMethod = function () {
  if (document.body.clientWidth < 768) {
    console.log('mobile device viewport');
  }
};

window.addEventListener('resize', resizeMethod, true);
```

This event is also triggered continuously and in large numbers, so it is best, like the `scroll` event above, to control the event trigger frequency through a `throttle` function.



#### 4.3 The fullscreenchange Event (Triggered When an Element Enters/Exits Fullscreen), the fullscreenerror Event (Triggered When Fullscreen Cannot Be Toggled)

The `fullscreenchange` event is **triggered when an element enters or exits the fullscreen state**, and this event **occurs on the `document` object**.

**Note that this event is not the event triggered when the browser itself enters/exits fullscreen.**

```js
document.addEventListener('fullscreenchange', function (event) {
    console.log(document.fullscreenElement); // the fullscreen element; if null, it means fullscreen has been exited
});
document.addEventListener('click', function (event) {
    document.querySelector('div').requestFullscreen(); // set the element to display fullscreen
});
```

The `fullscreenerror` event is **triggered when the browser cannot switch to the fullscreen state**.



### 5. Clipboard Events

The following three events are related to clipboard operations.

- `cut`: triggered when the selected content is removed from the document and added to the clipboard. **【Cut】**
- `copy`: triggered when a copy action is performed. **【Copy】**
- `paste`: triggered after the clipboard content is pasted into the document. **【Paste】**

These three events are all **instances of the `ClipboardEvent` interface**. `ClipboardEvent` has an instance property `clipboardData`, which is a DataTransfer object that holds the clipboard data. For the specific API interface and operation methods, please refer to the DataTransfer object part of the "Drag Events" section.

```js
document.addEventListener('copy', function (e) {
  e.clipboardData.setData('text/plain', 'Hello, world!');
  e.clipboardData.setData('text/html', '<b>Hello, world!</b>');
  e.preventDefault();
});
```

The code above makes what is copied into the clipboard the developer-specified data, rather than the data the user wanted to copy.



### 6. Focus Events

Focus events occur on element nodes and the `document` object, and are related to gaining or losing focus. They mainly include the following four events.

- `focus`: triggered after an element node **gains focus**; this event does not bubble.
- `blur`: triggered after an element node **loses focus**; this event does not bubble.
- `focusin`: triggered when an element node **is about to gain focus**, occurring before the `focus` event. This event bubbles.
- `focusout`: triggered when an element node **is about to lose focus**, occurring before the `blur` event. This event bubbles.



These four events all inherit the `FocusEvent` interface. A `FocusEvent` instance has the following properties.

- `FocusEvent.target`: the target node of the event.
- `FocusEvent.relatedTarget`: for the `focusin` event, returns the node losing focus; for the `focusout` event, returns the node about to receive focus; for the `focus` and `blur` events, returns `null`.

Since the `focus` and `blur` events do not bubble and can only be triggered during the capture phase, the third parameter of the `addEventListener` method needs to be set to `true`.

```js
form.addEventListener('focus', function (event) {
  event.target.style.background = 'pink';
}, true);

form.addEventListener('blur', function (event) {
  event.target.style.background = '';
}, true);
```

The code above, for the form's text input boxes, sets the background color when they receive focus and removes the background color when they lose focus.



### 7. The CustomEvent Interface (Custom Events)

The CustomEvent interface is **used to generate custom event instances**. The browser's predefined events, although they can be generated manually, often cannot have data bound to them. If you need to pass specified data at the same time as triggering an event, you can use a custom event object generated by the CustomEvent interface.

The browser natively provides a `CustomEvent()` constructor for generating CustomEvent event instances.

```js
new CustomEvent(type, options)
```

The `CustomEvent()` constructor accepts two parameters. The first parameter is a string representing the event name, which is required. The second parameter is the event's configuration object, which is optional. Besides accepting the configuration properties of the Event event, the configuration object of `CustomEvent` has only one of its own properties.

- `detail`: represents the data attached to the event, defaulting to `null`.

Here is an example.

```js
var event = new CustomEvent('build', { 'detail': 'hello' });

function eventHandler(e) {
  console.log(e.detail);
}

document.body.addEventListener('build', function (e) {
  console.log(e.detail);
});

document.body.dispatchEvent(event);
```

In the code above, we manually defined the `build` event. After this event is triggered, it is listened for, thereby outputting the `detail` property of the event instance (i.e., the string `hello`).

Here is another example.

```js
var myEvent = new CustomEvent('myevent', {
  detail: {
    foo: 'bar'
  },
  bubbles: true,
  cancelable: false
});

el.addEventListener('myevent', function (event) {
  console.log('Hello ' + event.detail.foo);
});

el.dispatchEvent(myEvent);
```

The code above also shows that a CustomEvent event instance, besides having the instance properties of the Event interface, also has a `detail` property.



## 11. The GlobalEventHandlers Interface (The Global Event Handling Interface)

To specify the callback function for an event, the recommended method is the element's `addEventListener` method.

```js
div.addEventListener('click', clickHandler, false);
```

Besides this, there is another way to directly specify the callback function for an event.

```js
div.onclick = clickHandler;
```

This interface is provided by the `GlobalEventHandlers` interface. Its advantage is that it is relatively convenient to use, and its disadvantage is that you can only specify one callback function per event, and you cannot specify the phase in which the event is triggered (the capture phase or the bubbling phase).

**`HTMLElement`, `Document`, and `Window` all inherit this interface, which means that the properties provided by the `GlobalEventHandlers` interface can be used on various HTML elements, the `document` object, and the `window` object.** Below are listed the main event properties provided by this interface.



### 1. GlobalEventHandlers.onabort (Abort Event)

When an object's `abort` event (stop loading) occurs, the callback function specified by the `onabort` property is called.

There is currently no unified specification for exactly how the stop-loading event of various elements is triggered. So in practice, this property is generally only used on `<img>` elements.

```js
// HTML code is as follows
// <img src="example.jpg" id="img">
var img = document.getElementById('img');
img.onabort = function () {
  console.log('image loading aborted');
}
```



### 2. GlobalEventHandlers.onerror (Error Event)

When the `error` event occurs, the callback function specified by the `onerror` property is called.

The `error` event is **divided into two kinds**.

**One is a JavaScript runtime error**, which propagates to the `window` object, causing `window.onerror()`.

```js
window.onerror = function (message, source, lineno, colno, error) {
  // ...
}
```

The handler function of `window.onerror` accepts a total of five parameters, with the following meanings.

- message: the error message string
- source: the URL of the script that threw the error
- lineno: the line number where the error occurred, an integer
- colno: the column number where the error occurred, an integer
- error: the error object

**The other is a resource loading error**, for example when a resource loaded by `<img>` or `<script>` has a loading error. In this case, the Error object propagates to the corresponding element, causing that element's `onerror` property to begin executing.

```js
element.onerror = function (event) {
  // ...
}
```

Note that, generally speaking, a resource loading error does not trigger `window.onerror`.



### 3. GlobalEventHandlers.onload (Load Complete Event), GlobalEventHandlers.onloadstart (Load Start Event)

When an element finishes loading, the `load` event is triggered, executing `onload()`. Its typical use scenarios are the `window` object and `<img>` elements. For the `window` object, the `load` event is only triggered when all the page's resources have finished loading (including images, scripts, style sheets, fonts, and all other external resources).

For elements such as `<img>` and `<video>`, the `loadstart` event is also triggered when loading starts, causing `onloadstart` to execute.



### 4. GlobalEventHandlers.onfocus (Focus Gain Event), GlobalEventHandlers.onblur (Focus Loss Event)

When the current element gains focus, `element.onfocus` is triggered; when it loses focus, `element.onblur` is triggered.

```js
element.onfocus = function () {
  console.log("onfocus event detected!");
};
element.onblur = function () {
  console.log("onblur event detected!");
};
```

Note that if it is not an element that can accept user input, then to trigger `onfocus`, that element must have a `tabindex` attribute.



### 5. GlobalEventHandlers.onscroll (Scroll Event)

When the page or an element scrolls, the `scroll` event is triggered, causing `onscroll()` to execute.



### 6. GlobalEventHandlers.oncontextmenu (Context Menu Event), GlobalEventHandlers.onshow (Triggered When the Context Menu Is Shown)

When the user presses the right mouse button on the page, the `contextmenu` event is triggered, causing `oncontextmenu()` to execute. If this property returns `false` after executing, it is equivalent to disabling the context menu. `document.oncontextmenu` and `window.oncontextmenu` have the same effect.

```js
document.oncontextmenu = function () {// disable the context menu
  return false;
};
```

In the code above, the `oncontextmenu` property returns `false` after executing, so the context menu will not appear.

When an element's context menu is shown, that element's `onshow` listener function is triggered.
