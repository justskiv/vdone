# Jest Installation and Configuration

## Jest Installation

Since our project was initialized with `typescript-library-starter`, Jest is already installed by default. However, the installed version is not the latest one, so we can update the relevant dependency versions in `package.json` and reinstall.

```json
{
  "@types/jest": "^24.0.13",
  "jest": "^24.8.0",
  "jest-config": "^24.8.0",
  "ts-jest": "^24.0.2",
  "typescript": "^3.4.5"
}
```

> Note that these are the latest versions at the time of writing. If newer versions are released in the future, feel free to update to the latest ones yourself.

After changing the versions, run `npm install` again on the command line to install the corresponding versions.

## Jest Configuration

In the `package.json` file there is a `jest` field, which corresponds to the Jest configuration:

```json
"jest": {
  "transform": {
    ".(ts|tsx)": "ts-jest"
  },
  "testEnvironment": "jsdom",
  "testRegex": "/test/.*\\.(test|spec)\\.(ts)$",
  "moduleFileExtensions": [
    "ts",
    "tsx",
    "js"
  ],
  "coverageThreshold": {
    "global": {
      "branches": 90,
      "functions": 95,
      "lines": 95,
      "statements": 95
    }
  },
  "collectCoverageFrom": [
    "src/*.{js,ts}",
    "src/**/*.{js,ts}"
  ],
  "setupFilesAfterEnv": [
    "<rootDir>/test/boot.ts"
  ]
},
```

Next, let's go through the meaning of each of these configuration options.

- [transform](https://jestjs.io/docs/en/configuration#transform-object-string-string)

Put simply, this is a transformer configuration. For example, here:

```json
"transform": {
  ".(ts|tsx)": "ts-jest"
},
```

This means using the `ts-jest` tool to transform the contents of `.ts` and `.tsx` files into JavaScript, because we also write our test code in TypeScript, and Node.js cannot support TypeScript directly, so a transformer needs to be configured.

- [testEnvironment](https://jestjs.io/docs/en/configuration#testenvironment-string)

The test environment.

```json
"testEnvironment": "jsdom"
```

This means it is a browser-like test environment, where we can use some of the APIs available in a browser environment.

- [testRegex](https://jestjs.io/docs/en/configuration#testregex-string-array-string)

The regular expression for files to test.

```json
"testRegex": "/test/.*\\.(test|spec)\\.(ts)$"
```

This means that all files ending in `.test.ts` and `.spec.ts` under the `test` directory should be run as tests.

- [moduleFileExtensions](https://jestjs.io/docs/en/configuration#modulefileextensions-array-string)

Module file extensions. When you import a module without specifying an extension, it will try adding these extensions one by one to find the module file you are importing.

```json
"moduleFileExtensions": [
  "ts",
  "tsx",
  "js"
]
```

This means it first looks for `.ts` modules, then `.tsx`, and finally `.js`.

- [coverageThreshold](https://jestjs.io/docs/en/configuration#coveragethreshold-object)

The test coverage threshold settings. When our test coverage falls below the threshold, the tests will fail.

```json
"coverageThreshold": {
  "global": {
    "branches": 90,
    "functions": 95,
    "lines": 95,
    "statements": 95
  }
}
```

This means that the global branch coverage must reach `90%`, function coverage must reach `95%`, line coverage must reach `95%`, and statement coverage must reach `95%`.

- [collectCoverageFrom](https://jestjs.io/docs/en/configuration#collectcoveragefrom-array)

Collects test coverage for the specified files (even if you haven't written tests for them). Its value is of the [glob patterns](https://github.com/jonschlinkert/micromatch) type.

```json
"collectCoverageFrom": [
  "src/*.{js,ts}",
  "src/**/*.{js,ts}"
]
```

This means collecting test coverage for the `js` and `ts` files in the `src` directory and all of its subdirectories.


- [setupFilesAfterEnv](https://jestjs.io/docs/en/configuration#setupfilesafterenv-array)

A list of code files to run immediately after the test framework is installed.

```json
"setupFilesAfterEnv": [
  "<rootDir>/test/boot.ts"
]
```

This means that before running each specific test, the code in `<rootDir>/test/boot.ts` is run first, where `<rootDir>` represents the root directory of the current project. We will cover this configuration in detail in a later chapter.

For other Jest configuration options, interested readers can head to the [official site](https://jestjs.io/docs/en/configuration) for further study.
