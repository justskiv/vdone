# Custom Valid Status Codes

## Requirement Analysis

Previously, when `ts-axios` processed the response result, it considered an HTTP [status](https://developer.mozilla.org/en-US/docs/Web/API/XMLHttpRequest/status) between 200 and 300 to be a valid value, and created an error for anything outside this range. Sometimes we want to customize this rule—for example, to also treat 304 as a valid status code. So we want `ts-axios` to provide a configuration option that allows us to customize the rule for valid status codes, as follows:

```typescript
axios.get('/more/304', {
  validateStatus(status) {
    return status >= 200 && status < 400
  }
}).then(res => {
  console.log(res)
}).catch((e: AxiosError) => {
  console.log(e.message)
})
```

By configuring a `validateStatus` function in the request config, we can customize the rule for valid status codes based on the `status` parameter.

## Implementation

First, let's modify the type definition.

`types/index.ts`:

```typescript
export interface AxiosRequestConfig {
  // ...
  validateStatus?: (status: number) => boolean
}
```

Then let's modify the default configuration rule.

`defaults.ts`:

```typescript
validateStatus(status: number): boolean {
  return status >= 200 && status < 300
}
```

This adds the default validation rule for valid status codes. Next, the logic for handling the response data after the request.

`core/xhr.ts`:

```typescript
const {
  /*...*/
  validateStatus
} = config

function handleResponse(response: AxiosResponse): void {
  if (!validateStatus || validateStatus(response.status)) {
    resolve(response)
  } else {
    reject(
      createError(
        `Request failed with status code ${response.status}`,
        config,
        null,
        request,
        response
      )
    )
  }
}
```

When `validateStatus` is not configured, or when the `validateStatus` function returns true, the response is considered valid and we normally `resolve(response)`; otherwise we create an error.

## Writing the demo

```typescript
axios.get('/more/304').then(res => {
  console.log(res)
}).catch((e: AxiosError) => {
  console.log(e.message)
})

axios.get('/more/304', {
  validateStatus(status) {
    return status >= 200 && status < 400
  }
}).then(res => {
  console.log(res)
}).catch((e: AxiosError) => {
  console.log(e.message)
})
```

In `server.js` we wrote this routing endpoint.

```javascript

router.get('/more/304', function(req, res) {
  res.status(304)
  res.end()
})
```

The endpoint returns a 304 status code, and for the default request we output an error message. In the second request, we configured a custom rule for valid status codes with the range between 200 and 400, so it won't raise an error and instead outputs the response object normally.

At this point `ts-axios` has implemented the custom valid status code feature, and users can configure `validateStatus` to customize the rule for valid status codes.
