# XSRF Protection

## Requirements Analysis

XSRF, also known as [CSRF](https://developer.mozilla.org/en-US/docs/Learn/Server-side/First_steps/Website_security#Cross-Site_Request_Forgery_(CSRF)) (Cross-Site Request Forgery), is a common type of front-end attack. Let's first understand its attack methods through a diagram.

![xsrf](https://jsd.cdn.zzko.cn/gh/xugaoyi/image_store/blog/20200105110743.png 'xsrf')

There are many defense mechanisms against CSRF, such as verifying the request's referer, but the referer can also be forged. So one way to eliminate this type of attack is for the server to require that every request includes a `token`. This `token` is not generated on the front end, but rather generated each time we visit the site and planted on the client via `set-cookie`. Then when the client sends a request, it reads the `token` from the corresponding field in the `cookie` and adds it to the request `headers`. This way the server can read this `token` from the request `headers` and verify it. Since this `token` is very difficult to forge, it can distinguish whether the request was legitimately initiated by the user.

For our `ts-axios` library, we want to handle these things automatically. Each time a request is sent, we read the corresponding `token` value from the `cookie` and add it to the request `headers`. We allow users to configure `xsrfCookieName` and `xsrfHeaderName`, where `xsrfCookieName` is the name of the `cookie` storing the `token`, and `xsrfHeaderName` is the name of the `header` in the request `headers` corresponding to the `token`.

```typescript
axios.get('/more/get',{
  xsrfCookieName: 'XSRF-TOKEN', // default
  xsrfHeaderName: 'X-XSRF-TOKEN' // default
}).then(res => {
  console.log(res)
})
```

We provide default values for `xsrfCookieName` and `xsrfHeaderName`. Of course, users can also configure `xsrfCookieName` and `xsrfHeaderName` in the request according to their own needs.

## Code Implementation

First, modify the type definition of `AxiosRequestConfig`.

`types/index.ts`:

```typescript
export interface AxiosRequestConfig {
  // ...
  xsrfCookieName?: string
  xsrfHeaderName?: string
}
```

Then modify the default configuration.

`defaults.ts`:

```typescript
const defaults: AxiosRequestConfig = {
  // ...
  xsrfCookieName: 'XSRF-TOKEN',

  xsrfHeaderName: 'X-XSRF-TOKEN',
}
```

Next, we need to do three things:

- First, determine whether `withCredentials` is configured as `true` or it is a same-origin request. Only then will we add the `xsrf`-related fields to the request `headers`.

- If the check passes, try to read the `xsrf` `token` value from the cookie.

- If it can be read, add it to the `xsrf`-related field in the request `headers`.

Let's first implement the same-origin request check.

`helpers/url.ts`:

```typescript
interface URLOrigin {
  protocol: string
  host: string
}


export function isURLSameOrigin(requestURL: string): boolean {
  const parsedOrigin = resolveURL(requestURL)
  return (
    parsedOrigin.protocol === currentOrigin.protocol && parsedOrigin.host === currentOrigin.host
  )
}

const urlParsingNode = document.createElement('a')
const currentOrigin = resolveURL(window.location.href)

function resolveURL(url: string): URLOrigin {
  urlParsingNode.setAttribute('href', url)
  const { protocol, host } = urlParsingNode

  return {
    protocol,
    host
  }
}
```

The same-origin check mainly uses a trick: create an `a` tag DOM element, then set its `href` attribute to the `url` we pass in, after which we can obtain the DOM's `protocol` and `host`. Both the current page's `url` and the request's `url` are obtained this way, and then we just compare whether their `protocol` and `host` are the same.

Next, implement reading the cookie.

`helpers/cookie.ts`:

```typescript
const cookie = {
  read(name: string): string | null {
    const match = document.cookie.match(new RegExp('(^|;\\s*)(' + name + ')=([^;]*)'))
    return match ? decodeURIComponent(match[3]) : null
  }
}

export default cookie
```

The logic for reading the `cookie` is very simple: it uses a regular expression to parse the value corresponding to `name`.

Finally, implement the complete logic.

`core/xhr.ts`:

```typescript
const {
  /*...*/
  xsrfCookieName,
  xsrfHeaderName
} = config

if ((withCredentials || isURLSameOrigin(url!)) && xsrfCookieName){
  const xsrfValue = cookie.read(xsrfCookieName)
  if (xsrfValue) {
    headers[xsrfHeaderName!] = xsrfValue
  }
}
```

## Writing the Demo

```typescript
const instance = axios.create({
  xsrfCookieName: 'XSRF-TOKEN-D',
  xsrfHeaderName: 'X-XSRF-TOKEN-D'
})

instance.get('/more/get').then(res => {
  console.log(res)
})
```

`examples/server.js`:

```javascript
app.use(express.static(__dirname, {
  setHeaders (res) {
    res.cookie('XSRF-TOKEN-D', '1234abc')
  }
}))
```

When the page is accessed, the server plants a `cookie` on the client via `set-cookie` with `key` `XSRF-TOKEN` and value `1234abc`, which serves as the `xsrf` `token` value.

Then when we send a request from the front end, we can read the value with `key` `XSRF-TOKEN` from the cookie and add it to the request `headers` with `key` `X-XSRF-TOKEN`.

At this point, we have implemented the automatic XSRF defense capability. In the next lesson, we'll implement ts-axios support for upload and download requests.
