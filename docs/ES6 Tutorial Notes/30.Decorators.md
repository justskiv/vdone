# Decorators

[Note] The Decorator proposal has undergone major revisions and has not yet been finalized; it is unclear whether the syntax will change again. The following content is based entirely on the earlier proposal and is already somewhat out of date. Once it is finalized, this section will need to be completely rewritten.

A decorator is a syntax related to classes, used to annotate or modify classes and class methods. Many object-oriented languages have this feature, and there is currently a [proposal](https://github.com/tc39/proposal-decorators) to introduce it into ECMAScript.
<!-- more -->
A decorator is a function written as `@ + function name`. It can be placed in front of the definition of a class or a class method.

```javascript
@frozen class Foo {
  @configurable(false)
  @enumerable(true)
  method() {}

  @throttle(500)
  expensiveMethod() {}
}
```

The code above uses four decorators in total: one applied to the class itself and the other three to class methods. They not only improve the readability of the code by clearly expressing intent, but also provide a convenient means of adding to or modifying the functionality of a class.

## Decorating Classes

A decorator can be used to decorate an entire class.

```javascript
@testable
class MyTestableClass {
  // ...
}

function testable(target) {
  target.isTestable = true;
}

MyTestableClass.isTestable // true
```

In the code above, `@testable` is a decorator. It modifies the behavior of the `MyTestableClass` class by adding the static property `isTestable` to it. The `target` parameter of the `testable` function is the `MyTestableClass` class itself.

Basically, the behavior of a decorator is as follows.

```javascript
@decorator
class A {}

// equivalent to

class A {}
A = decorator(A) || A;
```

In other words, a decorator is a function that processes a class. The first parameter of the decorator function is the target class to be decorated.

```javascript
function testable(target) {
  // ...
}
```

In the code above, the `target` parameter of the `testable` function is the class that will be decorated.

If one parameter is not enough, you can wrap another layer of function around the decorator.

```javascript
function testable(isTestable) {
  return function(target) {
    target.isTestable = isTestable;
  }
}

@testable(true)
class MyTestableClass {}
MyTestableClass.isTestable // true

@testable(false)
class MyClass {}
MyClass.isTestable // false
```

In the code above, the decorator `testable` can accept a parameter, which means you can modify the behavior of the decorator.

Note that a decorator changes the behavior of a class at compile time, not at runtime. This means a decorator can run code during the compilation phase. In other words, a decorator is essentially a function executed at compile time.

The previous example adds a static property to the class. If you want to add an instance property, you can do so via the `prototype` object of the target class.

```javascript
function testable(target) {
  target.prototype.isTestable = true;
}

@testable
class MyTestableClass {}

let obj = new MyTestableClass();
obj.isTestable // true
```

In the code above, the decorator function `testable` adds a property to the `prototype` object of the target class, so it can be called on instances.

Here is another example.

```javascript
// mixins.js
export function mixins(...list) {
  return function (target) {
    Object.assign(target.prototype, ...list)
  }
}

// main.js
import { mixins } from './mixins'

const Foo = {
  foo() { console.log('foo') }
};

@mixins(Foo)
class MyClass {}

let obj = new MyClass();
obj.foo() // 'foo'
```

The code above uses the `mixins` decorator to add the methods of the `Foo` object to the instances of `MyClass`. You can simulate this functionality with `Object.assign()`.

```javascript
const Foo = {
  foo() { console.log('foo') }
};

class MyClass {}

Object.assign(MyClass.prototype, Foo);

let obj = new MyClass();
obj.foo() // 'foo'
```

In real-world development, when using the React and Redux libraries together, you often need to write something like the following.

```javascript
class MyReactComponent extends React.Component {}

export default connect(mapStateToProps, mapDispatchToProps)(MyReactComponent);
```

With decorators, you can rewrite the code above.

```javascript
@connect(mapStateToProps, mapDispatchToProps)
export default class MyReactComponent extends React.Component {}
```

Comparatively speaking, the latter style is easier to understand.

## Decorating Methods

Decorators can decorate not only classes but also class properties.

```javascript
class Person {
  @readonly
  name() { return `${this.first} ${this.last}` }
}
```

In the code above, the decorator `readonly` is used to decorate the `name` method of the "class".

The decorator function `readonly` can accept three parameters in total.

```javascript
function readonly(target, name, descriptor){
  // the original value of the descriptor object is as follows
  // {
  //   value: specifiedFunction,
  //   enumerable: false,
  //   configurable: true,
  //   writable: true
  // };
  descriptor.writable = false;
  return descriptor;
}

readonly(Person.prototype, 'name', descriptor);
// similar to
Object.defineProperty(Person.prototype, 'name', descriptor);
```

The first parameter of the decorator is the prototype object of the class, which in the example above is `Person.prototype`. The original intent of a decorator is to "decorate" instances of the class, but at this point the instances have not yet been created, so it can only decorate the prototype (this differs from class decoration, where the `target` parameter refers to the class itself). The second parameter is the name of the property to be decorated, and the third parameter is the descriptor object of that property.

In addition, the code above shows that the decorator (readonly) modifies the property's descriptor object (descriptor), and then the modified descriptor object is used to define the property.

Here is another example, which modifies the `enumerable` property of the property descriptor object so that the property is non-enumerable.

```javascript
class Person {
  @nonenumerable
  get kidCount() { return this.children.length; }
}

function nonenumerable(target, name, descriptor) {
  descriptor.enumerable = false;
  return descriptor;
}
```

The `@log` decorator below can serve the purpose of outputting logs.

```javascript
class Math {
  @log
  add(a, b) {
    return a + b;
  }
}

function log(target, name, descriptor) {
  var oldValue = descriptor.value;

  descriptor.value = function() {
    console.log(`Calling ${name} with`, arguments);
    return oldValue.apply(this, arguments);
  };

  return descriptor;
}

const math = new Math();

// passed parameters should get logged now
math.add(2, 4);
```

In the code above, the purpose of the `@log` decorator is to run `console.log` once before performing the original operation, thereby achieving the goal of outputting logs.

Decorators serve as annotations.

```javascript
@testable
class Person {
  @readonly
  @nonenumerable
  name() { return `${this.first} ${this.last}` }
}
```

From the code above, we can tell at a glance that the `Person` class is testable, and that the `name` method is read-only and non-enumerable.

Below is a [component](https://github.com/ionic-team/stencil) written using the Decorator style, which looks clear at a glance.

```javascript
@Component({
  tag: 'my-component',
  styleUrl: 'my-component.scss'
})
export class MyComponent {
  @Prop() first: string;
  @Prop() last: string;
  @State() isVisible: boolean = true;

  render() {
    return (
      <p>Hello, my name is {this.first} {this.last}</p>
    );
  }
}
```

If the same method has multiple decorators, they work like peeling an onion: first entering from the outside in, then executing from the inside out.

```javascript
function dec(id){
  console.log('evaluated', id);
  return (target, property, descriptor) => console.log('executed', id);
}

class Example {
    @dec(1)
    @dec(2)
    method(){}
}
// evaluated 1
// evaluated 2
// executed 2
// executed 1
```

In the code above, the outer decorator `@dec(1)` is entered first, but the inner decorator `@dec(2)` is executed first.

Besides annotation, decorators can also be used for type checking. So, for classes, this feature is quite useful. In the long run, it will be an important tool for static analysis of JavaScript code.

## Why Can't Decorators Be Used on Functions?

Decorators can only be used on classes and class methods, not on functions, because of function hoisting.

```javascript
var counter = 0;

var add = function () {
  counter++;
};

@add
function foo() {
}
```

The intent of the code above is for `counter` to equal 1 after execution, but the actual result is that `counter` equals 0. This is because function hoisting makes the actually executed code look like the following.

```javascript
@add
function foo() {
}

var counter;
var add;

counter = 0;

add = function () {
  counter++;
};
```

Here is another example.

```javascript
var readOnly = require("some-decorator");

@readOnly
function foo() {
}
```

The code above also has a problem, because what is actually executed is the following.

```javascript
var readOnly;

@readOnly
function foo() {
}

readOnly = require("some-decorator");
```

In short, because of function hoisting, decorators cannot be used on functions. Classes are not hoisted, so this problem does not arise.

On the other hand, if you really must decorate a function, you can use the form of a higher-order function and execute it directly.

```javascript
function doSomething(name) {
  console.log('Hello, ' + name);
}

function loggingDecorator(wrapped) {
  return function() {
    console.log('Starting');
    const result = wrapped.apply(this, arguments);
    console.log('Finished');
    return result;
  }
}

const wrapped = loggingDecorator(doSomething);
```

## core-decorators.js

[core-decorators.js](https://github.com/jayphelps/core-decorators.js) is a third-party module that provides several common decorators, through which you can better understand decorators.

**(1) @autobind**

The `autobind` decorator binds the `this` object in a method to the original object.

```javascript
import { autobind } from 'core-decorators';

class Person {
  @autobind
  getPerson() {
    return this;
  }
}

let person = new Person();
let getPerson = person.getPerson;

getPerson() === person;
// true
```

**(2) @readonly**

The `readonly` decorator makes a property or method non-writable.

```javascript
import { readonly } from 'core-decorators';

class Meal {
  @readonly
  entree = 'steak';
}

var dinner = new Meal();
dinner.entree = 'salmon';
// Cannot assign to read only property 'entree' of [object Object]
```

**(3) @override**

The `override` decorator checks whether a subclass method correctly overrides the parent class method of the same name; if it does not, it throws an error.

```javascript
import { override } from 'core-decorators';

class Parent {
  speak(first, second) {}
}

class Child extends Parent {
  @override
  speak() {}
  // SyntaxError: Child#speak() does not properly override Parent#speak(first, second)
}

// or

class Child extends Parent {
  @override
  speaks() {}
  // SyntaxError: No descriptor matching Child#speaks() was found on the prototype chain.
  //
  //   Did you mean "speak"?
}
```

**(4) @deprecate (alias @deprecated)**

The `deprecate` or `deprecated` decorator displays a warning in the console indicating that the method will be removed.

```javascript
import { deprecate } from 'core-decorators';

class Person {
  @deprecate
  facepalm() {}

  @deprecate('We stopped facepalming')
  facepalmHard() {}

  @deprecate('We stopped facepalming', { url: 'http://knowyourmeme.com/memes/facepalm' })
  facepalmHarder() {}
}

let person = new Person();

person.facepalm();
// DEPRECATION Person#facepalm: This function will be removed in future versions.

person.facepalmHard();
// DEPRECATION Person#facepalmHard: We stopped facepalming

person.facepalmHarder();
// DEPRECATION Person#facepalmHarder: We stopped facepalming
//
//     See http://knowyourmeme.com/memes/facepalm for more details.
//
```

**(5) @suppressWarnings**

The `suppressWarnings` decorator suppresses the `console.warn()` calls caused by the `deprecated` decorator. However, calls issued from asynchronous code are excluded.

```javascript
import { suppressWarnings } from 'core-decorators';

class Person {
  @deprecated
  facepalm() {}

  @suppressWarnings
  facepalmWithoutWarning() {
    this.facepalm();
  }
}

let person = new Person();

person.facepalmWithoutWarning();
// no warning is logged
```

## Using Decorators to Automatically Publish Events

We can use a decorator to make an object's method automatically emit an event when it is called.

```javascript
const postal = require("postal/lib/postal.lodash");

export default function publish(topic, channel) {
  const channelName = channel || '/';
  const msgChannel = postal.channel(channelName);
  msgChannel.subscribe(topic, v => {
    console.log('channel: ', channelName);
    console.log('event: ', topic);
    console.log('data: ', v);
  });

  return function(target, name, descriptor) {
    const fn = descriptor.value;

    descriptor.value = function() {
      let value = fn.apply(this, arguments);
      msgChannel.publish(topic, value);
    };
  };
}
```

The code above defines a decorator named `publish`. By rewriting `descriptor.value`, it makes the original method automatically emit an event when it is called. The event "publish/subscribe" library it uses is [Postal.js](https://github.com/postaljs/postal.js).

It is used as follows.

```javascript
// index.js
import publish from './publish';

class FooComponent {
  @publish('foo.some.message', 'component')
  someMethod() {
    return { my: 'data' };
  }
  @publish('foo.some.other')
  anotherMethod() {
    // ...
  }
}

let foo = new FooComponent();

foo.someMethod();
foo.anotherMethod();
```

From now on, whenever `someMethod` or `anotherMethod` is called, an event will be emitted automatically.

```bash
$ bash-node index.js
channel:  component
event:  foo.some.message
data:  { my: 'data' }

channel:  /
event:  foo.some.other
data:  undefined
```

## Mixin

On the basis of decorators, you can implement the `Mixin` pattern. The so-called `Mixin` pattern is an alternative to object inheritance; it means mixing the methods of one object into another object.

Look at the following example.

```javascript
const Foo = {
  foo() { console.log('foo') }
};

class MyClass {}

Object.assign(MyClass.prototype, Foo);

let obj = new MyClass();
obj.foo() // 'foo'
```

In the code above, the object `Foo` has a `foo` method. Using the `Object.assign` method, the `foo` method can be "mixed into" the `MyClass` class, so that the instance `obj` of `MyClass` also has the `foo` method. This is a simple implementation of the "mixin" pattern.

Next, we deploy a general-purpose script `mixins.js` that writes Mixin as a decorator.

```javascript
export function mixins(...list) {
  return function (target) {
    Object.assign(target.prototype, ...list);
  };
}
```

Then, you can use the decorator above to "mix in" various methods to a class.

```javascript
import { mixins } from './mixins';

const Foo = {
  foo() { console.log('foo') }
};

@mixins(Foo)
class MyClass {}

let obj = new MyClass();
obj.foo() // "foo"
```

Through the `mixins` decorator, we have "mixed" the `foo` method of the `Foo` object into the `MyClass` class.

However, the method above rewrites the `prototype` object of the `MyClass` class. If you don't like this, you can also implement Mixin through class inheritance.

```javascript
class MyClass extends MyBaseClass {
  /* ... */
}
```

In the code above, `MyClass` inherits from `MyBaseClass`. If we want to "mix in" a `foo` method into `MyClass`, one approach is to insert a mixin class between `MyClass` and `MyBaseClass`. This class has the `foo` method and inherits all the methods of `MyBaseClass`, and then `MyClass` inherits from this class.

```javascript
let MyMixin = (superclass) => class extends superclass {
  foo() {
    console.log('foo from MyMixin');
  }
};
```

In the code above, `MyMixin` is a mixin class generator that accepts `superclass` as a parameter and then returns a subclass that inherits from `superclass`. This subclass contains a `foo` method.

Then, when the target class inherits from this mixin class, the goal of "mixing in" the `foo` method is achieved.

```javascript
class MyClass extends MyMixin(MyBaseClass) {
  /* ... */
}

let c = new MyClass();
c.foo(); // "foo from MyMixin"
```

If you need to "mix in" multiple methods, generate multiple mixin classes.

```javascript
class MyClass extends Mixin1(Mixin2(MyBaseClass)) {
  /* ... */
}
```

One benefit of this style is that you can call `super`, so you can avoid overriding the parent class's method of the same name during the "mixing in" process.

```javascript
let Mixin1 = (superclass) => class extends superclass {
  foo() {
    console.log('foo from Mixin1');
    if (super.foo) super.foo();
  }
};

let Mixin2 = (superclass) => class extends superclass {
  foo() {
    console.log('foo from Mixin2');
    if (super.foo) super.foo();
  }
};

class S {
  foo() {
    console.log('foo from S');
  }
}

class C extends Mixin1(Mixin2(S)) {
  foo() {
    console.log('foo from C');
    super.foo();
  }
}
```

In the code above, each time a `mixin` occurs, the parent class's `super.foo` method is called, so that the parent class's method of the same name is not overridden and its behavior is preserved.

```javascript
new C().foo()
// foo from C
// foo from Mixin1
// foo from Mixin2
// foo from S
```

## Trait

Trait is also a kind of decorator, with effects similar to Mixin, but it provides more features, such as preventing conflicts between methods of the same name, excluding certain methods from being mixed in, aliasing the mixed-in methods, and so on.

The following uses the third-party module [traits-decorator](https://github.com/CocktailJS/traits-decorator) as an example. The `traits` decorator provided by this module can accept not only objects but also ES6 classes as parameters.

```javascript
import { traits } from 'traits-decorator';

class TFoo {
  foo() { console.log('foo') }
}

const TBar = {
  bar() { console.log('bar') }
};

@traits(TFoo, TBar)
class MyClass { }

let obj = new MyClass();
obj.foo() // foo
obj.bar() // bar
```

In the code above, through the `traits` decorator, the `foo` method of the `TFoo` class and the `bar` method of the `TBar` object are "mixed into" the `MyClass` class.

Trait does not allow "mixing in" methods of the same name.

```javascript
import { traits } from 'traits-decorator';

class TFoo {
  foo() { console.log('foo') }
}

const TBar = {
  bar() { console.log('bar') },
  foo() { console.log('foo') }
};

@traits(TFoo, TBar)
class MyClass { }
// error
// throw new Error('Method named: ' + methodName + ' is defined twice.');
//        ^
// Error: Method named: foo is defined twice.
```

In the code above, both `TFoo` and `TBar` have a `foo` method, so the `traits` decorator throws an error.

One solution is to exclude the `foo` method of `TBar`.

```javascript
import { traits, excludes } from 'traits-decorator';

class TFoo {
  foo() { console.log('foo') }
}

const TBar = {
  bar() { console.log('bar') },
  foo() { console.log('foo') }
};

@traits(TFoo, TBar::excludes('foo'))
class MyClass { }

let obj = new MyClass();
obj.foo() // foo
obj.bar() // bar
```

The code above uses the bind operator (::) to exclude the `foo` method on `TBar`, so no error occurs during mixing in.

Another approach is to give an alias to the `foo` method of `TBar`.

```javascript
import { traits, alias } from 'traits-decorator';

class TFoo {
  foo() { console.log('foo') }
}

const TBar = {
  bar() { console.log('bar') },
  foo() { console.log('foo') }
};

@traits(TFoo, TBar::alias({foo: 'aliasFoo'}))
class MyClass { }

let obj = new MyClass();
obj.foo() // foo
obj.aliasFoo() // foo
obj.bar() // bar
```

The code above gives the `foo` method of `TBar` the alias `aliasFoo`, so `MyClass` can now also mix in the `foo` method of `TBar`.

The `alias` and `excludes` methods can be used together.

```javascript
@traits(TExample::excludes('foo','bar')::alias({baz:'exampleBaz'}))
class MyClass {}
```

The code above excludes the `foo` and `bar` methods of `TExample` and gives the `baz` method the alias `exampleBaz`.

The `as` method provides another way to write the code above.

```javascript
@traits(TExample::as({excludes:['foo', 'bar'], alias: {baz: 'exampleBaz'}}))
class MyClass {}
```
