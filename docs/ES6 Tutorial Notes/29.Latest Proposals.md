# Latest Proposals

This chapter introduces some promising recent proposals that have not yet entered the standard.

## do expressions

In essence, a block-level scope is a statement that wraps multiple operations together and has no return value.

```javascript
{
  let t = f();
  t = t * t + 1;
}
```
<!-- more -->
In the code above, the block-level scope wraps two statements together. However, outside the block-level scope there is no way to obtain the value of `t`, because a block-level scope does not return a value, unless `t` is a global variable.

There is now a [proposal](https://github.com/tc39/proposal-do-expressions) that makes block-level scopes into expressions, meaning they can return a value. The way to do this is to add `do` before the block-level scope, turning it into a `do` expression, which then returns the value of the last expression executed internally.

```javascript
let x = do {
  let t = f();
  t * t + 1;
};
```

In the code above, the variable `x` gets the return value of the entire block-level scope (`t * t + 1`).

The logic of a `do` expression is very simple: whatever it wraps is what it returns.

```javascript
// equivalent to <expression>
do { <expression>; }

// equivalent to <statement>
do { <statement> }
```

The benefit of `do` expressions is that they can wrap multiple statements, making the program more modular, assembled piece by piece like LEGO bricks.

```javascript
let x = do {
  if (foo()) { f() }
  else if (bar()) { g() }
  else { h() }
};
```

The essence of the code above is to call different functions based on the execution result of the function `foo`, and assign the returned result to the variable `x`. Using a `do` expression, the intent of this operation is expressed very concisely and clearly. Moreover, the `do` block-level scope provides a separate scope, so internal operations can be isolated from the global scope.

It is worth mentioning that `do` expressions are very handy in JSX syntax.

```javascript
return (
  <nav>
    <Home />
    {
      do {
        if (loggedIn) {
          <LogoutButton />
        } else {
          <LoginButton />
        }
      }
    }
  </nav>
)
```

In the code above, without `do` expressions you could only use the ternary conditional operator (`?:`). In that case, once the conditional logic becomes complex, the code becomes very hard to read.

## throw expressions

JavaScript syntax stipulates that `throw` is a command used to throw an error, and it cannot be used within an expression.

```javascript
// error
console.log(throw new Error());
```

In the code above, the argument to `console.log` must be an expression; if it is a `throw` statement, an error will be reported.

There is now a [proposal](https://github.com/tc39/proposal-throw-expressions) that allows `throw` to be used in expressions.

```javascript
// default parameter value
function save(filename = throw new TypeError("Argument required")) {
}

// return value of an arrow function
lint(ast, {
  with: () => throw new Error("avoid using 'with' statements.")
});

// conditional expression
function getEncoder(encoding) {
  const encoder = encoding === "utf8" ?
    new UTF8Encoder() :
    encoding === "utf16le" ?
      new UTF16Encoder(false) :
      encoding === "utf16be" ?
        new UTF16Encoder(true) :
        throw new Error("Unsupported encoding");
}

// logical expression
class Product {
  get id() {
    return this._id;
  }
  set id(value) {
    this._id = value || throw new Error("Invalid value");
  }
}
```

In the code above, `throw` appears inside expressions.

Syntactically, the `throw` inside a `throw` expression is no longer a command but an operator. To avoid confusion with the `throw` command, it is stipulated that a `throw` appearing at the start of a line is always interpreted as a `throw` statement, not a `throw` expression.

## Partial Application of Functions

### Syntax

A multi-argument function sometimes needs to bind one or more of its arguments and then return a new function.

```javascript
function add(x, y) { return x + y; }
function add7(x) { return x + 7; }
```

In the code above, the `add7` function is actually a special version of the `add` function; by binding one argument to `7`, you can derive `add7` from `add`.

```javascript
// bind method
const add7 = add.bind(null, 7);

// arrow function
const add7 = x => add(x, 7);
```

Both of the above approaches are somewhat redundant. Among them, the limitations of the `bind` method are more obvious: it must provide `this`, and it can only bind arguments one by one from front to back, with no way to bind only non-leading arguments.

There is now a [proposal](https://github.com/tc39/proposal-partial-application) that makes binding arguments and returning a new function easier. This is called partial application of functions.

```javascript
const add = (x, y) => x + y;
const addOne = add(1, ?);

const maxGreaterThanZero = Math.max(0, ...);
```

According to the new proposal, `?` is a placeholder for a single argument, and `...` is a placeholder for multiple arguments. All of the following forms are partial application of functions.

```javascript
f(x, ?)
f(x, ...)
f(?, x)
f(..., x)
f(?, x, ?)
f(..., x, ...)
```

`?` and `...` can only appear within a function call, and they return a new function.

```javascript
const g = f(?, 1, ...);
// equivalent to
const g = (x, ...y) => f(x, 1, ...y);
```

Partial application of functions can also be used with object methods.

```javascript
let obj = {
  f(x, y) { return x + y; },
};

const g = obj.f(?, 3);
g(1) // 4
```

### Points to Note

There are some special points to be aware of with partial application of functions.

(1) Partial application of a function is based on the original function. If the original function changes, the new function generated by partial application immediately reflects that change.

```javascript
let f = (x, y) => x + y;

const g = f(?, 3);
g(1); // 4

// replace function f
f = (x, y) => x * y;

g(1); // 3
```

In the code above, after the partial application of the function is defined, replacing the original function immediately affects the new function.

(2) If the value supplied in advance is an expression, that expression is not evaluated at definition time but is evaluated on each call.

```javascript
let a = 3;
const f = (x, y) => x + y;

const g = f(?, a);
g(1); // 4

// change the value of a
a = 10;
g(1); // 11
```

In the code above, the pre-supplied argument is the variable `a`, so `a` is evaluated only each time the function `g` is called.

(3) If the new function has more arguments than the number of placeholders, the extra arguments will be ignored.

```javascript
const f = (x, ...y) => [x, ...y];
const g = f(?, 1);
g(2, 3, 4); // [2, 1]
```

In the code above, the function `g` has only one placeholder, which means it can only accept one argument, and any extra arguments are ignored.

If written as below, the extra arguments are no problem.

```javascript
const f = (x, ...y) => [x, ...y];
const g = f(?, 1, ...);
g(2, 3, 4); // [2, 1, 3, 4];
```

(4) `...` is collected only once. If a partial application of a function uses multiple `...`, then the value of every `...` will be the same.

```javascript
const f = (...x) => x;
const g = f(..., 9, ...);
g(1, 2, 3); // [1, 2, 3, 9, 1, 2, 3]
```

In the code above, `g` defines two `...` placeholders, and when it actually runs, their values are the same.

## Pipeline Operator

The Unix operating system has a pipeline mechanism that can pass the value of a previous operation to the next operation. This mechanism is very useful, allowing simple operations to be combined into complex ones. Many languages have a pipeline implementation, and there is now a [proposal](https://github.com/tc39/proposal-pipeline-operator) to give JavaScript a pipeline mechanism as well.

The JavaScript pipeline is an operator, written as `|>`. Its left side is an expression and its right side is a function. The pipeline operator passes the value of the left-side expression into the right-side function for evaluation.

```javascript
x |> f
// equivalent to
f(x)
```

The biggest benefit of the pipeline operator is that it lets you write nested functions as a left-to-right chained expression.

```javascript
function doubleSay (str) {
  return str + ", " + str;
}

function capitalize (str) {
  return str[0].toUpperCase() + str.substring(1);
}

function exclaim (str) {
  return str + '!';
}
```

Above are three simple functions. To execute them nested, the traditional way and the pipeline way are as follows.

```javascript
// traditional way
exclaim(capitalize(doubleSay('hello')))
// "Hello, hello!"

// pipeline way
'hello'
  |> doubleSay
  |> capitalize
  |> exclaim
// "Hello, hello!"
```

The pipeline operator can only pass a single value, which means the function on its right side must be a single-argument function. If it is a multi-argument function, it must be curried into a single-argument version.

```javascript
function double (x) { return x + x; }
function add (x, y) { return x + y; }

let person = { score: 25 };
person.score
  |> double
  |> (_ => add(7, _))
// 57
```

In the code above, the `add` function needs two arguments. However, the pipeline operator can only pass one value, so you need to supply the other argument in advance and change it into a single-argument arrow function `_ => add(7, _)`. The underscore inside this function has no special meaning and can be replaced with another symbol; the underscore is used only because it vividly represents the placeholder here.

The pipeline operator also works with `await` functions.

```javascript
x |> await f
// equivalent to
await f(x)

const userAge = userId |> await fetchUserById |> getAgeFromUser;
// equivalent to
const userAge = getAgeFromUser(await fetchUserById(userId));
```

## Numeric Separators

In Western languages, longer numbers are allowed to have a separator (usually a comma) added every three digits to increase readability. For example, `1000` can be written as `1,000`.

There is now a [proposal](https://github.com/tc39/proposal-numeric-separator) that allows JavaScript numbers to use an underscore (`_`) as a separator.

```javascript
let budget = 1_000_000_000_000;
budget === 10 ** 12 // true
```

JavaScript's numeric separator does not specify the number of digits in an interval; that is, you can add a separator every three digits, or every one, two, or four digits.

```javascript
123_00 === 12_300 // true

12345_00 === 123_4500 // true
12345_00 === 1_234_500 // true
```

Decimals and scientific notation can also use numeric separators.

```javascript
// decimal
0.000_001
// scientific notation
1e10_000
```

There are a few points to note when using numeric separators.

- It cannot be at the very front (leading) or very end (trailing) of the number.
- Two or more separators cannot be placed together.
- There cannot be a separator before or after the decimal point.
- In scientific notation, there cannot be a separator before or after the `e` or `E` that denotes the exponent.

All of the following will report an error.

```javascript
// all error
3_.141
3._141
1_e12
1e_12
123__456
_1464301
1464301_
```

Besides decimal, numbers in other bases can also use separators.

```javascript
// binary
0b1010_0001_1000_0101
// hexadecimal
0xA0_B0_C0
```

Note that a separator cannot immediately follow a base prefix `0b`, `0B`, `0o`, `0O`, `0x`, `0X`.

```javascript
// error
0_b111111000
0b_111111000
```

The following three functions for converting strings to numbers do not support numeric separators. The main reason is that the proposal's designers believe numeric separators are mainly for the convenience of writing numbers when coding, not for processing externally input data.

- Number()
- parseInt()
- parseFloat()

```javascript
Number('123_456') // NaN
parseInt('123_456') // 123
```

## Math.signbit()

`Math.sign()` is used to determine whether a value is positive or negative, but if the argument is `-0`, it returns `-0`.

```javascript
Math.sign(-0) // -0
```

This makes `Math.sign()` not very useful for determining the sign bit. Internally, JavaScript uses 64-bit floating-point numbers (the international standard IEEE 754) to represent values. IEEE 754 stipulates that the first bit is the sign bit, where `0` denotes a positive number and `1` denotes a negative number. So there are two kinds of zero: `+0` is the zero value when the sign bit is `0`, and `-0` is the zero value when the sign bit is `1`. In actual programming, determining whether a value is `+0` or `-0` is very troublesome, because they are equal.

```javascript
+0 === -0 // true
```

Currently, there is a [proposal](http://jfbastien.github.io/papers/Math.signbit.html) that introduces the `Math.signbit()` method to determine whether a number's sign bit is set.

```javascript
Math.signbit(2) //false
Math.signbit(-2) //true
Math.signbit(0) //false
Math.signbit(-0) //true
```

As you can see, this method correctly reports that the sign bit of `-0` is set.

The algorithm of this method is as follows.

- If the argument is `NaN`, return `false`
- If the argument is `-0`, return `true`
- If the argument is negative, return `true`
- In all other cases, return `false`

## Double-Colon Operator

Arrow functions can bind the `this` object, greatly reducing the need to explicitly bind the `this` object (with `call`, `apply`, `bind`). However, arrow functions are not suitable for all situations, so there is now a [proposal](https://github.com/zenparsing/es-function-bind) introducing a "function bind" operator to replace `call`, `apply`, and `bind` calls.

The function bind operator is two side-by-side colons (`::`); the left side of the double colon is an object, and the right side is a function. This operator automatically binds the object on the left as the context (i.e., the `this` object) to the function on the right.

```javascript
foo::bar;
// equivalent to
bar.bind(foo);

foo::bar(...arguments);
// equivalent to
bar.apply(foo, arguments);

const hasOwnProperty = Object.prototype.hasOwnProperty;
function hasOwn(obj, key) {
  return obj::hasOwnProperty(key);
}
```

If the left side of the double colon is empty and the right side is a method of an object, it is equivalent to binding that method to that object.

```javascript
var method = obj::obj.foo;
// equivalent to
var method = ::obj.foo;

let log = ::console.log;
// equivalent to
var log = console.log.bind(console);
```

If the result of the double-colon operator is still an object, you can use chained notation.

```javascript
import { map, takeWhile, forEach } from "iterlib";

getPlayers()
::map(x => x.character())
::takeWhile(x => x.strength > 100)
::forEach(x => console.log(x));
```

## Realm API

The [Realm API](https://github.com/tc39/proposal-realms) provides a sandbox feature that allows code to be isolated, preventing the isolated code from accessing the global object.

In the past, `<iframe>` was often used as a sandbox.

```javascript
const globalOne = window;
let iframe = document.createElement('iframe');
document.body.appendChild(iframe);
const globalTwo = iframe.contentWindow;
```

In the code above, the global object of the `<iframe>` is independent (`iframe.contentWindow`). The Realm API can replace this feature.

```javascript
const globalOne = window;
const globalTwo = new Realm().global;
```

In the code above, the `Realm API` separately provides a global object `new Realm().global`.

The Realm API provides a `Realm()` constructor used to generate a Realm object. The object's `global` property points to a new top-level object that is similar to the original top-level object.

```javascript
const globalOne = window;
const globalTwo = new Realm().global;

globalOne.evaluate('1 + 2') // 3
globalTwo.evaluate('1 + 2') // 3
```

In the code above, the `evaluate()` method of the top-level object generated by Realm can run code.

The following code proves that the Realm top-level object and the original top-level object are two different objects.

```javascript
let a1 = globalOne.evaluate('[1,2,3]');
let a2 = globalTwo.evaluate('[1,2,3]');
a1.prototype === a2.prototype; // false
a1 instanceof globalTwo.Array; // false
a2 instanceof globalOne.Array; // false
```

In the code above, the prototype object of the array inside the Realm sandbox is different from that of the array in the original environment.

Inside the Realm sandbox, only APIs provided by ECMAScript syntax can run; APIs provided by the host environment cannot run.

```javascript
globalTwo.evaluate('console.log(1)')
// throw an error: console is undefined
```

In the code above, the Realm sandbox does not have a `console` object, which causes an error. This is because `console` is not part of the syntax standard but is provided by the host environment.

To solve this problem, you can use the following code.

```javascript
globalTwo.console = globalOne.console;
```

The `Realm()` constructor can accept a parameter object whose `intrinsics` property can specify that the Realm sandbox inherits methods from the original top-level object.

```javascript
const r1 = new Realm();
r1.global === this;
r1.global.JSON === JSON; // false

const r2 = new Realm({ intrinsics: 'inherit' });
r2.global === this; // false
r2.global.JSON === JSON; // true
```

In the code above, under normal circumstances the sandbox's `JSON` method differs from the original `JSON` object. However, after the `Realm()` constructor accepts `{ intrinsics: 'inherit' }` as a parameter, it inherits the methods of the original top-level object.

Users can define their own subclass of `Realm` to customize their own sandbox.

```javascript
class FakeWindow extends Realm {
  init() {
    super.init();
    let global = this.global;

    global.document = new FakeDocument(...);
    global.alert = new Proxy(fakeAlert, { ... });
    // ...
  }
}
```

In the code above, `FakeWindow` simulates a fake top-level object `window`.

## `#!` Command

Unix command-line scripts all support the `#!` command, also known as Shebang or Hashbang. This command is placed on the first line of the script and is used to specify the script's interpreter.

For example, the first line of a Bash script.

```bash
#!/bin/sh
```

The first line of a Python script.

```python
#!/usr/bin/env python
```

There is now a [proposal](https://github.com/tc39/proposal-hashbang) that introduces the `#!` command to JavaScript scripts, written on the first line of a script file or module file.

```javascript
// written on the first line of a script file
#!/usr/bin/env node
'use strict';
console.log(1);

// written on the first line of a module file
#!/usr/bin/env node
export {};
console.log(1);
```

With this line, the Unix command line can directly execute the script.

```bash
# the old way to execute the script
$ node hello.js

# the hashbang way
$ hello.js
```

For the JavaScript engine, `#!` is understood as a comment and this line is ignored.

## import.meta

When a developer uses a module, they sometimes need to know some information about the module itself (such as the module's path). There is now a [proposal](https://github.com/tc39/proposal-import-meta) that adds a meta-property `import.meta` to the import command, returning the meta-information of the current module.

`import.meta` can only be used inside a module; using it outside a module will report an error.

This property returns an object whose various properties are the meta-information of the currently running script. Exactly which properties it contains is not specified by the standard and is decided by each runtime environment itself. Generally speaking, `import.meta` will have at least the following two properties.

**(1) import.meta.url**

`import.meta.url` returns the URL path of the current module. For example, if the path of the current module's main file is `https://foo.com/main.js`, then `import.meta.url` returns this path. If the module also has a data file `data.txt`, you can use the following code to get the path of this data file.

```javascript
new URL('data.txt', import.meta.url)
```

Note that in a Node.js environment, `import.meta.url` always returns a local path, that is, a string of the `file:URL` protocol, such as `file:///home/user/foo.js`.

**(2) import.meta.scriptElement**

`import.meta.scriptElement` is a browser-specific meta-property that returns the `<script>` element that loaded the module, equivalent to the `document.currentScript` property.

```javascript
// the HTML code is
// <script type="module" src="my-module.js" data-foo="abc"></script>

// run the following code inside my-module.js
import.meta.scriptElement.dataset.foo
// "abc"
```
