# Coding Style

This chapter explores how to apply the new ES6 syntax to coding practice, combining it with traditional JavaScript syntax to write reasonable code that is easy to read and maintain.

Many companies and organizations have published their own style guidelines. The following content mainly references the JavaScript style guide from [Airbnb](https://github.com/airbnb/javascript).
<!-- more -->
## Block-Level Scope

**(1) Replace var with let**

ES6 introduced two new commands for declaring variables: `let` and `const`. Among them, `let` can completely replace `var`, because the two are semantically equivalent, and `let` has no side effects.

```javascript
'use strict';

if (true) {
  let x = 'hello';
}

for (let i = 0; i < 10; i++) {
  console.log(i);
}
```

If the code above used `var` instead of `let`, it would actually declare two global variables, which is obviously not the intention. A variable should only be valid within the code block in which it is declared, something that the `var` command cannot do.

The `var` command has the variable hoisting effect, while the `let` command does not have this problem.

```javascript
'use strict';

if (true) {
  console.log(x); // ReferenceError
  let x = 'hello';
}
```

If the code above used `var` instead of `let`, the `console.log` line would not throw an error but would instead output `undefined`, because the variable declaration is hoisted to the top of the code block. This violates the principle that a variable should be declared before it is used.

Therefore, it is recommended to no longer use the `var` command and to use the `let` command instead.

**(2) Global Constants and Thread Safety**

Between `let` and `const`, it is recommended to prefer `const`, especially in the global environment, where you should not set variables but only constants.

There are several reasons why `const` is preferable to `let`. One is that `const` can remind people reading the program that this variable should not be changed; another is that `const` is more in line with functional programming ideas, where computation does not change values but only creates new ones, which is also beneficial for future distributed computing; the last reason is that the JavaScript **compiler optimizes `const`**, so using `const` more often helps improve the running efficiency of the program. In other words, the essential difference between `let` and `const` is actually the difference in how the compiler handles them internally.

```javascript
// bad
var a = 1, b = 2, c = 3;

// good
const a = 1;
const b = 2;
const c = 3;

// best
const [a, b, c] = [1, 2, 3];
```

Declaring constants with `const` has two additional benefits: first, the person reading the code will immediately realize that this value should not be modified; second, it prevents errors caused by unintentionally modifying the variable's value.

All functions should be set as constants.

In the long run, JavaScript may have multithreaded implementations (such as projects like Intel's River Trail). At that point, the variable represented by `let` should only appear in code that runs on a single thread and cannot be shared across multiple threads, which helps ensure thread safety.

## Strings

Static strings should always use single quotes or backticks, never double quotes. Dynamic strings should use backticks.

```javascript
// bad
const a = "foobar";
const b = 'foo' + a + 'bar';

// acceptable
const c = `foobar`;

// good
const a = 'foobar';
const b = `foo${a}bar`;
```

## Destructuring Assignment

When assigning array members to variables, prefer destructuring assignment.

```javascript
const arr = [1, 2, 3, 4];

// bad
const first = arr[0];
const second = arr[1];

// good
const [first, second] = arr;
```

If a function's parameter is a member of an object, prefer destructuring assignment.

```javascript
// bad
function getFullName(user) {
  const firstName = user.firstName;
  const lastName = user.lastName;
}

// good
function getFullName(obj) {
  const { firstName, lastName } = obj;
}

// best
function getFullName({ firstName, lastName }) {
}
```

If a function returns multiple values, prefer object destructuring over array destructuring. This makes it easier to add return values later and to change the order of return values.

```javascript
// bad
function processInput(input) {
  return [left, right, top, bottom];
}

// good
function processInput(input) {
  return { left, right, top, bottom };
}

const { left, right } = processInput(input);
```

## Objects

For objects defined on a single line, the last member should not end with a comma. For objects defined across multiple lines, **the last member should end with a comma**.

```javascript
// bad
const a = { k1: v1, k2: v2, };
const b = {
  k1: v1,
  k2: v2
};

// good
const a = { k1: v1, k2: v2 };
const b = {
  k1: v1,
  k2: v2,
};
```

Objects should be kept as static as possible; once defined, no new properties should be added arbitrarily. If adding properties is unavoidable, use the `Object.assign` method.

```javascript
// bad
const a = {};
a.x = 3;

// if reshape unavoidable
const a = {};
Object.assign(a, { x: 3 });

// good
const a = { x: null };
a.x = 3;
```

If an object's property name is dynamic, you can use a property expression to define it when creating the object.

```javascript
// bad
const obj = {
  id: 5,
  name: 'San Francisco',
};
obj[getKey('enabled')] = true;

// good
const obj = {
  id: 5,
  name: 'San Francisco',
  [getKey('enabled')]: true,
};
```

In the code above, the last property name of the object `obj` needs to be computed. In this case, it is best to use a property expression and define this property together with the other properties when creating `obj`. This way, all properties are defined in one place.

In addition, the properties and methods of an object should **use shorthand notation as much as possible**, which makes them easier to describe and write.

```javascript
var ref = 'some value';

// bad
const atom = {
  ref: ref,

  value: 1,

  addValue: function (value) {
    return atom.value + value;
  },
};

// good
const atom = {
  ref,

  value: 1,

  addValue(value) {
    return atom.value + value;
  },
};
```

## Arrays

Use the **spread operator (...) to copy arrays**.

```javascript
// bad
const len = items.length;
const itemsCopy = [];
let i;

for (i = 0; i < len; i++) {
  itemsCopy[i] = items[i];
}

// good
const itemsCopy = [...items];
```

Use the **Array.from method to convert an array-like object into an array**.

```javascript
const foo = document.querySelectorAll('.foo');
const nodes = Array.from(foo);
```

## Functions

An immediately invoked function can be written as an arrow function.

```javascript
(() => {
  console.log('Welcome to the Internet.');
})();
```

In cases where an anonymous function is used as an argument, **try to use an arrow function instead**, because it is more concise and binds `this`.

```javascript
// bad
[1, 2, 3].map(function (x) {
  return x * x;
});

// good
[1, 2, 3].map((x) => {
  return x * x;
});

// best
[1, 2, 3].map(x => x * x);
```

Use arrow functions instead of `Function.prototype.bind`; you should no longer use self/\_this/that to bind `this`.

```javascript
// bad
const self = this;
const boundMethod = function(...params) {
  return method.apply(self, params);
}

// acceptable
const boundMethod = method.bind(this);

// best
const boundMethod = (...params) => method.apply(this, params);
```

For simple, single-line functions that will not be reused, arrow functions are recommended. **If the function body is relatively complex with many lines, you should still use the traditional function syntax.**

**All configuration options should be gathered into a single object, placed as the last parameter, and boolean values should not be passed directly as parameters.**

```javascript
// bad
function divide(a, b, option = false ) {
}

// good
function divide(a, b, { option = false } = {}) {
}
```

Do not use the `arguments` variable inside a function body; use the rest operator (...) instead. Because the rest operator explicitly shows that you want to obtain the parameters, and `arguments` is an array-like object whereas the rest operator provides a real array.

```javascript
// bad
function concatenateAll() {
  const args = Array.prototype.slice.call(arguments);
  return args.join('');
}

// good
function concatenateAll(...args) {
  return args.join('');
}
```

Use the default value syntax to set default values for function parameters.

```javascript
// bad
function handleThings(opts) {
  opts = opts || {};
}

// good
function handleThings(opts = {}) {
  // ...
}
```

## Map Structure

Be careful to distinguish between Object and Map. Only use Object when modeling real-world entity objects. If you simply need a `key: value` data structure, use the Map structure, because Map has a built-in traversal mechanism.

```javascript
let map = new Map(arr);

for (let key of map.keys()) {
  console.log(key);
}

for (let value of map.values()) {
  console.log(value);
}

for (let item of map.entries()) {
  console.log(item[0], item[1]);
}
```

## Class

Always use Class instead of operations that require prototype, because the Class syntax is more concise and easier to understand.

```javascript
// bad
function Queue(contents = []) {
  this._queue = [...contents];
}
Queue.prototype.pop = function() {
  const value = this._queue[0];
  this._queue.splice(0, 1);
  return value;
}

// good
class Queue {
  constructor(contents = []) {
    this._queue = [...contents];
  }
  pop() {
    const value = this._queue[0];
    this._queue.splice(0, 1);
    return value;
  }
}
```

Use `extends` to implement inheritance, because it is simpler and carries no risk of breaking the `instanceof` operation.

```javascript
// bad
const inherits = require('inherits');
function PeekableQueue(contents) {
  Queue.apply(this, contents);
}
inherits(PeekableQueue, Queue);
PeekableQueue.prototype.peek = function() {
  return this._queue[0];
}

// good
class PeekableQueue extends Queue {
  peek() {
    return this._queue[0];
  }
}
```

## Modules

First, the Module syntax is the standard way to write JavaScript modules, so stick with this approach. Use `import` instead of `require`.

```javascript
// bad
const moduleA = require('moduleA');
const func1 = moduleA.func1;
const func2 = moduleA.func2;

// good
import { func1, func2 } from 'moduleA';
```

Use `export` instead of `module.exports`.

```javascript
// commonJS syntax
var React = require('react');

var Breadcrumbs = React.createClass({
  render() {
    return <nav />;
  }
});

module.exports = Breadcrumbs;

// ES6 syntax
import React from 'react';

class Breadcrumbs extends React.Component {
  render() {
    return <nav />;
  }
};

export default Breadcrumbs;
```

If a module has only one output value, use `export default`. If a module has multiple output values, do not use `export default`. Do not use `export default` and ordinary `export` at the same time.

Do not use wildcards in module imports, because this ensures that your module has a default export (`export default`).

```javascript
// bad
import * as myObject from './importModule';

// good
import myObject from './importModule';
```

If a module's default output is a function, the first letter of the function name should be lowercase.

```javascript
function makeStyleGuide() {
}

export default makeStyleGuide;
```

If a module's default output is an object, the first letter of the object name should be uppercase.

```javascript
const StyleGuide = {
  es6: {
  }
};

export default StyleGuide;
```

## Using ESLint

ESLint is a tool for checking syntax rules and code style, which can be used to ensure that you write code that is syntactically correct and stylistically consistent.

First, install ESLint.

```bash
$ npm i -g eslint
```

Then, install the Airbnb syntax rules, along with the import, a11y, and react plugins.

```bash
$ npm i -g eslint-config-airbnb
$ npm i -g eslint-plugin-import eslint-plugin-jsx-a11y eslint-plugin-react
```

Finally, create a `.eslintrc` file in the project's root directory to configure ESLint.

```javascript
{
  "extends": "eslint-config-airbnb"
}
```

Now you can check whether the current project's code conforms to the preset rules.

The code of the `index.js` file is as follows.

```javascript
var unusued = 'I have no purpose!';

function greet() {
    var message = 'Hello, World!';
    alert(message);
}

greet();
```

Using ESLint to check this file will report errors.

```bash
$ eslint index.js
index.js
  1:1  error  Unexpected var, use let or const instead          no-var
  1:5  error  unusued is defined but never used                 no-unused-vars
  4:5  error  Expected indentation of 2 characters but found 4  indent
  4:5  error  Unexpected var, use let or const instead          no-var
  5:5  error  Expected indentation of 2 characters but found 4  indent

✖ 5 problems (5 errors, 0 warnings)
```

The code above shows that the original file has five errors: two are that the `var` command should not be used and `let` or `const` should be used instead; one is that a variable is defined but never used; and the other two are that the line indentation is 4 spaces instead of the prescribed 2 spaces.
