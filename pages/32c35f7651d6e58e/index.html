Reading the ECMAScript Specification

# Reading the ECMAScript Specification

# Overview

The specification document is the official standard of a computer language, describing the syntax rules and implementation methods in detail.

Generally speaking, there is no need to read the specification unless you are writing a compiler. The specification is written in a very abstract and condensed way, lacks examples, is hard to understand, and is of little help for solving practical application problems. However, if you run into a thorny syntax problem and really cannot find an answer, you can turn to the specification document to learn what the language standard actually says. The specification is the "last resort" for solving problems. This is very necessary for the JavaScript language. Because its usage scenarios are complex, its syntax rules are not unified, there are many exceptions, and the behavior across different runtime environments is inconsistent, strange syntax problems keep emerging endlessly, and no syntax book can possibly cover every case. Consulting the specification is the most reliable and authoritative ultimate method for solving syntax problems.

This chapter introduces how to read the specification document of ECMAScript 6.

The ECMAScript 6 specification can be downloaded for free and read online on the official website of the ECMA International standards organization (www.ecma-international.org/ecma-262/6.0/ (opens new window)).

This specification document is quite large, with 26 chapters in total. Printed on A4 paper, it comes to a full 545 pages. Its distinguishing feature is that it specifies everything in great detail: every syntactic behavior and the implementation of every function is described thoroughly and clearly. Basically, a compiler author only needs to translate each step into code. To a large extent, this guarantees that all ES6 implementations exhibit consistent behavior.

Among the 26 chapters of the ECMAScript 6 specification, Chapters 1 through 3 introduce the document itself and have little to do with the language. Chapter 4 describes the overall design of the language, and interested readers may give it a read. Chapters 5 through 8 describe the language at a macro level. Chapter 5 explains the terminology and notation of the specification, Chapter 6 introduces data types, Chapter 7 introduces the abstract operations used internally by the language, and Chapter 8 explains how code is executed. Chapters 9 through 26 introduce the concrete syntax.

For the average user, apart from Chapter 4, the other chapters all deal with details of one aspect or another. There is no need to read them through; you only need to consult the relevant chapter when you need it.

# Terminology

The ES6 specification uses some specialized terms. Understanding these terms can help you read the specification. This section introduces a few of them.

# Abstract operations

So-called "abstract operations" are some of the engine's internal methods, which cannot be called from the outside. The specification defines a series of abstract operations, specifies their behavior, and leaves it to each engine to implement them itself.

For example, the algorithm of Boolean(value) has the following first step.

  1. Let b be ToBoolean(value).

Here ToBoolean is an abstract operation, the engine's internal algorithm for computing a boolean value.

The algorithms of many functions use the same steps multiple times, so the ES6 specification extracts them and defines them as "abstract operations" for convenience of description.

# Record and field

The ES6 specification calls the data structure of key-value maps a Record, and each key-value pair within it a field. That is to say, a Record is composed of multiple fields, and each field contains a key and a value.

# [[Notation]]

The ES6 specification makes extensive use of the [[Notation]] writing style, such as [[Value]], [[Writable]], [[Get]], [[Set]], and so on. It is used to refer to the key of a field.

For example, obj is a Record that has a Prototype property. The ES6 specification does not write obj.Prototype, but instead writes obj.[[Prototype]]. Generally speaking, properties written with the [[Notation]] style are internal properties of an object.

All JavaScript functions have an internal property [[Call]], which is used to run the function.

F.[[Call]](V, argumentsList)
1

In the code above, F is a function object, [[Call]] is its internal method, F.[[call]]() means running the function, V represents the value of this when [[Call]] runs, and argumentsList is the arguments passed to the function when it is called.

# Completion Record

Every statement returns a Completion Record, representing the result of execution. Each Completion Record has a [[Type]] property, indicating the type of the execution result.

The [[Type]] property has five possible values.

  • normal
  • return
  • throw
  • break
  • continue

If the value of [[Type]] is normal, it is called a normal completion, indicating that execution proceeded normally. The other values are all called abrupt completions. Among them, developers only need to pay attention to the case where [[Type]] is throw, that is, an execution error; the three values break, continue, and return only appear in specific scenarios and can be disregarded.

# The standard procedure of abstract operations

The execution flow of an abstract operation is generally as follows.

  1. Let resultCompletionRecord be AbstractOp().
  2. If resultCompletionRecord is an abrupt completion, return resultCompletionRecord.
  3. Let result be resultCompletionRecord.[[Value]].
  4. return result.

The first step above is to call the abstract operation AbstractOp() and obtain resultCompletionRecord, which is a Completion Record. In the second step, if this Record is an abrupt completion, then resultCompletionRecord is returned to the user. If there is no return here, it means the execution result is normal, and the resulting value is stored in the resultCompletionRecord.[[Value]] property. In the third step, this value is recorded as result. In the fourth step, result is returned to the user.

The ES6 specification expresses this standard procedure in a shorthand form.

  1. Let result be AbstractOp().
  2. ReturnIfAbrupt(result).
  3. return result.

The ReturnIfAbrupt(result) in this shorthand represents the second and third steps above, that is: if there is an error, return the error, otherwise extract the value.

There is even a further shorthand format.

  1. Let result be ? AbstractOp().
  2. return result.

The ? in the flow above represents that AbstractOp() may report an error. Once an error occurs, the error is returned; otherwise the value is extracted.

In addition to ?, the ES6 specification also uses another shorthand symbol, !.

  1. Let result be ! AbstractOp().
  2. return result.

The ! in the flow above represents that AbstractOp() will not report an error; what it returns is always a normal completion, and the value can always be extracted.

# The equality operator

Below we use a few examples to introduce how to use this specification.

The equality operator (==) is a very headache-inducing operator. Its syntactic behavior is variable and counterintuitive. This section looks at how the specification defines its behavior.

Look at the following expression. What is its value?

0 == null
1

If you are unsure of the answer, or want to know how the language handles it internally, you can consult the specification. Section 7.2.12 (opens new window) is the description of the equality operator (==).

The specification's description of each syntactic behavior is divided into two parts: first the overall behavior description, then the detailed implementation algorithm. The overall description of the equality operator is just one sentence.

"The comparison x == y, where x and y are values, produces true or false."

The meaning of the sentence above is that the equality operator is used to compare two values and returns true or false.

Below are the algorithm details.

  1. ReturnIfAbrupt(x).
  2. ReturnIfAbrupt(y).
  3. If Type(x) is the same as Type(y), then
    1. Return the result of performing Strict Equality Comparison x === y.
  4. If x is null and y is undefined, return true.
  5. If x is undefined and y is null, return true.
  6. If Type(x) is Number and Type(y) is String, return the result of the comparison x == ToNumber(y).
  7. If Type(x) is String and Type(y) is Number, return the result of the comparison ToNumber(x) == y.
  8. If Type(x) is Boolean, return the result of the comparison ToNumber(x) == y.
  9. If Type(y) is Boolean, return the result of the comparison x == ToNumber(y).
  10. If Type(x) is either String, Number, or Symbol and Type(y) is Object, then return the result of the comparison x == ToPrimitive(y).
  11. If Type(x) is Object and Type(y) is either String, Number, or Symbol, then return the result of the comparison ToPrimitive(x) == y.
  12. Return false.

The algorithm above has 12 steps in total, translated as follows.

  1. If x is not a normal value (for example, it throws an error), interrupt execution.
  2. If y is not a normal value, interrupt execution.
  3. If Type(x) is the same as Type(y), perform the strict equality operation x === y.
  4. If x is null and y is undefined, return true.
  5. If x is undefined and y is null, return true.
  6. If Type(x) is a number and Type(y) is a string, return the result of x == ToNumber(y).
  7. If Type(x) is a string and Type(y) is a number, return the result of ToNumber(x) == y.
  8. If Type(x) is a boolean, return the result of ToNumber(x) == y.
  9. If Type(y) is a boolean, return the result of x == ToNumber(y).
  10. If Type(x) is a string, number, or Symbol value, and Type(y) is an object, return the result of x == ToPrimitive(y).
  11. If Type(x) is an object, and Type(y) is a string, number, or Symbol value, return the result of ToPrimitive(x) == y.
  12. Return false.

Since the type of 0 is Number and the type of null is Null (this is specified in Section 4.3.13 (opens new window) of the specification; it is the result of the internal Type operation and has nothing to do with the typeof operator), the first 11 steps above all fail to produce a result, and only at the 12th step is false obtained.

0 == null // false
1

# Holes in arrays

Now let's look at another example.

const a1 = [undefined, undefined, undefined];
const a2 = [, , ,];

a1.length // 3
a2.length // 3

a1[0] // undefined
a2[0] // undefined

a1[0] === a2[0] // true

In the code above, the members of array a1 are three undefined, while the members of array a2 are three holes. These two arrays are very similar: both have a length of 3, and the member read out at each position is undefined.

However, there is actually a major difference between them.

0 in a1 // true
0 in a2 // false

a1.hasOwnProperty(0) // true
a2.hasOwnProperty(0) // false

Object.keys(a1) // ["0", "1", "2"]
Object.keys(a2) // []

a1.map(n => 1) // [1, 1, 1]
a2.map(n => 1) // [, , ,]

The code above lists four operations in total, and arrays a1 and a2 produce different results for all of them. The first three operations (the in operator, the array's hasOwnProperty method, and the Object.keys method) all show that the property names cannot be obtained for array a2. The last operation (the array's map method) shows that no iteration occurred for array a2.

Why is the behavior of the members of a1 and a2 inconsistent? What exactly is the difference between an array member being undefined and being a hole?

Section 12.2.5 "Array Initializer" (opens new window) of the specification gives the answer.

"Array elements may be elided at the beginning, middle or end of the element list. Whenever a comma in the element list is not preceded by an AssignmentExpression (i.e., a comma at the beginning or after another comma), the missing array element contributes to the length of the Array and increases the index of subsequent elements. Elided array elements are not defined. If an element is elided at the end of an array, that element does not contribute to the length of the Array."

Translated as follows.

"Array members may be omitted. As long as there is no expression before a comma, the array's length property increases by 1, and the position index of subsequent members increases accordingly. An omitted member is not defined. If the omitted member is the last member of the array, it does not cause the array's length property to increase."

The specification above states it very clearly: holes in an array are reflected in the length property, which means a hole has its own position, but the value at this position is undefined, that is, the value does not exist. If you insist on reading it, the result is undefined (because undefined means "does not exist" in the JavaScript language).

This explains why the in operator, the array's hasOwnProperty method, and the Object.keys method all fail to obtain the property name of a hole. Because this property name simply does not exist: the specification does not say that a property name (position index) should be assigned to a hole; it only says to increase the position index of the next element by 1.

As for why the array's map method skips holes, see the next section.

# The array's map method

Section 22.1.3.15 (opens new window) of the specification defines the array's map method. This section first gives an overall description of the behavior of the map method, in which array holes are not mentioned.

The algorithm description that follows is as follows.

  1. Let O be ToObject(this value).
  2. ReturnIfAbrupt(O).
  3. Let len be ToLength(Get(O, "length")).
  4. ReturnIfAbrupt(len).
  5. If IsCallable(callbackfn) is false, throw a TypeError exception.
  6. If thisArg was supplied, let T be thisArg; else let T be undefined.
  7. Let A be ArraySpeciesCreate(O, len).
  8. ReturnIfAbrupt(A).
  9. Let k be 0.
  10. Repeat, while k < len
    1. Let Pk be ToString(k).
    2. Let kPresent be HasProperty(O, Pk).
    3. ReturnIfAbrupt(kPresent).
    4. If kPresent is true, then
      1. Let kValue be Get(O, Pk).
      2. ReturnIfAbrupt(kValue).
      3. Let mappedValue be Call(callbackfn, T, «kValue, k, O»).
      4. ReturnIfAbrupt(mappedValue).
      5. Let status be CreateDataPropertyOrThrow (A, Pk, mappedValue).
      6. ReturnIfAbrupt(status).
    5. Increase k by 1.
  11. Return A.

Translated as follows.

  1. Obtain the this object of the current array.
  2. If there is an error, return.
  3. Compute the length property of the current array.
  4. If there is an error, return.
  5. If the map method's argument callbackfn is not callable, throw an error.
  6. If this is specified among the arguments of the map method, let T equal that argument; otherwise T is undefined.
  7. Generate a new array A, keeping it consistent with the length property of the current array.
  8. If there is an error, return.
  9. Set k equal to 0.
  10. As long as k is less than the length property of the current array, repeat the following steps.
    1. Set Pk equal to ToString(k), i.e., convert K to a string.
    2. Set kPresent equal to HasProperty(O, Pk), i.e., determine whether the current array has the specified property.
    3. If there is an error, return.
    4. If kPresent equals true, then carry out the following steps.
      1. Set kValue equal to Get(O, Pk), fetching the specified property of the current array.
      2. If there is an error, return.
      3. Set mappedValue equal to Call(callbackfn, T, «kValue, k, O»), i.e., execute the callback function.
      4. If there is an error, return.
      5. Set status equal to CreateDataPropertyOrThrow (A, Pk, mappedValue), i.e., place the value of the callback function into the specified position of array A.
      6. If there is an error, return.
    5. Increase k by 1.
  11. Return A.

Examining the algorithm above carefully, you can find that when processing an array that is entirely holes, the earlier steps have no problem. When the procedure reaches step 2 within step 10, kPresent will report an error, because the property name corresponding to a hole does not exist as far as the array is concerned, so it returns and does not carry out the subsequent steps.

const arr = [, , ,];
arr.map(n => {
  console.log(n);
  return 1;
}) // [, , ,]

In the code above, arr is an array that is entirely holes. When the map method iterates over the members and finds a hole, it skips it directly and does not enter the callback function. Therefore, the console.log statement inside the callback function is never executed at all, and the whole map method returns a new array that is entirely holes.

The V8 engine's implementation (opens new window) of the map method is as follows, and you can see that it is completely consistent with the algorithm description in the specification.

function ArrayMap(f, receiver) {
  CHECK_OBJECT_COERCIBLE(this, "Array.prototype.map");

  // Pull out the length so that modifications to the length in the
  // loop will not affect the looping and side effects are visible.
  var array = TO_OBJECT(this);
  var length = TO_LENGTH_OR_UINT32(array.length);
  return InnerArrayMap(f, receiver, array, length);
}

function InnerArrayMap(f, receiver, array, length) {
  if (!IS_CALLABLE(f)) throw MakeTypeError(kCalledNonCallable, f);

  var accumulator = new InternalArray(length);
  var is_array = IS_ARRAY(array);
  var stepping = DEBUG_IS_STEPPING(f);
  for (var i = 0; i < length; i++) {
    if (HAS_INDEX(array, i, is_array)) {
      var element = array[i];
      // Prepare break slots for debugger step in.
      if (stepping) %DebugPrepareStepInIfStepping(f);
      accumulator[i] = %_Call(f, receiver, element, i, array);
    }
  }
  var result = new GlobalArray();
  %MoveArrayContents(accumulator, result);
  return result;
}
