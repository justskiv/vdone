Basic syntax of Class

# Basic syntax of Class

# Introduction

# The origin of classes

In JavaScript, the traditional way to generate instance objects is through constructor functions. Here is an example.

function Point(x, y) {
  this.x = x;
  this.y = y;
}

Point.prototype.toString = function () {
  return '(' + this.x + ', ' + this.y + ')';
};

var p = new Point(1, 2);

The syntax above is very different from traditional object-oriented languages (such as C++ and Java), which can easily confuse programmers who are new to this language.

ES6 provides a syntax that is closer to traditional languages, introducing the concept of Class as a template for objects. With the class keyword, you can define a class.

Basically, ES6's class can be seen as just syntactic sugar; the vast majority of its functionality can be achieved with ES5, and the new class syntax merely makes object prototype syntax clearer and more like object-oriented programming syntax. Rewriting the code above with ES6's class looks like the following.

class Point {
  constructor(x, y) {
    this.x = x;
    this.y = y;
  }

  toString() {
    return '(' + this.x + ', ' + this.y + ')';
  }
}

The code above defines a "class". You can see there is a constructor method inside it, which is the constructor method, and the this keyword represents the instance object. In other words, the ES5 constructor function Point corresponds to the constructor method of the ES6 Point class.

Besides the constructor method, the Point class also defines a toString method. Note that when defining a method of a "class", you do not need to add the function keyword in front; you can just put the function definition directly inside. Also, methods do not need to be separated by commas, and adding them will cause an error.

An ES6 class can be entirely regarded as another way of writing a constructor function.

class Point {
  // ...
}

typeof Point // "function"
Point === Point.prototype.constructor // true

# The constructor method

The constructor method is the default method of a class. When an object instance is generated through the new command, this method is automatically called. A class must have a constructor method; if not explicitly defined, an empty constructor method is added by default.

class Point {
}

// equivalent to
class Point {
  constructor() {}
}
1

In the code above, an empty class Point is defined, and the JavaScript engine automatically adds an empty constructor method to it.

The constructor method returns the instance object (i.e., this) by default, but you can specify that it return another object instead.

class Foo {
  constructor() {
    return Object.create(null);
  }
}

new Foo() instanceof Foo
// false

In the code above, the constructor function returns a completely new object, with the result that the instance object is not an instance of the Foo class.

A class must be called with new, otherwise an error will be thrown. This is one of the main differences from an ordinary constructor function, which can be executed without new.

  constructor() {
    return Object.create(null);
  }
}

// TypeError: Class constructor Foo cannot be invoked without 'new'

# Instances of a class

The syntax for generating an instance of a class is exactly the same as in ES5, also using the new command. As mentioned earlier, if you forget to add new and call Class like a function, an error will be thrown.

class Point {
  // ...
}

// error
var point = Point(2, 3);

// correct
var point = new Point(2, 3);

As in ES5, unless the properties of an instance are explicitly defined on the instance itself (i.e., on the this object), they are all defined on the prototype (i.e., on the class).

// define the class
class Point {

  constructor(x, y) { // the constructor property is defined on the prototype
    this.x = x; // the x property is defined on the instance
    this.y = y; // the y property is defined on the instance
  }

  toString() { // the toString property is defined on the prototype
    return '(' + this.x + ', ' + this.y + ')';
  }

}

var point = new Point(2, 3);

point.toString() // (2, 3)

point.hasOwnProperty('x') // true
point.hasOwnProperty('y') // true
point.hasOwnProperty('toString') // false
point.__proto__.hasOwnProperty('toString') // true

These are all consistent with ES5 behavior.

As in ES5, all instances of a class share a single prototype object.

var p1 = new Point(2,3);
var p2 = new Point(3,2);

//true

In the code above, both p1 and p2 are instances of Point, and their prototypes are both Point.prototype, so the __proto__ properties are equal.

This also means that you can add methods to a "class" through the __proto__ property of an instance.

__proto__ is not a feature of the language itself; it is a private property added by the various vendors in their specific implementations. Although many modern browsers' JS engines currently provide this private property, it is still not recommended to use this property in production, to avoid creating a dependency on the environment. In a production environment, we can use the Object.getPrototypeOf method to get the prototype of an instance object, and then add methods/properties to the prototype.

var p1 = new Point(2,3);
var p2 = new Point(3,2);

p1.__proto__.printName = function () { return 'Oops' };

p1.printName() // "Oops"
p2.printName() // "Oops"

var p3 = new Point(4,2);
p3.printName() // "Oops"

The code above adds a printName method on the prototype of p1. Since the prototype of p1 is the prototype of p2, p2 can also call this method. Moreover, the newly created instance p3 can also call this method afterwards. This means that you must be quite careful when using an instance's __proto__ property to rewrite the prototype; it is not recommended, because this changes the original definition of the "class" and affects all instances.

# Getter and setter functions

As in ES5, you can use the get and set keywords inside a "class" to set a setter function and a getter function for a property, intercepting the get/set behavior of that property.

class MyClass {
  constructor() {
    // ...
  }
  get prop() {
    return 'getter';
  }
  set prop(value) {
    console.log('setter: '+value);
  }
}

let inst = new MyClass();

inst.prop = 123;
// setter: 123

inst.prop
// 'getter'

# Class expressions

Like functions, classes can also be defined in the form of an expression.

const MyClass = class Me {
  getClassName() {
    return Me.name;
  }
};

# Points to note

(1) Strict mode

The inside of classes and modules is in strict mode by default, so you do not need to use use strict to specify the running mode. As long as your code is written inside a class or module, only strict mode is available. Considering that all code in the future will actually run inside modules, ES6 has actually upgraded the entire language to strict mode.

(2) No hoisting

Classes are not subject to variable hoisting, which is completely different from ES5.

new Foo(); // ReferenceError
class Foo {}
1
2

In the code above, the Foo class is used before it is defined, and this will throw an error, because ES6 does not hoist the class declaration to the top of the code. The reason for this rule is related to inheritance, which will be mentioned below; it must be guaranteed that a subclass is defined after the parent class.

{
  let Foo = class {};
  class Bar extends Foo {
  }
}
1
2
3
4
5

The code above will not throw an error, because when Bar inherits from Foo, Foo is already defined. However, if class were hoisted, the code above would throw an error, because class would be hoisted to the top of the code while the let command is not hoisted, so when Bar inherits from Foo, Foo would not yet be defined.

(3) The name property

Since an ES6 class is essentially just a wrapper around an ES5 constructor function, many features of functions are inherited by Class, including the name property.

class Point {}
Point.name // "Point"
1
2

The name property always returns the class name that immediately follows the class keyword.

(4) Generator methods

If an asterisk (*) is added before a method, it indicates that the method is a Generator function.

class Foo {
  constructor(...args) {
    this.args = args;
  }
  * [Symbol.iterator]() {
    for (let arg of this.args) {
      yield arg;
    }
  }
}

for (let x of new Foo('hello', 'world')) {
  console.log(x);
}
// hello
// world

In the code above, there is an asterisk before the Symbol.iterator method of the Foo class, indicating that the method is a Generator function. The Symbol.iterator method returns the default iterator of the Foo class, and the for...of loop automatically calls this iterator.

(5) The reference of this

If a method of a class contains this, it points to the instance of the class by default. However, you must be very careful, as an error is likely to occur once the method is used on its own.

class Logger {
  printName(name = 'there') {
    this.print(`Hello ${name}`);
  }

  print(text) {
    console.log(text);
  }
}

const logger = new Logger();
const { printName } = logger;
printName(); // TypeError: Cannot read property 'print' of undefined
1
2

In the code above, the this in the printName method points to the instance of the Logger class by default. However, if this method is extracted and used on its own, this points to the environment where the method runs (since the inside of a class is in strict mode, this actually points to undefined), which results in not being able to find the print method and throwing an error.

A relatively simple solution is to bind this in the constructor method, so that the print method will not be missing.

class Logger {
  constructor() {
    this.printName = this.printName.bind(this);
  }

  // ...
}
1

Another solution is to use an arrow function.

class Obj {
  constructor() {
    this.getThis = () => this;
  }
}

const myObj = new Obj();
myObj.getThis() === myObj // true
1

The this inside an arrow function always points to the object where it was defined. In the code above, the arrow function is located inside the constructor function, and its definition takes effect when the constructor function is executed. At this point, the running environment where the arrow function resides is definitely the instance object, so this will always point to the instance object.

There is also another solution that uses Proxy to automatically bind this when a method is obtained.

function selfish (target) {
  const cache = new WeakMap();
  const handler = {
    get (target, key) {
      const value = Reflect.get(target, key);
      if (typeof value !== 'function') {
        return value;
      }
      if (!cache.has(value)) {
        cache.set(value, value.bind(target));
      }
      return cache.get(value);
    }
  };
  const proxy = new Proxy(target, handler);
  return proxy;
}

const logger = selfish(new Logger());

# Static methods

A class is equivalent to the prototype of its instances, and all methods defined in the class are inherited by the instances. If you add the static keyword before a method, it indicates that the method will not be inherited by instances, but is instead called directly through the class. This is called a "static method".

class Foo {
  static classMethod() {
    return 'hello';
  }
}

Foo.classMethod() // 'hello'

var foo = new Foo();
foo.classMethod()
// TypeError: foo.classMethod is not a function
1
2
3
4
5
6
7
8

In the code above, the classMethod method of the Foo class is preceded by the static keyword, indicating that the method is a static method that can be called directly on the Foo class (Foo.classMethod()), rather than on an instance of the Foo class. If a static method is called on an instance, an error is thrown, indicating that the method does not exist.

Note that if a static method contains the this keyword, this this refers to the class, not the instance.

class Foo {
  static bar() {
    this.baz();
  }
  static baz() {
    console.log('hello');
  }
  baz() {
    console.log('world');
  }
}

Foo.bar() // hello

In the code above, the static method bar calls this.baz, and here this refers to the Foo class, not an instance of Foo, which is equivalent to calling Foo.baz. In addition, this example also shows that a static method can have the same name as a non-static method.

class Foo {
  static classMethod() {

# New syntax for instance properties

Besides being defined on this inside the constructor() method, instance properties can also be defined at the top level of the class.

class IncreasingCounter {
  constructor() {
    this._count = 0;
  }
  get value() {
    console.log('Getting the current value!');
    return this._count;
  }
  increment() {
    this._count++;
  }
}

In the code above, the instance property this._count is defined inside the constructor() method. Another syntax is that this property can also be defined at the top level of the class, with everything else unchanged.

class IncreasingCounter {
  _count = 0;
  get value() {
    console.log('Getting the current value!');
    return this._count;
  }
  increment() {
    this._count++;
  }
}

In the code above, the instance property _count is at the same level as the getter function value() and the increment() method. In this case, you do not need to add this before the instance property.

The benefit of this new syntax is that all of an instance object's own properties are defined at the head of the class, which looks tidier, and you can tell at a glance which instance properties the class has.

class foo {

# Static properties

A static property refers to a property of the Class itself, i.e., Class.propName, rather than a property defined on the instance object (this).

class Foo {
}

Foo.prop = 1;
Foo.prop // 1
1
2
3
4
5

The syntax above defines a static property prop for the Foo class.

Currently, only this syntax is feasible, because ES6 explicitly stipulates that there are only static methods inside a Class, not static properties. There is now a proposal (opens new window) that provides static properties for classes, with the syntax being to add the static keyword in front of an instance property.

class MyClass {
  static myStaticProp = 42;

  constructor() {
    console.log(MyClass.myStaticProp); // 42
  }
}
1
2
3
4
5
6
7

This new syntax greatly facilitates the expression of static properties.

// old syntax
class Foo {
  // ...
}
Foo.prop = 1;

// new syntax
class Foo {
  static prop = 1;
}
1

4

6
7
8
9
10

In the code above, with the old syntax the static property is defined outside the class. After the entire class is generated, the static property is then generated. This makes it easy to overlook this static property, and it also does not conform to the code organization principle that related code should be placed together. In addition, the new syntax is declarative (declarative), rather than assignment processing, with better semantics.

# Private methods and private properties

# Existing solutions

Private methods and private properties are methods and properties that can only be accessed inside the class and cannot be accessed externally. This is a common requirement that helps with code encapsulation, but ES6 does not provide it, so it can only be simulated through workarounds.

One approach is to distinguish them in naming.

class Widget {

  // public method
  foo (baz) {
    this._bar(baz);
  }

  // private method
  _bar(baz) {
    return this.snaf = baz;
  }

  // ...
}
1

In the code above, the underscore before the _bar method indicates that this is a private method intended for internal use only. However, this naming is not foolproof, and this method can still be called from outside the class.

Another approach is to simply move the private method out of the module, because all methods inside a module are externally visible.

class Widget {
  foo (baz) {
    bar.call(this, baz);
  }

  // ...
}

function bar(baz) {
  return this.snaf = baz;
}

In the code above, foo is a public method that internally calls bar.call(this, baz). This makes bar effectively a private method of the current module.

const bar = Symbol('bar');
const snaf = Symbol('snaf');

export default class myClass{

  // public method
  foo(baz) {
    this[bar](baz);
  }

  // private method
  [bar](baz) {
    return this[snaf] = baz;
  }

  // ...
};

# Proposal for private properties

Currently, there is a proposal (opens new window) that adds private properties to class. The method is to use # before the property name.

class IncreasingCounter {
  #count = 0;
  get value() {
    console.log('Getting the current value!');
    return this.#count;
  }
  increment() {
    this.#count++;
  }
}

The reason for introducing a new prefix # to denote private properties, rather than adopting a private keyword, is that JavaScript is a dynamic language without type declarations, and using a separate symbol seems to be the only relatively convenient and reliable way to accurately distinguish whether a property is private. In addition, the Ruby language uses @ to denote private properties; ES6 did not use this symbol but used # because @ has already been reserved for Decorators.

This syntax can be used not only to write private properties, but also to write private methods.

  constructor(a, b) {
    this.#a = a;
    this.#b = b;
  }
  #sum() {
    return #a + #b;
  }
  printSum() {
    console.log(this.#sum());
  }
}

# The new.target property

new is the command for generating an instance object from a constructor function. ES6 introduces a new.target property for the new command. This property is generally used inside a constructor function and returns the constructor function on which the new command operates. If the constructor function is not called via the new command or Reflect.construct(), new.target returns undefined, so this property can be used to determine how a constructor function was called.

function Person(name) {
  if (new.target !== undefined) {
    this.name = name;
  } else {
    throw new Error('must use the new command to generate an instance');
  }
}

// another way of writing
function Person(name) {
  if (new.target === Person) {
    this.name = name;
  } else {
    throw new Error('must use the new command to generate an instance');
  }
}

var person = new Person('Zhang San'); // correct
var notAPerson = Person.call(person, 'Zhang San');  // error
