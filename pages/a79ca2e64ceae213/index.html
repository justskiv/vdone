Module Loading Implementation

# Module Loading Implementation

The previous chapter covered the syntax of modules. This chapter explains how to load ES6 modules in the browser and in Node.js, along with some problems frequently encountered in real-world development (such as circular loading).

# Loading in the Browser

# Traditional Approach

In an HTML page, the browser loads JavaScript scripts via the <script> tag.

<!-- Script embedded in the page -->
<script type="application/javascript">
  // module code
</script>

<!-- External script -->
<script type="application/javascript" src="path/to/myModule.js">
</script>
1
2
3
4
5
6
7
8

In the code above, since the default language of browser scripts is JavaScript, type="application/javascript" can be omitted.

By default, the browser loads JavaScript scripts synchronously: the rendering engine stops when it encounters a <script> tag, waits until the script has finished executing, and then continues rendering. For an external script, the download time must also be added.

If the script is large, downloading and executing it can take a long time, which blocks the browser. The user feels that the browser has "frozen" and is unresponsive. This is obviously a poor experience, so the browser allows scripts to load asynchronously. Below are the two asynchronous-loading syntaxes.

<script src="path/to/myModule.js" defer></script>
<script src="path/to/myModule.js" async></script>
1
2

In the code above, when the <script> tag enables the defer or async attribute, the script loads asynchronously. When the rendering engine encounters this line, it begins downloading the external script, but instead of waiting for it to download and execute, it directly executes the commands that follow.

The difference between defer and async is this: defer waits until the entire page has been rendered normally in memory (the DOM structure is fully generated and other scripts have finished executing) before it executes; async, once downloaded, causes the rendering engine to interrupt rendering, execute the script, and then continue rendering. In a nutshell, defer is "execute after rendering is done," and async is "execute as soon as the download is done." In addition, if there are multiple defer scripts, they are loaded in the order they appear on the page, whereas the load order of multiple async scripts is not guaranteed.

# Loading Rules

To load an ES6 module, the browser also uses the <script> tag, but the type="module" attribute must be added.

<script type="module" src="./foo.js"></script>
1

The code above inserts a module foo.js into the web page. Since the type attribute is set to module, the browser knows this is an ES6 module.

For a <script> with type="module", the browser always loads it asynchronously, without blocking the browser. That is, it waits until the entire page has been rendered before executing the module script, which is equivalent to enabling the defer attribute of the <script> tag.

<script type="module" src="./foo.js"></script>
<!-- equivalent to -->
<script type="module" src="./foo.js" defer></script>
1
2
3

If a web page has multiple <script type="module"> elements, they execute one after another in the order they appear on the page.

The async attribute of the <script> tag can also be enabled. In that case, as soon as loading completes, the rendering engine interrupts rendering and executes the script immediately. After execution finishes, rendering resumes.

<script type="module" src="./foo.js" async></script>
1

Once the async attribute is used, <script type="module"> no longer executes in the order it appears on the page; instead, the module is executed as soon as it finishes loading.

ES6 modules can also be embedded inline in a web page, with syntax and behavior identical to loading an external script.

<script type="module">
  import utils from "./utils.js";

  // other code
</script>
1
2
3
4
5

For example, jQuery supports module loading.

<script type="module">
  import $ from "./jquery/src/jquery.js";
  $('#message').text('Hi from jQuery!');
</script>
1
2
3
4

There are a few things to note about external module scripts (in the example above, foo.js).

  • The code runs within the module scope, not the global scope. Top-level variables inside the module are not visible to the outside.
  • A module script automatically uses strict mode, whether or not use strict is declared.
  • Within a module, you can use the import command to load other modules (the .js suffix cannot be omitted, and an absolute or relative URL must be provided), and you can use the export command to expose interfaces.
  • Within a module, the top-level this keyword returns undefined, rather than pointing to window. In other words, using the this keyword at the top level of a module is meaningless.
  • If the same module is loaded multiple times, it executes only once.

Below is an example module.

import utils from 'https://example.com/js/utils.js';

const x = 1;

console.log(x === window.x); //false
console.log(this === undefined); // true
1
2
3
4
5
6

Exploiting the fact that the top-level this equals undefined, you can detect whether the current code is inside an ES6 module.

const isNotModuleScript = this !== undefined;
1

# Differences Between ES6 Modules and CommonJS Modules

Before discussing how Node.js loads ES6 modules, you must understand that ES6 modules are completely different from CommonJS modules.

There are two major differences between them.

  • A CommonJS module outputs a copy of a value, while an ES6 module outputs a reference to the value.
  • CommonJS modules are loaded at runtime, while ES6 modules export their interface at compile time.

The second difference is because what CommonJS loads is an object (i.e., the module.exports property), and that object is generated only after the script has finished running. An ES6 module, by contrast, is not an object; its external interface is merely a static definition that is generated during the static parsing phase of the code.

Let us focus on explaining the first difference.

A CommonJS module outputs a copy of a value, which means that once a value is output, changes inside the module no longer affect that value. Consider the following example, the module file lib.js.

// lib.js
var counter = 3;
function incCounter() {
  counter++;
}
module.exports = {
  counter: counter,
  incCounter: incCounter,
};
1

The code above outputs the internal variable counter and the internal method incCounter that modifies it. Then, in main.js, we load this module.

// main.js
var mod = require('./lib');

console.log(mod.counter);  // 3
mod.incCounter();
console.log(mod.counter); // 3
1
2
3
4
5
6

The code above shows that after lib.js is loaded, its internal changes no longer affect the output mod.counter. This is because mod.counter is a primitive value and gets cached. Only by writing it as a function can you obtain the value after the internal change.

// lib.js
var counter = 3;
function incCounter() {
  counter++;
}
module.exports = {
  get counter() {
    return counter
  },
  incCounter: incCounter,
};
1

In the code above, the exported counter property is in fact a getter function. Now if you run main.js again, you can correctly read the change in the internal variable counter.

$ node main.js
3
4

The runtime mechanism of ES6 modules is different from CommonJS. When the JS engine performs static analysis of a script and encounters a module-loading command import, it generates a read-only reference. When the script is actually executed, it uses this read-only reference to fetch the value from the loaded module. In other words, ES6's import is somewhat like a "symbolic link" in Unix systems: when the original value changes, the value loaded by import changes accordingly. Therefore, ES6 modules are dynamic references and do not cache values; the variables inside a module are bound to the module they belong to.

Let's use the previous example again.

// lib.js
export let counter = 3;
export function incCounter() {
  counter++;
}

// main.js
import { counter, incCounter } from './lib';
console.log(counter); // 3
incCounter();
console.log(counter); // 4
1

The code above shows that the variable counter imported through the ES6 module is live and fully reflects the internal changes of the module lib.js it belongs to.

// m1.js
export var foo = 'bar';
setTimeout(() => foo = 'baz', 500);

// m2.js
import {foo} from './m1.js';
console.log(foo);

In the code above, the variable foo in m1.js equals bar when it is first loaded, and after 500 milliseconds it changes to baz.

Let's see whether m2.js can correctly read this change.

The code above shows that an ES6 module does not cache the run result; instead, it dynamically fetches the value from the loaded module, and the variable is always bound to the module it belongs to.

Because a module variable imported via ES6 is merely a "symbolic link," this variable is read-only, and reassigning it throws an error.

// lib.js
export let obj = {};

// main.js
import { obj } from './lib';

obj.prop = 123; // OK
obj = {}; // TypeError
1

In the code above, main.js imports the variable obj from lib.js. You can add properties to obj, but reassigning it throws an error. This is because the address that the variable obj points to is read-only and cannot be reassigned, just as if main.js had created a const variable named obj.

Finally, what export outputs through an interface is the same value. Different scripts that load this interface all get the same instance.

// mod.js
function C() {

  this.add = function () {
    this.sum += 1;

  this.show = function () {
    console.log(this.sum);
  };
}

export let c = new C();

# Loading in Node.js

# Overview

Node.js handles ES6 modules in a rather cumbersome way, because it has its own CommonJS module format, which is incompatible with the ES6 module format. The current solution is to separate the two, with ES6 modules and CommonJS using their own respective loading schemes. Starting from version v13.2, Node.js has ES6 module support enabled by default.

Node.js requires ES6 modules to use the .mjs filename suffix. That is, as long as a script file uses the import or export command, it must use the .mjs suffix. When Node.js encounters an .mjs file, it treats it as an ES6 module, enables strict mode by default, and there is no need to specify "use strict" at the top of each module file.

If you don't want to change the suffix to .mjs, you can specify the type field as module in the project's package.json file.

{
   "type": "module"
}
1
2
3

Once this is set, the JS scripts in that directory are interpreted as ES6 modules.

# interpreted as an ES6 module
$ node my-app.js
1
2

If you still need to use CommonJS modules at this point, then you need to change the suffix of all CommonJS scripts to .cjs. If there is no type field, or the type field is commonjs, then .js scripts are interpreted as CommonJS modules.

To sum it up in one sentence: .mjs files are always loaded as ES6 modules, .cjs files are always loaded as CommonJS modules, and the loading of .js files depends on the setting of the type field in package.json.

Note: try not to mix ES6 modules and CommonJS modules. The require command cannot load .mjs files and will throw an error; only the import command can load .mjs files. Conversely, you cannot use the require command inside an .mjs file either; you must use import.

# The main Field

The package.json file has two fields that can specify the entry file of a module: main and exports. For relatively simple modules, you can use only the main field to specify the entry file for module loading.

// ./node_modules/es-module-package/package.json
{
  "type": "module",
  "main": "./src/index.js"
}
1
2
3
4
5

The code above specifies the project's entry script as ./src/index.js, in ES6 module format. If there were no type field, index.js would be interpreted as a CommonJS module.

Then, the import command can load this module.

// ./my-app.mjs

import { something } from 'es-module-package';
// what is actually loaded is ./node_modules/es-module-package/src/index.js
1
2
3
4

In the code above, after running this script, Node.js goes to the ./node_modules directory to look for the es-module-package module, and then executes the entry file according to the main field of that module's package.json.

At this point, if you use the CommonJS require() command to load the es-module-package module, it will throw an error, because CommonJS modules cannot handle the export command.

# The exports Field

The exports field has higher priority than the main field. It has several uses.

(1) Subdirectory aliases

The exports field of the package.json file can specify aliases for scripts or subdirectories.

// ./node_modules/es-module-package/package.json
{
  "exports": {
    "./submodule": "./src/submodule.js"
  }
}
1
2
3
4
5
6

The code above specifies submodule as the alias for src/submodule.js, and you can then load this file via the alias.

import submodule from 'es-module-package/submodule';
// loads ./node_modules/es-module-package/src/submodule.js
1
2

Below is an example of subdirectory aliases.

// ./node_modules/es-module-package/package.json
{
  "exports": {
    "./features/": "./src/features/"
  }
}

import feature from 'es-module-package/features/x.js';
// loads ./node_modules/es-module-package/src/features/x.js
1
2
3
4
5

7
8
9

If no alias is specified, you cannot load a script using this "module + script name" form.

// error
import submodule from 'es-module-package/private-module.js';

// no error
import submodule from './node_modules/es-module-package/private-module.js';
1
2

(2) Alias for main

If the alias in the exports field is ., it represents the module's main entry, has higher priority than the main field, and can be shortened directly to the value of the exports field.

{
  "exports": {
    ".": "./main.js"
  }
}

// equivalent to
{
  "exports": "./main.js"
}
1

10

Since the exports field is only recognized by Node.js versions that support ES6, it can be used to maintain compatibility with older Node.js versions.

{
  "main": "./main-legacy.cjs",
  "exports": {
    ".": "./main-modern.cjs"
  }
}
1
2
3
4
5
6

In the code above, the entry file for older Node.js versions (which don't support ES6 modules) is main-legacy.cjs, while the entry file for newer Node.js versions is main-modern.cjs.

(3) Conditional loading

Using the . alias, you can specify different entries for ES6 modules and CommonJS. Currently, this feature requires the --experimental-conditional-exports flag to be enabled when running Node.js.

{
  "type": "module",
  "exports": {
    ".": {
      "require": "./main.cjs",
      "default": "./main.js"
    }
  }
}
1

In the code above, the require condition of the alias . specifies the entry file for the require() command (i.e., the CommonJS entry), and the default condition specifies the entry for all other cases (i.e., the ES6 entry).

The code above can be shortened as follows.

{
  "exports": {
    "require": "./main.cjs",
    "default": "./main.js"
  }
}

{
  // error
  "exports": {
    "./feature": "./lib/feature.js",

    "default": "./main.js"
  }

# Loading a CommonJS Module from an ES6 Module

Currently, the common way for a single module to support both ES6 and CommonJS formats is this: the main field of the package.json file specifies the CommonJS entry, for use by Node.js; the module field specifies the ES6 module entry, for use by bundlers, because Node.js does not recognize the module field.

With the conditional loading from the previous section, Node.js itself can handle both module types simultaneously.

// ./node_modules/pkg/package.json
{
  "type": "module",
  "main": "./index.cjs",
  "exports": {
    "require": "./index.cjs",
    "default": "./wrapper.mjs"
  }
}
1
2
3
4
5
6
7
8
9

The code above specifies the CommonJS entry file index.cjs. Below is the code of this file.

// ./node_modules/pkg/index.cjs
exports.name = 'value';
1
2

Then, an ES6 module can load this file.

// ./node_modules/pkg/wrapper.mjs
import cjsModule from './index.cjs';
export const name = cjsModule.name;
1
2
3

Note that when the import command loads a CommonJS module, it can only load it as a whole; it cannot load just a single output item.

// correct
import packageMain from 'commonjs-package';

// error
import { method } from 'commonjs-package';
1
2
3
4
5

There is also an alternative loading approach, which is to use Node.js's built-in module.createRequire() method.

// cjs.cjs
module.exports = 'cjs';

// esm.mjs
import { createRequire } from 'module';

const require = createRequire(import.meta.url);

const cjs = require('./cjs.cjs');
cjs === 'cjs'; // true

In the code above, the ES6 module can load a CommonJS module via the module.createRequire() method.

# Loading Paths

The loading path of an ES6 module must give the full path of the script; the script's suffix cannot be omitted. If the import command or the main field of the package.json file omits the script suffix, an error is thrown.

// throws an error in an ES6 module
import { something } from './index';
1
2

To match the browser's import loading rules, Node.js's .mjs files support URL paths.

import './foo.mjs?query=1'; // loads ./foo with the parameter ?query=1
1

In the code above, the script path carries the parameter ?query=1, and Node interprets it according to URL rules. The same script will be loaded multiple times as long as the parameters differ, and saved as separate caches. For this reason, if a filename contains special characters such as :, %, #, or ?, it is best to escape these characters.

Currently, Node.js's import command only supports loading local modules (the file: protocol) and the data: protocol; it does not support loading remote modules. In addition, the script path only supports relative paths, not absolute paths (i.e., paths beginning with / or //).

Finally, Node's import command loads asynchronously, which is the same as the browser's handling.

# Internal Variables

ES6 modules are meant to be universal: the same module, without modification, should work in both the browser environment and the server environment. To achieve this goal, Node stipulates that some of CommonJS's special internal variables cannot be used inside ES6 modules.

First is the this keyword. Inside an ES6 module, the top-level this points to undefined; the top-level this of a CommonJS module points to the current module, which is a major difference between the two.

Second, the following top-level variables do not exist inside ES6 modules.

  • arguments
  • require
  • module
  • exports
  • __filename
  • __dirname

# Circular Loading

"Circular loading" (circular dependency) refers to the situation where the execution of script a depends on script b, while the execution of script b in turn depends on script a.

// a.js
var b = require('b');

// b.js
var a = require('a');
1
2
3
4
5

Usually, "circular loading" indicates strong coupling. If not handled well, it can even lead to recursive loading, making the program unable to run, so it should be avoided.

But in practice, it is hard to avoid, especially in large projects with complex dependency relationships, where it's easy to encounter situations like a depending on b, b depending on c, and c in turn depending on a. This means that the module loading mechanism must take "circular loading" into account.

For the JavaScript language, the two most common module formats currently, CommonJS and ES6, handle "circular loading" differently, and return different results.

# The Loading Principle of CommonJS Modules

Before explaining how ES6 handles "circular loading," let's first introduce the loading principle of the currently most popular module format, CommonJS.

A CommonJS module is a single script file. The first time the require command loads the script, it executes the entire script and then generates an object in memory.

{
  id: '...',
  exports: { ... },
  loaded: true,
  ...
}
1
2
3
4
5
6

The code above is an object generated by Node internally after loading a module. The object's id property is the module name, the exports property holds the various interfaces output by the module, and the loaded property is a boolean indicating whether the module's script has finished executing. There are many other properties, all omitted here.

Whenever this module is needed later, its value is fetched from the exports property. Even if the require command is executed again, the module will not be executed again; instead, the value is fetched from the cache. In other words, no matter how many times a CommonJS module is loaded, it runs only once on the first load; subsequent loads return the result of the first run, unless the system cache is manually cleared.

# Circular Loading of CommonJS Modules

An important characteristic of CommonJS modules is that they execute on load: the script code is fully executed at the moment of require. Once a module is "circularly loaded," only the part that has already been executed is output; the part that has not yet been executed will not be output.

Let's look at the example in Node's official documentation (opens new window). The script file a.js has the following code.

exports.done = false;
var b = require('./b.js');
console.log('in a.js, b.done = %j', b.done);
exports.done = true;
console.log('a.js finished executing');
1
2
3
4
5

In the code above, the a.js script first outputs a done variable and then loads another script file, b.js. Note that at this point the a.js code stops here, waiting for b.js to finish executing before continuing.

Now look at the code of b.js.

exports.done = false;
var a = require('./a.js');
console.log('in b.js, a.done = %j', a.done);
exports.done = true;
console.log('b.js finished executing');
1
2
3
4
5

In the code above, when b.js executes its second line, it goes to load a.js, and at this point "circular loading" occurs. The system goes to fetch the value from the exports property of the object corresponding to the a.js module, but because a.js has not finished executing, only the part that has already been executed can be retrieved from the exports property, rather than the final value.

The part of a.js that has already been executed is only one line.

exports.done = false;
1

Therefore, as far as b.js is concerned, it imports from a.js only a single variable done, with the value false.

Then, b.js continues to execute downward, and once it has fully finished, it hands control back to a.js. So a.js continues to execute downward until it finishes. Let's write a script main.js to verify this process.

var a = require('./a.js');
var b = require('./b.js');
console.log('in main.js, a.done=%j, b.done=%j', a.done, b.done);
1
2
3

Run main.js, and the result is as follows.

$ node main.js

in b.js, a.done = false
b.js finished executing
in a.js, b.done = true
a.js finished executing
in main.js, a.done=true, b.done=true
1
2
3
4
5
6
7

The code above proves two things. First, in b.js, a.js had not finished executing; only its first line had been executed. Second, when main.js reaches its second line, it does not execute b.js again, but instead outputs the cached execution result of b.js, i.e., its fourth line.

exports.done = true;
1

In short, what CommonJS imports is a copy of the output value, not a reference.

In addition, because a CommonJS module, when encountering circular loading, returns the value of the part that has currently been executed rather than the value after the code has fully run, the two may differ. So you must be very careful when importing variables.

var a = require('a'); // safe approach
var foo = require('a').foo; // dangerous approach

exports.good = function (arg) {
  return a.foo('good', arg); // uses the latest value of a.foo
};

exports.bad = function (arg) {
  return foo('bad', arg); // uses a value from a partial load
};

# Circular Loading of ES6 Modules

ES6 handles "circular loading" in a fundamentally different way from CommonJS. ES6 modules are dynamic references. If you use import to load variables from a module (i.e., import foo from 'foo'), those variables are not cached; instead, they become a reference pointing to the loaded module, and the developer must ensure that a value can actually be obtained when it is actually fetched.

Consider the following example.

// a.mjs
import {bar} from './b';
console.log('a.mjs');
console.log(bar);
export let foo = 'foo';

// b.mjs
import {foo} from './a';
console.log('b.mjs');
console.log(foo);
export let bar = 'bar';

$ node --experimental-modules a.mjs
b.mjs
ReferenceError: foo is not defined

In the code above, executing a.mjs throws an error: the variable foo is not defined. Why?

Let's go through it line by line to see how ES6 circular loading is handled. First, after executing a.mjs, the engine discovers that it loads b.mjs, so it executes b.mjs first and then executes a.mjs. Next, when executing b.mjs, since it is known that it imports the foo interface from a.mjs, the engine does not execute a.mjs at this point; instead, it assumes this interface already exists and continues executing downward. Only when it reaches the third line, console.log(foo), does it discover that this interface is not defined at all, and therefore throws an error.

The way to solve this problem is to make foo already defined when b.mjs runs. This can be solved by writing foo as a function.

// a.mjs
import {bar} from './b';
console.log('a.mjs');
console.log(bar());
function foo() { return 'foo' }
export {foo};

// b.mjs
import {foo} from './a';
console.log('b.mjs');
console.log(foo());
function bar() { return 'bar' }
export {bar};
